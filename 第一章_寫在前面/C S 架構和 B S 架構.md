# C/S 架構和 B/S 架構

> 所屬章節：第一章｜寫在前面
> 關鍵字：C/S、B/S、Client/Server、Browser/Server、客戶端、服務端、伺服器、瀏覽器、專用客戶端、桌面應用、Web 應用、部署、更新、跨平台
> 建議回查情境：想分清桌面軟體和 Web 系統的差別、想理解為什麼有些系統需要安裝專用客戶端、想比較 C/S 與 B/S 的適用情境、想知道它們為什麼不是完全對立的概念

## 本節導讀

這一節整理兩種常見的系統使用型態：`C/S` 與 `B/S`。

很多入門教材會把它們講成完全對立的兩個世界：

- `C/S` 是一種架構。
- `B/S` 是另一種架構。
- 兩者優缺點完全相反。

這樣講方便記憶，但容易過度簡化。

閱讀時先抓住這句話：

**C/S 和 B/S 都是在談客戶端與服務端如何分工；兩者最明顯的差別，是使用者這一端的客戶端形式不同。**

更進一步說：

**B/S 可以看成一種常見的 C/S 形式，只是它把客戶端主要固定成瀏覽器。**

## 你會在這篇學到什麼

- `C/S` 與 `B/S` 的基本意思
- 兩者在客戶端形式上的核心差異
- 為什麼 `B/S` 可以看成廣義 `C/S` 的一種形式
- `C/S` 和 `B/S` 在部署、更新、裝置能力、跨平台上的差異
- 兩種型態各自常見的適用情境
- 為什麼現代系統常常不是純 C/S 或純 B/S

## 這篇在解決什麼問題？

初學時常見混亂有幾種：

- 以為 `C/S` 和 `B/S` 是完全對立、互不相關的概念。
- 以為 `C/S` 一定比 `B/S` 快。
- 以為 `B/S` 一定比 `C/S` 簡單。
- 以為 `B/S` 就一定是三層架構。
- 以為瀏覽器什麼都不能做，所以複雜系統只能用 `C/S`。
- 以為只要要安裝軟體就一定是傳統 `C/S`，只要開網頁就一定沒有客戶端邏輯。

這篇要建立的是更穩定的判斷方式：

**先看客戶端是專用程式還是瀏覽器，再看系統如何部署、更新、使用本機能力和分配邏輯。**

## 30 秒先講結論

`C/S` 是 `Client/Server`，也就是客戶端 / 服務端架構。

`B/S` 是 `Browser/Server`，也就是瀏覽器 / 服務端架構。

先用一張表抓核心：

| 類型 | 客戶端形式 | 使用者怎麼用 | 核心特點 |
|---|---|---|---|
| C/S | 通常是專用客戶端程式 | 安裝軟體後使用 | 客戶端能力較強，可能深度整合本機能力 |
| B/S | 通常是瀏覽器 | 打開網址使用 | 部署與更新較集中，跨平台存取通常較方便 |

可以這樣理解：

```text
C/S：Client / Server
使用者透過某種客戶端使用服務

B/S：Browser / Server
使用者主要透過瀏覽器這個客戶端使用服務
```

所以：

**B/S 不是完全脫離 C/S 的另一個宇宙；它可以理解成把 Client 這一端主要限定為 Browser 的一種常見模式。**

## 先複習：Client 和 Server 是什麼？

在前面的 [客戶端、終端、服務端是什麼意思](./客户端、终端、服务端是什么意思.md) 已經講過：

- **客戶端**：發出請求、使用服務的一端。
- **服務端**：接收請求、提供服務的一端。

`C/S` 和 `B/S` 都離不開這組關係。

差別在於：

- `C/S` 比較泛指「客戶端 / 服務端」這種分工。
- `B/S` 則特別強調「瀏覽器 / 服務端」這種形式。

## C/S 架構是什麼？

`C/S` 架構中的 `C` 是 `Client`，也就是客戶端。

在常見教材語境裡，`C/S` 通常指：

**使用者需要透過專用客戶端程式，和服務端系統互動。**

這類客戶端可能需要：

- 安裝
- 登入
- 更新版本
- 連接服務端
- 在本機儲存部分設定或資料
- 使用本機硬體或系統能力

常見例子包含：

- 桌面版聊天軟體
- 桌面版辦公軟體
- 企業內部專用系統
- 遊戲客戶端
- 交易或金融專用客戶端
- 工業控制或硬體管理工具
- 資料庫管理工具

在 `C/S` 裡，客戶端通常不只是顯示畫面。
它可能會負責：

- 較完整的使用者介面
- 本機資料暫存
- 較複雜的互動邏輯
- 本機檔案操作
- 硬體或系統能力整合
- 離線或弱網路環境下的部分功能

所以可以先記：

**C/S 的重點，是使用者透過專用客戶端程式來使用服務。**

## B/S 架構是什麼？

`B/S` 架構中的 `B` 是 `Browser`，也就是瀏覽器。

`B/S` 通常指：

**使用者主要透過瀏覽器和服務端互動。**

使用者通常不需要安裝某個專用客戶端，只要：

- 打開瀏覽器
- 輸入網址
- 登入系統
- 操作頁面

常見例子包含：

- 公司官網
- 部落格
- 線上商城
- 管理後台
- 線上表單
- 線上文件工具
- 企業內部 Web 系統
- 大多數透過瀏覽器使用的 SaaS 工具

在 `B/S` 裡，瀏覽器就是最主要的客戶端。
它負責：

- 請求網頁資源
- 解析 `HTML/CSS/JavaScript`
- 顯示頁面
- 執行前端互動邏輯
- 向服務端 API 請求資料

所以可以先記：

**B/S 的重點，是把瀏覽器當成主要客戶端來交付系統。**

## C/S 和 B/S 到底差在哪裡？

最核心差異是：

**客戶端是專用程式，還是通用瀏覽器。**

| 比較面向 | C/S 架構 | B/S 架構 |
|---|---|---|
| 客戶端形式 | 專用應用程式 | 瀏覽器 |
| 使用入口 | 安裝並打開客戶端 | 打開網址 |
| 安裝成本 | 通常需要安裝 | 通常不需要安裝專用程式 |
| 更新方式 | 可能需要更新客戶端版本 | 多數更新集中在服務端與前端資源 |
| 本機能力 | 較容易整合硬體、檔案、系統能力 | 受瀏覽器安全模型限制較多 |
| 跨平台 | 常需不同平台版本 | 通常只要瀏覽器支援即可使用 |
| 離線能力 | 較容易做深度離線能力 | 可做部分離線，但受技術與場景限制 |
| 安全控制 | 可在受控客戶端中做更多限制 | 需面對開放瀏覽器環境 |
| 維護成本 | 客戶端發行與版本管理成本較高 | 集中部署較方便，但仍需處理瀏覽器相容性 |
| 常見情境 | 專用工具、重度互動、硬體整合、受控環境 | 網站、後台、資訊系統、廣泛存取 |

這張表不是絕對規則，而是常見傾向。

## C/S 架構常見優點與限制

![C/S 架構示意圖](./images/1775985638808_NCBoiy2ejj.png)

### 常見優點

`C/S` 在下面這些方面通常比較有優勢：

- 可以打造更完整的專用操作體驗。
- 較容易整合本機硬體，例如掃描器、印表機、讀卡機、攝影機。
- 較容易存取本機檔案或作業系統能力。
- 可以把部分運算或資料處理放在客戶端。
- 在受控環境中，版本、權限和設備可以統一管理。
- 對高效能、低延遲或特殊設備支援的場景可能更合適。

### 常見限制

`C/S` 也常有這些成本：

- 使用前通常需要安裝。
- 不同作業系統可能需要不同版本。
- 客戶端更新、發版、回滾和相容性管理成本較高。
- 使用者環境越複雜，維護成本越高。
- 如果客戶端版本太舊，可能和服務端不相容。
- 安全更新需要確保使用者真的更新到新版。

所以 `C/S` 不是「一定比較高級」，它只是更適合某些需要專用客戶端能力的場景。

## B/S 架構常見優點與限制

![B/S 架構示意圖](./images/1775985638808_0vtutEUUC0.png)

### 常見優點

`B/S` 在下面這些方面通常比較有優勢：

- 使用門檻低，通常打開瀏覽器即可使用。
- 不需要為每位使用者單獨安裝專用客戶端。
- 服務端和前端資源集中部署，更新較方便。
- 更容易面向大量使用者提供服務。
- 跨平台通常較容易，只要瀏覽器支援即可。
- 使用者換設備時，也能透過同一網址存取。

### 常見限制

`B/S` 也有自己的限制：

- 本機檔案、硬體、系統能力存取受瀏覽器安全模型限制。
- 複雜互動或高效能需求可能需要更多前端工程處理。
- 仍然要處理不同瀏覽器、不同版本、不同裝置的相容性。
- 網路狀態會直接影響使用體驗。
- 對深度離線、本機設備控制、特殊安全環境的支援可能不如專用客戶端自然。

所以 `B/S` 不是「一定比較簡單」，它只是把使用入口和部署方式變得更通用、更集中。

## 什麼時候比較適合 C/S？

通常在下面這些情況，`C/S` 會比較常見：

- 需要深度整合本機硬體。
- 需要大量本機檔案處理。
- 需要專用操作體驗。
- 需要高效能圖形或即時互動。
- 使用環境比較受控，例如企業內部固定設備。
- 對離線能力有較高要求。
- 對客戶端安全策略、設備控制或特殊協議有較高需求。

例如：

- 工廠設備控制系統
- 專業剪輯或設計軟體
- 大型遊戲客戶端
- 銀行或交易專用客戶端
- 需要讀取本機設備的業務系統

## 什麼時候比較適合 B/S？

通常在下面這些情況，`B/S` 會比較常見：

- 希望使用者不用安裝專用軟體。
- 需要快速面向大量使用者提供服務。
- 希望部署、更新與維護集中化。
- 使用者裝置和平台很多樣。
- 系統主要是資料查詢、填寫、管理、展示與一般互動。
- 希望只要有瀏覽器和網路就能使用。

例如：

- 官網
- 管理後台
- 線上表單
- 電商網站
- 內容管理系統
- 線上文件與協作工具
- 一般企業資訊系統

## 現代系統常常是混合型

現代應用不一定能簡單分成純 `C/S` 或純 `B/S`。

很多產品會混合兩者特點。

例如：

### 1. 桌面 App 內嵌 Web 技術

有些桌面應用看起來像專用客戶端，但內部大量使用 Web 技術。

例如某些 Electron 類型應用：

- 使用者需要安裝桌面 App。
- 但介面可能用 `HTML/CSS/JavaScript` 製作。
- 它同時有專用客戶端和 Web 技術特徵。

### 2. Web App 越來越像傳統客戶端

現代瀏覽器能力越來越強，Web App 也可以做很多以前需要客戶端才容易做的事。

例如：

- 前端路由
- 離線快取
- 本地儲存
- PWA
- 即時通訊
- 音視頻處理
- 複雜圖形互動

所以 B/S 不代表瀏覽器端一定很弱。

### 3. 手機 App + 後端服務

手機 App 通常也是一種 C/S 模式：

- App 是客戶端。
- 後端服務是服務端。

但 App 裡也可能嵌入 WebView，顯示某些 Web 頁面。

### 4. Web 系統也可能有本地輔助程式

有些 Web 系統主要透過瀏覽器使用，但為了讀卡、列印、掃描或安全控制，會額外安裝本地輔助程式。

這就不是純粹的 B/S，而是 B/S 加上本地能力補充。

所以判斷時不要只問「它是不是網站」或「它是不是軟體」。
更好的問法是：

**使用者主要透過什麼客戶端使用？系統哪些能力放在客戶端？哪些能力放在服務端？**

## B/S 和三層架構不是同一件事

這是常見混淆點。

`B/S` 描述的是：

**客戶端形式是瀏覽器，服務由服務端提供。**

三層架構通常描述的是：

**系統內部分成展示層、業務邏輯層、資料存取層等不同責任層。**

它們不是同一個維度。

一個 `B/S` 系統可以設計成三層架構，也可以不是。
一個 `C/S` 系統內部也可能有多層架構。

所以不要把：

```text
B/S = 三層架構
```

直接當成固定公式。

更精準地說：

**B/S 是使用入口和客戶端形式；三層架構是系統內部設計分層。**

## C/S 和 B/S 怎麼判斷？

遇到一個系統時，可以按下面幾個問題判斷。

### 1. 使用者主要透過什麼使用？

- 專用 App 或桌面軟體：偏 `C/S`
- 瀏覽器網址：偏 `B/S`
- 兩者都有：可能是混合型

### 2. 是否需要安裝專用客戶端？

- 需要安裝特定客戶端：偏 `C/S`
- 只需要瀏覽器：偏 `B/S`

### 3. 是否深度使用本機能力？

- 需要讀寫本機檔案、硬體設備、特殊安全控制：偏 `C/S`
- 主要是頁面展示、表單、資料查詢、一般互動：偏 `B/S`

### 4. 更新主要在哪裡完成？

- 使用者端客戶端也需要頻繁更新：偏 `C/S`
- 多數更新集中在伺服器端與頁面資源：偏 `B/S`

### 5. 使用者環境是否高度受控？

- 企業內部固定設備與固定版本：`C/S` 可能可控
- 大量外部使用者、多平台、多裝置：`B/S` 通常更方便

## 常見混淆點

| 混淆說法 | 更清楚的理解 |
|---|---|
| C/S 和 B/S 是完全對立的 | B/S 可以看成一種常見 C/S 形式，只是客戶端主要是瀏覽器 |
| C/S 一定比 B/S 快 | 效能取決於場景、實作、網路、資料量與設備能力 |
| B/S 一定比 C/S 簡單 | B/S 部署入口簡單，但前端工程、相容性和服務端設計也可能很複雜 |
| B/S 等於三層架構 | B/S 是客戶端形式，三層架構是系統內部分層 |
| 瀏覽器端不能做複雜功能 | 現代瀏覽器能做很多事情，但仍受安全模型與平台能力限制 |
| 安裝軟體就一定是傳統 C/S | 有些桌面 App 內部也大量使用 Web 技術 |
| 網頁系統就完全沒有客戶端邏輯 | 現代 Web 前端可能有大量狀態管理和互動邏輯 |
| B/S 不需要考慮客戶端 | 瀏覽器本身就是客戶端，仍要考慮瀏覽器版本、性能和相容性 |

## 初學時最該記住的六句話

1. `C/S` 是 `Client/Server`，重點是客戶端和服務端分工。
2. `B/S` 是 `Browser/Server`，重點是瀏覽器作為主要客戶端。
3. `B/S` 可以看成廣義 `C/S` 的一種常見形式。
4. 兩者最大差別通常是客戶端形式：專用程式 vs 瀏覽器。
5. `C/S` 和 `B/S` 的優缺點不是絕對，要看場景。
6. 現代系統常常混合兩者特徵，判斷時要看使用入口、部署方式和能力分配。

## 一句話抓核心

`C/S` 和 `B/S` 都是在談客戶端與服務端如何分工；`C/S` 通常強調專用客戶端，`B/S` 則強調瀏覽器作為主要客戶端。兩者不是絕對對立，真正判斷時要看客戶端形式、部署更新方式、本機能力需求、使用場景與系統能力分配。
