# C/S 架構和 B/S 架構

> 所屬章節：第一章｜寫在前面  
> 關鍵字：C/S、B/S、Client/Server、Browser/Server、客戶端、伺服器、瀏覽器、桌面應用、Web 應用  
> 建議回查情境：想分清楚桌面軟體和 Web 系統的差別、想理解為什麼有些系統需要專用客戶端、想快速比較 C/S 與 B/S 的適用情境

## 本節導讀

這一節整理兩種常見的系統使用型態：`C/S` 與 `B/S`。  
很多入門教材會把它們講成完全對立的兩個世界，但更好的理解方式是：

- 它們都在談客戶端與伺服器怎麼分工
- 差別主要在於客戶端的形式不同

閱讀時先抓住一句核心：

- **`B/S` 可以看成一種常見的 `C/S` 形式，只是它把客戶端固定成瀏覽器。**

## 你會在這篇學到什麼

- `C/S` 與 `B/S` 的基本意思
- 兩者在客戶端形式與系統分工上的差異
- 兩種架構各自較適合的情境
- 為什麼它們不是絕對對立的概念

## 這篇在解決什麼問題？

- 很多入門筆記會把 `C/S` 和 `B/S` 講成完全對立的兩種世界。
- 也常會把兩者的優缺點講得過度絕對，例如「C/S 一定快」「B/S 一定簡單」。
- 這篇要先把核心講清楚：它們都在描述客戶端和伺服器如何分工，而最明顯的差別，在於客戶端是不是專用程式。

## 先講結論

- `C/S` 是 `Client/Server`，也就是客戶端 / 伺服器架構。
- `B/S` 是 `Browser/Server`，也就是瀏覽器 / 伺服器架構。
- 可以把 `B/S` 看成一種常見的 `C/S` 形式，只是它把客戶端固定成瀏覽器，因此常被教材單獨拿出來比較。

## C/S 架構是什麼？

- `C/S` 架構中的客戶端通常是專用應用程式，需要安裝後才能使用。
- 這類客戶端除了顯示畫面，也常會承擔一部分互動邏輯、資料處理，甚至存取本機裝置能力。
- 常見例子包含桌面辦公軟體、企業內部系統、專用工具軟體、部分遊戲啟動器等。

### 你可以怎麼理解？

在 `C/S` 架構裡：

- 使用者會透過某個專用客戶端來使用服務
- 客戶端和伺服器通常有比較明確的分工
- 客戶端往往不是瀏覽器，而是專門為這套系統設計的程式

## B/S 架構是什麼？

- `B/S` 架構把瀏覽器當成客戶端。
- 使用者通常不需要安裝專用客戶端，只要透過瀏覽器輸入網址或開啟頁面，就能使用系統。
- 常見例子包含網站、管理後台、線上表單、許多企業內部 Web 系統等。

### 你可以怎麼理解？

在 `B/S` 架構裡：

- `Browser` 就是客戶端
- 使用者主要透過瀏覽器與伺服器互動
- 系統的更新通常集中在伺服器端與前端部署內容上

## C/S 和 B/S 到底差在哪裡？

| 比較面向 | C/S 架構 | B/S 架構 |
| --- | --- | --- |
| 客戶端形式 | 專用應用程式 | 瀏覽器 |
| 安裝方式 | 通常需要安裝 | 通常不需安裝專用程式 |
| 更新方式 | 可能需要更新客戶端版本 | 多數情況主要更新伺服器端與頁面資源 |
| 裝置能力 | 較容易整合本機硬體、檔案與系統功能 | 受瀏覽器環境與安全限制較多 |
| 跨平台便利性 | 常需針對不同平台提供版本 | 只要瀏覽器支援，通常更容易跨平台 |
| 常見情境 | 專用工具、重度互動、硬體整合、受控環境 | 網站、資訊系統、快速部署與廣泛存取 |

## C/S 架構常見優點與限制

![C/S 架構示意圖](./images/1775985638808_NCBoiy2ejj.png)

### 常見優點

- 可以依需求打造更完整的專用使用體驗。
- 較容易整合本機硬體、檔案系統與作業系統能力。
- 在特定場景下，可把部分運算與邏輯放在客戶端處理。
- 在受控環境中，客戶端功能可以做得很深。

### 常見限制

- 需要安裝、部署與維護客戶端。
- 多平台支援通常比 Web 系統更麻煩。
- 版本更新時，客戶端管理成本可能較高。
- 若使用者環境很多樣，發行與維護成本往往會上升。

## B/S 架構常見優點與限制

![B/S 架構示意圖](./images/1775985638808_0vtutEUUC0.png)

### 常見優點

- 使用門檻低，通常打開瀏覽器就能使用。
- 比較容易統一部署與更新。
- 對面向大量使用者的系統來說，推廣與存取通常更方便。
- 在跨平台與快速交付方面通常更有優勢。

### 常見限制

- 受瀏覽器安全模型與能力限制，本機資源存取通常沒有專用客戶端那麼直接。
- 複雜互動、效能要求高或需要深度整合硬體的場景，做起來可能較受限制。
- 仍然需要處理跨瀏覽器支援與相容性問題。

## 什麼時候比較適合 C/S？

通常在下面這些情況，`C/S` 會比較常見：

- 需要深度整合本機硬體
- 需要專用操作體驗
- 使用環境相對可控
- 對效能、離線能力或特定設備支援有較高要求

## 什麼時候比較適合 B/S？

通常在下面這些情況，`B/S` 會比較常見：

- 希望使用者不用安裝專用軟體
- 希望跨平台存取更容易
- 希望部署、更新與維護集中化
- 需要快速面向大量使用者提供服務

## 常見混淆點

- `C/S` 不只用在局域網，也可以透過網際網路提供服務。
- `B/S` 不等於「一定是三層架構」；`B/S` 描述的是客戶端形式，而三層架構描述的是系統如何分層，兩者不是同一個維度。
- 不是因為瀏覽器「完全不能做運算」，所以所有系統才需要 `C/S`；真正的差別通常在於裝置能力、部署方式、使用體驗、效能需求與安全控制。
- `B/S` 不是「比較低級的 `C/S`」，它只是更偏向透過瀏覽器交付服務的做法。

## 一句話抓核心

- `C/S` 和 `B/S` 都是在談客戶端與伺服器怎麼分工，而它們最主要的差別，在於客戶端是專用程式還是瀏覽器。
