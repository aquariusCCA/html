# C/S 架構和 B/S 架構

> 所屬章節：第一章｜寫在前面  
> 關鍵字：C/S、B/S、Client/Server、Browser/Server、客戶端、瀏覽器、服務器、局域網、廣域網  
> 建議回查情境：想分清楚桌面軟體和 Web 系統的差別、想理解為什麼有些系統必須安裝客戶端、想快速比較 C/S 與 B/S 的優缺點

## 本節導讀

這一節整理兩種常見的系統架構：`C/S` 與 `B/S`。  
閱讀時可以先掌握兩者的基本差異，再看各自的優點、限制與適用情境。

## 你會在這篇學到什麼

- C/S 架構與 B/S 架構的基本意思
- 兩種架構在客戶端與服務器分工上的差異
- C/S 與 B/S 各自的優點、缺點與常見使用情境

## 正文筆記

- 可以先用一句話理解這兩種架構的差別：瀏覽器能承擔的運算與權限控制有限，因此某些系統仍然需要安裝專用客戶端，也就是 `C/S` 架構。
- 例如像導彈系統這類需要嚴格授權、不能安裝在任意電腦上的軟體，就不適合只靠瀏覽器存取；即使拿到安裝包，沒有被授權的設備也無法安裝與使用。

## C/S 架構

- `C/S` 是 `Client/Server` 的縮寫，也就是客戶端 / 服務器架構。
- 這類軟體通常需要先下載並安裝客戶端，之後才能使用，例如 `QQ`、`Office` 等。
- 客戶端不只是接收畫面或做簡單操作，它也會負責一部分運算與業務邏輯處理。也就是說，有些原本可以由服務器完成的事情，也會分配到客戶端執行。

![C/S 架構示意圖](./images/1775985638808_NCBoiy2ejj.png)

### C/S 架構的優點

- 界面和操作可以做得很豐富，客戶端介面能依需求自由設計。
- 安全性較容易保證，因為通常只涉及兩層傳輸。
- 由於交互鏈路較短，回應速度通常較快。以 `QQ` 這類即時通訊工具來看，即使多人同時在線，也不一定會感到明顯延遲。

### C/S 架構的缺點

- 適用面較窄，通常更常見於局域網或特定環境。
- 用戶群相對固定，因為需要安裝程式，所以不適合面向完全未知的大量用戶。
- 維護成本較高，一旦版本升級，所有客戶端通常都需要同步更新。

## B/S 架構

> B/S 架構的系統通常不需要另外安裝客戶端，只要有 Web 瀏覽器就能使用。

> 可以先把它理解成常見的網站系統：前端主要負責頁面呈現、資料請求與少量邏輯，主要業務邏輯仍放在後端服務器。

- `B/S` 是 `Browser/Server` 的縮寫，也就是瀏覽器 / 服務器架構。
- 其中 `Browser` 指的是 Web 瀏覽器。這種架構下，少量事務邏輯會在前端處理，但主要事務邏輯仍由服務器完成。
- 和通常只有兩層的 `C/S` 架構不同，`B/S` 架構常可先理解為三層：

![B/S 架構示意圖](./images/1775985638808_0vtutEUUC0.png)

- 第一層是表現層，主要負責使用者和後台的互動，以及最終查詢結果的呈現。
- 第二層是邏輯層，主要透過服務器完成客戶端所需的應用邏輯。
- 第三層是數據層，主要接收客戶端請求後，獨立完成各種資料運算與處理。

### B/S 架構的優點

- 客戶端無需安裝，只要有 Web 瀏覽器即可使用。
- 可以直接部署在廣域網上，並透過權限控制讓多個使用者存取，互動性較強。
- 系統升級時通常只需更新服務器，不必要求所有使用者重新安裝新版客戶端。

### B/S 架構的缺點

- 跨瀏覽器相容性常常不夠理想。
- 如果想把表現做得接近 C/S 程式，往往需要投入更多開發成本。
- 在速度與安全性方面，也需要投入較高的設計成本，這通常是 B/S 架構最明顯的挑戰之一。
