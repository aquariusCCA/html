# 甚麼是 Web 的前端和後端

> 所屬章節：第一章｜寫在前面
> 關鍵字：前端、後端、客戶端、服務端、瀏覽器、伺服器、資料庫、API、使用者介面、業務邏輯、資料流、前後端分工
> 建議回查情境：剛開始學 Web 時分不清前端和後端、看到客戶端與服務端想確認差異、想理解一次請求裡資料如何流動、想分清伺服器、後端、資料庫與 API 的關係

## 本節導讀

這一節要釐清 Web 入門最容易混在一起的一組概念：

- 前端
- 後端
- 客戶端
- 服務端
- 伺服器
- 資料庫
- API

先抓住最重要的兩句話：

- **前端 / 後端是在談系統分工。**
- **客戶端 / 服務端是在談互動角色。**

它們常常一起出現，但不是同一件事。

在很多 Web 系統裡，前端通常在瀏覽器中執行，後端通常在伺服器上執行。
但這只是最常見的情況，不代表「前端 = 客戶端」或「後端 = 服務端」永遠可以直接畫上等號。

## 你會在這篇學到什麼

- 前端與後端最基本的分工
- 客戶端 / 服務端 和 前端 / 後端 的差別
- 伺服器、後端、資料庫、API 之間的關係
- 一次 Web 請求中資料大致怎麼流動
- 為什麼前端不只是畫面，後端也不只是資料庫
- 現代 Web 中前後端邊界為什麼有時會交疊

## 這篇在解決什麼問題？

初學 Web 時，最常見的混亂是把不同層次的詞混成一團：

- 以為前端就是「看得到的東西」。
- 以為後端就是「看不到的東西」。
- 以為伺服器就是後端。
- 以為資料庫就是後端。
- 以為客戶端一定等於前端。
- 以為服務端一定等於後端。
- 以為 API 是另一個獨立於前後端之外的神祕東西。

這些說法有些能當粗略記憶，但都不夠精準。
這篇要做的是把它們拆開，再放回同一條 Web 流程中理解。

## 30 秒先講結論

![前端和後端示意圖](./images/1775985638800_rWD8cWXIws.png)

可以先用這張表建立整體概念：

| 名詞 | 先這樣理解 | 重點 |
|---|---|---|
| 前端 | 面向使用者的頁面與互動部分 | 使用者看到什麼、怎麼操作、資料怎麼呈現 |
| 後端 | 支撐系統運作的服務與資料處理部分 | 接收請求、處理邏輯、存取資料、回傳結果 |
| 客戶端 | 發出請求、使用服務的一端 | 誰在使用服務 |
| 服務端 | 接收請求、提供服務的一端 | 誰在提供服務 |
| 伺服器 | 承載服務的機器、環境或服務角色 | 程式在哪裡持續運行 |
| 資料庫 | 儲存和管理資料的系統 | 資料放在哪裡、怎麼查詢與更新 |
| API | 前後端或不同系統之間交換資料的約定入口 | 用什麼格式、路徑和規則溝通 |

最常見的 Web 流程可以先記成：

```text
使用者操作前端
  -> 前端發出請求
  -> 後端接收請求並處理邏輯
  -> 後端必要時讀寫資料庫
  -> 後端回傳結果
  -> 前端把結果顯示給使用者
```

## 前端在做什麼？

前端主要負責使用者直接接觸到的部分。

它關心的是：

- 使用者看到什麼
- 使用者怎麼操作
- 資料怎麼呈現
- 畫面狀態怎麼變化
- 使用者操作如何轉成請求
- 回來的資料如何更新到頁面上

以網站為例，前端通常包含：

- `HTML`：頁面結構與語意
- `CSS`：畫面樣式與排版
- `JavaScript`：互動行為與動態更新
- 表單、按鈕、導覽列、彈窗、提示訊息
- 從後端取得資料後的畫面呈現
- 基本的輸入檢查與互動回饋

前端不是只有「畫面好不好看」。
更精準地說：

**前端負責把系統功能轉成使用者能理解、能操作、能得到回饋的介面。**

例如一個商品列表頁，前端會處理：

- 商品卡片怎麼排列
- 價格和圖片怎麼顯示
- 使用者點篩選條件後畫面怎麼變
- 載入中、成功、失敗時要顯示什麼
- 從後端拿到資料後如何更新列表

## 後端在做什麼？

後端主要負責使用者不直接看到、但系統必須依賴的服務邏輯。

它關心的是：

- 接收前端或其他系統送來的請求
- 判斷請求是否合法
- 處理業務邏輯
- 驗證使用者身分與權限
- 查詢、建立、更新或刪除資料
- 呼叫其他服務
- 回傳前端需要的結果

後端通常會處理：

- 登入與註冊
- 權限檢查
- 訂單建立
- 商品查詢
- 表單資料儲存
- 檔案上傳處理
- 付款流程串接
- API 回應資料

後端不是單純「看不到的東西」。
更精準地說：

**後端負責系統規則、資料處理、權限控制和服務能力。**

例如一個登入功能，後端會處理：

- 帳號是否存在
- 密碼是否正確
- 是否允許登入
- 是否產生登入狀態或 token
- 回傳登入成功或失敗的結果

## 前端和後端怎麼合作？

前端和後端不是兩個完全無關的世界，它們通常透過請求與回應合作。

最常見的合作方式如下：

1. 使用者在前端頁面操作。
2. 前端把操作轉成請求。
3. 請求送到後端提供的 API 或路由。
4. 後端根據請求處理邏輯。
5. 後端必要時讀寫資料庫。
6. 後端把結果回傳給前端。
7. 前端根據結果更新畫面。

可以把它理解成：

```text
前端負責「把操作送出去、把結果顯示出來」
後端負責「接住請求、處理規則、回傳結果」
```

## 一個完整例子：登入流程

用登入流程來看會更清楚。

### 1. 使用者操作前端

使用者在頁面上輸入：

- 帳號
- 密碼

然後點擊「登入」按鈕。

### 2. 前端整理資料並發出請求

前端可能先做基本檢查：

- 帳號是否為空
- 密碼是否為空
- 格式是否明顯不對

然後把資料送到後端，例如：

```text
POST /api/login
```

請求內容可能類似：

```json
{
  "account": "alice",
  "password": "123456"
}
```

### 3. 後端接收並處理

後端收到請求後，會處理更關鍵的邏輯：

- 檢查帳號是否存在
- 比對密碼是否正確
- 判斷帳號是否被停用
- 建立登入狀態
- 記錄必要的安全資訊

### 4. 後端查詢資料庫

後端通常需要向資料庫查詢使用者資料，例如：

- 使用者帳號
- 加密後的密碼
- 使用者角色
- 帳號狀態

注意：在一般 Web 系統中，瀏覽器通常不會直接連到正式資料庫。
通常是後端負責和資料庫互動，前端透過後端提供的 API 取得結果。

### 5. 後端回傳結果

如果登入成功，後端可能回傳：

```json
{
  "success": true,
  "message": "登入成功"
}
```

如果登入失敗，後端可能回傳：

```json
{
  "success": false,
  "message": "帳號或密碼錯誤"
}
```

### 6. 前端更新畫面

前端根據後端結果：

- 顯示登入成功
- 跳轉到首頁
- 顯示錯誤訊息
- 清空密碼欄位
- 讓按鈕恢復可點擊

這個流程可以看出：

**前端負責互動與呈現，後端負責規則與資料；兩者透過請求與回應合作。**

## API 是什麼？

`API` 可以先理解成：

**前端和後端溝通時約定好的入口與規則。**

在 Web 系統裡，API 常常會約定：

- 請求要送到哪個路徑
- 使用什麼 HTTP 方法
- 要帶哪些資料
- 回傳什麼格式
- 什麼情況算成功
- 什麼情況算失敗

例如：

| 功能 | API 路徑 | 方法 | 作用 |
|---|---|---|---|
| 登入 | `/api/login` | `POST` | 送出帳號密碼，取得登入結果 |
| 取得商品列表 | `/api/products` | `GET` | 取得商品資料 |
| 新增文章 | `/api/articles` | `POST` | 建立一篇文章 |
| 刪除文章 | `/api/articles/123` | `DELETE` | 刪除指定文章 |

前端不需要知道後端內部怎麼查資料庫。
後端也不需要知道前端畫面長什麼樣。

只要 API 約定清楚，兩邊就能合作。

所以 API 可以看成前後端之間的「契約」。

## 資料庫、伺服器、後端的關係

這三個概念很容易混淆，需要分開看。

### 1. 資料庫是管理資料的系統

資料庫負責：

- 儲存資料
- 查詢資料
- 更新資料
- 刪除資料
- 維持資料的一致性

例如：

- 使用者資料
- 商品資料
- 訂單資料
- 文章資料
- 留言資料

資料庫不是後端本身。
後端通常會使用資料庫，但不能把兩者畫上等號。

### 2. 伺服器是承載服務的環境或角色

伺服器可以指：

- 一台實體主機
- 一台雲端虛擬機
- 一個容器
- 一個持續運行服務的環境
- 在某個互動中提供服務的一端

伺服器不等於後端程式本身。
後端程式通常跑在伺服器上，但伺服器也可能承載很多其他東西。

### 3. 後端是應用程式邏輯

後端是負責處理請求與業務規則的應用程式邏輯。

它可能會：

- 接收瀏覽器請求
- 查詢資料庫
- 呼叫第三方服務
- 檢查權限
- 整理回傳資料

可以用一句話整理：

- **資料庫管資料。**
- **伺服器提供運行環境。**
- **後端負責業務邏輯與資料處理。**

## 客戶端與服務端是什麼？

客戶端 / 服務端 是從互動角色來看系統。

- **客戶端**：發出請求、使用服務的一端。
- **服務端**：接收請求、提供服務的一端。

在 Web 裡：

- 瀏覽器通常扮演客戶端。
- 提供網站資料與功能的系統通常扮演服務端。

在手機 App 場景裡：

- 手機上的 App 可以是客戶端。
- 背後提供資料、帳號、訂單、通知等功能的系統通常是服務端。

重點是：

**客戶端 / 服務端 描述的是「誰請求、誰提供服務」，不是固定哪一台機器比較大，也不是固定哪一邊比較重要。**

## 前端 / 後端 和 客戶端 / 服務端 差在哪？

這裡是本篇最重要的地方。

### 前端 / 後端：看功能分工

前端 / 後端是在問：

**這個部分負責什麼工作？**

例如：

- 負責畫面與互動的是前端。
- 負責規則與資料處理的是後端。

### 客戶端 / 服務端：看互動角色

客戶端 / 服務端是在問：

**這一次互動裡，誰發請求？誰提供服務？**

例如：

- 瀏覽器向伺服器請求網頁，瀏覽器是客戶端，伺服器是服務端。
- 手機 App 向後端請求訂單資料，App 是客戶端，後端系統是服務端。

### 常見情況

在很多傳統 Web 系統裡：

| 分工角度 | 常見對應 |
|---|---|
| 前端 | 跑在瀏覽器裡，接近客戶端 |
| 後端 | 跑在伺服器上，接近服務端 |

所以這兩組詞常常一起出現。

### 但不要完全畫上等號

因為現代 Web 裡有一些情況會讓邊界變得更靈活。

例如：

- 伺服器端渲染：部分前端頁面產生工作可能在伺服器上完成。
- 靜態網站：可能沒有傳統後端，只把靜態檔案部署出去。
- 前端框架：前端可能負責大量狀態管理和資料呈現邏輯。
- BFF：後端可能專門為某個前端介面整理資料。
- 全端開發：同一個人或同一個專案同時處理前端與後端。

所以更穩定的理解是：

**前端 / 後端是分工概念；客戶端 / 服務端是互動角色。它們常重疊，但不是同義詞。**

## 前端和後端各自關心什麼？

可以用這張表快速比較：

| 面向 | 前端更關心 | 後端更關心 |
|---|---|---|
| 使用者 | 使用者看到什麼、怎麼操作 | 使用者是否有權限、資料是否正確 |
| 畫面 | 排版、樣式、互動、狀態 | 通常不直接處理畫面，除非產生 HTML |
| 資料 | 如何呈現、如何送出 | 如何驗證、處理、儲存、查詢 |
| 安全 | 避免不安全操作、基本輸入檢查 | 身分驗證、權限控制、資料保護 |
| 效能 | 載入速度、互動流暢度 | 查詢效率、服務穩定性、併發處理 |
| 錯誤處理 | 顯示錯誤提示與回饋 | 判斷錯誤原因並回傳合適狀態 |
| 維護 | 元件、樣式、狀態管理 | API、資料模型、業務規則 |

這不是絕對分界，而是常見關注點。

## 前端不只是畫面，後端不只是資料庫

這兩個誤解很常見。

### 前端不只是畫面

前端還會處理：

- 使用者操作
- 狀態變化
- 資料呈現
- 表單互動
- 載入與錯誤回饋
- 響應式版面
- 可存取性
- 和 API 溝通

所以「前端 = 切畫面」是不完整的說法。

### 後端不只是資料庫

後端還會處理：

- 業務規則
- 權限判斷
- 資料驗證
- 安全控制
- API 設計
- 檔案處理
- 任務排程
- 外部服務串接

所以「後端 = 查資料庫」也是不完整的說法。

## 常見混淆點

| 混淆說法 | 更清楚的理解 |
|---|---|
| 前端就是看得到的東西 | 前端負責使用者介面、互動、資料呈現與使用者回饋 |
| 後端就是看不到的東西 | 後端負責請求處理、業務邏輯、資料存取與權限控制 |
| 伺服器就是後端 | 伺服器是承載服務的環境或角色，後端是應用程式邏輯 |
| 資料庫就是後端 | 資料庫負責資料管理，後端通常負責使用資料庫 |
| 客戶端就是前端 | 常見 Web 中通常重疊，但客戶端/服務端是在談互動角色 |
| 服務端就是後端 | 常見情況下接近，但服務端是提供服務的一端，後端是系統分工 |
| API 是另一個獨立層 | API 通常是前後端或不同系統溝通的約定入口 |
| 前端驗證就夠安全 | 前端驗證能改善體驗，但重要規則仍應由後端驗證 |
| 前後端一定完全分離 | 有些專案前後端分離，有些由後端產生頁面，有些是混合模式 |

## 初學時最該記住的六句話

1. 前端 / 後端是在談系統分工。
2. 客戶端 / 服務端是在談互動角色。
3. 前端負責使用者介面、互動與資料呈現。
4. 後端負責請求處理、業務邏輯、權限控制與資料存取。
5. 資料庫、伺服器、後端不是同一件事。
6. API 是前後端或不同系統之間溝通的約定入口。

## 自測問題

讀完這篇後，可以試著回答：

1. 為什麼說前端 / 後端是在談分工？
2. 為什麼說客戶端 / 服務端是在談互動角色？
3. 前端除了顯示畫面，還會負責哪些事情？
4. 後端除了查資料庫，還會負責哪些事情？
5. 登入流程中，前端、後端、資料庫分別做了什麼？
6. API 在前後端合作中扮演什麼角色？
7. 為什麼不能把伺服器、後端、資料庫當成同義詞？

如果這些問題都能說清楚，前端、後端、客戶端、服務端這組概念就基本穩了。

## 一句話抓核心

前端 / 後端是在談系統分工，客戶端 / 服務端是在談互動角色；在常見 Web 系統裡，前端多半負責瀏覽器中的介面與互動，後端多半負責伺服器上的邏輯與資料處理，兩者通常透過 API 和請求回應流程合作。
