> 所屬章節：第一章｜寫在前面

## 本節導讀

這一節要回答一個很容易被講成口號的問題：為什麼前端開發需要 `Web 標準`？

初學時常會聽到：

- 寫網頁要符合標準。
- `HTML`、`CSS`、`JavaScript` 要分離。
- 不同瀏覽器要盡量顯示一致。
- 不要只為某一款瀏覽器寫頁面。

這些說法都和 Web 標準有關，但不能只背口號。
閱讀時先抓住這句話：

**Web 標準是一組讓瀏覽器、開發者、工具與網站能用共同規則溝通的技術規範；它的核心價值是讓 Web 更一致、更可互通、更安全、更容易被不同人和不同裝置使用。**

入門階段常用 `HTML`、`CSS`、`JavaScript` 來理解 Web 的三層分工，但要注意：

**三層分工是學習 Web 標準的入口，不是 Web 標準的全部。**

## 你會在這篇學到什麼

- Web 標準的基本意思
- 為什麼 Web 需要共同標準
- Web 標準不只包含 `HTML`、`CSS`、`JavaScript`
- `HTML`、`CSS`、`JavaScript` 在頁面中的常見分工
- 為什麼會強調結構、表現、行為三層分工
- Web 標準和瀏覽器相容性的關係
- 初學時怎麼把「符合標準」落到實際寫法上

## 這篇在解決什麼問題？

如果沒有先理解 Web 標準，後面很容易出現幾種誤解：

- 以為 Web 標準只是「程式碼排版比較漂亮」。
- 以為 Web 標準只等於 `HTML`、`CSS`、`JavaScript` 三個檔案。
- 以為只要自己瀏覽器能看，程式就算沒問題。
- 以為「三層分離」就是任何情況都不能混在一起。
- 以為有標準就代表所有瀏覽器一定完全一樣。

這篇的目標，是把 Web 標準理解成一套「共同契約」：

- 規範告訴瀏覽器應該怎麼理解內容。
- 規範告訴開發者什麼寫法比較可靠。
- 規範讓不同工具和平台有共同依據。
- 規範降低不同瀏覽器各做各的機率。

## 30 秒先講結論

Web 標準可以先理解成：

**一套描述 Web 技術應該如何被寫出來、被解析、被執行、被呈現、被互動與被互通的共同規範。**

最常見的入門理解是三層分工：

| 層次 | 對應技術 | 負責什麼 | 常見例子 |
|---|---|---|---|
| 結構 | `HTML` | 描述頁面內容和語意結構 | 標題、段落、圖片、表單、列表 |
| 表現 | `CSS` | 控制外觀與版面 | 顏色、字體、間距、排版、動畫 |
| 行為 | `JavaScript` | 處理互動與動態邏輯 | 點擊事件、表單驗證、資料請求 |

但 Web 標準的範圍比這張表更大，還可能包含：

- `HTML` 如何解析
- `CSS` 如何計算樣式
- `JavaScript` 語言本身如何運作
- `DOM` 如何表示頁面結構
- `HTTP/HTTPS` 如何請求和回應資源
- `URL` 如何定位資源
- 各種 Web API 如何讓頁面操作瀏覽器能力
- 可存取性、安全、隱私、國際化等要求

所以：

**三層分工幫你理解頁面怎麼寫；Web 標準則是讓整個 Web 平台能穩定合作的一整套規則。**

## 為什麼需要 Web 標準？

![為甚麼需要了解 Web 標準示意圖 1](./images/1775987332667_NnwBDqRUnD.png)

![為甚麼需要了解 Web 標準示意圖 2](./images/1775987332672_Meoxt7ECsx.png)

Web 的特點是開放、多平台、多瀏覽器。

同一個網站可能被不同使用者用不同環境打開：

- 桌面瀏覽器
- 手機瀏覽器
- 平板
- 不同作業系統
- 不同瀏覽器版本
- 輔助工具，例如螢幕閱讀器

如果沒有共同標準，各家瀏覽器可能會用完全不同方式理解同一份程式碼。

結果可能是：

- 同一個頁面在 A 瀏覽器正常，在 B 瀏覽器跑版。
- 某個按鈕在這裡能點，在另一個環境不能點。
- 表單在某些瀏覽器能送出，在某些瀏覽器行為不同。
- 輔助工具無法正確理解頁面結構。
- 開發者必須針對每個瀏覽器寫一套特殊版本。

Web 標準的價值就在於：

- 讓瀏覽器有共同實作依據。
- 讓開發者有共同寫作規則。
- 讓工具能檢查和理解程式碼。
- 讓使用者在不同裝置上有更一致的體驗。
- 讓網站更容易維護、擴充與長期使用。

## Web 標準不是只有三個檔案

很多入門課會說：

- `HTML` 負責結構
- `CSS` 負責表現
- `JavaScript` 負責行為

這個說法非常適合入門，但它不是 Web 標準的完整範圍。

更完整地看，Web 標準是一整組規範與協作結果。常見範圍包含：

| 範圍 | 解決的問題 | 入門理解 |
|---|---|---|
| `HTML` | 頁面內容、語意、表單、媒體、文件結構 | 頁面有哪些東西 |
| `CSS` | 樣式、排版、響應式、動畫、視覺呈現 | 頁面看起來怎麼樣 |
| `ECMAScript` | `JavaScript` 語言核心語法與行為 | JS 語言本身怎麼運作 |
| `DOM` | 程式如何操作文件結構 | JS 如何改變頁面 |
| Web APIs | 瀏覽器提供給網頁使用的能力 | 例如事件、儲存、Fetch、Canvas |
| `URL/URI` | 如何定位資源 | 瀏覽器要去哪裡找資料 |
| `HTTP` | 請求與回應如何溝通 | 瀏覽器和伺服器怎麼說話 |
| 可存取性 | 讓更多使用者能理解與操作內容 | 螢幕閱讀器、鍵盤操作、語意標籤 |
| 安全與隱私 | 限制危險行為，保護使用者資料 | 權限、同源限制、HTTPS |
| 國際化 | 支援不同語言、文字方向與地區格式 | `lang`、編碼、文字方向 |

所以，當你聽到「Web 標準」時，不要只想到三個檔案。
更精準地說，它是一整個 Web 平台的共同規則集合。

## 誰在制定 Web 標準？

初學階段不需要背所有標準組織，但要知道：Web 標準不是某一家瀏覽器公司自己說了算。

常見相關組織包含：

| 組織 | 你可以先這樣理解 | 常見關聯 |
|---|---|---|
| W3C | 制定和推動多種 Web 標準與指南的組織 | CSS、SVG、WCAG、Web 平台多項規範 |
| WHATWG | 維護 HTML 等 Living Standard 的組織 | HTML、DOM、Fetch 等 |
| TC39 / Ecma | 推動 ECMAScript，也就是 JavaScript 語言標準 | JS 語言語法與內建能力 |
| IETF | 制定許多網際網路協議標準 | HTTP、URI、TLS 相關協議等 |

這裡最重要的不是背組織名稱，而是理解：

**Web 標準是多方協作的結果，包含標準組織、瀏覽器廠商、開發者、工具作者、測試與實際使用經驗。**

也因為如此，Web 標準不是靜止不變的。
新功能通常會經歷提案、討論、規範、實作、測試、瀏覽器支援、開發者採用等過程。

## Web 標準的入門理解：三個核心分工

![Web 標準的構成](./images/1775987332672_A1ZXaM9J2f.png)

雖然 Web 標準範圍很大，但初學前端時，最實用的入口仍然是三層分工。

### 1. 結構：HTML

`HTML` 負責描述頁面中有哪些內容與元素。

例如：

- 標題
- 段落
- 圖片
- 連結
- 列表
- 表格
- 表單
- 影片與音訊

它關心的是：

**這個頁面有哪些內容？這些內容之間是什麼關係？**

例如一個登入頁面，`HTML` 會描述：

- 這裡有一個標題
- 這裡有帳號輸入框
- 這裡有密碼輸入框
- 這裡有送出按鈕
- 輸入框和文字標籤之間有對應關係

好的 `HTML` 不只是讓畫面看起來有東西，而是讓瀏覽器、搜尋引擎、輔助工具都能理解內容結構。

### 2. 表現：CSS

`CSS` 負責控制頁面的外觀與視覺呈現。

例如：

- 顏色
- 字體
- 字級
- 間距
- 排版
- 邊框
- 背景
- 動畫
- 響應式版面

它關心的是：

**這些內容看起來怎麼樣？在不同螢幕上如何排列？**

同一份 `HTML`，可以透過不同 `CSS` 呈現出完全不同的視覺風格。
這也是為什麼把內容結構和視覺樣式分清楚很重要。

### 3. 行為：JavaScript

`JavaScript` 負責頁面的互動與動態邏輯。

例如：

- 點擊按鈕後展開內容
- 表單輸入時檢查格式
- 向後端請求資料
- 根據資料更新畫面
- 顯示彈窗、提示或錯誤訊息
- 控制動畫或互動狀態

它關心的是：

**頁面如何回應使用者操作？資料如何改變畫面？**

注意：嚴格說，`JavaScript` 是一個常用名稱；語言標準本身通常稱為 `ECMAScript`。
在日常前端開發中，大家通常仍會說「寫 JavaScript」。

## 三層分工不等於死板拆開

三層分工的目的不是把程式碼拆到越碎越好，也不是說任何情況下都不能有交集。

它真正強調的是：

- 結構盡量由 `HTML` 表達。
- 外觀盡量由 `CSS` 控制。
- 互動邏輯盡量由 `JavaScript` 處理。
- 不要用錯工具解決問題。

例如：

| 需求 | 較合理的負責層 |
|---|---|
| 這段文字是主標題 | `HTML`，用合適的標題元素 |
| 這個標題要變成藍色 | `CSS` |
| 點按鈕後顯示更多內容 | `JavaScript` |
| 表單欄位要有對應說明文字 | `HTML`，用語意和關聯 |
| 手機版改成單欄排列 | `CSS` |
| 送出表單前檢查輸入格式 | `HTML` 屬性與 `JavaScript` 都可能參與 |

所以：

**三層分工是一種讓程式更清楚、更好維護、更容易被瀏覽器和工具理解的設計方向，不是死板的教條。**

## 標準、瀏覽器實作、實際支援是三件事

這是理解 Web 標準時很重要的一點。

### 1. 標準

標準描述某個技術應該怎麼設計、怎麼解析、怎麼執行。

例如某個 HTML 元素可以有哪些屬性、某個 CSS 屬性應該如何計算、某個 JavaScript 語法應該如何運作。

### 2. 瀏覽器實作

瀏覽器廠商需要把標準落實到瀏覽器裡。

但不同瀏覽器：

- 開發進度可能不同
- 實作細節可能不同
- 修 bug 的時間可能不同
- 對舊功能的保留策略可能不同

### 3. 實際支援

即使某個功能已經進入規範，也不代表所有使用者的瀏覽器都能立刻使用。

你還要看：

- 哪些瀏覽器支援
- 哪些版本支援
- 是否需要開啟特殊設定
- 行動端是否支援
- 是否存在已知 bug
- 目標使用者是否真的使用這些瀏覽器版本

所以：

**有標準，不等於所有瀏覽器立刻完全一致；標準提供方向，實作和支援還需要時間。**

## Web 標準和相容性有什麼關係？

Web 標準的存在，可以讓不同瀏覽器盡量往共同規範靠攏。

但相容性問題仍然可能出現，原因包括：

- 不同瀏覽器使用不同引擎或實作策略
- 使用者瀏覽器版本不同
- 某些新功能尚未普遍支援
- 某些舊功能仍被保留但不建議使用
- 同一功能在不同平台可能有細節差異
- 開發者使用了非標準或過時寫法

所以學 Web 標準，不是為了幻想「所有瀏覽器永遠完全一樣」，而是為了：

- 優先使用有共同規範的寫法
- 避免依賴某個瀏覽器的特殊行為
- 更有依據地排查相容性問題
- 知道何時需要替代方案或漸進增強

## 初學者怎麼寫得更符合 Web 標準？

入門階段可以先從下面幾件事做起。

### 1. 用正確的 HTML 表達語意

不要只想「看起來像不像」，也要想「它本來是什麼」。

例如：

- 主標題用 `h1`
- 段落用 `p`
- 列表用 `ul` / `ol` / `li`
- 表單標籤用 `label`
- 按鈕行為用 `button`
- 頁面區塊可逐步學習 `header`、`nav`、`main`、`section`、`footer`

這能讓瀏覽器、搜尋引擎和輔助工具更容易理解內容。

### 2. 把樣式交給 CSS

不要只靠 HTML 元素或行內寫法硬做視覺效果。

比較好的方向是：

- 用 `HTML` 表達內容結構
- 用 `CSS` 控制字體、顏色、間距、排版
- 讓同一份結構可以比較容易換樣式

這會讓頁面更容易維護。

### 3. 把互動交給 JavaScript

如果需求是「使用者操作後頁面要改變」，通常就會進入 `JavaScript` 的範圍。

例如：

- 點擊後展開選單
- 輸入時即時提示錯誤
- 從後端取得資料後更新畫面

但也要記得：不是所有事情都要用 JavaScript。
如果 `HTML` 或 `CSS` 已經能自然解決，就不必硬加 JS。

### 4. 避免使用過時或非標準寫法

例如某些早期教材會出現已不建議使用的標籤或屬性。
學習時要逐步建立判斷：

- 這個寫法現在是否仍建議使用？
- 它是標準的一部分，還是某個瀏覽器早期的特殊功能？
- 是否有更語意化、更現代的替代寫法？

### 5. 使用驗證工具與開發者工具

符合標準不只靠感覺。

你可以使用：

- HTML 驗證工具檢查標籤結構
- CSS 檢查工具檢查語法
- 瀏覽器開發者工具檢查錯誤
- 相容性資料確認某個功能是否可用

初學階段不需要一次掌握所有工具，但要建立「可以檢查」的意識。

## 一個小例子：三層分工怎麼看？

假設要做一個登入表單。

### HTML 負責結構

```html
<form>
  <label for="account">帳號</label>
  <input id="account" name="account" type="text">

  <label for="password">密碼</label>
  <input id="password" name="password" type="password">

  <button type="submit">登入</button>
</form>
```

這裡的重點是：

- 使用 `form` 表示表單。
- 使用 `label` 說明輸入欄位。
- 使用 `input` 表示輸入控制項。
- 使用 `button` 表示按鈕。

### CSS 負責外觀

```css
form {
  max-width: 360px;
}

input {
  width: 100%;
  margin-bottom: 12px;
}
```

這裡的重點是控制寬度、間距與排版，而不是改變內容語意。

### JavaScript 負責互動

```js
const form = document.querySelector("form");

form.addEventListener("submit", (event) => {
  event.preventDefault();
  console.log("準備送出登入資料");
});
```

這裡的重點是處理送出事件，讓頁面能回應使用者操作。

這個例子不追求完整功能，而是幫你看出：

**HTML 描述內容，CSS 控制樣式，JavaScript 處理互動。**

## 常見混淆點

| 混淆說法 | 更清楚的理解 |
|---|---|
| Web 標準只是 HTML/CSS/JS 三件事 | 三層分工是入門框架，Web 標準還包含 DOM、HTTP、URL、Web API、可存取性、安全等 |
| 有標準就不會有相容性問題 | 標準提供共同方向，但瀏覽器實作與版本支援仍可能不同 |
| 能在我電腦打開就代表符合標準 | 單一瀏覽器正常不代表語法正確、語意合理或跨環境可靠 |
| 三層分離就是絕對不能混在一起 | 重點是分工清楚，不是形式上死板切割 |
| HTML 只負責把字放上去 | HTML 也負責語意與結構，會影響搜尋、可存取性和工具理解 |
| CSS 只是美化 | CSS 也負責排版、響應式、狀態呈現和使用體驗 |
| JavaScript 等於所有互動都要靠它 | 有些互動與表單能力可以先由 HTML/CSS 解決，JS 用在需要程式邏輯時 |
| 標準是固定不變的 | Web 標準會隨瀏覽器、需求、安全與平台能力持續演進 |

## 初學時最該記住的六句話

1. Web 標準是讓瀏覽器、工具和開發者有共同依據的一套規範。
2. `HTML` 負責內容結構，`CSS` 負責外觀表現，`JavaScript` 負責互動行為。
3. 三層分工是入門理解框架，不是 Web 標準的全部。
4. 符合標準能降低相容性問題，但不能保證所有瀏覽器完全一樣。
5. 寫前端時要優先使用語意清楚、標準化、可維護的寫法。
6. 判斷新功能能不能用，要同時看規範、瀏覽器支援和目標使用者環境。

## 自測問題

讀完這篇後，可以試著回答：

1. 為什麼 Web 需要共同標準？
2. Web 標準為什麼不只等於 `HTML`、`CSS`、`JavaScript`？
3. `HTML`、`CSS`、`JavaScript` 各自主要負責什麼？
4. 三層分工的目的，是不是要求任何情況都絕對拆開？
5. 為什麼有標準仍然可能出現瀏覽器相容性問題？
6. 初學者可以從哪些實作習慣開始寫得更符合 Web 標準？

如果這六題都能說清楚，這篇的核心就掌握住了。

## 一句話抓核心

Web 標準是一套讓 Web 技術能被共同理解、實作與使用的規範；入門時可先用 `HTML` 負責結構、`CSS` 負責表現、`JavaScript` 負責行為來理解，但要知道真正的 Web 標準還包含更大的平台規則、瀏覽器實作與相容性判斷。
