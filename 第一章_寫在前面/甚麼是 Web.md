> 所屬章節：第一章｜寫在前面

> 建議回查情境：剛開始學前端時想釐清 Web 是什麼、分不清 Web 與網際網路的關係、想理解瀏覽器如何取得網頁、想快速確認 URL、HTTP、資源與 JavaWeb 的基本意思、遇到網頁或資源載入問題時想知道先從哪裡排查

## 本節導讀

這一節要先回答一個最基礎的問題：`Web` 到底是什麼。

很多初學者會把 `Web`、網際網路、網站、網頁、瀏覽器、`HTTP`、伺服器全部混在一起。
它們確實常常一起出現，但不是同一件事。

閱讀時先抓住這條主線：

**Web 不是整個網際網路，也不是瀏覽器本身；Web 是建立在網際網路之上的一套資訊與資源存取系統。**

換成更白話的說法：

- 網際網路負責把世界各地的裝置連起來。
- Web 負責讓人透過網址、連結和通訊規則存取資源。
- 瀏覽器是使用 Web 最常見的工具。
- 伺服器負責接收請求並回傳對應內容或結果。

## 你會在這篇學到什麼

- `Web` 的基本定義
- `Web`、網際網路、瀏覽器、網站、網頁之間的差別
- `URL`、`HTTP/HTTPS`、`HTML`、資源在 Web 裡各自扮演什麼角色
- 瀏覽器打開一個網頁時，大致發生了什麼事
- `HTTP request / response` 在 Web 流程裡的最低限度結構
- 為什麼 `Web` 不等於瀏覽器，也不等於整個網際網路
- 遇到網頁打不開、圖片載不到、API 沒回應時，可以先從哪裡排查
- `JavaWeb` 的基本意思，以及它和 Web 的關係

## 這篇在解決什麼問題？

如果一開始沒有分清這些詞，後面學前端、後端、伺服器、路徑、表單、API 時很容易卡住。

常見混亂大概是這樣：

- 以為「有網路」就等於「在使用 Web」。
- 以為瀏覽器就是 Web。
- 以為網站、網頁、HTML、URL 是同一層概念。
- 以為打開一個頁面時，瀏覽器只拿到一個檔案。
- 以為 `HTTP` 就是 Web 本身。

這篇的目標不是背名詞，而是建立一張簡單、穩定的概念地圖。

## 30 秒先講結論

`Web` 是 `World Wide Web` 的簡稱，中文常譯為「全球資訊網」。

它可以先理解成：

**一套建立在網際網路之上的資源存取方式。使用者通常透過瀏覽器輸入 URL 或點擊連結，瀏覽器再透過 HTTP/HTTPS 向伺服器請求資源，最後把取得的內容解析並呈現出來。**

最簡化的分工如下：

| 名詞 | 先這樣理解 | 它解決的問題 |
|---|---|---|
| 網際網路 | 連接全球裝置的底層網路 | 裝置之間能不能連線 |
| Web | 建立在網際網路上的資訊存取系統 | 如何用統一方式取得與連結資源 |
| URL | 資源的位置或入口 | 要去哪裡找資源 |
| HTTP/HTTPS | 瀏覽器和伺服器常用的通訊規則 | 怎麼請求與回應資源 |
| request / response | HTTP 交換資料時的兩種基本訊息 | 誰提出需求、誰回傳結果 |
| HTML | 描述網頁結構的標記語言 | 頁面內容如何組織 |
| 瀏覽器 | 使用 Web 的常見客戶端 | 請求、解析、執行並顯示資源 |
| 伺服器 | 提供資源或服務的一端 | 接收請求並回傳結果 |
| 資源 | 被 URL/URI 識別、被請求指向的目標 | 實際想取得、呈現或操作的對象 |

## 先分清楚幾個核心名詞

### 1. 網際網路是底層連線基礎

網際網路負責把世界各地的電腦、手機、伺服器和其他裝置連接起來。

你可以把它想成道路或交通網：

- 它讓不同地方的裝置有機會互相傳資料。
- 它解決的是「能不能連起來」。
- 但它本身不等於「網頁怎麼被找到、請求、回傳、顯示」。

所以：

**網際網路是基礎設施，Web 是建立在這個基礎設施上的一種應用。**

有網際網路，不代表一定在使用 Web。
你也可能在使用電子郵件、即時通訊、檔案傳輸、遊戲連線或其他網路服務。

### 2. Web 是建立在網際網路上的資訊存取系統

`Web` 的重點不是「有網路」而已，而是：

- 可以透過 `URL` 指向資源。
- 可以透過超連結把不同資源連起來。
- 可以透過 `HTTP/HTTPS` 請求與回應資料。
- 可以用 `HTML` 等格式描述與呈現內容。
- 可以讓使用者從一個頁面跳到另一個頁面，形成一張互相連結的資訊網。

這也是為什麼它叫 `World Wide Web`。

`Web` 的核心不是某一個網站，也不是某一款瀏覽器，而是一整套讓資源可以被定位、請求、連結與呈現的系統。

### 3. 瀏覽器是使用 Web 的常見工具

瀏覽器是使用者最常用來存取 Web 的客戶端軟體。

常見瀏覽器包含：

- `Chrome`
- `Safari`
- `Firefox`
- `Edge`

瀏覽器通常會做這些事：

- 根據 `URL` 找到要請求的目標。
- 向伺服器發送 `HTTP/HTTPS` 請求。
- 接收伺服器回傳的內容。
- 解析 `HTML`、`CSS`、`JavaScript`。
- 顯示頁面並處理使用者互動。

所以瀏覽器不是 Web 本身，而是使用 Web 的常見入口。

### 4. URL 用來定位資源

`URL` 可以先理解成「資源的位置或入口」。

例如：

```text
https://example.com/articles/web-intro.html?from=nav#summary
```

這個 URL 大致可以拆成：

| 部分 | 例子 | 意思 |
|---|---|---|
| 協議 | `https` | 用什麼規則溝通 |
| 網域 | `example.com` | 要找哪個網站或服務 |
| 路徑 | `/articles/web-intro.html` | 要取得哪個具體資源 |
| 查詢參數 | `?from=nav` | 額外帶給伺服器或應用程式的資訊 |
| 片段識別 | `#summary` | 指向同一份資源裡的某個位置，通常不會送到伺服器 |

入門階段不用先背完整 URL 語法，但要知道：

**瀏覽器不是憑空知道要去哪裡拿資料，而是依照 URL 找到目標資源。**

另外也要先建立一個實務觀念：

**URL 的路徑不一定真的對應伺服器上的實體檔案。**

在早期或靜態網站裡，`/articles/web-intro.html` 可能真的對應某個 HTML 檔案。
但在現代後端或 Web App 裡，這個路徑也可能只是伺服器、框架或前端路由用來判斷「要回傳哪個頁面或資料」的入口。

### 5. HTTP / HTTPS 是常見的通訊規則

`HTTP` 和 `HTTPS` 是 Web 裡最常見的通訊規則。

可以先這樣理解：

- 瀏覽器發出請求，說明自己想要什麼。
- 伺服器收到請求，回傳對應結果。
- 這個請求與回應的過程，常見情況下會依照 `HTTP/HTTPS` 進行。

稍微展開來看：

- `request` 是客戶端送出的請求，常見資訊包含 method、目標 URL 或路徑、headers，以及可選的 body。
- `response` 是伺服器回傳的回應，常見資訊包含 status code、headers，以及可選的 body。
- `GET`、`POST`、`200`、`404`、`500`、`Content-Type`、Cookie、JSON 這些後面常見的詞，都會接在這條 request / response 主線上。

`HTTPS` 可以理解成透過 `TLS` 加上安全保護機制的 `HTTP`。
實務上，正式網站通常應該使用 `HTTPS`，因為它能保護傳輸過程中的資料安全。
但要注意：`HTTPS` 主要保護傳輸過程，不代表網站內容一定正確，也不代表後端程式一定沒有安全漏洞。

入門時先記住：

**瀏覽器和伺服器不是隨便傳資料，而是依照通訊規則交換請求與回應。**

### 6. 資源是被識別與請求的目標

在 Web 裡，資源不只是一整個網頁。

嚴格一點說：

- `URL` 或 `URI` 用來識別資源。
- `HTTP request` 會指向某個目標資源。
- 伺服器回傳的通常不是「資源這個抽象概念本身」，而是這個資源在某個時間點、某種格式下的表示資料，也就是 `representation`。

入門階段可以先把「資源」理解成你想取得、呈現或操作的對象。
在日常開發語境裡，大家也常把下面這些可被請求的內容直接稱為資源：

資源可以是：

- 一份 `HTML` 文件
- 一份 `CSS` 樣式檔
- 一支 `JavaScript` 腳本
- 一張圖片
- 一段影片或音訊
- 一個字型檔
- 一份 PDF
- API 回傳的 `JSON` 資料

所以當你打開一個網站時，瀏覽器通常不是只拿到一個檔案。
它常常會先取得 `HTML`，再根據 `HTML` 裡引用的內容，繼續請求樣式、腳本、圖片、字型或資料。

### 7. 網站與網頁是 Web 的常見呈現形式

這兩個詞也要分開：

- **網頁**：通常指單一頁面，例如一篇文章頁、一個登入頁、一個商品頁。
- **網站**：通常指多個網頁與相關資源組成的一個整體，例如公司官網、部落格、商城。

也就是說：

- 網頁是比較小的單位。
- 網站是比較大的整體。
- Web 則是比單一網站更大的資源存取系統。

## Web 最常見的運作流程

用最簡化的流程來看：

1. 使用者在瀏覽器輸入網址，或點擊一個連結。
2. 瀏覽器根據 `URL` 判斷要請求哪個目標。
3. 瀏覽器透過 `HTTP/HTTPS` 向伺服器發出請求。
4. 伺服器收到請求後，找到或產生對應資源。
5. 伺服器把資源回傳給瀏覽器。
6. 瀏覽器解析回傳內容，例如 `HTML`、`CSS`、`JavaScript`。
7. 瀏覽器把結果呈現在畫面上。
8. 如果頁面還需要圖片、樣式、腳本或 API 資料，瀏覽器會繼續發出其他請求。

可以把整個流程記成：

```text
使用者操作
  -> 瀏覽器根據 URL 找目標
  -> 透過 HTTP/HTTPS 發出請求
  -> 伺服器回傳內容或結果
  -> 瀏覽器解析並顯示
```

這裡先不用追太深，例如 DNS、TCP、TLS、快取、CDN 等細節。
那些是之後理解網路與部署時會逐步補上的內容。

## 一個具體例子

假設你在瀏覽器輸入：

```text
https://example.com/about.html
```

可以先這樣理解：

1. `https` 表示瀏覽器會使用 HTTPS 這類安全通訊方式。
2. `example.com` 表示要找這個網域對應的服務。
3. `/about.html` 表示想取得其中一個頁面資源。
4. 伺服器回傳 `about.html` 這份 HTML 文件。
5. 瀏覽器讀到 HTML 之後，可能又發現它需要 `style.css`、`main.js` 和 `logo.png`。
6. 瀏覽器再分別請求這些資源。
7. 最後瀏覽器把結構、樣式、圖片和互動整合成你看到的頁面。

所以「打開一個網頁」背後，其實是一連串資源請求與解析過程。

## 實務回查：網頁出問題時先看哪裡？

這篇還不用深入除錯，但要先知道一個前端與全端開發都會反覆使用的入口：瀏覽器開發者工具裡的 `Network` 面板。

當你遇到下面問題時，通常可以先看 `Network`：

- 網頁打不開。
- 圖片、CSS 或 JavaScript 沒載入。
- 按下按鈕後沒有資料回來。
- 表單送出後沒有成功。
- 前端顯示錯誤，但不確定是前端問題還是後端問題。

入門階段先看幾個欄位就夠：

| 要看什麼 | 可以幫你判斷什麼 |
|---|---|
| `URL` | 瀏覽器到底請求了哪個資源 |
| Method | 這次請求是 `GET`、`POST` 還是其他方法 |
| Status | 伺服器回應是否成功，例如 `200`、`404`、`500` |
| Type / Content-Type | 回來的是 HTML、CSS、JS、圖片還是 JSON |
| Response | 伺服器實際回傳了什麼內容 |

例如：

- `404` 常表示請求的資源或路徑找不到。
- `500` 常表示伺服器處理請求時發生錯誤。
- CSS、JS 或圖片顯示失敗，不代表整個網站都沒回應，可能只是其中一個資源請求失敗。
- API 沒回資料時，要先確認 request 真的有送出去、URL 是否正確、status code 是什麼、response body 寫了什麼。

之後學到表單、`fetch`、API、Spring MVC、Controller、伺服器日誌時，這裡的 request / response 觀念會一直重複出現。

## Web 不等於什麼？

### Web 不等於瀏覽器

瀏覽器只是使用 Web 的工具之一。

除了瀏覽器之外，其他工具也可以請求 Web 資源，例如：

- 命令列工具
- App 內部的網路請求
- 程式中的 HTTP client
- 自動化測試工具

所以：

- **瀏覽器是 Web 的常見入口。**
- **Web 是一套資源定位、請求、連結與呈現的系統。**

兩者有關係，但不能畫上等號。

### Web 不等於整個網際網路

網際網路是底層網路基礎，Web 是其中一種常見應用。

你可以這樣分：

- 使用網站、網頁、瀏覽器打開文章：通常是在使用 Web。
- 使用電子郵件、某些即時通訊、線上遊戲連線：可能是在使用網際網路，但不一定是在使用 Web。

所以：

- **網際網路解決的是連線。**
- **Web 解決的是用統一方式存取與連結資訊。**

### Web 不等於單一網站

網站是 Web 上的一組資源。
Web 則是許多網站、頁面、資源、協議與連結共同形成的更大系統。

你可以把關係想成：

```text
網際網路
  -> Web
      -> 網站
          -> 網頁
              -> HTML、CSS、JavaScript、圖片等資源
```

這不是嚴格的技術分層圖，而是幫助初學者理解大小關係的學習模型。

## Web、前端、後端的關係

在 Web 系統裡，前端與後端通常會一起合作。

前端偏向：

- 使用者看到的頁面
- 使用者操作的介面
- 資料如何呈現
- 瀏覽器中的互動行為

後端偏向：

- 接收請求
- 處理業務邏輯
- 查詢或儲存資料
- 控制權限
- 回傳結果

以登入為例：

1. 使用者在網頁輸入帳號密碼。
2. 前端把資料送到後端。
3. 後端檢查帳號密碼是否正確。
4. 後端把結果回傳給前端。
5. 前端根據結果顯示登入成功或失敗。

這就是 Web 學習後面會一路遇到的基本互動模式。

## 什麼是 JavaWeb？

`JavaWeb` 通常是指使用 `Java` 技術開發 Web 應用。

它的重點不是「只要是 Java 程式，又能被瀏覽器碰到，就一定叫 JavaWeb」。
比較準確的理解是：

- 它屬於 Web 應用開發的一種技術路線。
- 程式通常執行在伺服器端。
- 它會接收瀏覽器或其他客戶端送來的請求。
- 它會處理業務邏輯、資料庫操作、權限驗證等工作。
- 最後再把 HTML、JSON 或其他結果回傳出去。

所以 `JavaWeb` 和這篇的關係是：

**先理解 Web 是一套資源請求與回應系統，再理解 JavaWeb 是用 Java 參與這套系統的伺服器端開發方式之一。**

## 常見混淆點

| 混淆說法 | 更清楚的理解 |
|---|---|
| Web 就是網際網路 | Web 是建立在網際網路上的一種常見應用 |
| Web 就是瀏覽器 | 瀏覽器是使用 Web 的常見客戶端 |
| 網站就是 Web | 網站是 Web 上的一組資源，Web 是更大的系統 |
| 網頁就是 HTML | HTML 常用來描述網頁結構，但網頁通常還會包含 CSS、JavaScript、圖片等資源 |
| HTTP 就是 Web | HTTP/HTTPS 是 Web 中常見的通訊規則，不是 Web 的全部 |
| URL 就是網站名稱 | URL 可以包含協議、網域、路徑、查詢參數等，用來定位資源 |
| 伺服器就是後端 | 伺服器偏承載服務的環境或角色，後端偏應用程式邏輯 |

## 初學時最該記住的六句話

1. `Web` 是 `World Wide Web`，中文常譯為全球資訊網。
2. `Web` 不是整個網際網路，而是建立在網際網路上的資源存取系統。
3. 瀏覽器是使用 Web 的常見工具，負責請求、解析、執行與顯示資源。
4. `URL` 用來定位資源，`HTTP/HTTPS` 用來請求與回應資源。
5. 打開網頁時，瀏覽器通常會取得多種資源，而不是只拿到一個檔案。
6. 真實開發遇到載入或 API 問題時，先從瀏覽器 `Network` 面板確認 request / response。

## 一句話抓核心

`Web` 是建立在網際網路上的資源存取系統；最常見的使用方式，是由瀏覽器根據 `URL`，透過 `HTTP/HTTPS` 向伺服器請求資源，再把取得的內容解析並呈現給使用者。

## 來源與查證

- [W3C：Architecture of the World Wide Web](https://www.w3.org/TR/webarch/)：查證 Web 架構中的資源、URI、互動、表示資料與 Web page 概念。
- [IETF：RFC 9110 HTTP Semantics](https://www.ietf.org/rfc/rfc9110.html)：查證 HTTP 的 request / response、client / server、resource 與 representation 概念。
- [MDN：What is a URL?](https://developer.mozilla.org/en-US/docs/Learn_web_development/Howto/Web_mechanics/What_is_a_URL)：查證 URL 的用途與 scheme、authority、path、query、fragment 等組成。
- [MDN：Overview of HTTP](https://developer.mozilla.org/en-US/docs/Web/HTTP/Guides/Overview)：查證 HTTP 在 Web 資源取得與多資源文件載入中的角色。
- [MDN：HTTPS](https://developer.mozilla.org/en-US/docs/Glossary/HTTPS)：查證 HTTPS 與 TLS 的基本關係。
- [WHATWG：HTML Standard, Edition for Web Developers](https://html.spec.whatwg.org/dev/introduction.html)：查證 HTML 作為 Web 核心標記語言的定位。
