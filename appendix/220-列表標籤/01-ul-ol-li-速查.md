# ul ol li 速查

> 查表內容整理自 `notes/220-列表標籤/01-列表標籤入門與-ul-ol.md`。

## 三種常見列表

| 類型 | 主要標籤 | 適合內容 |
| --- | --- | --- |
| 無序列表 | `<ul>`、`<li>` | 並列項目，例如功能特色、食物清單、新聞列表 |
| 有序列表 | `<ol>`、`<li>` | 有順序的項目，例如排行榜、操作步驟、流程 |
| 描述列表 | `<dl>`、`<dt>`、`<dd>` | 名詞與說明，例如術語解釋、規格描述 |

## ul 與 ol 選擇

| 情境 | 建議標籤 | 原因 |
| --- | --- | --- |
| 商品特色清單 | `<ul>` | 每個特色並列，沒有固定順序 |
| 新聞標題列表 | `<ul>` | 通常只是同類項目集合 |
| 排行榜 | `<ol>` | 名次順序是內容的一部分 |
| 安裝步驟 | `<ol>` | 步驟先後不能任意交換 |
| 帳單明細 | 視情況使用 `<ul>` 或表格 | 若只是項目集合可用列表；若有多欄資料，表格更適合 |

## li 結構規則

| 規則 | 說明 |
| --- | --- |
| `<ul>` 的直接子元素應該是 `<li>` | 列表中的每個項目都要放進 `<li>` |
| `<ol>` 的直接子元素應該是 `<li>` | 有序列表的每個步驟或排名仍然由 `<li>` 表示 |
| `<li>` 是列表項目的容器 | 可以包含文字與多數常見元素 |
| 外觀交給 CSS 調整 | 不應只因為想要圓點或數字來決定標籤 |

## 實務判斷準則

| 問題 | 判斷 |
| --- | --- |
| 內容是不是一組相關項目？ | 是，才適合使用列表 |
| 項目順序是否重要？ | 不重要用 `<ul>`；重要用 `<ol>` |
| 是否只是想做縮排或符號效果？ | 應用 CSS，不應硬套列表 |
| 列表項目是否都放在 `<li>` 裡？ | 應確認 `<ul>`、`<ol>` 的直接子元素是 `<li>` |

## 常見錯誤與修正

| 錯誤 | 問題 | 建議 |
| --- | --- | --- |
| 用外觀決定 `<ul>` 或 `<ol>` | 標籤選擇應先看語意 | 看內容是否有順序意義 |
| 把文字直接放在 `<ul>` 或 `<ol>` 裡 | 列表結構不清楚，也不符合內容模型 | 把每個項目放進 `<li>` |
| 把列表當成排版工具 | 列表適合表達一組相關項目，不是單純做縮排 | 若只是排列外觀，使用 CSS |
