# FAQ Index

## 用途

用具體問題快速找到本篇筆記中能回答的段落。

## 常見問題索引

| 問題 | 簡短回答 | 對應段落 |
| --- | --- | --- |
| `contenteditable` 屬性做什麼？ | 它讓使用者可以直接修改頁面上的元素內容。 | [`contenteditable`：讓元素變成可編輯](../../../notes/第05章_全局屬性/03-可編輯內容與拼寫檢查屬性.md#contenteditable讓元素變成可編輯) |
| `contenteditable` 有哪些常見值？ | 常見值包含 `true`、`false` 與 `plaintext-only`。 | [`contenteditable`：讓元素變成可編輯](../../../notes/第05章_全局屬性/03-可編輯內容與拼寫檢查屬性.md#contenteditable讓元素變成可編輯) |
| `plaintext-only` 代表什麼？ | 內容可以被編輯，但貼上內容會保留純文字並移除格式。 | [`contenteditable`：讓元素變成可編輯](../../../notes/第05章_全局屬性/03-可編輯內容與拼寫檢查屬性.md#contenteditable讓元素變成可編輯) |
| `contenteditable` 會自動儲存內容嗎？ | 不會。它只讓 DOM 內容可編輯，保存資料仍需要額外處理。 | [可編輯不等於自動儲存](../../../notes/第05章_全局屬性/03-可編輯內容與拼寫檢查屬性.md#可編輯不等於自動儲存) |
| 如果要保存可編輯內容，需要怎麼做？ | 通常需要用 JavaScript 讀取內容，再送到後端或存到本地儲存空間。 | [可編輯不等於自動儲存](../../../notes/第05章_全局屬性/03-可編輯內容與拼寫檢查屬性.md#可編輯不等於自動儲存) |
| `contenteditable` 內容會隨表單提交嗎？ | 不會。它不是表單控制項，不會像 `<input>` 或 `<textarea>` 一樣自動帶出欄位值。 | [以為 `contenteditable` 會自動送出表單](../../../notes/第05章_全局屬性/03-可編輯內容與拼寫檢查屬性.md#以為-contenteditable-會自動送出表單) |
| `spellcheck` 屬性做什麼？ | 它提示瀏覽器是否要對元素中的文字進行拼寫與語法檢查。 | [`spellcheck`：控制拼寫與語法檢查](../../../notes/第05章_全局屬性/03-可編輯內容與拼寫檢查屬性.md#spellcheck控制拼寫與語法檢查) |
| 哪些內容可以被瀏覽器拼寫檢查？ | 文字輸入值、`<textarea>` 文字與可編輯元素中的文字通常可以被檢查。 | [`spellcheck`：控制拼寫與語法檢查](../../../notes/第05章_全局屬性/03-可編輯內容與拼寫檢查屬性.md#spellcheck控制拼寫與語法檢查) |
| `spellcheck="true"` 一定會生效嗎？ | 不一定。實際行為會受瀏覽器、系統語言與使用者設定影響。 | [以為 `spellcheck` 一定會生效](../../../notes/第05章_全局屬性/03-可編輯內容與拼寫檢查屬性.md#以為-spellcheck-一定會生效) |
| 什麼情況適合停用拼寫檢查？ | 使用者名稱、帳號或專有字串等不是自然語言文字的欄位，通常適合停用。 | [搭配使用範例](../../../notes/第05章_全局屬性/03-可編輯內容與拼寫檢查屬性.md#搭配使用範例) |
| 一般文字輸入應該用 `contenteditable` 嗎？ | 一般文字輸入應優先使用具備表單語意與瀏覽器支援的輸入欄位。 | [實務使用場景](../../../notes/第05章_全局屬性/03-可編輯內容與拼寫檢查屬性.md#實務使用場景) |
