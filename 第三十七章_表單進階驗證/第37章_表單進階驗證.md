# 表單進階驗證

> - 所屬章節：第三十七章｜表單進階驗證  
> - 關鍵字：表單驗證、constraint validation、`required`、`pattern`、`min`、`max`、`step`、`minlength`、`maxlength`、`readonly`、`disabled`、`autocomplete`、`novalidate`  
> - 建議回查情境：想限制必填、格式、字數、數值範圍、日期範圍，或想分清前端驗證和後端驗證時

## 本節導讀

表單不只讓使用者輸入資料，也可以描述「什麼樣的輸入才合理」。

HTML 內建了一些表單驗證能力，例如必填、字數限制、數值範圍、email 格式、url 格式、正則格式等。

但這一章一開始就要先記住：

> HTML 表單驗證是使用者體驗輔助，不是安全邊界。真正重要的資料仍然必須在後端重新驗證。

## 30 秒複習入口

- `required` 表示欄位必填。
- `type="email"`、`type="url"` 會帶來基本格式檢查。
- `minlength` / `maxlength` 限制字元長度。
- `min` / `max` / `step` 常用於 number、range、date 等型別。
- `pattern` 用正則表達式描述輸入格式。
- `readonly` 不能改但通常會提交。
- `disabled` 不能操作，通常不會提交。
- `novalidate` 可以關閉整個表單的內建驗證。

## 速查區

### 常見驗證屬性

| 屬性 | 用途 | 常見元素 |
|---|---|---|
| `required` | 必填 | `input`、`select`、`textarea` |
| `minlength` / `maxlength` | 文字長度限制 | 文字類輸入 |
| `min` / `max` | 最小值 / 最大值 | `number`、`date`、`range` |
| `step` | 步進間隔 | `number`、`range`、日期時間類 |
| `pattern` | 正則格式限制 | text、tel、email、url、password、search |
| `readonly` | 只能讀不能改 | 多數文字類輸入 |
| `disabled` | 禁用控件 | 多數表單控件 |

### 狀態差異

| 屬性 | 可聚焦 | 可修改 | 通常會提交 |
|---|---:|---:|---:|
| `readonly` | 是 | 否 | 是 |
| `disabled` | 否 | 否 | 否 |

## 正文筆記

### 1. 最基本的必填欄位

```html
<form>
  <label for="email">Email</label>
  <input id="email" name="email" type="email" required>

  <button type="submit">送出</button>
</form>
```

這裡有兩層驗證：

- `required`：不能空白。
- `type="email"`：瀏覽器會做基本 email 格式檢查。

如果使用者沒填或格式不符合，瀏覽器會阻止提交並提示錯誤。

### 2. `required` 要注意什麼？

`required` 是布林屬性。

```html
<input type="text" required>
```

不需要寫成：

```html
<input type="text" required="required">
```

雖然很多瀏覽器也能理解，但現代 HTML 常寫簡潔布林屬性。

單選按鈕要特別注意：

```html
<label><input type="radio" name="level" value="beginner" required> 初學</label>
<label><input type="radio" name="level" value="advanced"> 進階</label>
```

同一組 `name` 的 radio，只要其中一個帶 `required`，就表示這組必須選一個。

### 3. 文字長度限制

```html
<label for="username">使用者名稱</label>
<input
  id="username"
  name="username"
  type="text"
  minlength="3"
  maxlength="16"
  required>
```

這裡表示：

- 至少 3 個字元。
- 最多 16 個字元。
- 不可空白。

`maxlength` 常見於帳號、暱稱、短留言。  
但不要只靠前端限制，後端仍應確認資料長度。

### 4. 數值範圍與步進

```html
<label for="quantity">數量</label>
<input
  id="quantity"
  name="quantity"
  type="number"
  min="1"
  max="10"
  step="1">
```

這裡表示：

- 最小值 1。
- 最大值 10。
- 每次增減以 1 為單位。

`min`、`max`、`step` 也常用於日期、時間、range 等型別。

### 5. `pattern` 如何使用？

`pattern` 用正則表達式描述欄位格式。

```html
<label for="code">邀請碼</label>
<input
  id="code"
  name="code"
  type="text"
  pattern="[A-Z0-9]{6}"
  title="請輸入 6 位大寫英文字母或數字">
```

這裡希望使用者輸入 6 位大寫英文字母或數字。

常搭配 `title` 或旁邊的提示文字，讓使用者知道格式要求。  
否則使用者只會看到錯誤，卻不知道應該怎麼修正。

### 6. `placeholder` 不能取代 `<label>`

```html
<label for="phone">手機號碼</label>
<input id="phone" name="phone" type="tel" placeholder="0912-345-678">
```

`placeholder` 是輸入前的提示文字。  
使用者一旦輸入內容，它通常就消失。

`<label>` 是欄位名稱，應該穩定存在，讓使用者和輔助技術知道這個欄位在問什麼。

所以不要這樣做：

```html
<input name="phone" type="tel" placeholder="手機號碼">
```

這對可用性與可存取性都比較弱。

### 7. `readonly` 和 `disabled`

`readonly` 表示欄位不能修改，但值通常仍會提交。

```html
<input name="email" value="user@example.com" readonly>
```

`disabled` 表示控件被禁用，通常不能聚焦、不能操作，也不會被提交。

```html
<input name="coupon" value="SPRING" disabled>
```

初學時可以這樣記：

- 還要提交，只是不讓改：`readonly`。
- 整個控件失效，不要提交：`disabled`。

### 8. `autocomplete` 在做什麼？

`autocomplete` 告訴瀏覽器是否可以協助自動填入，以及這個欄位期待什麼資料。

```html
<input name="email" type="email" autocomplete="email">
<input name="given-name" autocomplete="given-name">
<input name="family-name" autocomplete="family-name">
```

它不只是 `on` / `off`，也可以用更具體的值描述欄位用途。

這對使用者體驗很重要，尤其是登入、註冊、結帳、地址填寫等表單。

### 9. `accept` 和 `multiple`

檔案上傳常用：

```html
<label for="avatar">上傳頭像</label>
<input id="avatar" name="avatar" type="file" accept="image/png, image/jpeg">
```

`accept` 提示瀏覽器檔案類型限制。  
但它不是安全保證，後端仍要檢查檔案類型與內容。

允許多檔：

```html
<input name="photos" type="file" accept="image/*" multiple>
```

`multiple` 也可以用在部分其他輸入型別，例如 email。

### 10. 關閉內建驗證

整個表單關閉內建驗證：

```html
<form novalidate>
  <input type="email" required>
  <button type="submit">送出</button>
</form>
```

單一提交按鈕關閉驗證：

```html
<button type="submit" formnovalidate>草稿儲存</button>
```

常見情境：

- 儲存草稿時不要求所有欄位完整。
- 專案改用自訂 JavaScript 驗證流程。
- 測試表單提交流程。

### 11. 前端驗證不能取代後端驗證

HTML 驗證可以被繞過，例如：

- 使用者關閉驗證。
- 透過開發者工具改 HTML。
- 直接送 HTTP 請求。
- 瀏覽器支援差異或腳本錯誤。

所以資料進到伺服器後，後端仍然必須重新驗證。

可以這樣分工：

- HTML 驗證：即時提示、減少錯誤、改善體驗。
- 後端驗證：保護資料完整性與安全。

## 常見問法

### `required` 可以保證資料一定有值嗎？

不能作為安全保證。

它只能在瀏覽器端提供提交前檢查。  
後端仍然要檢查必填資料。

### `pattern` 可以處理所有格式驗證嗎？

不建議。

太複雜的格式驗證會讓規則難維護，也可能讓錯誤提示不清楚。  
能用明確 `type`、`min`、`max`、`maxlength` 解決的，通常先用這些。

### `disabled` 欄位為什麼沒有送到後端？

因為禁用控件通常不會參與表單提交。  
如果需要提交值但不讓使用者改，應考慮 `readonly` 或其他資料處理方式。

### `placeholder` 寫了欄位名稱，還需要 label 嗎？

需要。

`placeholder` 只是提示，不是穩定欄位名稱。  
尤其在可存取性和長表單情境中，`label` 很重要。
