表單進階驗證 Demo

這個範例對應筆記中的 HTML Constraint Validation：`required`、`type`、`minlength`、`pattern`、`min/max/step`、`readonly`、`disabled`、`accept`、`novalidate`、`formnovalidate`，以及 `checkValidity()`、`reportValidity()`、`setCustomValidity()`。

觀察方式

  • 輸入不合法資料，觀察右側 `ValidityState`。
  • 按 `reportValidity()` 觸發瀏覽器原生提示。
  • 切換 `novalidate`，比較原生驗證是否被跳過。
  • 用「儲存草稿」觀察 `formnovalidate`。

程度

同一組 radio 使用同一個 `name`；其中一個加 `required` 即可要求整組必選。

興趣，至少選 2 項

HTML 不能直接限制 checkbox 群組至少選幾項，這裡用 `setCustomValidity()` 補上。

檔案與狀態

`accept` 只是提示檔案挑選器，不等於安全驗證；後端仍要檢查。

`readonly` 可送出，也會進入 `FormData`。

`disabled` 不會送出，也不會參與原生驗證。

pattern

`pattern` 寫法重點

<input
  type="tel"
  inputmode="numeric"
  pattern="09[0-9]{8}"
  maxlength="10"
  autocomplete="tel">

HTML 的 `pattern` 不要包 `/ /`。筆記也提醒：手機、OTP、郵遞區號這類文字型數字，通常用 `type="tel"` 或 `type="text"` 搭配 `inputmode`，不要用 `type="number"`。

novalidate

`novalidate` 與 `formnovalidate`

<form novalidate>
  ...
</form>

<button type="submit" formnovalidate>
  儲存草稿
</button>

`novalidate` 放在 `form` 上，表示整張表單送出時不跑瀏覽器原生驗證。`formnovalidate` 放在特定送出按鈕上，適合草稿、預覽等流程。

client vs server

前端驗證不等於安全驗證

層級 適合處理 限制
HTML 原生驗證 必填、格式、長度、數值範圍、快速 UX 回饋。 可被停用或繞過，不可作為資料安全依據。
JavaScript 驗證 跨欄位規則、checkbox 至少 N 項、即時自訂訊息。 同樣可被關閉或繞過。
後端驗證 所有重要資料規則、權限、唯一性、檔案型別與大小。 不可省略；前端驗證只能改善體驗。
