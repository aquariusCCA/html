# Notes 轉 Appendix 工作流程提示詞

> 檔案位置：`prompts/workflows/notes-to-appendix.workflow.md`  
> 用途：定義如何從 `notes/` 正式筆記生成 `appendix/` 查表型資料，例如名詞表、元素表、屬性表、API 表、指令表與設定檔範例。  
> 建議搭配：
>
> - `prompts/system/base-role.md`
> - `prompts/system/appendix-generator.md`
> - `prompts/formats/appendix-format.md`
> - `prompts/requests/generate-appendix-from-notes.request.md`

---

## 一、工作流程定位

本工作流程負責定義「如何從 `notes/` 生成查表型附錄資料」。

它只負責生成流程，不負責：

- 定義通用教學角色
- 取代 `notes/` 正式筆記
- 生成 `review/` 複習材料
- 生成 `practice/` 練習題
- 生成 `demos/` 範例程式專案
- 補完 `notes/` 沒有提到的外部知識

生成時應同時遵守：

1. `base-role.md` 的 Knowledge Pack 基本規則。
2. `appendix-generator.md` 的附錄資料生成規則。
3. `appendix-format.md` 的固定輸出格式。

---

## 二、工作流程目標

你要根據 `notes/` 中的正式筆記，生成可以放入 `appendix/` 的查表型資料。

生成結果必須幫助學習者做到：

1. 快速查找名詞定義、語法、屬性、API、指令或設定。
2. 快速比較相似概念、相似語法或相似使用情境。
3. 從表格中找到最小可用範例。
4. 根據來源索引回到 `notes/` 深入閱讀。
5. 將正式筆記轉換成可長期維護的參考資料。

---

## 三、輸入資料

請根據使用者提供的資料進行生成。

### 必要輸入

至少需要提供以下其中一種：

1. `notes/` 的某一份正式筆記內容
2. 某章節的正式筆記片段
3. 某個明確知識點的筆記內容

### 建議輸入

若使用者有提供，請一併參考：

| 資料 | 說明 |
|---|---|
| notes 路徑 | 用來標示附錄條目的來源 |
| 章節名稱 | 用來決定附錄資料所屬主題 |
| 目標輸出路徑 | 通常為 `appendix/{chapter-slug}/` 或 `appendix/{chapter-slug}-appendix.md` |
| 附錄類型 | 名詞表、元素表、屬性表、API 表、指令表、設定檔範例、程式碼片段 |
| 條目粒度 | 章節級、主題級、單一語法級 |
| 是否保留範例 | 預設保留最小可用範例 |
| 是否建立來源索引 | 預設建立 |

若部分資料未提供，應根據 `notes/` 內容合理推斷；無法推斷時，標示為「待確認」。

---

## 四、整體流程

請按照以下順序處理。

```text
讀取 notes
  ↓
界定附錄範圍
  ↓
辨識可查表資訊
  ↓
分類附錄條目
  ↓
設計表格欄位
  ↓
生成附錄內容
  ↓
建立來源索引
  ↓
執行品質檢查
  ↓
輸出最終結果
```

---

## 五、流程 1：讀取 notes 並界定範圍

請先完整理解使用者提供的 `notes/`，判斷這份筆記可以支撐哪些附錄資料。

你需要辨識：

1. 這份 `notes` 的主題是什麼。
2. 主要章節與小節有哪些。
3. 是否有明確名詞定義。
4. 是否有 HTML 元素、屬性、語法或規則。
5. 是否有 API、方法、事件、指令或設定。
6. 是否有可抽成常用片段的程式碼。
7. 是否有適合對照比較的概念。
8. 是否有常見錯誤或注意事項。
9. 是否有資訊不足、不適合生成的附錄類型。

如果 `notes` 資訊不足，請明確標註：

```text
此 notes 內容不足以生成「{附錄類型}」，原因：{不足原因}
```

不要為了湊表格而生成與 `notes` 無關的條目。

---

## 六、流程 2：辨識可查表資訊

請把 `notes` 內容轉換成可查詢、可排序、可維護的附錄條目。

| notes 內容類型 | 適合附錄類型 | 轉換方式 |
|---|---|---|
| 定義與核心概念 | 名詞表 | 整理成術語、定義、用途、注意事項、來源 |
| HTML 標籤或元素 | 元素表 | 整理成元素、用途、常用屬性、使用情境、範例 |
| 屬性或參數 | 屬性表、參數表 | 整理成名稱、適用對象、值、用途、注意事項 |
| API、方法或事件 | API 表 | 整理成名稱、用途、語法、回傳值、範例 |
| 指令或 CLI 操作 | 指令表 | 整理成指令、用途、常用參數、範例、注意事項 |
| 設定檔內容 | 設定檔範例 | 整理成最小範例、欄位說明、修改位置、注意事項 |
| 程式碼範例 | 常用片段 | 整理成場景、片段、使用前提、來源 |
| 常見錯誤 | 錯誤速查表 | 整理成錯誤現象、原因、修正方向、來源 |
| 概念差異 | 對照表 | 整理成比較項目、概念 A、概念 B、選用建議 |

每個條目都應保留與 `notes` 的對應關係，避免附錄失去來源。

---

## 七、流程 3：分類附錄條目

請先規劃再生成，不要直接輸出零散表格。

### 附錄類型選擇原則

| 條件 | 建議輸出 |
|---|---|
| `notes` 主要是概念說明 | 名詞表、相似概念對照表 |
| `notes` 主要介紹 HTML 元素 | 元素表、屬性表、常用片段 |
| `notes` 主要介紹語法規則 | 語法速查表、錯誤速查表 |
| `notes` 主要介紹 API 或方法 | API 表、參數表、回傳值表 |
| `notes` 主要介紹工具或環境 | 指令表、設定檔範例 |
| `notes` 包含多種內容 | 依類型分區輸出，不硬塞到同一張表 |

### 條目數量建議

| notes 內容狀況 | 建議條目數量 |
|---|---:|
| 單一小知識點 | 3～8 筆 |
| 一個小節，含 2～3 個概念 | 6～15 筆 |
| 一份完整筆記 | 10～30 筆 |
| 一個完整章節 | 20～60 筆 |
| notes 資訊不足 | 只生成可支撐的條目，並說明不足處 |

不需要每次都填滿所有表格。若某區塊缺乏 `notes` 支撐，請標示原因。

---

## 八、流程 4：設計表格欄位

請根據附錄類型選擇穩定欄位。

### 1. 名詞表

建議欄位：

| 欄位 | 說明 |
|---|---|
| 名詞 | 技術詞或概念名稱 |
| 英文 | 常見英文原文，若沒有則填「無」 |
| 簡要定義 | 一句話說明 |
| 用途 | 何時需要知道或使用 |
| 注意事項 | 常見誤解或限制 |
| 來源 | 對應 notes 章節或小節 |

### 2. 元素表

建議欄位：

| 欄位 | 說明 |
|---|---|
| 元素 | HTML 標籤或元素名稱 |
| 用途 | 元素負責表達的語意或功能 |
| 常用屬性 | `notes` 中提到的屬性 |
| 使用情境 | 適合出現的場景 |
| 最小範例 | 最短可理解程式碼 |
| 注意事項 | 使用限制或常見錯誤 |
| 來源 | 對應 notes 章節或小節 |

### 3. 屬性表

建議欄位：

| 欄位 | 說明 |
|---|---|
| 屬性 | attribute 名稱 |
| 適用元素 | 可用在哪些元素上 |
| 常見值 | `notes` 中提到的值 |
| 用途 | 此屬性的目的 |
| 範例 | 最小可用片段 |
| 注意事項 | 使用限制或錯誤提醒 |
| 來源 | 對應 notes 章節或小節 |

### 4. API 表

建議欄位：

| 欄位 | 說明 |
|---|---|
| API / 方法 | 名稱 |
| 類型 | 方法、屬性、事件、物件等 |
| 用途 | 解決什麼問題 |
| 語法 | 最小語法 |
| 回傳值 | 若 notes 有提到則填寫 |
| 範例 | 最小可用片段 |
| 注意事項 | 限制或常見錯誤 |
| 來源 | 對應 notes 章節或小節 |

### 5. 指令表

建議欄位：

| 欄位 | 說明 |
|---|---|
| 指令 | CLI 指令 |
| 用途 | 執行目的 |
| 常用參數 | `notes` 中提到的參數 |
| 範例 | 可直接參考的指令 |
| 注意事項 | 執行前提或風險 |
| 來源 | 對應 notes 章節或小節 |

### 6. 設定檔範例

建議欄位：

| 欄位 | 說明 |
|---|---|
| 設定名稱 | 設定項目或檔名 |
| 使用情境 | 為什麼需要這段設定 |
| 最小範例 | 可複用的設定片段 |
| 欄位說明 | 各欄位用途 |
| 修改位置 | 若 notes 有提到則填寫 |
| 注意事項 | 限制或常見錯誤 |
| 來源 | 對應 notes 章節或小節 |

---

## 九、流程 5：生成附錄內容

生成時請遵守：

1. 表格內容要短，不把完整教學段落塞進表格。
2. 若需要較長補充，放在表格後的「補充說明」區。
3. 程式碼範例要最小化，且必須標示語言。
4. 每個範例只能展示一個主要知識點。
5. 若條目來自多個 `notes` 小節，來源欄可以列出多個來源。
6. 不確定來源時標示「來源待確認」，不要假裝知道。

---

## 十、流程 6：建立來源索引

附錄資料最後應建立來源索引，方便回查。

來源索引建議包含：

| 附錄區塊 | 對應 notes 路徑 | 對應章節或小節 | 備註 |
|---|---|---|---|
| 名詞表 | `notes/...` | `{小節名稱}` |  |
| 元素表 | `notes/...` | `{小節名稱}` |  |

若使用者沒有提供 `notes` 路徑，請標示：

```text
notes 路徑未提供，來源索引只能標示章節或小節名稱。
```

---

## 十一、品質檢查

輸出前請自行檢查：

1. 是否完全根據 `notes/` 內容生成？
2. 是否沒有加入無根據的外部知識？
3. 是否偏向查表型資料，而不是重新教學？
4. 是否有保留來源索引？
5. 表格欄位是否穩定、清楚、可維護？
6. 範例是否短小、可理解、可複用？
7. 是否明確標示資訊不足的區塊？
8. 是否適合保存成 Markdown？
