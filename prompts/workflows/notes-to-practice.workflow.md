# Notes 轉 Practice 工作流程提示詞

> 檔案位置：`prompts/workflows/notes-to-practice.workflow.md`  
> 用途：定義如何從 `notes/` 正式筆記生成 `practice/` 練習題、實作任務、改錯題與重構題。  
> 建議搭配：
>
> - `prompts/system/learning-coach.md`
> - `prompts/system/notes-practice-generator-rules.md`
> - `prompts/formats/practice-format.md`
> - `prompts/requests/generate-practice-from-note.request.md`

---

## 一、工作流程定位

本工作流程負責定義「如何從 `notes/` 生成練習材料」。

它只負責生成流程，不負責：

- 定義通用教學角色
- 定義所有題型的完整輸出格式
- 實際指定某一次生成任務的 notes 內容
- 取代 `notes/` 原始筆記
- 生成 `demos/` 範例程式

生成時應同時遵守：

1. `learning-coach.md` 的教學角色與回答原則。
2. `notes-practice-generator-rules.md` 的題目生成規則。
3. `practice-format.md` 的固定輸出格式。

---

## 二、工作流程目標

你要根據 `notes/` 中的正式筆記，生成可以放入 `practice/` 的練習材料。

生成結果必須幫助學習者做到：

1. 檢查是否理解 notes 中的核心概念。
2. 練習把概念轉成具體程式或判斷。
3. 透過改錯題訓練 debug 能力。
4. 透過重構題訓練程式碼品質與設計能力。
5. 透過實作任務銜接真實開發情境。
6. 透過提示、參考答案與評分標準完成自我檢查。

---

## 三、輸入資料

請根據使用者提供的資料進行生成。

### 必要輸入

至少需要提供以下其中一種：

1. `notes/` 的某一份正式筆記內容
2. 某章節的正式筆記片段
3. 某個明確知識點的筆記內容

### 建議輸入

若使用者有提供，請一併參考：

| 資料 | 說明 |
|---|---|
| notes 路徑 | 用來標示每題對應來源 |
| 章節名稱 | 用來決定練習題所屬主題 |
| 目標輸出路徑 | 通常為 `practice/{chapter-slug}/` |
| 題型偏好 | 概念理解題、練習題、實作任務、改錯題、重構題 |
| 題目數量 | 若未提供，請依 notes 內容量合理判斷 |
| 難度範圍 | 使用 Lv1～Lv5 |
| 是否提供答案 | 預設提供提示、參考答案與評分標準 |
| 是否限制技術 | 例如只使用原生 HTML / CSS / JavaScript |

若部分資料未提供，應根據 notes 內容合理推斷；無法推斷時，標示為「待確認」。

---

## 四、整體流程

請按照以下順序處理。

```text
讀取 notes
  ↓
界定主題與範圍
  ↓
萃取可出題知識點
  ↓
判斷適合的題型
  ↓
設計題目清單與難度順序
  ↓
生成題目區
  ↓
生成提示、參考答案與評分標準
  ↓
執行品質檢查
  ↓
輸出最終結果
```

---

## 五、流程 1：讀取 notes 並界定範圍

請先完整理解使用者提供的 notes，判斷這份筆記可以支撐哪些練習材料。

你需要辨識：

1. 這份 notes 的主題是什麼。
2. 主要知識點有哪些。
3. 哪些內容適合轉成概念理解題。
4. 哪些內容適合轉成練習題或實作任務。
5. 是否有常見錯誤可以轉成改錯題。
6. 是否有可改善的程式碼或設計情境可以轉成重構題。
7. 是否有 notes 資訊不足、無法合理出題的題型。

如果 notes 資訊不足，請明確標註：

```text
此 notes 內容不足以生成「{題型}」，原因：{不足原因}
```

不要為了湊數量而生成與 notes 無關的題目。

---

## 六、流程 2：萃取可出題知識點

請把 notes 內容轉換成可檢查、可操作、可評分的知識點。

| notes 內容類型 | 適合題型 | 轉換方式 |
|---|---|---|
| 定義與核心概念 | 概念理解題 | 要求比較、判斷適用場景、解釋原因 |
| 語法與屬性 | 練習題、改錯題 | 要求補完程式、辨識錯誤、修正寫法 |
| 範例程式 | 練習題、實作任務、改錯題 | 改寫成新需求、加入限制或錯誤版本 |
| 常見錯誤 | 改錯題 | 提供錯誤程式碼，要求找出問題並修正 |
| 實務場景 | 實作任務 | 設計具體功能需求與驗收標準 |
| 程式碼品質提醒 | 重構題 | 提供重構前程式碼與重構目標 |
| 概念關係 | 概念理解題、重構題 | 要求說明取捨、選擇適合寫法 |

每個知識點都應保留與 notes 的對應關係，避免題目失去來源。

---

## 七、流程 3：設計題目清單

題目清單應先規劃再生成，不要直接輸出零散題目。

### 題目數量建議

| notes 內容狀況 | 建議題目數量 |
|---|---:|
| 單一小知識點 | 3～5 題 |
| 一個小節，含 2～3 個概念 | 5～8 題 |
| 一份完整筆記 | 8～15 題 |
| 一個完整章節 | 12～25 題 |
| notes 資訊不足 | 先生成可支撐的題目，並說明不足處 |

### 題型配置建議

若使用者沒有指定題型比例，請依 notes 內容選擇合適題型。

預設可使用以下配置：

| 題型 | 建議用途 |
|---|---|
| 概念理解題 | 先檢查基本理解 |
| 練習題 | 讓學習者套用語法或 API |
| 實作任務 | 將多個知識點組合成小功能 |
| 改錯題 | 檢查是否能辨識錯誤與修正原因 |
| 重構題 | 訓練可讀性、維護性與設計取捨 |

不需要每次都硬塞所有題型。若某題型缺乏 notes 支撐，請明確說明。

---

## 八、流程 4：產生各類題型

### 1. 概念理解題

概念理解題不能只是問「請說明 XXX 是什麼」。

應優先使用：

1. 比較差異。
2. 判斷適用場景。
3. 解釋程式行為。
4. 找出錯誤觀念。
5. 說明某個寫法的原因與影響。

每題都應附上明確評分重點。

### 2. 練習題

練習題應要求學習者實際操作或補完一小段程式。

每題應包含：

1. 題目目標。
2. 初始條件。
3. 作答要求。
4. 限制條件。
5. 預期結果或驗收標準。

若題目涉及程式碼，請提供足夠上下文，讓學習者知道要修改哪裡。

### 3. 實作任務

實作任務應接近真實開發場景，但範圍要能被 notes 支撐。

每題應包含：

1. 任務背景。
2. 功能需求。
3. 技術限制。
4. 完成條件。
5. 驗收標準。
6. 可選加分項目。

不要把小型實作題擴張成大型專案。

### 4. 改錯題

改錯題必須提供有問題的程式碼。

每題應包含：

1. 錯誤情境。
2. 有問題的程式碼。
3. 預期行為。
4. 修正任務。
5. 至少一個提示。
6. 參考修正方向或答案。

錯誤應與 notes 的知識點直接相關，不要加入 notes 沒教過的複雜錯誤。

### 5. 重構題

重構題必須提供重構前程式碼與重構目標。

每題應包含：

1. 重構前程式碼。
2. 目前程式碼的問題。
3. 重構目標。
4. 不可改變的外部行為。
5. 驗收標準。
6. 評分標準。

重構題的重點不是炫技，而是讓學習者知道如何讓程式更清楚、更穩定、更容易維護。

---

## 九、流程 5：安排難度順序

題目應按照 Lv1 到 Lv5 由淺入深排序。

| 難度 | 定位 | 適合題型 |
|---|---|---|
| Lv1 | 基礎理解題 | 概念理解題、簡單判斷題 |
| Lv2 | 應用題 | 練習題、簡單改寫題 |
| Lv3 | 小型實作題 | 練習題、實作任務、改錯題 |
| Lv4 | 綜合實作題 | 實作任務、複合改錯題 |
| Lv5 | 設計與重構題 | 重構題、設計取捨題 |

同一份練習中，建議先讓學習者建立信心，再逐步增加整合度。

---

## 十、流程 6：生成提示、參考答案與評分標準

預設輸出應分成：

1. 題目區
2. 提示區
3. 參考答案區
4. 評分標準區

### 提示區

提示應幫助學習者靠近答案，但不要直接解題。

可以使用：

- 觀察方向
- 關鍵知識點提醒
- 應檢查的程式碼位置
- 容易忽略的限制條件

### 參考答案區

參考答案應清楚但不過度發散。

程式題的參考答案應：

1. 可執行或可合理套用。
2. 對應題目限制。
3. 避免使用 notes 沒有鋪陳的大量進階技巧。
4. 必要時補充簡短說明。

### 評分標準區

評分標準應讓學習者知道如何自我檢查。

可包含：

- 是否達成功能需求
- 是否使用正確語法或概念
- 是否符合限制條件
- 是否能說明原因
- 是否處理常見錯誤或邊界情境
- 程式碼是否清楚、可維護

若使用者要求「不要給答案」，請省略參考答案區，並把驗收標準寫得更具體。

---

## 十一、建議輸出路徑

若使用者沒有指定輸出位置，建議使用：

```text
practice/
  {chapter-slug}/
    {note-slug}-practice.md
```

若同一章節拆成多份練習，可使用：

```text
practice/
  {chapter-slug}/
    01-{topic-slug}-practice.md
    02-{topic-slug}-practice.md
```

檔名應能看出來源主題，不要只命名為 `practice.md`。

---

## 十二、品質檢查

生成後，請自行檢查：

| 檢查項目 | 判斷標準 |
|---|---|
| 是否對應 notes | 每題都能追溯到 notes 的知識點 |
| 是否具體 | 題目有明確任務、限制或評分重點 |
| 是否排序合理 | 題目由淺入深，不突然跳到過高難度 |
| 是否題型合適 | 沒有硬塞 notes 無法支撐的題型 |
| 是否避免重複 | 同一批題目沒有高度相似的練習 |
| 是否答案策略一致 | 有答案或無答案模式符合使用者要求 |
| 是否可操作 | 程式題有足夠上下文與驗收標準 |
| 是否標示延伸 | notes 之外的內容有標示「補充延伸內容」 |

若品質檢查發現某題不合格，應修正題目，而不是只在最後提醒。
