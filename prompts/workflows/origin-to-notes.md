# origin-to-notes.md

## 工作流程定位

本工作流程用於將 `origin/` 中的原始資料，轉換成 `notes/` 中的正式教書型筆記。

此流程應搭配以下提示詞使用：

```text
prompts/system/base-role.md
prompts/system/html-teacher-role.md
prompts/formats/teaching-note-format.md
```

本檔案只負責「如何轉換」，不負責定義通用角色、不負責定義 HTML 教學角色，也不負責完整筆記格式。

---

## 適用情境

當輸入資料屬於以下類型時，可以使用此流程：

- HTML 課程逐字稿
- HTML 教材段落
- 官方文件片段
- 文章內容
- PDF 整理內容
- 程式碼範例
- 舊筆記
- 零散知識點
- AI 生成的初稿

目標是將這些資料整理成可長期維護的正式筆記。

---

## 輸入資料

使用此流程時，應提供以下資料：

```text
來源路徑：origin/...
目標主題：...
原始內容：...
補充要求：...
預期輸出路徑：notes/...
```

若部分資料未提供，應根據內容合理推斷；無法推斷時，標示為「待確認」。

### 輸入欄位規格

本流程的輸入欄位由 workflow 統一定義，`prompts/requests/` 中的 request 檔只負責套用這些欄位，不應自行新增未定義的 placeholder。

| 欄位 | Placeholder | 是否必填 | 說明 | 範例 |
|---|---|---|---|---|
| 來源路徑 | `{{origin_path}}` | 建議必填 | 原始資料所在路徑，用於追溯來源與填入正式筆記的 `source` 欄位。 | `origin/第02章_兼容性問題/01-兼容性問題.md` |
| 目標主題 | `{{target_topic}}` | 建議必填 | 本次整理的筆記主題，用於限制內容範圍與決定 H1 標題方向。 | `HTML 瀏覽器兼容性處理` |
| 原始內容 | `{{origin_content}}` | 必填 | 從來源路徑取得的原始 Markdown 或文字內容，是產生筆記的主要依據。 | 原始教材內容 |
| 補充要求 | `{{extra_requirements}}` | 否 | 本次任務的額外限制、偏好或注意事項。若沒有額外要求，可填「無」。 | `請補充初學者常見錯誤` |
| 預期輸出路徑 | `{{output_path}}` | 建議必填 | 預期產生的正式筆記位置，用於維持 `origin/` 到 `notes/` 的資料流一致性。 | `notes/第02章_兼容性問題/01-兼容性問題.md` |

若 request 檔使用 placeholder，應優先使用上表定義的名稱。若未來需要新增欄位，應先更新本節，再更新對應的 request 範本。

---

## 輸出目標

輸出結果應是一份可放入 `notes/` 的正式教書型筆記。

輸出內容應符合：

1. `base-role.md` 的通用規則。
2. `html-teacher-role.md` 的 HTML 教學規則。
3. `teaching-note-format.md` 的筆記格式。
4. `origin/` 原始資料的核心內容。

---

## 核心轉換流程

### Step 1：理解原始資料

先閱讀原始資料，判斷：

- 這份資料的主題是什麼？
- 它主要想教什麼？
- 是否包含多個主題？
- 哪些內容是核心知識？
- 哪些內容只是背景、雜訊或重複描述？
- 是否有範例程式？
- 是否有錯誤、過時或需要補充的地方？

不要急著改寫，必須先理解資料結構。

---

### Step 2：清理原始內容

移除或弱化不適合放入正式筆記的內容，例如：

- 廣告文字
- 課程寒暄
- 重複段落
- 無意義導覽
- 與主題無關的連結
- 不完整或無上下文的片段
- 明顯不影響理解的口語贅詞

但不得刪除重要概念、條件、限制或範例。

---

### Step 3：建立主題範圍

根據原始資料決定筆記主題。

每份正式筆記應盡量聚焦在一個清楚主題，例如：

```text
HTML 文件基本結構
HTML 語意化標籤
HTML 表單基礎
HTML input 元素
HTML table 表格
HTML image 圖片元素
HTML metadata
```

若原始資料同時涵蓋多個大主題，應建議拆成多份筆記。

---

### Step 4：章節拆分規則

優先根據「大標題」拆分，但不能只看標題文字，還要判斷內容是否形成完整學習單元。

適合拆成獨立筆記的條件：

1. 有明確主題。
2. 可以獨立學習。
3. 有足夠內容支撐一份筆記。
4. 與其他章節不是單純前後文依賴。
5. 後續可以單獨生成練習題、範例或複習材料。

不適合拆成獨立筆記的情況：

1. 只有一小段補充說明。
2. 只是某個主題底下的細節。
3. 內容不足以形成完整知識單元。
4. 拆開後會造成概念斷裂。
5. 標題只是排版用，不代表真正主題。

建議判斷方式：

```text
大標題可以作為第一層候選拆分依據，
但最終仍要以「是否能形成一份可獨立學習的教學筆記」為準。
```

---

### Step 5：萃取知識點

從原始資料中整理出：

- 核心概念
- 關鍵術語
- 重要語法
- 常用屬性
- 範例程式
- 注意事項
- 常見錯誤
- 實務場景
- 與其他概念的關係

整理時不要只複製原文，應重新組織成教學順序。

---

### Step 6：補充必要背景

若原始資料太簡略，可以補充必要背景。

補充內容應符合以下條件：

1. 有助於理解原始主題。
2. 不偏離筆記主線。
3. 不取代原始資料的核心內容。
4. 不把筆記擴張成另一個主題。
5. 必要時標示為「補充說明」。

HTML 筆記常見可補充背景：

- DOM 與 HTML 的關係
- 語意化的目的
- 表單資料流
- 可訪問性注意事項
- SEO 相關影響
- 與 CSS / JavaScript / Vue 的關係

---

### Step 7：改寫成正式筆記

將整理後的知識點改寫成正式筆記。

要求：

1. 使用繁體中文。
2. 標題層級清楚。
3. 說明順序符合學習邏輯。
4. 程式碼範例完整且不過度複雜。
5. 重要概念有定義。
6. 抽象概念有例子。
7. 常見錯誤有修正方式。
8. 補充內容不喧賓奪主。
9. 符合 `teaching-note-format.md`。

---

### Step 8：檢查筆記品質

產生筆記後，必須自我檢查：

| 檢查項目 | 判斷標準 |
|---|---|
| 主題是否清楚 | 是否一眼能看出本章在教什麼 |
| 結構是否合理 | 是否從概念到範例再到注意事項 |
| 是否保留原意 | 是否沒有扭曲原始資料 |
| 是否過度補充 | 是否加入太多非本章必要內容 |
| 範例是否正確 | HTML 語法是否合理 |
| 是否有常見錯誤 | 是否提醒初學者容易踩坑之處 |
| 是否可回查 | 之後複習時是否容易定位重點 |
| 是否可延伸 | 是否能用於生成 demos、practice、review |

---

## 來源與補充標示規則

正式筆記可以包含以下標示：

```md
> 來源重點：此段整理自 origin/... 的核心內容。
```

```md
> 補充說明：以下內容是為了幫助理解而補充，原始資料未必完整提及。
```

```md
> 注意：原始資料此處資訊不足，以下結論需要再確認。
```

若內容完全來自合理改寫，不需要每段都標示來源；但若加入原始資料以外的推論或背景，應讓讀者看得出來。

## 檔名建議規則

筆記檔名建議使用英文小寫與連字號：

```text
notes/html-document-structure.md
notes/html-semantic-elements.md
notes/html-forms.md
notes/html-input-element.md
notes/html-table.md
notes/html-image-element.md
notes/html-metadata.md
```

檔名應符合：

1. 看得出主題。
2. 不使用空白。
3. 不使用過長名稱。
4. 不使用不穩定章節編號作為唯一識別。
5. 若需要排序，可在外部索引檔處理，不強迫檔名帶序號。

---

## 輸出格式

執行本流程時，請輸出以下內容：

```text
1. 建議輸出路徑
2. 正式筆記 Markdown 內容
3. 若需要拆分，列出拆分建議
4. 若有待確認資訊，列出待確認清單
```

若使用者要求「只輸出 Markdown 檔案內容」，則不要額外輸出說明。

---

## 禁止事項

1. 不要只做摘要。
2. 不要只重排原文標題。
3. 不要把多個大主題硬塞進同一份筆記。
4. 不要加入與 HTML 主題無關的內容。
5. 不要省略重要範例。
6. 不要將補充內容偽裝成原始資料內容。
7. 不要在工作流程檔案中重複定義完整筆記格式。
