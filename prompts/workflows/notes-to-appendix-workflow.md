# Notes 轉 Appendix 工作流程提示詞

> 檔案位置：`prompts/workflows/notes-to-appendix-workflow.md`
> 用途：定義如何從 `notes/` 正式筆記生成 `appendix/` 查表型資料。
> 建議搭配：
>
> - `prompts/system/base-role.md`
> - `prompts/system/appendix-generator.md`
> - `prompts/formats/appendix-format.md`
> - `prompts/requests/generate-appendix-from-notes-request.md`

---

## 一、工作流程定位

本檔只負責「生成流程與判斷順序」。

它不負責：

- 定義通用教學角色，請交給 `base-role.md`。
- 定義附錄生成原則，請交給 `appendix-generator.md`。
- 定義表格欄位與輸出格式，請交給 `appendix-format.md`。
- 補完 `notes/` 沒有提到的外部知識。
- 生成 `review/`、`practice/` 或 `demos/` 內容。

生成時應同時遵守：

1. `base-role.md` 的 Knowledge Pack 基本規則。
2. `appendix-generator.md` 的附錄資料生成規則。
3. `appendix-format.md` 的可選區塊格式。

---

## 二、輸入資料

至少需要提供以下其中一種：

1. `notes/` 的某一份正式筆記內容。
2. 某章節的正式筆記片段。
3. 某個明確知識點的筆記內容。

若是來源索引重建模式，需同時提供既有 `Appendix 內容` 與對應的 `notes` 內容。

若使用者有提供，請一併參考：

| 資料 | 用途 |
|---|---|
| notes 路徑 | 建立來源索引 |
| 章節名稱 | 決定 appendix 標題與範圍 |
| 目標輸出路徑 | 判斷輸出位置 |
| 指定附錄類型 | 限制只生成特定區塊 |
| 條目粒度 | 決定章節級、主題級或單一語法級 |
| 生成完整度 | 判斷輸出應為完整附錄或裁切式附錄 |
| 範例策略 | 決定範例是否只保留最小可用片段 |
| 來源索引需求 | 決定是否建立、補上或重建來源索引 |
| Appendix 內容 | 僅用於已存在附錄的來源索引重建模式 |

缺少但可由 `notes` 合理推斷的資訊可以推斷；無法推斷時標示「待確認」。

---

## 三、整體流程

請按照以下五段流程處理：

```text
讀取 notes 並界定主題
  ↓
判斷可生成的 appendix 區塊
  ↓
過濾不適用區塊
  ↓
生成查表型內容
  ↓
建立來源索引並做品質檢查
```

---

## 四、流程 1：讀取 notes 並界定主題

先完整理解使用者提供的 `notes/`，判斷這份筆記的主題與可支撐範圍。

需要確認：

1. 本份 `notes` 的主題是概念、元素、屬性、語法、API、工具，還是綜合章節。
2. 主要章節與小節有哪些。
3. 是否有明確名詞定義。
4. 是否有 HTML 元素、屬性、語法規則或常見錯誤。
5. 是否有明確教 API、方法、事件、CLI 指令或設定檔。
6. 是否有可抽成最小片段的 HTML、CSS 或 JavaScript 範例。
7. 是否有適合做相似概念對照的內容。
8. 哪些常見 appendix 區塊缺乏 `notes` 支撐。

若資訊不足，使用以下寫法：

```text
此 notes 內容不足以生成「{附錄類型}」，原因：{不足原因}
```

不要為了湊表格而加入 `notes` 沒有提到的條目。

---

## 五、流程 2：判斷可生成的 appendix 區塊

先在內部規劃可生成區塊，再決定正式輸出內容。除非使用者要求，不要輸出 `Appendix 規劃` 表。

固定保留：

| 區塊 | 原因 |
|---|---|
| 附錄定位 | 說明本 appendix 對應的 notes 範圍 |
| 附錄索引 | 讓讀者快速找到可查資料 |
| 來源索引 | 讓讀者能回到 notes 深入閱讀 |
| 維護備註 | 提醒後續同步更新方向 |

依內容選用：

| notes 內容特徵 | 建議生成區塊 |
|---|---|
| 明確定義、術語、核心概念 | 名詞表 |
| 章節主題是 HTML 元素、文件骨架、語意結構或標籤關係 | HTML 元素表 |
| 章節主題是 HTML 屬性、屬性值或屬性規則 | 屬性表 |
| 明確教 API、DOM 方法、事件或物件 | API / 方法表 |
| 明確教 CLI 指令或工具操作 | 指令表 |
| 明確提供設定檔內容 | 設定檔範例 |
| 有可複用的最小 HTML、CSS 或 JavaScript 範例 | 常用片段 |
| 有錯誤現象、原因與修正方向 | 錯誤速查表 |
| 有容易混淆的概念或寫法 | 相似概念對照表 |

HTML appendix 特別判斷：

1. `HTML 元素表` 只在元素本身是章節主題時完整生成。
2. 若 `<p>`、`button`、`div` 等元素只是範例載體，不要擴寫成完整元素表。
3. `API / 方法表` 只在 notes 明確教 API、方法或事件時生成。
4. HTML 範例中的輔助 JavaScript 不等於本章 API 重點。
5. 不適用的區塊集中列在「未生成區塊」，不要保留完整空標題與空表格。

---

## 六、流程 3：過濾不適用區塊

正式輸出前，請先排除不適用區塊。

排除原則：

1. 沒有 `notes` 來源支撐的區塊不生成。
2. 只在範例中附帶出現、不是章節主題的內容不擴寫。
3. 使用者指定只生成某些區塊時，不輸出未指定區塊。
4. 區塊缺乏資料時，不輸出空表格。
5. 若某區塊通常會被期待但本章不適用，放入「未生成區塊」並簡短說明原因。

建議格式：

```md
## 未生成區塊

| 區塊 | 不生成原因 |
|---|---|
| API / 方法表 | 原筆記未明確教 API、方法或事件。 |
| 指令表 | 原筆記沒有 CLI 指令。 |
```

---

## 七、流程 4：生成查表型內容

生成正文時請依照 `appendix-format.md` 的欄位與可選區塊格式，不要在 workflow 內重新設計欄位。

內容要求：

1. 只根據 `notes/` 生成。
2. 內容要偏向查表、速查、對照與複用，不要重新寫成教學筆記。
3. 表格內容要短，避免把完整段落塞進表格。
4. 表格中的程式碼用短句或 inline code。
5. 多行 HTML、CSS 或 JavaScript 範例放到「常用片段」。
6. 每個範例只展示一個主要知識點。
7. 條目要保留來源欄位或能對應來源索引。
8. 需要補充推論時，明確標示「補充理解」或「延伸提醒」。
9. 不確定來源時標示「來源待確認」，不要假裝知道。

---

## 八、流程 5：建立來源索引並做品質檢查

最後建立來源索引，讓每個已生成的主要區塊能回到對應 `notes`。

來源索引至少包含：

| 附錄區塊 | 對應 notes 路徑 | 對應章節或小節 | 備註 |
|---|---|---|---|

若使用者沒有提供 `notes` 路徑，請標示：

```text
notes 路徑未提供，來源索引只能標示章節或小節名稱。
```

輸出前檢查：

1. 是否完全根據 `notes/` 內容生成？
2. 是否避免加入無來源的外部知識？
3. 是否偏向查表型資料，而不是重新教學？
4. 是否只輸出有支撐、且與章節主題相關的區塊？
5. 是否避免將範例載體誤擴寫成主題條目？
6. 是否避免輸出空表格與空區塊？
7. 是否有保留來源索引？
8. 範例是否短小、可理解、可複用？
9. 是否適合保存成 Markdown？
