# Notes 轉 Demos 工作流程提示詞

> 檔案位置：`prompts/workflows/notes-to-demos.workflow.md`  
> 用途：定義如何從 `notes/` 正式筆記生成 `demos/` 範例程式。  
> 建議搭配：
>
> - `prompts/system/demo-generator.system.md`
> - `prompts/formats/demo-format.md`
> - `prompts/requests/generate-demo-from-note.request.md`

---

## 一、工作流程目標

你要根據 `notes/` 中的正式筆記，生成對應的 `demos/` 示範程式。

生成結果必須讓學習者可以做到：

1. 看完筆記後，透過 demo 觀察概念如何落地。
2. 直接打開範例程式，看到可執行結果。
3. 修改 demo 中的程式碼，驗證自己是否理解。
4. 從靜態知識逐步銜接到 `practice/` 的練習題。

---

## 二、輸入資料

請根據使用者提供的資料進行生成。

### 必要輸入

至少需要提供以下其中一種：

1. `notes/` 的某一份正式筆記內容
2. 某章節的正式筆記片段
3. 某個明確知識點的筆記內容

### 建議輸入

若使用者有提供，請一併參考：

| 資料 | 說明 |
|---|---|
| 章節名稱 | 用來決定 demo 所屬資料夾 |
| 筆記檔案路徑 | 用來標示 demo 對應來源 |
| 目標知識點 | 用來避免 demo 發散 |
| 希望產生的 demo 數量 | 若未提供，請依筆記內容合理判斷 |
| 難度 | 可分為基礎、進階、整合 |
| 是否拆檔 | 決定使用單一 `index.html` 或拆成 `index.html`、`style.css`、`main.js`、`README.md` |

---

## 三、整體流程

請按照以下順序處理。

```text
讀取 notes
  ↓
萃取知識點
  ↓
判斷適合生成哪些 demo
  ↓
設計 demo 清單
  ↓
決定檔案結構
  ↓
生成 README 與程式碼
  ↓
執行自我檢查
  ↓
輸出最終結果
```

---

## 四、流程 1：讀取 notes 並界定範圍

請先理解使用者提供的 notes，判斷該筆記的範圍。

你需要辨識：

1. 這份 notes 的主題是什麼
2. 主要概念有哪些
3. 哪些概念適合轉成 demo
4. 是否有範例程式可以改寫或延伸
5. 是否有不適合生成 demo 的純觀念內容
6. 是否有資訊不足的地方

如果 notes 資訊不足，請明確標註：

```text
此處需要補充筆記內容，才能生成更完整的 demo。
```

不要為了湊數量而生成與 notes 無關的 demo。

---

## 五、流程 2：萃取可示範的知識點

請從 notes 中萃取「可以被觀察、操作、修改」的知識點。

### 適合生成 demo 的內容

| 類型 | 說明 | 範例 |
|---|---|---|
| 結構型概念 | 可以透過 HTML 結構呈現 | 標題、段落、列表、表格、表單 |
| 對照型概念 | 不同寫法會造成不同效果 | `block` 與 `inline`、語意標籤與非語意標籤 |
| 屬性型概念 | 屬性改變會影響行為或呈現 | `href`、`target`、`alt`、`for`、`name` |
| 表單型概念 | 使用者可以輸入或互動 | input、label、select、textarea |
| 常見錯誤 | 可以示範錯誤與修正差異 | 標籤未閉合、label 未綁定、圖片缺少 alt |
| 小型整合 | 多個相關概念可合成一個頁面 | 文章頁、產品卡片、註冊表單 |

### 不適合直接生成 demo 的內容

| 類型 | 處理方式 |
|---|---|
| 純歷史背景 | 可放在 README 補充，不必硬做 demo |
| 純定義 | 可轉成最小觀察範例 |
| 過於抽象的原理 | 先拆成可觀察的小概念 |
| notes 未提到的框架內容 | 不要擴充到 Vue、React 或其他框架 |

---

## 六、流程 3：選擇 demo 類型

每個 demo 應盡量對應一種主要類型。

### 1. 單一概念 demo

用來示範一個最核心的知識點。

適合：

- 初學概念
- HTML 標籤用途
- 單一屬性效果
- 最小可執行範例

命名建議：

```text
01-basic-{concept}
```

---

### 2. 對照型 demo

用來比較兩種或多種寫法的差異。

適合：

- 語意標籤 vs 非語意標籤
- 有 `alt` vs 沒有 `alt`
- 有 `label for` vs 沒有 `label for`
- 不同 input type 的行為差異

命名建議：

```text
02-compare-{concept-a}-and-{concept-b}
```

---

### 3. 常見錯誤 demo

用來示範錯誤寫法、造成的問題、修正方式。

適合：

- 巢狀結構錯誤
- 表單欄位沒有 name
- label 沒有正確綁定
- 連結缺少必要資訊
- 表格語意結構不完整

命名建議：

```text
03-common-mistake-{problem}
```

---

### 4. 實務應用 demo

用來示範某個知識點在真實頁面中的使用方式。

適合：

- 導覽列
- 商品卡片
- 文章頁
- 登入表單
- FAQ 區塊
- 聯絡我們表單

命名建議：

```text
04-practical-{scenario}
```

---

### 5. 小型整合 demo

用來整合一章節內多個相關知識點。

適合：

- 章節總結
- 多個表單元素整合
- HTML 基本頁面結構整合
- 語意化頁面整合

命名建議：

```text
05-mini-integration-{topic}
```

---

## 七、流程 4：決定 demo 數量

請根據 notes 的內容量決定 demo 數量。

| notes 內容狀況 | 建議 demo 數量 |
|---|---:|
| 單一小知識點 | 1 個 |
| 一個小節，含 2～3 個概念 | 1～2 個 |
| 一個完整章節 | 3～5 個 |
| 整合型章節 | 3～6 個 |
| notes 資訊不足 | 先生成 1 個最小 demo，並標註不足處 |

不要為了數量產生低品質 demo。

---

## 八、流程 5：決定檔案結構

### 預設結構

若 demo 包含 HTML、CSS、JavaScript 或需要完整教學說明，請使用拆檔結構：

```text
demos/
  {chapter-slug}/
    {demo-slug}/
      index.html
      style.css
      main.js
      README.md
```

### 小型 demo 結構

若 demo 很小，且只需要展示 HTML 結構，可以使用單一檔案：

```text
demos/
  {chapter-slug}/
    {demo-slug}/
      index.html
      README.md
```

### 命名規則

| 項目 | 建議 |
|---|---|
| 資料夾名稱 | 小寫英文、數字與連字號 |
| demo 編號 | 使用 `01`、`02`、`03` 排序 |
| 檔名 | 固定使用 `index.html`、`style.css`、`main.js`、`README.md` |
| 中文章節 | 可轉成英文 slug，或保留章節編號 |

範例：

```text
demos/
  html-form/
    01-basic-input-types/
    02-label-for-binding/
    03-common-mistake-missing-name/
```

---

## 九、流程 6：設計每個 demo

每個 demo 都必須先設計，再輸出程式碼。

請先確認：

1. Demo 名稱
2. 對應 notes 知識點
3. Demo 類型
4. 學習目標
5. 使用者可以觀察到什麼
6. 使用者可以修改什麼
7. 是否需要 HTML / CSS / JavaScript
8. 是否需要 README

### Demo 設計表

輸出 demo 前，建議先整理成以下格式：

| Demo | 類型 | 對應知識點 | 學習目標 | 檔案 |
|---|---|---|---|---|
| 01-basic-xxx | 單一概念 | xxx | 理解 xxx 的基本用法 | `index.html`, `README.md` |
| 02-compare-xxx | 對照型 | xxx / yyy | 比較兩種寫法差異 | `index.html`, `style.css`, `README.md` |

---

## 十、流程 7：生成程式碼

生成程式碼時，請遵守以下規則。

### HTML

1. 必須有完整基本結構。
2. 必須設定 `lang="zh-Hant"`。
3. 必須設定 `<meta charset="UTF-8">`。
4. 必須設定 viewport。
5. 標籤結構要清楚。
6. 不要塞入與 demo 主題無關的內容。
7. 若示範語意標籤，請讓語意結構清楚可讀。

### CSS

1. 只寫輔助觀察的樣式。
2. 不要做過度華麗的視覺設計。
3. class 命名要清楚。
4. 若樣式只是為了讓差異更容易觀察，請在 README 說明。
5. 不要引入 CSS 框架。

### JavaScript

1. 只有在 demo 需要互動或觀察行為時才加入。
2. 優先使用原生 JavaScript。
3. 程式碼保持短小。
4. 事件命名與變數命名要清楚。
5. 不要加入與該 HTML 知識點無關的複雜邏輯。

### README

每個 demo 建議包含：

1. Demo 目標
2. 對應 notes 知識點
3. 如何執行
4. 觀察重點
5. 修改練習
6. 常見錯誤
7. 自我檢查

---

## 十一、流程 8：自我檢查

輸出前請檢查以下項目。

### 內容對齊檢查

- [ ] Demo 是否主要根據 notes 生成？
- [ ] 是否避免混入 appendix、practice、review 的內容？
- [ ] 是否有標註 notes 資訊不足之處？
- [ ] 是否避免擴充到 notes 未提及的大型功能？

### Demo 品質檢查

- [ ] 每個 demo 是否只聚焦一個主要概念？
- [ ] 是否可以直接打開執行？
- [ ] 是否有清楚的檔案結構？
- [ ] 是否有 README 或必要說明？
- [ ] 是否有觀察點與修改練習？
- [ ] 是否包含常見錯誤或注意事項？

### 程式碼檢查

- [ ] HTML 結構是否完整？
- [ ] CSS 是否只是輔助觀察，不喧賓奪主？
- [ ] JavaScript 是否必要且簡潔？
- [ ] 是否避免不必要的第三方套件？
- [ ] 程式碼是否容易修改？

---

## 十二、最終輸出順序

請依照以下順序輸出：

1. Notes 內容理解摘要
2. Demo 規劃清單
3. 建議檔案結構
4. 每個 demo 的完整檔案內容
5. 執行方式
6. 觀察重點
7. 修改練習
8. 自我檢查結果

如果使用者要求「只輸出檔案內容」，可以省略規劃說明，但仍要保持檔案完整。

---

## 十三、特殊情況處理

### 情況 1：notes 太短

若 notes 太短，請生成最小 demo，並補上提醒：

```text
此 notes 目前資訊較少，本 demo 只能示範最基本概念。
若要生成更完整的 demo，建議補充：定義、使用場景、常見錯誤、範例程式。
```

### 情況 2：notes 有多個不相關主題

請先拆分主題，再分別生成 demo。

不要把不相關主題塞進同一個 demo。

### 情況 3：notes 本身已經有範例程式

請優先改寫成更適合學習的 demo：

1. 保留核心概念
2. 補上可觀察點
3. 補上修改練習
4. 簡化不必要的內容
5. 修正不清楚的命名

### 情況 4：使用者要求大量 demo

請先規劃清單，再生成前 3～5 個最重要的 demo。

除非使用者明確要求全部生成，否則不要一次輸出過大的內容。

---

## 十四、生成品質標準

一個合格的 demo 應符合：

1. 來源清楚：知道它對應 notes 的哪個知識點。
2. 目標清楚：知道它要示範什麼。
3. 結果可見：打開後可以觀察到效果。
4. 修改可驗證：改一兩行程式碼就能看到差異。
5. 範圍可控：不混入太多額外概念。
6. 可銜接練習：可以自然延伸成 `practice/` 題目。
