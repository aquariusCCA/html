# 更新影響範圍與初次產製判斷提問模板

## 用途

用來請 AI 依照 `meta/update-rules.md` 判斷一次改動會影響哪些下游內容、哪些內容需要檢查或重生成，以及 `meta/chapter-status.md` 應如何同步標記。

這份模板只負責「判斷影響範圍」或「初次產製流程」，不負責直接重生成內容。

## 適用場景

當 `origin/`、`atomic/`、`notes/`、`appendix/`、`demos/`、`practice/`、`review/`、`supplements/` 或章節資產有改動，需要先確認下游是否過期時使用。第一次從 `origin/` 製作產物，需要先確認資料流與狀態標記時，也使用本模板。

適合用在以下情況：

1. 已修改某個章節內容，想確認哪些下游需要檢查。
2. 準備重生成下游材料前，想先縮小處理範圍。
3. 不確定 `meta/chapter-status.md` 哪些欄位應標記為「已完成」、「待確認」或「不適用」。
4. 下游內容疑似過期，需要判斷最新正確來源在哪一層。
5. 第一次從 `origin/` 製作某章節產物，需要確認應該從哪一層開始、做到哪一層，以及狀態表如何標記。

## 不適用場景

不適合用來直接產生 atomic notes、正式教學筆記、範例、練習題、複習材料或 appendix。若要執行實際重生成，應改用對應的 `prompts/requests/` 提問模板。

不適合在沒有章節名稱、改動位置、改動摘要或初次產製目標時直接使用。資訊不足時，AI 應列出待確認項目，不要猜測影響範圍。

## 使用方式

1. 如果是既有內容更新後的影響判斷，使用「既有內容更新用提問」。
2. 如果是第一次從 `origin/` 製作產物，使用「初次產物製作用提問」。
3. 將提問中的可替換欄位改成實際資訊。
4. 若已有 `git diff`、檔案路徑、相關段落、檔案清單或資產狀態，放入細節欄位。
5. 依照 AI 的判斷，再決定是否執行其他重生成或檢查用的 request template。

## 既有內容更新用提問

```text
請依照 meta/update-rules.md 判斷這次改動的影響範圍。

請先閱讀 meta/update-rules.md，並根據其中的資料流、上下游規則、重生成判斷、狀態同步標準做判斷。

本次改動資訊：

1. 章節名稱：<章節名稱>
2. 改動位置：<origin / origin/<章節>/assets / atomic / notes / appendix / demos / practice / review / supplements / prompts>
3. 改動類型：<小修 / 技術概念 / 結構 / 範例 / 資產 / 標題 / 狀態同步 / 其他>
4. 改動摘要：<改動摘要>
5. 希望 AI 判斷：<希望 AI 判斷的事項>

改動細節：

<git diff、檔案路徑、相關段落或變更清單；如果沒有，請填「無」>

請遵守以下要求：

1. 只判斷影響範圍，不要直接修改、重生成或刪除任何檔案。
2. 先找最新正確來源，再判斷下游是否過期。
3. 不要因為上游有改動就直接要求全部重跑，必須說明必要範圍。
4. 如果資訊不足，請列入「待確認」，不要假設內容一定是最新的。
5. `meta/chapter-status.md` 只提出應標記的狀態，不要修改欄位結構。

請只輸出：

1. 改動摘要
2. 最新正確來源
3. 改動類型
4. 需要檢查的下游
5. 需要重生成的下游
6. meta/chapter-status.md 應標記的狀態
7. 是否有過期內容風險
8. 下一步建議
```

## 初次產物製作用提問

```text
請依照 meta/update-rules.md 判斷這個章節第一次從 origin/ 製作產物時，應該採用的資料流、產出範圍與狀態標記。

請先閱讀 meta/update-rules.md，並根據其中的資料流、上下游規則、標準工作流程、完成標準與狀態同步標準做判斷。

本次產製資訊：

1. 章節名稱：<章節名稱>
2. origin 章節路徑：<origin章節路徑>
3. 目前已有產物：<無 / atomic / notes / appendix / demos / practice / review / 其他>
4. 目標產出層級：<atomic / atomic review / notes / appendix / demos / practice / review / 完整章節>
5. 希望 AI 判斷：<希望 AI 判斷的事項>

補充細節：

<origin 檔案清單、資產狀態、既有產物路徑或其他限制；如果沒有，請填「無」>

請遵守以下要求：

1. 只判斷初次產製流程，不要直接建立、修改、重生成或刪除任何檔案。
2. 不要把「第一次製作產物」誤判成「既有下游過期」；只有已存在舊產物時，才判斷是否需要檢查或覆蓋。
3. 必須依照資料流順序判斷，不要跳過必要階段。
4. 如果 `origin/<章節>/assets/` 有資產，必須先納入資產路徑標準化與 alt / 連結文字整理的判斷。
5. 如果資訊不足，請列入「待確認」，不要假設某一層已完成。
6. `meta/chapter-status.md` 只提出應標記的狀態，不要修改欄位結構。

請只輸出：

1. 產製目標摘要
2. 應採用的資料流順序
3. 需要先執行的 request template
4. 這次應產出的範圍
5. 這次不應產出的範圍
6. 已有產物是否需要檢查或覆蓋
7. meta/chapter-status.md 應標記的狀態
8. 待確認事項
9. 下一步建議
```

## 可替換欄位

| 欄位 | 說明 | 範例 |
| --- | --- | --- |
| `<章節名稱>` | 受影響的章節名稱 | `表單` |
| `<改動位置>` | 改動發生的資料層或路徑 | `notes/表單/01-表單入門與form基本結構.md` |
| `<改動類型>` | 改動性質 | `標題`、`技術概念`、`資產` |
| `<改動摘要>` | 用一到三句話描述實際改了什麼 | `調整 form/action/method 的說明，補上 GET 與 POST 的差異。` |
| `<希望 AI 判斷的事項>` | 這次最需要 AI 協助判斷的問題 | `哪些下游需要重生成，以及 status 要怎麼標記。` |
| `<改動細節>` | 可協助判斷的具體差異 | `git diff`、檔案清單、修改前後段落 |
| `<origin章節路徑>` | 第一次產製時的原始章節目錄 | `origin/表單/` |
| `<目前已有產物>` | 目前已存在的下游或中間產物 | `無`、`atomic/表單/` |
| `<目標產出層級>` | 這次希望做到哪一層 | `atomic review`、`notes`、`完整章節` |
