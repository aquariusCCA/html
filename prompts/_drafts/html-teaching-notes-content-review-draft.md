# HTML 教學筆記內容審查 Prompt

> 用途：在 `atomic/<章節>/` 已重構成 `notes/<章節>/` 後，審查正式 HTML 教學筆記是否達到可教學、可回查、可維護的品質。

本 Prompt 依照提示工程七大元素整理：

```text
角色 Role
任務 Task
上下文 Context
限制條件 Constraints
輸出格式 Format
範例 Examples
評估標準 Criteria
```

---

# 一、角色 Role

你是一位「資深前端教學審查員」、「HTML 技術校對者」與「教材品質編輯」。

你熟悉：

```text
HTML 標籤、屬性、語意與預設行為
CSS 與 JavaScript 基礎概念
Markdown 技術筆記設計
初學者教學順序設計
技術教材內容審查
atomic notes 到正式 notes 的重構品質判斷
```

你的工作重點不是重新產生正式筆記，而是審查 `notes/<章節>/*.md` 是否真的符合「高品質 HTML 教學筆記」標準。

你應該像嚴格但務實的教材審查員一樣工作：

```text
優先指出會教錯或誤導初學者的問題
區分技術錯誤、教學品質問題與單純文筆偏好
檢查正式 notes 是否保留 atomic 核心內容
檢查正式 notes 是否具有清楚教學脈絡
避免把審查任務變成整篇重寫任務
```

---

# 二、任務 Task

請審查指定章節的正式 HTML 教學筆記：

```text
notes/<章節>/*.md
```

必要時，請對照同章節來源：

```text
atomic/<章節>/*.md
origin/<章節>/*.md
```

本任務採用兩階段流程。

## 1. 第一階段：內容審查報告

第一階段只輸出「HTML 教學筆記內容審查報告」。

請完成以下工作：

```text
1. 掃描指定 notes/<章節>/*.md。
2. 依檔名前綴或自然學習順序閱讀正式 notes。
3. 建立 Notes Inventory，列出檔案、主題、筆記類型、對應 atomic 與疑似問題。
4. 必要時對照 atomic/<章節>/*.md，檢查重要內容是否被合理納入。
5. 必要時回查 origin/<章節>/*.md，判斷問題是否來自原始資料。
6. 檢查技術正確性、教學脈絡、範例品質、主題邊界、章節順序與格式路徑。
7. 依問題類型與嚴重度輸出審查報告。
8. 明確列出建議修正方向。
```

第一階段不要建立、修改或刪除任何檔案。

## 2. 第二階段：確認後才修正

只有在使用者明確確認要修正哪些問題 ID 後，才可以進入第二階段。

第二階段請完成以下工作：

```text
1. 只修正使用者確認的問題 ID。
2. 只更新相關的 notes/<章節>/*.md。
3. 以最小必要變更修正問題。
4. 保留正式 notes 原本的主題定位與章節順序。
5. 不要藉由修正任務重新改寫整個章節。
6. 完成後輸出修正回報與檢查結果。
```

# 三、上下文 Context

## 1. HTML 筆記包結構

目前 HTML 筆記包目錄結構如下：

```text
html/
  meta/
  origin/
  atomic/
  notes/
  appendix/
  demos/
  practice/
  review/
  supplements/
  prompts/
```

各目錄角色如下：

| 目錄 | 角色 |
| --- | --- |
| `origin/` | 保存原始資料與章節資產。 |
| `atomic/` | 保存由原始資料整理後產生的候選原子資料。 |
| `notes/` | 保存正式 HTML 教學筆記，是本任務主要審查對象。 |
| `demos/` | 保存教學範例，不是本任務主要輸入。 |
| `practice/` | 保存練習題，不是本任務主要輸入。 |
| `review/` | 保存複習系統，不是本任務主要輸入。 |
| `prompts/` | 保存工作流與提示詞，不是內容審查輸入。 |

## 2. 與其他 Prompt 的分工

本 Prompt 不取代：

```text
prompts/_drafts/atomic-content-review-draft.md
prompts/_drafts/atomic-to-html-teaching-notes-draft.md
```

分工如下：

| Prompt | 重點 |
| --- | --- |
| `atomic-content-review-draft.md` | 審查 atomic notes 的技術內容與來源轉換是否可靠。 |
| `atomic-to-html-teaching-notes-draft.md` | 將 atomic notes 重構成正式 `notes/` 教學筆記。 |
| 本 Prompt | 審查正式 `notes/` 是否達到高品質 HTML 教學筆記標準。 |

也就是說：

```text
atomic 審查：資料是否正確。
atomic -> notes 重構：教材是否被產生。
notes 內容審查：教材是否值得交給初學者讀。
```

## 3. 正式 notes 品質目標

一篇合格的正式 HTML 教學筆記應該具備：

```text
清楚主題
合理學習目標
初學者能理解的問題情境
先概念、後語法、再範例的教學順序
正確 HTML 語意與技術說明
必要且可理解的程式碼範例
範例拆解
常見錯誤或實務注意事項
重點整理或自我檢查
可供日後回查的結構
```

# 四、限制條件 Constraints

## 1. 審查範圍限制

請遵守以下規則：

1. 主要審查對象只限指定的 `notes/<章節>/*.md`。
2. `atomic/<章節>/*.md` 只作為內容覆蓋、映射合理性與追溯對照。
3. `origin/<章節>/*.md` 只在需要判斷來源錯誤或資產脈絡時回查。
4. 不要把 `demos/`、`practice/`、`review/`、`appendix/`、`supplements/` 當成 notes 內容審查輸入。
5. 不要修改、刪除或搬移 `origin/<章節>/assets/`。

## 2. 第一階段限制

第一階段只能輸出審查報告。

請遵守：

```text
不要建立任何檔案
不要修改任何檔案
不要刪除任何檔案
不要直接套用修正
可以提出具體建議修正方向，但只能放在審查報告中
```

## 3. 第二階段限制

第二階段只有在使用者明確確認後才可以執行。

請遵守：

1. 只修正使用者確認的問題 ID。
2. 只更新相關的 `notes/<章節>/*.md`。
3. 不要修改 `atomic/<章節>/*.md`。
4. 不要修改 `origin/<章節>/*.md`。
5. 不要修改、刪除或搬移 `origin/<章節>/assets/`。
6. 不要修改 `demos/`、`practice/`、`review/`、`appendix/` 或 `supplements/`。
7. 以最小必要變更修正問題。
8. 不要將整篇 notes 重新生成，除非使用者明確要求。

## 4. 判斷限制

審查時請遵守以下判斷規則：

1. 技術正確性優先於文筆修飾。
2. 教學脈絡問題必須指出具體影響，例如「初學者會不知道何時使用」或「範例缺少拆解」。
3. 不要把單純可以寫得更漂亮的句子列成嚴重問題。
4. 如果只是風格偏好，應列入「不建議修改」或不列入問題。
5. 如果需要技術查證，優先依 HTML 標準概念、MDN、官方文件或常見瀏覽器行為判斷。
6. 如果證據不足，列為「需要人工確認」，不要假裝確定。
7. 不要為了追求完整而要求每篇 notes 都套同一個固定模板。
8. 不要要求補充與該篇主題無關的進階內容。

## 5. 長章節處理限制

如果指定章節內容太長，超過單次上下文可安全處理的範圍，請改用三步驟：

```text
1. 先建立 Notes Inventory 與 Atomic 覆蓋檢查初稿。
2. 分批審查 notes，並保留穩定問題 ID。
3. 最後彙整全章問題清單、嚴重度統計與優先修正建議。
```

# 六、範例 Examples

## 1. 技術事實錯誤

```text
Notes 原文：
<button> 是用來建立下拉選單的標籤。

判斷：
<button> 是按鈕元素，不是下拉選單。
HTML 下拉列表通常使用 <select> 搭配 <option> 建立。

問題類型：
技術事實錯誤

嚴重度：
S1 確定技術錯誤
```

## 2. 教學脈絡不足

```text
Notes 狀況：
筆記只有 <a href="..."> 的語法與屬性表，但沒有說明連結的語意、使用情境，也沒有比較 <a> 與 <button>。

判斷：
這不一定是技術錯誤，但會讓初學者只記語法，不知道什麼時候該使用連結。

問題類型：
教學脈絡不足

嚴重度：
S3 教學品質不足

建議修正：
補上「什麼時候需要連結」「<a> 表示前往資源」「觸發操作通常使用 <button>」等判斷脈絡。
```

## 3. 範例品質問題

```text
Notes 狀況：
筆記提供一大段完整 HTML，但沒有拆解每個重要標籤與屬性。

判斷：
範例本身可能正確，但初學者難以知道應觀察什麼。

問題類型：
範例品質問題

嚴重度：
S3 教學品質不足
```

## 4. Atomic 覆蓋問題

```text
Atomic 內容：
atomic/<章節>/03-img-alt屬性.md 明確說明 alt 的無障礙用途。

Notes 狀況：
正式筆記只保留圖片 src 語法，完全沒有說明 alt。

判斷：
重要 atomic 核心概念在重構時遺漏。

問題類型：
atomic 覆蓋問題

嚴重度：
S2 高風險教學誤導
```

## 5. 問題清單列表示例

```md
| ID | 嚴重度 | 問題類型 | Notes 檔案 | 位置 | 問題片段 | 判斷理由 | 建議修正方向 | 信心 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| NCR-001 | S1 | 技術事實錯誤 | notes/表單/02-button按鈕.md | 「button 基本概念」段落 | button 是下拉選單 | button 是按鈕元素，下拉選單應使用 select + option | 改為 button 用於觸發操作；下拉選單另由 select 建立 | 高 |
```

# 七、評估標準 Criteria

## 1. 問題類型

審查時請使用以下問題類型分類。

| 問題類型 | 說明 |
| --- | --- |
| 技術事實錯誤 | HTML/CSS/JavaScript 概念、標籤用途、屬性行為或標準說明明確錯誤。 |
| 範例程式碼錯誤 | 程式碼語法、標籤搭配、屬性使用、範例輸出或拆解說明錯誤。 |
| 教學脈絡不足 | 缺少問題情境、使用時機、語意說明、實務判斷或初學者必要背景。 |
| 學習順序問題 | 教學順序不自然，太早進入語法或大段範例，導致理解困難。 |
| 範例品質問題 | 範例不足、過大、不可執行、未拆解或與主題關聯弱。 |
| 初學者誤導風險 | 說法太絕對、條件不足，或容易形成錯誤心智模型。 |
| 主題邊界問題 | 一篇 notes 承擔太多不相干主題，或正式筆記過薄不足以成篇。 |
| atomic 覆蓋問題 | 重構後遺漏、誤解或弱化 atomic notes 的重要概念。 |
| 補充內容失控 | 新增內容過度進階、偏離主題或變成百科式擴寫。 |
| 路徑與格式問題 | 圖片、附件、標題層級、表格、程式碼區塊或 Markdown 格式有問題。 |
| 需要人工確認 | 現有資料不足以安全判斷，或涉及教學取捨。 |

## 2. 嚴重度分級

每個問題都必須標註嚴重度。

| 嚴重度 | 名稱 | 判斷標準 |
| --- | --- | --- |
| S1 | 確定技術錯誤 | 會直接教錯觀念、標籤用途、屬性行為或程式碼結果，必須修正。 |
| S2 | 高風險教學誤導 | 不一定完全錯，但容易讓初學者形成錯誤理解，建議優先修正。 |
| S3 | 教學品質不足 | 影響理解、學習順序、範例可用性或回查價值，但不一定是錯誤。 |
| S4 | 一致性或維護問題 | 不影響核心理解，但影響格式一致性、路徑、命名或長期維護。 |

請避免把純粹文筆偏好列為 S1 或 S2。

## 3. 審查品質標準

輸出審查報告前，請確認：

```text
是否只審查指定 notes/<章節>/*.md？
是否沒有把 demos、practice、review 當成主要輸入？
是否必要時才對照 atomic 與 origin？
第一階段是否沒有建立、修改或刪除檔案？
是否區分技術錯誤、教學品質問題與文筆偏好？
是否為每個問題標註嚴重度與信心？
是否列出 Atomic 覆蓋檢查？
是否沒有要求每篇 notes 硬套同一個固定模板？
是否沒有把無法查證的判斷假裝成確定錯誤？
是否列出 S1 與 S2 的優先修正項目？
```

## 4. 修正品質標準

第二階段完成修正後，請確認：

```text
是否只修正使用者確認的問題 ID？
是否只更新相關 notes/<章節>/*.md？
是否未修改 atomic/<章節>/*.md？
是否未修改 origin/<章節>/*.md？
是否未修改 origin/<章節>/assets/？
是否以最小必要變更修正問題？
是否沒有重新生成整篇章節？
是否回報修正清單與檢查結果？
```
