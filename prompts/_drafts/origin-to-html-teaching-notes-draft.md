# 一、任務背景

我目前有一個 HTML 筆記包，目錄結構如下：

```text
html/
  origin/
  notes/
  appendix/
  demos/
  practice/
  review/
  supplements/
  prompts/
```

其中：

```text
origin/
  <章節>/
    assets/
      images/
      pdfs/
      excels/
      word/
      files/
      ...
    *.md
```

規則如下：

1. `origin/` 只存放 `.md` 原始資料文件。
2. `origin/<章節>/` 下面的 `.md` 是原始資料。
3. `origin/<章節>/assets/` 存放該章節使用到的圖片、PDF、Excel、Word 或其他附件。
4. 不會有 `origin/<章節>/notes/`。
5. 重構後的正式教學筆記要輸出到：

```text
notes/<章節>/<原檔名>.md
```

例如：

```text
origin/HTML簡介/網頁網站.md
```

要重構成：

```text
notes/HTML簡介/網頁網站.md
```

# 二、你的任務

請將我提供的 `origin/<章節>/<檔名>.md` 原始資料，重構成一篇「高品質 HTML 教學筆記」。

這不是單純整理、改寫或美化排版，而是要把原始資料轉換成真正適合學習的教學內容。

請遵守以下核心原則：

```text
固定的是教學品質標準，
不是每篇文章都硬套同一個標題模板。
```

也就是說，你要根據主題類型選擇合適的教學結構，而不是每篇都機械套用相同目錄。

# 四、重構原則

請依照以下原則重構筆記。

## 1. 保留原始資料的核心內容

原始資料中的重要概念、定義、範例、圖片、連結都要盡量保留。

但不要逐字照抄，而是要重新整理成更適合教學的邏輯。

## 2. 補足教學脈絡

如果原始資料只有零散知識點，請補上：

```text
為什麼要學這個
它解決什麼問題
什麼時候會用到
初學者容易誤解什麼
實務上怎麼判斷
```

## 3. 先講概念，再講語法

不要一開始就丟大量程式碼。

請優先採用：

```text
問題情境 → 白話理解 → HTML 語法 → 程式碼拆解 → 常見錯誤 → 實務使用
```

## 4. 不要做成單純整理型筆記

請避免只寫成：

```text
定義
語法
屬性
範例
```

這種格式只能查資料，不一定能教會讀者。

高品質教學筆記要能讓讀者理解：

```text
為什麼需要它
它代表什麼語意
它怎麼使用
它容易錯在哪裡
實務上如何判斷
```

## 5. 根據主題彈性安排段落

可以使用以下教學模組，但不必每篇都全部使用：

```text
學習目標
問題情境
一句話理解
核心概念
基本語法
範例拆解
常用屬性
錯誤寫法與正確寫法
相似概念比較
實務使用場景
常見錯誤
小練習
重點整理
自我檢查
```

請根據筆記主題挑選合適模組。

# 五、HTML 教學筆記品質標準

重構後的筆記必須符合以下標準：

## 1. 初學者能理解

請用清楚、白話、循序漸進的方式說明。

避免只使用抽象名詞。

## 2. 工程師能回查

筆記也要適合作為日後查詢資料使用。

因此需要有清楚標題、表格、程式碼區塊與重點整理。

## 3. 有完整範例

如果主題涉及 HTML 標籤或結構，請盡量提供完整、可理解的 HTML 範例。

範例不要只有孤立片段，除非該主題真的只需要片段。

## 4. 有錯誤對比

如果該主題容易寫錯，請提供：

```text
錯誤寫法
為什麼錯
正確寫法
```

## 5. 有實務判斷

請說明真實開發時應該如何使用。

例如：

```text
什麼時候用 a
什麼時候用 button
什麼時候用 section
什麼時候不該濫用 div
```

## 6. 有語意觀念

HTML 筆記不能只講畫面效果。

請優先說明標籤的語意、文件結構與可讀性。

必要時可補充 SEO、可維護性、無障礙等觀念。

## 7. 不過度補充

可以補充必要的教學內容，但不要把一篇簡單筆記擴寫成百科全書。

補充內容要服務於該篇主題。

## 8. 不硬套固定模板

每篇筆記要有清楚教學邏輯，但標題與段落可以根據主題調整。

# 九、Few-Shot 風格範例

請參考以下教學風格。

如果主題是 `a 標籤`，不要只寫：

````md
# a 標籤

a 標籤是超連結標籤。

```html
<a href="https://example.com">連結</a>
```

````

應該寫成教學型：

````md
# a 標籤：讓網頁可以連到另一個資源

## 問題情境

假設你有一個個人網站，想讓使用者點擊「查看 GitHub」後前往你的 GitHub 頁面。

這時候就需要使用 `<a>` 標籤。

## 一句話理解

`<a>` 是用來建立「連結」的 HTML 語意標籤。

它的重點不是讓文字看起來可以點，而是讓目前文件可以連到另一個資源。

## 基本語法

```html
<a href="https://github.com">查看 GitHub</a>
```

這段程式碼中：

* `<a>`：建立連結
* `href`：指定連結目標
* `查看 GitHub`：使用者看到並點擊的文字

## 常見錯誤

錯誤寫法：

```html
<a>查看 GitHub</a>
```

這樣雖然有 `<a>` 標籤，但沒有 `href`，因此不是一個真正能前往目標的連結。

## 實務判斷

如果目的是「前往某個網址」，使用 `<a>`。

如果目的是「觸發操作」，例如送出表單、打開彈窗，通常使用 `<button>`。

````

# 十、重構流程

請按照以下流程執行：

1. 讀取原始 Markdown。
2. 判斷這篇筆記屬於哪一種類型。
3. 找出原始資料中的核心概念。
4. 刪除重複、鬆散、低價值的內容。
5. 補上必要的教學脈絡。
6. 根據主題選擇合適的教學模組。
7. 重寫成完整 Markdown 教學筆記。
8. 修正圖片與附件相對路徑。
9. 檢查標題層級、程式碼區塊、表格與格式。
10. 確認輸出到正確路徑。
