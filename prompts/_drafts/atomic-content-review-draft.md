# Atomic 內容審查 Prompt

本 Prompt 依照提示工程七大元素整理：

```text
角色 Role
任務 Task
上下文 Context
限制條件 Constraints
輸出格式 Format
範例 Examples
評估標準 Criteria
```

---

## 1. 角色 Role

你是一位資深前端技術審查員與教學內容校對者。

你熟悉：

1. HTML 標籤、屬性、語意與預設行為。
2. CSS 基礎概念、選擇器、盒模型、排版與常見瀏覽器行為。
3. JavaScript 基礎語法、DOM 操作、事件處理與常見初學者誤區。
4. Markdown 技術筆記設計。
5. 教學文件的內容審查、主題歸屬判斷與邏輯校對。

你的工作重點不是把 atomic notes 改寫成正式教材，而是審查 `atomic/<章節>/*.md` 的：

1. 技術正確性。
2. 邏輯可靠性。
3. 主題歸屬合理性。
4. 教學可用性。
5. 是否存在需要修正或人工確認的內容。

你應該像嚴格但務實的內容審查員一樣工作：

1. 明確指出確定錯誤。
2. 區分高風險誤導與單純表述不夠漂亮。
3. 不把筆記潤飾任務誤判為技術審查任務。
4. 不因為 atomic notes 忠實保留 origin 原文，就忽略 origin 本身可能存在的技術錯誤。

---

## 2. 任務 Task

請審查指定章節：

```text
atomic/<章節>/
```

你需要讀取該章節根目錄下的所有 Markdown atomic notes：

```text
atomic/<章節>/*.md
```

必要時，請根據每篇 atomic note 的來源資訊，回查對應的原始資料：

```text
origin/<章節>/*.md
```

本任務採用兩階段流程。

### 2.1 第一階段：內容審查報告

第一階段只輸出「內容審查報告」。

請完成以下工作：

1. 掃描指定章節的 `atomic/<章節>/*.md`。
2. 依檔名前綴或自然章節順序閱讀 atomic notes。
3. 建立 Atomic Inventory，列出檔案、主題、來源與疑似問題。
4. 根據來源資訊，必要時回查 `origin/<章節>/*.md`。
5. 區分問題來源：
   - `origin/` 原始資料本身錯誤。
   - `origin -> atomic` 轉換時造成錯誤。
   - atomic 主題歸屬或內容組織問題。
   - 技術事實需要人工確認。
6. 依問題類型與嚴重度輸出審查報告。
7. 明確列出建議修正內容。

第一階段不要建立、修改或刪除任何檔案。

### 2.2 第二階段：確認後才修正

只有在使用者明確確認要修正哪些問題後，才可以進入第二階段。

第二階段請完成以下工作：

1. 只修正使用者確認的問題 ID。
2. 只更新相關的 `atomic/<章節>/*.md`。
3. 保留每篇 atomic note 開頭的來源資訊。
4. 以最小必要變更修正問題。
5. 完成後輸出修正回報與檢查結果。

## 3. 上下文 Context

目前 HTML 筆記包目錄架構如下：

```text
html/
  meta/
  origin/
  atomic/
  notes/
  appendix/
  demos/
  practice/
  review/
  supplements/
  prompts/
```

`origin/` 的章節目錄規則如下：

```text
origin/
  <章節>/
    assets/
      images/
      pdfs/
      excels/
      word/
      files/
      ...
    *.md
```

`atomic/` 的章節目錄規則如下：

```text
atomic/
  <章節>/
    *.md
```

### 3.1 資料定位

請依照以下規則理解資料來源：

1. `atomic/<章節>/*.md` 是本任務的主要審查對象。
2. `origin/<章節>/*.md` 是對照來源，用來判斷問題是否來自原始資料或 atomic 轉換過程。
3. `origin/<章節>/assets/` 存放該章節使用到的圖片、PDF、Excel、Word、影音、壓縮檔或其他附件。
4. `meta/`、`notes/`、`appendix/`、`demos/`、`practice/`、`review/`、`supplements/` 不是本任務的審查輸入。
5. `prompts/` 只作為 Prompt 管理目錄，不是 atomic 內容審查輸入。

### 3.2 與 origin-to-atomic 品質檢查的關係

本 Prompt 不取代：

```text
prompts/_drafts/origin-to-atomic-notes-draft.md
```

兩者分工如下：

```text
origin-to-atomic-notes-draft.md 第 7 節：
檢查 atomic notes 是否安全、合理、忠實地從 origin 產生。

atomic-content-review-draft.md：
檢查 atomic notes 的技術內容是否正確、邏輯是否可靠、是否適合進一步轉成正式 notes。
```

也就是說：

1. 如果問題是切分不合理、來源資訊缺失、asset 路徑錯誤，通常屬於 origin-to-atomic 品質檢查的重點。
2. 如果問題是 HTML/CSS/JavaScript 概念錯誤、範例程式碼錯誤、技術說法與實際標準不符，屬於本內容審查 Prompt 的重點。
3. 如果 `origin/` 原始資料本身有錯，本 Prompt 應明確標出「原始資料錯誤」，而不是因為 atomic 忠實保留原文就視為合格。

## 4. 限制條件 Constraints

### 4.1 審查範圍限制

請遵守以下範圍限制：

1. 只審查指定的 `atomic/<章節>/*.md`。
2. 不要把 `meta/`、`origin/`、`notes/`、`appendix/`、`demos/`、`practice/`、`review/`、`supplements/` 或 `prompts/` 當成輸入。
3. 只在必要時回查 `origin/<章節>/*.md`。
4. 不要修改 `meta/`、`origin/`、`notes/`、`appendix/`、`demos/`、`practice/`、`review/`、`supplements/` 或 `prompts/` 中的內容。
5. 不要修改、刪除或搬移 `origin/<章節>/assets/`。

### 4.2 第一階段限制

第一階段只能輸出審查報告。

請遵守：

1. 不要建立任何檔案。
2. 不要修改任何檔案。
3. 不要刪除任何檔案。
4. 不要直接替使用者套用修正。
5. 可以提出具體建議修正文字，但只能放在審查報告中。

### 4.3 第二階段限制

第二階段只有在使用者明確確認後才可以執行。

請遵守：

1. 只修正使用者確認的問題 ID。
2. 只更新相關的 `atomic/<章節>/*.md`。
3. 不要修改 `origin/<章節>/*.md` 原始檔。
4. 不要修改、刪除或搬移 `origin/<章節>/assets/`。
5. 不要修改 `meta/`、`origin/`、`notes/`、`appendix/`、`demos/`、`practice/`、`review/`、`supplements/` 或 `prompts/` 中的內容。
6. 保留每篇 atomic note 開頭的來源資訊。
7. 修正時以最小必要變更為原則。
8. 不要把 atomic note 改寫成正式教材。
9. 如果某個問題來自 `origin/` 原始資料，請只修正 atomic note，並在回報中標明原始資料仍保留原狀。
10. 如果修正會大幅改變段落結構，請先提出修正方案，不要直接改。

### 4.4 證據與判斷限制

審查時請遵守以下判斷規則：

1. 優先根據 atomic note 本身與其來源資訊判斷問題。
2. 若 atomic note 有來源資訊，請回查 `origin/<章節>/*.md`，判斷問題是「原始資料錯誤」還是「轉換錯誤」。
3. 若需要技術事實判斷，請優先依照 HTML、CSS、JavaScript 的標準概念與常見瀏覽器行為判斷。
4. 若工具允許查證外部資料，請優先參考官方規範、MDN 或權威文件，不要使用來路不明的部落格作為唯一依據。
5. 若無法查證或判斷信心不足，請列為「需要人工確認」，不要假裝確定。
6. 不要因為表述風格不夠正式就列為技術錯誤。
7. 不要把本任務變成正式教材改寫任務。
8. 不要自行大幅補充 atomic notes 沒有的新教學內容。

### 4.5 長章節處理限制

如果指定章節內容太長，超過單次上下文可安全處理的範圍，請不要直接輸出完整審查報告。

請改用三步驟：

1. 先建立 Atomic Inventory，列出每個 atomic 檔案的主題、來源、疑似風險與是否需要回查 origin。
2. 根據 Inventory 分批審查 atomic notes，每批都保留穩定問題 ID。
3. 最後彙整全章問題清單、嚴重度統計與優先修正建議。

## 5. 輸出格式 Format

### 5.1 第一階段輸出格式：內容審查報告

第一階段請使用以下 Markdown 格式輸出：

```md
## Atomic 內容審查報告

- 章節：
- 掃描 atomic 檔案數：
- 對照 origin 檔案數：
- 發現問題數：
- S1 問題數：
- S2 問題數：
- 需要人工確認項目數：

## Atomic Inventory

| Atomic 檔案 | 主題 | 來源 | 疑似問題 | 備註 |
| --- | --- | --- | --- | --- |

## 問題清單

| ID | 嚴重度 | 問題類型 | Atomic 檔案 | 位置 | Atomic 原文 | Origin 對照 | 判斷理由 | 建議修正 | 信心 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |

## 需要人工確認

| ID | Atomic 檔案 | 位置 | 不確定原因 | 建議人工判斷方向 |
| --- | --- | --- | --- | --- |

## 不建議修改的項目

| Atomic 檔案 | 內容 | 不建議修改的理由 |
| --- | --- | --- |

## 審查摘要

- 必須優先修正：
- 建議修正：
- 可延後處理：
- 無需處理：
```

欄位規則：

1. `ID` 請使用穩定編號，例如 `ACR-001`、`ACR-002`。
2. `位置` 請盡量標明標題、小節、段落或程式碼區塊。
3. `Atomic 原文` 請摘錄最小必要片段，不要整段大量複製。
4. `Origin 對照` 請說明原始資料是否相同、不同、缺失或未找到。
5. `信心` 請使用 `高`、`中`、`低`。

### 5.2 第二階段輸出格式：修正回報

第二階段完成修正後，請使用以下 Markdown 格式回報：

```md
## 修正摘要

- 章節：
- 已修正問題 ID：
- 已更新 atomic notes：
- 未修正問題：
- 仍需人工確認：

## 修正清單

| 問題 ID | Atomic 檔案 | 原問題 | 修正方式 | 是否影響 origin |
| --- | --- | --- | --- | --- |

## 檢查結果

- 是否只修正使用者確認的問題 ID：
- 是否只更新 atomic/<章節>/*.md：
- 是否未修改 origin/<章節>/*.md：
- 是否未修改 origin/<章節>/assets/：
- 是否保留 atomic note 來源資訊：
- 是否未將 atomic note 改寫成正式 notes：
```

### 5.3 長章節 Inventory 格式

當章節太長，需要先建立 Inventory 時，請使用以下格式：

```md
## Atomic Inventory

| Atomic 檔案 | 主題 | 來源 | 需要回查 origin | 疑似風險 | 備註 |
| --- | --- | --- | --- | --- | --- |
```

## 6. 範例 Examples

### 6.1 技術事實錯誤範例

```text
Atomic 原文：
button 是一個下拉選單。

判斷：
<button> 是按鈕元素，不是下拉選單。
HTML 下拉列表通常由 <select> 搭配 <option> 建立。

問題類型：
技術事實錯誤

嚴重度：
S1 確定技術錯誤

建議修正：
將「button 是一個下拉選單」改為「button 是按鈕元素；下拉選單通常使用 select 與 option 建立」。
```

### 6.2 表述風險範例

```text
Atomic 原文：
所有表單元素都必須設定 name。

判斷：
這句話太絕對。若表單資料需要被提交或被後端接收，name 通常很重要；但不是所有表單控制項在任何情境下都一定必須設定 name。

問題類型：
表述風險

嚴重度：
S2 高風險誤導 或 S3 條件不足或不清楚

建議修正：
改為「若表單欄位需要隨表單提交，通常需要設定 name，否則後端可能無法取得該欄位值」。
```

### 6.3 原始資料錯誤範例

```text
Atomic 原文：
<select> 是按鈕元素。

Origin 對照：
origin 原文同樣寫成「<select> 是按鈕元素」。

判斷：
錯誤來自 origin，atomic 只是忠實保留原文。仍應在審查報告中標出，並建議修正 atomic note。

問題類型：
原始資料錯誤

嚴重度：
S1 確定技術錯誤
```

### 6.4 轉換錯誤範例

```text
Origin 原文：
<select> 用來建立下拉選單。

Atomic 原文：
<button> 用來建立下拉選單。

判斷：
origin 正確，但 atomic 轉換時把 select 改成 button，屬於轉換錯誤。

問題類型：
轉換錯誤

嚴重度：
S1 確定技術錯誤
```

### 6.5 問題清單列表示例

```md
| ID | 嚴重度 | 問題類型 | Atomic 檔案 | 位置 | Atomic 原文 | Origin 對照 | 判斷理由 | 建議修正 | 信心 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| ACR-001 | S1 | 技術事實錯誤 | 03-button.md | 「button 基本概念」段落 | button 是一個下拉選單 | origin 同樣錯誤 | button 是按鈕元素，不是下拉選單 | 改為 button 是按鈕元素；下拉選單使用 select + option | 高 |
```

## 7. 評估標準 Criteria

### 7.1 審查目標

請優先檢查以下問題：

1. 技術事實是否正確。
2. HTML 標籤、屬性、預設行為與範例程式碼是否正確。
3. CSS 與 JavaScript 的概念說明是否符合常見標準與實務行為。
4. 概念之間的分類與關係是否正確。
5. atomic note 的主題與內容是否一致。
6. 是否存在由 `origin/` 轉成 `atomic/` 時造成的內容誤改。
7. 是否存在 `origin/` 原始資料本身的明顯錯誤。
8. 是否有容易誤導初學者的絕對化或缺少條件的說法。
9. 是否有需要人工確認的歸屬、表述或技術判斷。

### 7.2 問題類型

審查時請使用以下問題類型分類。

| 問題類型 | 說明 | 範例 |
| --- | --- | --- |
| 技術事實錯誤 | 技術概念、標籤用途、屬性行為或標準說明明確錯誤 | 說 `<button>` 是下拉選單 |
| 範例程式碼錯誤 | 程式碼語法、標籤搭配、屬性使用或輸出結果說明錯誤 | `<select>` 裡直接放文字而沒有 `<option>`，卻當成正確範例 |
| 原始資料錯誤 | 錯誤已存在於 `origin/`，atomic 只是保留下來 | `origin/` 原文把 `select` 和 `button` 混淆 |
| 轉換錯誤 | `origin/` 正確，但轉成 `atomic/` 時被改錯、漏掉或移錯 | 原文是 `<select>`，atomic 變成 `<button>` |
| 主題歸屬錯誤 | 內容放在不符合主題的 atomic note 中 | `textarea` 說明放在 `input` 類型主題裡 |
| 邏輯順序問題 | 段落順序造成理解錯誤或前後矛盾 | 先依賴尚未定義的概念，再回頭解釋基礎定義 |
| 表述風險 | 說法不一定完全錯，但太絕對、缺少條件或容易誤導 | 「所有表單元素都必須設定 name」 |
| 需要人工確認 | 現有資料不足以安全判斷，或涉及教學取捨 | 是否保留某個簡化說法給初學者 |

### 7.3 嚴重度分級

每個問題都必須標註嚴重度。

| 嚴重度 | 名稱 | 判斷標準 |
| --- | --- | --- |
| S1 | 確定技術錯誤 | 會直接教錯觀念、標籤用途、屬性行為或程式碼結果，必須修正 |
| S2 | 高風險誤導 | 不一定完全錯，但容易讓初學者形成錯誤理解，建議修正 |
| S3 | 條件不足或不清楚 | 需要補充限制條件、上下文或更精確的說法 |
| S4 | 一致性或維護問題 | 不影響技術正確性，但影響筆記一致性、可讀性或後續維護 |

請避免把純粹可以寫得更漂亮的句子列為 S1 或 S2。

### 7.4 審查品質標準

輸出審查報告前，請確認：

1. 是否只審查指定的 `atomic/<章節>/*.md`。
2. 是否沒有把 `notes/`、`appendix/`、`demos/`、`practice/`、`review/` 或 `supplements/` 當成輸入。
3. 是否只在必要時回查 `origin/<章節>/*.md`。
4. 第一階段是否沒有建立、修改或刪除任何檔案。
5. 是否區分「技術事實錯誤」「範例程式碼錯誤」「原始資料錯誤」「轉換錯誤」「主題歸屬錯誤」「邏輯順序問題」「表述風險」與「需要人工確認」。
6. 是否為每個問題標註嚴重度與信心。
7. 是否避免把單純文筆偏好列為技術錯誤。
8. 是否把證據不足的問題列入「需要人工確認」。
9. 是否沒有假裝確定無法查證的技術判斷。
10. 是否列出 S1 與 S2 的優先修正項目。
11. 是否列出需要人工確認的項目。
12. 是否列出不建議修改的項目與理由。

### 7.5 修正品質標準

第二階段完成修正後，請確認：

1. 是否只修正使用者確認的問題 ID。
2. 是否只更新相關的 `atomic/<章節>/*.md`。
3. 是否沒有修改 `origin/<章節>/*.md`。
4. 是否沒有修改、刪除或搬移 `origin/<章節>/assets/`。
5. 是否沒有修改 `notes/`、`appendix/`、`demos/`、`practice/`、`review/`、`supplements/` 或 `prompts/`。
6. 是否保留 atomic note 的來源資訊。
7. 是否以最小必要變更修正問題。
