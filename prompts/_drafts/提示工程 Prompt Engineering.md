# 提示工程 Prompt Engineering 教學筆記

## 一、什麼是提示工程？

提示工程，英文是 **Prompt Engineering**，指的是透過設計清楚、具體、有結構的指令，引導 AI 產生更符合需求的回覆。

簡單來說，提示工程不是單純「問 AI 問題」，而是學會把你的需求整理成 AI 容易理解、容易執行、容易輸出正確結果的形式。

---

## 二、為什麼需要提示工程？

AI 的回覆品質，很大程度取決於你給它的資訊是否完整。

如果提示太模糊，AI 只能根據一般情境猜測你的需求，容易產生以下問題：

* 回答太空泛
* 沒有符合你的程度
* 格式不符合需求
* 回答方向偏掉
* 缺少實際範例
* 產生看似正確但不適合你的內容

例如：

```text
請介紹 Spring Boot。
```

這個提示太廣，AI 不知道你要的是：

* 初學者入門？
* 面試重點？
* 專案實戰？
* 原理解析？
* 和 Spring MVC 的比較？
* Java 8 環境下的使用方式？

因此，高品質提示的核心是：

```text
把「你要什麼」說清楚，
把「背景條件」說完整，
把「輸出形式」限制清楚。
```

---

## 三、提示工程的核心觀念

提示工程的本質不是背模板，而是學會控制 AI 回覆的方向。

可以把它理解成：

```text
模糊需求
→ 結構化需求
→ 明確限制
→ 可用輸出
```

也就是說，好的提示不是越長越好，而是要包含足夠的「任務資訊」。

---

## 四、經典提示結構：CREAT 原則

一個高品質提示通常可以包含以下五個要素：

```text
C：Character 角色
R：Request 請求
E：Environment 背景
A：Action 行動
T：Target 格式
```

### 1. Character：角色

角色是指你希望 AI 以什麼身份來回答。

例如：

```text
你是一位資深 Java 後端工程師。
```

```text
你是一位熟悉 Vue 3、TypeScript、Vite 的前端架構師。
```

```text
你是一位擅長把複雜知識整理成教學筆記的技術講師。
```

角色可以幫助 AI 決定回答角度。

例如，同樣是解釋 Spring Security：

* 資深後端工程師：會偏向架構、認證授權流程、安全設計
* 面試官：會偏向常見面試題、答題重點
* 初學者老師：會偏向白話解釋、範例、學習順序

---

### 2. Request：請求

請求是指你明確要 AI 做什麼。

例如：

```text
請解釋這段程式碼的作用。
```

```text
請幫我整理成教學筆記。
```

```text
請幫我設計一份學習計畫。
```

```text
請比較 Spring MVC 和 Spring Boot 的差異。
```

請求要避免太模糊。

不好的提示：

```text
幫我看一下。
```

較好的提示：

```text
請幫我檢查這段 Java 程式碼是否有設計問題，並指出可以優化的地方。
```

---

### 3. Environment：背景

背景是指 AI 完成任務前需要知道的上下文。

例如：

```text
我是 Java 初學者，目前剛學完流程控制，還沒學泛型。
```

```text
我目前有兩年 Vue 開發經驗，正在轉向 Java 全端工程師。
```

```text
這份筆記是給未來複習用，希望內容清楚、可維護、可延伸。
```

背景越清楚，AI 越能產生符合你程度與場景的回答。

---

### 4. Action：行動

行動是指你希望 AI 用什麼方式處理任務。

例如：

```text
請先用白話解釋，再用程式碼示範，最後整理重點。
```

```text
請按照「概念 → 使用場景 → 範例 → 常見錯誤 → 練習題」的順序說明。
```

```text
請先指出問題，再給出修改後版本，最後說明為什麼這樣改。
```

行動可以幫助 AI 產生更穩定、更有步驟的內容。

---

### 5. Target：格式

格式是指你希望 AI 用什麼形式輸出。

例如：

```text
請使用 Markdown 格式。
```

```text
請用表格比較。
```

```text
請條列式說明。
```

```text
請輸出成可以直接放進 Obsidian 的教學筆記。
```

格式限制可以讓 AI 的輸出更容易閱讀、整理與複用。

---

## 五、完整提示範例

### 範例一：學習筆記型提示

```text
你是一位擅長技術教學的 Java 講師。

我目前正在學習 Spring Boot，已經懂基本 Java 語法，但對 Spring 的自動配置還不熟。

請幫我整理一篇「Spring Boot 自動配置」的教學筆記。

請按照以下結構輸出：

1. 什麼是自動配置
2. 為什麼需要自動配置
3. 傳統 Spring 和 Spring Boot 的差異
4. 自動配置的簡單範例
5. 常見誤解
6. 總結
```

這個提示包含：

| 元素 | 內容                        |
| -- | ------------------------- |
| 角色 | Java 講師                   |
| 請求 | 整理 Spring Boot 自動配置筆記     |
| 背景 | 使用者懂 Java，但不熟 Spring 自動配置 |
| 行動 | 依照指定結構說明                  |
| 格式 | 條列式 Markdown 筆記           |

---

### 範例二：程式碼檢查型提示

```text
你是一位資深 Java 後端工程師。

我正在練習設計模式，目前想檢查這段程式碼是否符合開閉原則。

請你幫我做三件事：

1. 說明目前程式碼的問題
2. 指出它違反開閉原則的地方
3. 給我一個較好的重構版本

請用初學者能理解的方式說明，不要只給結論。
```

這個提示比「幫我優化程式碼」更好，因為它明確指定了檢查方向：開閉原則。

## 六、四個進階提示技巧

## 1. Few-Shot Prompting：少次提示

Few-Shot Prompting 是指在提示中提供幾個範例，讓 AI 模仿你的格式、風格或判斷標準。

適合用在：

* 固定格式輸出
* 分類任務
* 改寫任務
* 筆記整理
* 程式碼註解風格
* 面試回答格式

### 範例

```text
請將以下技術概念整理成固定格式。

範例：

輸入：JavaScript
輸出：
- 定義：瀏覽器與 Node.js 常用的程式語言
- 用途：前端互動、後端服務、工具開發
- 學習重點：語法、DOM、非同步、模組化

現在請整理：

輸入：TypeScript
```

AI 會根據範例產生類似格式的輸出。

Few-Shot 的重點是：

```text
與其只說「請照這種風格」，
不如直接給 AI 看範例。
```

---

## 2. Chain of Thought：思維鏈

思維鏈的概念是讓 AI 在處理複雜問題時，不要直接跳到答案，而是先進行分解與推理。

但在實際使用時，不一定要要求 AI 展示完整內部思考過程。更好的做法是要求 AI 提供：

* 解題步驟
* 判斷依據
* 推理摘要
* 檢查流程
* 最終結論

### 建議提示方式

```text
請先拆解問題，再給出結論，最後列出判斷依據。
```

```text
請用「問題分析 → 解法設計 → 程式碼 → 測試案例」的方式回答。
```

```text
請不要直接給答案，先說明你會如何判斷。
```

適合用在：

* 數學題
* 程式除錯
* 架構設計
* 投資分析
* 法規理解
* 流程狀態設計

---

## 3. Constraints：設定限制條件

限制條件是提示工程中非常重要的一環。

因為 AI 很容易回答過多、過廣，或者使用不符合需求的方式。

你可以明確指定：

```text
不要使用太多專業術語。
```

```text
請控制在 500 字以內。
```

```text
請不要直接給答案，先引導我思考。
```

```text
請使用 Java 8 語法，不要使用 Java 17 的新特性。
```

```text
請不要使用太複雜的設計模式。
```

限制條件可以讓回答更貼近你的實際需求。

---

## 4. Reverse Prompting：逆向提示

當你不知道怎麼問時，可以反過來請 AI 告訴你需要提供什麼資訊。

### 範例

```text
我想請你幫我設計一份 Java 學習計畫。

但我不知道該提供哪些資訊。

請先列出你需要我回答的問題，等我補充後，再幫我設計。
```

適合用在：

* 學習計畫
* 專案規劃
* 履歷修改
* 系統設計
* 投資規劃
* 面試準備

逆向提示的核心是：

```text
當你不知道怎麼問，
就先讓 AI 幫你定義問題。
```

## 七、提示品質比較

### 不好的提示

```text
請教我 Vue。
```

問題：

* 範圍太大
* 沒有學習背景
* 沒有指定目標
* 沒有輸出格式
* 沒有說明想學到什麼程度

---

### 較好的提示

```text
你是一位資深 Vue 3 前端工程師。

我已經學過 HTML、CSS、JavaScript、TypeScript，目前想系統性複習 Vue 3。

請幫我整理一份 Vue 3 學習路線，目標是能看懂 View UI Plus 這類組件庫源碼。

請按照以下格式輸出：

1. 必備基礎
2. Vue 核心概念
3. 組件設計能力
4. 原始碼閱讀重點
5. 練習建議

請用初中階前端工程師能理解的方式說明。
```

這個提示更好，因為它有：

* 角色
* 背景
* 目標
* 任務
* 格式
* 輸出程度限制

## 八、萬用提示模板

### 模板一：教學筆記模板

```text
你是一位擅長技術教學的講師。

我目前的背景是：
- [填入你的程度]
- [填入你已經學過的內容]
- [填入你目前卡住的地方]

請幫我整理一篇關於「[主題]」的教學筆記。

請按照以下結構輸出：

1. 核心概念
2. 為什麼需要它
3. 使用場景
4. 基本範例
5. 常見錯誤
6. 與其他概念的差異
7. 總結
8. 練習題

限制條件：
- 請用初學者能理解的方式說明
- 不要只給定義，要搭配例子
- 請使用 Markdown 格式
```

---

### 模板二：程式碼解釋模板

```text
你是一位資深 [技術領域] 工程師。

我正在學習 [主題]，目前程度是 [你的程度]。

請幫我解釋以下程式碼：

[貼上程式碼]

請按照以下順序說明：

1. 這段程式碼整體在做什麼
2. 每個重要區塊的作用
3. 執行流程
4. 可能的問題
5. 可以如何優化
6. 總結我應該學到什麼
```

---

### 模板三：比較型模板

```text
你是一位熟悉 [領域] 的專家。

請幫我比較 [A] 和 [B] 的差異。

請用表格整理，欄位包含：

1. 定義
2. 使用場景
3. 優點
4. 缺點
5. 適合什麼情況
6. 初學者應該如何理解

最後請用一段話總結：
什麼情況下應該選 A，什麼情況下應該選 B。
```

---

### 模板四：專案設計模板

```text
你是一位資深全端工程師。

我想開發一個 [專案名稱]。

我的背景：
- [技術能力]
- [目前會的框架]
- [希望練習的能力]

請你幫我設計這個專案。

請按照以下結構輸出：

1. 專案目標
2. 核心功能
3. 頁面設計
4. 資料結構
5. 狀態設計
6. 流程設計
7. 前端技術重點
8. 後端技術重點
9. 可以分階段完成的開發順序

限制條件：
- 不要一次設計得太複雜
- 請適合我的目前程度
- 請以可實作為主
```

## 九、提示工程常見錯誤

### 錯誤一：只給任務，沒有背景

不佳：

```text
請幫我學 Java。
```

較佳：

```text
我已經學過 Java 基本語法，但還不熟物件導向。
請幫我設計一份物件導向的學習順序。
```

---

### 錯誤二：要求太多，但沒有優先順序

不佳：

```text
請教我 Java、Spring、MySQL、Redis、Docker、Linux。
```

較佳：

```text
我想成為 Java 全端工程師。
請先幫我規劃前三個月最重要的學習順序，並說明哪些可以先學，哪些可以之後再學。
```

---

### 錯誤三：只說不要什麼，沒有說要什麼

不佳：

```text
不要太難，不要太長，不要太理論。
```

較佳：

```text
請用初學者能理解的方式說明，控制在 800 字內，並搭配一個實際例子。
```

---

### 錯誤四：沒有指定輸出格式

不佳：

```text
請比較 Vue 和 React。
```

較佳：

```text
請用表格比較 Vue 和 React，欄位包含：學習曲線、核心思想、適合場景、生態系、初學者建議。
```

---

### 錯誤五：把複雜任務一次丟給 AI

不佳：

```text
請幫我從零學會 Java 全端。
```

較佳：

```text
請先幫我把 Java 全端能力拆成 5 個階段。
每個階段請列出核心目標、必學知識、練習項目。
```

---

## 十、提示工程的實戰心法

高品質提示可以用一句話總結：

```text
角色決定回答角度，
背景決定回答深度，
請求決定任務方向，
行動決定處理流程，
格式決定輸出品質。
```

寫提示時，可以先問自己五個問題：

1. 我希望 AI 扮演什麼角色？
2. 我到底要 AI 做什麼？
3. AI 需要知道哪些背景？
4. 我希望 AI 怎麼處理？
5. 我希望 AI 用什麼格式輸出？

只要這五個問題回答清楚，提示品質通常就會大幅提升。

## 十一、練習題

### 練習一：改寫模糊提示

請把以下提示改寫成高品質提示：

```text
請教我 MySQL。
```

提示方向：

* 補上角色
* 補上學習背景
* 補上學習目標
* 補上輸出格式
* 補上限制條件

---

### 練習二：設計程式碼解釋提示

請設計一個提示，讓 AI 幫你解釋一段 Java 程式碼。

提示中至少要包含：

* AI 角色
* 你的學習程度
* 想理解的重點
* 輸出格式

---

### 練習三：設計筆記整理提示

請設計一個提示，讓 AI 幫你把一篇文章整理成 Obsidian 筆記。

提示中至少要包含：

* 筆記用途
* 目標讀者
* 整理格式
* 是否需要補充範例
* 是否需要保留原文重點

---

## 十二、總結

提示工程的重點不是使用華麗的指令，而是把需求講清楚。

對初學者來說，最實用的提示公式是：

```text
你是誰？
我要你做什麼？
我目前的背景是什麼？
請你怎麼做？
請用什麼格式輸出？
```

也就是：

```text
角色 + 請求 + 背景 + 行動 + 格式
```

當你能穩定使用這個結構，就能讓 AI 從「泛泛而談的回答工具」，變成「能配合你學習、工作、整理知識的輔助系統」。
