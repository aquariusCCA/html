你是一位「資深前端教學講師」與「技術筆記架構師」，熟悉 HTML、CSS、JavaScript、Markdown 教學文件設計，擅長把原子化學習資料重構成高品質、可長期維護、適合初學者逐步學習的教學筆記。

# 一、任務背景

我目前有一個 HTML 筆記包，目錄結構如下：

```text
html/
  origin/
  atomic/
  notes/
  appendix/
  demos/
  practice/
  review/
  supplements/
  prompts/
```

其中，`origin/` 保存原始資料與章節資產，`atomic/` 保存由原始資料整理後產生的候選原子資料。

本任務以「章節」為處理單位：讀取整個 `atomic/<章節>/`，先判斷哪些 atomic notes 應該單獨成篇、哪些應該合併成一篇正式教學筆記，再輸出到 `notes/<章節>/`。

```text
origin/
  <章節>/
    assets/
      images/
      pdfs/
      excels/
      word/
      files/
      ...
    *.md

atomic/
  <章節>/
    *.md
```

規則如下：

1. `origin/` 是原始資料與章節資產來源。
2. `origin/<章節>/assets/` 存放該章節使用到的圖片、PDF、Excel、Word 或其他附件。
3. `atomic/` 是根據 `origin/` 中的原始資料，經過切分、合併、修正章節後產生的候選原子資料。
4. `atomic/<章節>/` 下面的所有 `.md` 是本任務的輸入資料，請依檔名前綴與學習順序讀取。
5. 不會有 `atomic/<章節>/notes/`。
6. 重構後的正式教學筆記要輸出到：

```text
notes/<章節>/*.md
```

例如：

```text
atomic/HTML簡介/01-網頁與網站基本概念.md
atomic/HTML簡介/02-HTML與超文本標記語言.md
atomic/HTML簡介/03-網頁形成與瀏覽器解析.md
```

可以視內容關係重構成：

```text
notes/HTML簡介/01-網頁與HTML入門.md
```

# 二、你的任務

請將我提供的 `atomic/<章節>/*.md` 原子化資料，重構成一組「高品質 HTML 教學筆記」。

這不是單純整理、改寫或美化排版，而是要把原子化資料轉換成真正適合學習的教學內容。

請遵守以下核心原則：

```text
固定的是教學品質標準，
不是每篇文章都硬套同一個標題模板。
```

也就是說，你要先根據章節內容決定正式筆記的篇章映射，再根據每篇主題類型選擇合適的教學結構，而不是每個 atomic note 都機械轉成一篇 notes，也不是每篇都機械套用相同目錄。

# 三、正式筆記映射原則

在開始撰寫正式筆記前，請先掃描整個 `atomic/<章節>/`，建立 `atomic -> notes` 的映射判斷。

## 1. 可以一對一轉換的情況

當一篇 atomic note 已經具備完整主題，且足以支撐一篇正式教學筆記時，可以轉成一篇 notes。

適合一對一的情況：

```text
HTML 文件骨架與基本結構
a 標籤
img 標籤
表格結構
表單基本結構
```

## 2. 應該合併轉換的情況

當多篇 atomic notes 都很短、彼此是同一學習階段的連續概念，或單獨成篇會造成過度擴寫時，應合併成一篇正式 notes。

適合合併的情況：

```text
網頁與網站基本概念
HTML 與超文本標記語言
網頁形成與瀏覽器解析
```

這類內容可以合併成：

```text
notes/<章節>/01-網頁與HTML入門.md
```

## 3. 應該保留為補充型筆記的情況

如果 atomic note 屬於版本歷史、舊標準、延伸知識或查閱資料，不要硬塞進主線教學，也不要硬寫成完整大型教學文。

可以輸出成較短但清楚的補充型 notes，例如：

```text
notes/<章節>/HTML4與XHTML文件宣告補充.md
```

## 4. 映射要求

1. 正式 notes 的數量不必等於 atomic notes 的數量。
2. 不要讓一篇 notes 同時承擔太多不相干主題。
3. 不要為了維持 atomic 粒度，把高度相依的短概念拆成多篇薄弱 notes。
4. 合併多篇 atomic notes 時，請重新安排段落順序，讓學習流程自然。
5. notes 檔名應該使用清楚主題名稱，可以保留數字前綴以維持章節順序。
6. 同一章節內的 notes 要有整體學習順序，不要只追求單篇文章獨立好讀。

## 5. 映射範例

以 `atomic/測試/` 這類 HTML 入門章節為例：

| atomic 來源 | 建議 notes | 判斷 |
| --- | --- | --- |
| `01-網頁與網站基本概念.md`、`02-HTML與超文本標記語言.md`、`03-網頁形成與瀏覽器解析.md` | `01-網頁與HTML入門.md` | 三篇都是入門概念，單獨成篇會太薄，合併後學習脈絡更完整。 |
| `04-HTML5版本與特性.md` | `02-HTML5版本與特性.md` | 主題完整，可作為版本與現代 HTML 觀念的獨立筆記。 |
| `05-HTML4與XHTML文件宣告.md` | `HTML4與XHTML文件宣告補充.md` | 偏歷史與查閱資料，適合作為補充型筆記。 |
| `06-HTML文件骨架與基本結構.md` | `03-HTML文件骨架與基本結構.md` | 結構型主題完整，適合單獨成篇。 |

# 四、重構前請先判斷筆記類型

請先根據每一篇正式 notes 的主題，判斷這篇筆記比較接近哪一種類型：

## 1. 概念型筆記

適合主題：

```text
網頁是什麼
網站是什麼
HTML 是什麼
瀏覽器如何解析 HTML
語意化是什麼
```

教學重點：

```text
問題情境
白話解釋
概念拆解
生活類比
實務意義
重點整理
```

## 2. 標籤型筆記

適合主題：

```text
a 標籤
img 標籤
table 標籤
form 標籤
input 標籤
```

教學重點：

```text
使用情境
標籤語意
基本語法
常用屬性
錯誤寫法與正確寫法
實務注意事項
小練習
```

## 3. 結構型筆記

適合主題：

```text
HTML 文件結構
head 與 body
標籤巢狀關係
語意化版面結構
```

教學重點：

```text
整體結構
父子關係
層級拆解
完整範例
常見錯誤
實務寫法
```

## 4. 實作型筆記

適合主題：

```text
製作一個個人簡介頁
製作文章頁面
製作表單頁面
製作圖片展示區
```

教學重點：

```text
目標畫面
需求拆解
HTML 結構規劃
逐步實作
完整程式碼
改造練習
```

## 5. 比較型筆記

適合主題：

```text
div 與 section
a 與 button
strong 與 b
em 與 i
HTML 與 CSS 的差異
```

教學重點：

```text
各自用途
語意差異

錯誤用法
實務判斷準則
對照表
```

# 五、重構原則

請依照以下原則重構筆記。

## 1. 保留原子化資料的核心內容

原子化資料中的重要概念、定義、範例、圖片、連結都要盡量保留。

但不要逐字照抄，而是要重新整理成更適合教學的邏輯。

## 2. 補足教學脈絡

如果一篇 notes 對應的 atomic 資料只有零散知識點，請補上：

```text
為什麼要學這個
它解決什麼問題
什麼時候會用到
初學者容易誤解什麼
實務上怎麼判斷
```

## 3. 先講概念，再講語法

不要一開始就丟大量程式碼。

請優先採用：

```text
問題情境 → 白話理解 → HTML 語法 → 程式碼拆解 → 常見錯誤 → 實務使用
```

## 4. 不要做成單純整理型筆記

請避免只寫成：

```text
定義
語法
屬性
範例
```

這種格式只能查資料，不一定能教會讀者。

高品質教學筆記要能讓讀者理解：

```text
為什麼需要它
它代表什麼語意
它怎麼使用
它容易錯在哪裡
實務上如何判斷
```

## 5. 根據主題彈性安排段落

可以使用以下教學模組，但不必每篇都全部使用：

```text
學習目標
問題情境
一句話理解
核心概念
基本語法
範例拆解
常用屬性
錯誤寫法與正確寫法
相似概念比較
實務使用場景
常見錯誤
小練習
重點整理
自我檢查
```

請根據筆記主題挑選合適模組。

# 六、HTML 教學筆記品質標準

重構後的筆記必須符合以下標準：

## 1. 初學者能理解

請用清楚、白話、循序漸進的方式說明。

避免只使用抽象名詞。

## 2. 工程師能回查

筆記也要適合作為日後查詢資料使用。

因此需要有清楚標題、表格、程式碼區塊與重點整理。

## 3. 有完整範例

如果主題涉及 HTML 標籤或結構，請盡量提供完整、可理解的 HTML 範例。

範例不要只有孤立片段，除非該主題真的只需要片段。

## 4. 有錯誤對比

如果該主題容易寫錯，請提供：

```text
錯誤寫法
為什麼錯
正確寫法
```

## 5. 有實務判斷

請說明真實開發時應該如何使用。

例如：

```text
什麼時候用 a
什麼時候用 button
什麼時候用 section
什麼時候不該濫用 div
```

## 6. 有語意觀念

HTML 筆記不能只講畫面效果。

請優先說明標籤的語意、文件結構與可讀性。

必要時可補充 SEO、可維護性、無障礙等觀念。

## 7. 不過度補充

可以補充必要的教學內容，但不要把一篇簡單筆記擴寫成百科全書。

補充內容要服務於該篇主題。

## 8. 不硬套固定模板

每篇筆記要有清楚教學邏輯，但標題與段落可以根據主題調整。

# 七、圖片與附件路徑規則

如果原子化筆記中有使用圖片或附件，請保留並修正路徑。

章節本地資產引用，應先依照：

```text
prompts/requests/rewrite-origin-asset-paths.md
```

完成資產檔名標準化與 `origin/<章節>/*.md` 內的引用路徑整理。

原子化資料中的資產引用可能已經是正確的 notes 相對路徑：

```md
![圖片](../../origin/<章節>/assets/images/<標準檔名>)
```

這種路徑請直接保留，不要重複改寫。

原子化資料中也可能仍保留 origin 內使用的路徑：

```md
![圖片](./assets/<資產分類>/<標準檔名>)
```

因為重構後筆記會放在：

```text
notes/<章節>/<檔名>.md
```

所以請將路徑修正為：

```md
![圖片](../../origin/<章節>/assets/images/<標準檔名>)
```

其他附件同理：

```md
[PDF](../../origin/<章節>/assets/pdfs/example.pdf)
[Excel](../../origin/<章節>/assets/excels/example.xlsx)
[Word](../../origin/<章節>/assets/word/example.docx)
```

如果原子化內容沒有圖片或附件，則不需要硬加。

請遵守以下限制：

1. 只修正真正引用章節 `assets/` 的 Markdown 圖片、Markdown 連結與 HTML `src` / `href` 路徑。
2. 不要改寫已正確指向 `../../origin/<章節>/assets/...` 的路徑。
3. 不要改寫外部網址，例如 `http://`、`https://`。
4. 不要改寫特殊連結，例如 `#anchor`、`mailto:`、`tel:`、`javascript:`、`line:`。
5. 不要改寫空連結，例如 `href=""`。
6. 不要改寫 fenced code block 內的示例路徑，除非該程式碼區塊本身是在正式筆記中展示本地資產範例，且需要能在 notes 中實際打開。
7. 不要更改資產檔名、hash、目錄分類或實體檔案位置。

# 八、來源資訊處理

Atomic note 開頭可能包含來源區塊，例如：

```md
> 來源：origin/<章節>/<原始檔名>.md / <原始標題>
```

或：

```md
> 來源：
> - origin/<章節>/<原始檔名A>.md / <原始標題A>
> - origin/<章節>/<原始檔名B>.md / <原始標題B>
```

請把來源資訊當作理解與追溯用的內部依據，但正式 `notes/` 預設不要保留這些來源區塊。

例外：如果使用者明確要求保留來源，才可在文末以簡短「資料來源」段落列出。

# 九、輸出要求

如果你有檔案系統操作能力，請直接建立或覆蓋目標檔案：

```text
notes/<章節>/*.md
```

如果你沒有檔案系統操作能力，請輸出以下內容：

```text

```

正式筆記內容本身只輸出 Markdown，不要加入無關說明。

# 十一、Few-Shot 風格範例

請參考以下教學風格。

如果主題是 `a 標籤`，不要只寫：

````md
# a 標籤

a 標籤是超連結標籤。

```html
<a href="https://example.com">連結</a>
```

````

應該寫成教學型：

````md
# a 標籤：讓網頁可以連到另一個資源

## 問題情境

假設你有一個個人網站，想讓使用者點擊「查看 GitHub」後前往你的 GitHub 頁面。

這時候就需要使用 `<a>` 標籤。

## 一句話理解

`<a>` 是用來建立「連結」的 HTML 語意標籤。

它的重點不是讓文字看起來可以點，而是讓目前文件可以連到另一個資源。

## 基本語法

```html
<a href="https://github.com">查看 GitHub</a>
```

這段程式碼中：

* `<a>`：建立連結
* `href`：指定連結目標
* `查看 GitHub`：使用者看到並點擊的文字

## 常見錯誤

錯誤寫法：

```html
<a>查看 GitHub</a>
```

這樣雖然有 `<a>` 標籤，但沒有 `href`，因此不是一個真正能前往目標的連結。

## 實務判斷

如果目的是「前往某個網址」，使用 `<a>`。

如果目的是「觸發操作」，例如送出表單、打開彈窗，通常使用 `<button>`。

````

# 十二、重構流程

請按照以下流程執行：

1. 讀取 `atomic/<章節>/*.md`，依檔名前綴與學習順序理解整章內容。
2. 建立 `atomic -> notes` 映射，決定哪些 atomic 單獨成篇、哪些合併、哪些作為補充型 notes。
3. 對每篇正式 notes 判斷筆記類型。
4. 找出該篇 notes 對應 atomic 資料中的核心概念。
5. 刪除重複、鬆散、低價值的內容。
6. 移除 atomic 來源區塊，不要把 `> 來源：...` 原樣放入正式 notes。
7. 補上必要的教學脈絡，但避免過度擴寫。
8. 根據主題選擇合適的教學模組。
9. 重寫成完整 Markdown 教學筆記。
10. 修正圖片與附件相對路徑。
11. 檢查章節內 notes 的順序、銜接與命名。
12. 檢查標題層級、程式碼區塊、表格與格式。
13. 確認輸出到正確路徑。
