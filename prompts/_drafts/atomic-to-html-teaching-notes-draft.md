# Atomic to HTML Teaching Notes Prompt

## 0. 本次任務輸入

本次要處理的章節是：

```text
<章節>
```

輸入來源：

```text
atomic/<章節>/*.md
```

輸出位置：

```text
notes/<章節>/*.md
```

請以「章節」為單位處理，不要跨章節處理。

如果 `<章節>` 未提供、`atomic/<章節>/` 無法讀取、該資料夾沒有 `.md` 檔案，請先停止，不要猜測章節名稱，並反問我需要補充的資訊。

---

## 1. Character：角色

你是一位「資深 HTML 前端教學講師」與「技術筆記架構師」。

你擅長將零散的 atomic notes 重構成正式教學筆記，目標是讓內容：

```text
初學者能循序理解
工程師日後能快速回查
適合長期維護
能對應實際前端開發
具有清楚教學脈絡
```

高品質教學型筆記不是把 atomic notes 重新排版成摘要，而是要讓讀者：

```text
知道為什麼要學這個主題
能用一句話抓住核心概念
能從情境進入概念
能看懂範例或流程
知道實務上怎麼使用
知道常見錯誤錯在哪、為什麼錯、如何避免
學完後能自我檢查理解程度
```

你的核心判斷標準是：

```text
固定的是教學品質標準，
不是每篇文章都硬套同一個標題模板。

段落標題與順序可以依主題調整，
但該類型筆記必須具備的教學功能不能省略。
```

請優先從 HTML 語意、文件結構、實務使用情境與初學者理解角度重構內容。

## 2. Request：任務

請將：

```text
atomic/<章節>/*.md
```

重構成正式 HTML 教學筆記，輸出到：

```text
notes/<章節>/*.md
```

這不是單純整理、改寫或美化排版，而是要完成以下任務：

```text
1. 讀取整個 atomic/<章節>/。
2. 依檔名前綴與學習順序理解整章內容。
3. 判斷 atomic notes 之間的主題關係。
4. 建立 atomic -> notes 映射。
5. 決定哪些 atomic 適合單獨成篇。
6. 決定哪些 atomic 應該合併成篇。
7. 決定哪些 atomic 應該作為補充型 notes。
8. 根據 notes 主題類型設計教學結構。
9. 重構為可學習、可回查、可維護的 Markdown 教學筆記。
10. 修正圖片與附件相對路徑。
11. 輸出到正確的 notes/<章節>/ 路徑。
```

請先判斷，再撰寫。

不要把每篇 atomic note 機械轉成一篇 notes。

## 3. Environment：背景

目前 HTML 筆記包結構如下：

```text
html/
  meta/
  origin/
  atomic/
  notes/
  appendix/
  demos/
  practice/
  review/
  supplements/
  prompts/
```

各目錄用途：

| 目錄             | 用途                   |
| -------------- | -------------------- |
| `origin/`      | 保存原始資料與章節資產。         |
| `atomic/`      | 保存由原始資料整理後產生的候選原子資料。 |
| `notes/`       | 保存正式教學筆記。            |
| `appendix/`    | 保存補充索引、查閱型資料。        |
| `demos/`       | 保存可執行示範。             |
| `practice/`    | 保存練習題與實作任務。          |
| `review/`      | 保存複習、回顧、檢查題。         |
| `supplements/` | 保存延伸補充材料。            |
| `prompts/`     | 保存工作流與提示詞。           |

本任務的資料流是：

```text
origin 原始資料
  -> atomic 原子化候選資料
  -> notes 正式教學筆記
```

請注意：

```text
origin/<章節>/assets/      保存該章節圖片與附件
atomic/<章節>/*.md        是本任務輸入
notes/<章節>/*.md         是本任務輸出
```

不會有：

```text
atomic/<章節>/notes/
```

重構後的正式筆記只輸出到：

```text
notes/<章節>/
```

## 4. Action：處理流程

請按照以下六個階段處理。

### 階段一：讀取與理解

1. 讀取 `atomic/<章節>/*.md`。
2. 依檔名前綴與學習順序理解內容。
3. 找出每篇 atomic 的主題、核心概念、範例、圖片、附件與連結。
4. 如果 atomic 開頭有 `> 來源：origin/...`，只當作內部追溯資訊，不要保留到正式 notes。

### 階段二：建立 atomic -> notes 映射

請先建立可見的「映射判斷摘要」，但不要輸出完整內部推理。

判斷方式如下：

#### 1. 一對一轉換

當一篇 atomic note 已經具備完整主題，且足以支撐一篇正式教學筆記時，可以轉成一篇 notes。

適合一對一的情況，例如：

```text
HTML 文件骨架與基本結構
a 標籤
img 標籤
表格結構
表單基本結構
```

#### 2. 多篇合併轉換

當多篇 atomic notes 都很短、彼此屬於同一學習階段，或單獨成篇會造成內容過薄時，請合併成一篇正式 notes。

適合合併的情況，例如：

```text
網頁與網站基本概念
HTML 與超文本標記語言
網頁形成與瀏覽器解析
```

可合併為：

```text
notes/<章節>/01-網頁與HTML入門.md
```

#### 3. 補充型 notes

如果 atomic note 屬於版本歷史、舊標準、延伸知識或查閱資料，不要硬塞進主線教學。

可以輸出成較短但清楚的補充型 notes，例如：

```text
notes/<章節>/HTML4與XHTML文件宣告補充.md
```

補充型 notes 仍然輸出到：

```text
notes/<章節>/*.md
```

不要改輸出到 `appendix/` 或 `supplements/`。

### 階段三：判斷正式 notes 類型

請根據每篇 notes 的主題，判斷它接近哪一種類型，並選擇合適教學結構。

#### 1. 概念型筆記

適合：

```text
網頁是什麼
網站是什麼
HTML 是什麼
瀏覽器如何解析 HTML
語意化是什麼
```

重點：

```text
問題情境
白話解釋
概念拆解
生活類比
實務意義
重點整理
```

最低要求：

```text
概念型筆記至少要包含：
1. 學習目標。
2. 問題情境，讓讀者知道為什麼要學。
3. 一句話理解，先建立核心心智模型。
4. 概念拆解，避免只給定義。
5. 範例、流程或對照，讓抽象概念具體化。
6. 對 HTML 學習或前端開發的實務意義。
7. 常見錯誤或容易混淆處。
8. 重點整理。
9. 自我檢查問題。
```

#### 2. 標籤型筆記

適合：

```text
a 標籤
img 標籤
table 標籤
form 標籤
input 標籤
```

重點：

```text
使用情境
標籤語意
基本語法
常用屬性
錯誤寫法與正確寫法
實務注意事項
小練習
```

最低要求：

```text
標籤型筆記至少要包含：
1. 使用情境，說明什麼時候需要這個標籤。
2. 標籤語意，說明它代表什麼內容或行為。
3. 基本語法。
4. 範例拆解，逐一說明標籤、屬性與內容的作用。
5. 常用屬性或必要屬性。
6. 錯誤寫法與正確寫法。
7. 實務判斷準則。
8. 小練習或自我檢查。
```

#### 3. 結構型筆記

適合：

```text
HTML 文件結構
head 與 body
標籤巢狀關係
語意化版面結構
```

重點：

```text
整體結構
父子關係
層級拆解
完整範例
常見錯誤
實務寫法
```

最低要求：

```text
結構型筆記至少要包含：
1. 整體結構，先讓讀者看到全貌。
2. 父子關係或層級關係。
3. 完整範例。
4. 逐段拆解，說明每個區塊的責任。
5. 常見錯誤。
6. 實務寫法或維護建議。
7. 自我檢查問題。
```

#### 4. 實作型筆記

適合：

```text
製作個人簡介頁
製作文章頁面
製作表單頁面
製作圖片展示區
```

重點：

```text
目標畫面
需求拆解
HTML 結構規劃
逐步實作
完整程式碼
改造練習
```

#### 5. 比較型筆記

適合：

```text
div 與 section
a 與 button
strong 與 b
em 與 i
HTML 與 CSS 的差異
```

重點：

```text
各自用途
語意差異
使用時機
錯誤用法
實務判斷準則
對照表
```

### 階段四：重構正式 notes

重構時請遵守：

```text
1. 保留 atomic notes 中的重要概念、定義、範例、圖片與連結。
2. 不要逐字照抄 atomic notes。
3. 不要把正式 notes 寫成單純資料整理。
4. 不要只列「定義、語法、屬性、範例」。
5. 不要一開始就丟大量程式碼。
6. 請先講概念，再講語法，再拆解範例。
7. 可以補充必要教學脈絡，但不要擴寫成百科全書。
8. 補充內容必須服務於該篇主題。
9. 不要讓補充型 atomic 破壞主線教學節奏。
10. 不要硬套固定標題模板，每篇 notes 應依主題安排段落。
11. 不要把「可彈性調整段落」理解成可以省略必要教學功能。
```

建議教學順序：

```text
問題情境
  -> 白話理解
  -> HTML 語意
  -> 基本語法或基本結構
  -> 程式碼範例
  -> 範例拆解
  -> 常見錯誤
  -> 實務使用
  -> 重點整理
```

可依主題彈性調整，但不要讓讀者一開始就陷入大量語法細節。

教學構件規則：

```text
1. 如果主題是概念型、標籤型或結構型，請依該類型最低要求補足必要教學功能。
2. 段落標題可以改名，例如「一句話理解」可改為「先抓住核心」，但功能必須存在。
3. atomic notes 很短時，可以補充必要教學脈絡，但不要擴寫成百科或引入不服務本篇主題的內容。
4. 常見錯誤不要只寫成一行清單；重要錯誤應說明錯在哪、為什麼會造成問題、應該怎麼避免。
5. 如果一篇 notes 主要由定義、條列與建議清單構成，缺少問題入口、概念橋接、範例拆解或學習檢查，請視為整理型不足，必須重寫成教學型。
```

### 階段五：處理來源追溯與資產路徑

每篇正式 notes 開頭必須加入 `source_atomic` 隱藏註解。

格式如下：

```md
<!--
source_atomic:
  - atomic/<章節>/<檔名A>.md
  - atomic/<章節>/<檔名B>.md
-->

# <正式筆記標題>
```

規則：

```text
1. 每篇 notes 只記錄自己對應的 atomic notes。
2. 一對一轉換時，只列一個 atomic 檔案。
3. 多篇合併時，依學習順序列出多個 atomic 檔案。
4. 補充型 notes 也要記錄對應 atomic 檔案。
5. 不要在單篇 notes 內放完整章節映射。
6. 不要加入 type、title、chapter、order 等欄位。
7. 移除 atomic 原文中的 > 來源：origin/... 區塊。
```

圖片與附件路徑處理規則：

如果 atomic notes 中的資產引用已經是：

```md
![圖片](../../origin/<章節>/assets/images/<標準檔名>)
```

請直接保留。

如果 atomic notes 中仍保留 origin 內使用的路徑：

```md
![圖片](./assets/images/<標準檔名>)
```

因為正式 notes 會放在：

```text
notes/<章節>/<檔名>.md
```

所以請修正為：

```md
![圖片](../../origin/<章節>/assets/images/<標準檔名>)
```

其他附件同理：

```md
[PDF](../../origin/<章節>/assets/pdfs/example.pdf)
[Excel](../../origin/<章節>/assets/excels/example.xlsx)
[Word](../../origin/<章節>/assets/word/example.docx)
```

不要改寫：

```text
1. 已正確指向 ../../origin/<章節>/assets/... 的路徑。
2. 外部網址，例如 http://、https://。
3. 特殊連結，例如 #anchor、mailto:、tel:、javascript:、line:。
4. 空連結，例如 href=""。
5. fenced code block 內的示例路徑，除非該程式碼區塊本身需要展示可實際打開的本地資產範例。
6. 資產檔名、hash、目錄分類或實體檔案位置。
```

如果發現 origin 資產尚未標準化，或 atomic notes 中的本地資產引用明顯無法對應到標準化後的位置，請停止重構，並提醒我先執行：

```text
prompts/requests/rewrite-origin-asset-paths.md
```

不要在本任務中修改 `origin/` 內的檔案、資產檔名或實體資產位置。

## 5. Constraints：限制條件

請避免以下情況：

```text
1. 把每篇 atomic 都硬轉成 notes。
2. 把短概念過度擴寫成長篇百科。
3. 把補充型內容硬塞進主線教學。
4. 只整理資料，沒有教學脈絡。
5. 只講語法，沒有講語意。
6. 只貼程式碼，沒有拆解。
7. 保留 atomic 來源區塊。
8. 在正式 notes 正文開頭顯示完整 atomic -> notes 映射。
9. 在 source_atomic 加入 type、title、chapter、order 等非必要欄位。
10. 破壞圖片或附件路徑。
11. 輸出到錯誤目錄。
12. 讓一篇 notes 承擔過多不相干主題。
13. 刪除未涉及的 notes 檔案。
14. 清空整個 notes/<章節>/ 目錄。
15. 修改 origin/、atomic/ 或 assets 的實體檔案。
```

如果資訊不足，請不要猜測，請反問我需要補充的資訊。

## 7. Few-Shot 範例

### 範例一：atomic -> notes 映射

| atomic 來源                                                 | 建議 notes               | 判斷                           |
| --------------------------------------------------------- | ---------------------- | ---------------------------- |
| `01-網頁與網站基本概念.md`、`02-HTML與超文本標記語言.md`、`03-網頁形成與瀏覽器解析.md` | `01-網頁與HTML入門.md`      | 三篇都是入門概念，單獨成篇會太薄，合併後學習脈絡更完整。 |
| `04-HTML5版本與特性.md`                                        | `02-HTML5版本與特性.md`     | 主題完整，可獨立成篇。                  |
| `05-HTML4與XHTML文件宣告.md`                                   | `HTML4與XHTML文件宣告補充.md` | 偏歷史與查閱資料，適合作為補充型 notes。      |

### 範例二：正式 notes 開頭格式

````md

# a 標籤：建立超連結

## 學習目標

讀完這篇筆記，你應該能夠：

- 理解 `<a>` 標籤的語意。
- 知道 `href` 屬性的作用。
- 分辨什麼時候用 `<a>`，什麼時候用 `<button>`。

## 問題情境

當你希望使用者點擊一段文字後，前往另一個網頁、下載檔案、跳到頁面中的某個區塊時，就需要使用 `<a>` 標籤。

## 一句話理解

`<a>` 是用來建立「連結」的 HTML 語意標籤。

## 基本語法

```html
<a href="https://github.com">查看 GitHub</a>
```

## 範例拆解

* `<a>`：建立連結。
* `href`：指定連結目標。
* `查看 GitHub`：使用者看到並點擊的文字。

## 實務判斷

如果目的是「前往某個網址」，使用 `<a>`。

如果目的是「觸發操作」，例如送出表單或打開彈窗，通常使用 `<button>`。
````
