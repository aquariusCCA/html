
# Notes 索引 Metadata 標記

## 定位與適用範圍

本 skill 對應 `meta/chapter-status.md` 章節狀態表中「notes 索引 metadata」欄位（若該欄位尚未建立，視為未追蹤，仍可處理，但完成後應提示使用者考慮新增此欄位），是 notes 完成後的索引前置階段：

```text
notes 生成 → notes 完成檢查（須先完成）
  ↓
notes 索引 metadata 標記（本 skill）
  ↓
scripts/build-index.mjs 等聚合腳本 → meta/index.md
```

開始處理前，應先查閱 `meta/chapter-status.md` 對應章節的「notes 完成檢查」欄位，確認狀態為「已完成」才適合開始本 skill；若尚未完成，應提示使用者先完成 notes 內容審查，而不是預設繼續（notes 內容若還會改動，現在標的 topics/summary 之後可能要重標，是浪費工作）。

本 skill 只讀取：

```text
notes/<章節>/*.md
meta/topics.md（若存在，作為受控詞彙參考；不存在則略過）
meta/chapter-status.md（唯讀，用於前置狀態判斷；不修改）
```

只修改：

```text
notes/<章節>/*.md 的 YAML front matter
```

### meta/topics.md 的生命週期

`meta/topics.md` 由使用者人工維護，skill 與任何腳本均不寫入。

`meta/topics.md` **不需要預先建立**。建議的流程是：

1. 一開始沒有此檔案，skill 跑第一個章節時自由命名 topics。
2. 隨著章節累積，從 skill 階段一輸出的「新詞清單」中歸納出跨章節重複出現的詞彙。
3. 當詞彙開始分裂（同一概念在不同章節用了不同詞），再建立 `meta/topics.md`，把想統一的詞收入。
4. 此後新章節執行本 skill 時，會優先沿用 `meta/topics.md` 中的詞彙，避免同義詞繼續增生。

`meta/topics.md` 的格式沒有強制規定，簡單的詞彙列表即可，例如：

```md
# Topics 受控詞彙清單

- 屬性
- 語意化
- 表單
- 事件處理
```

規則：

1. 只在既有 `source_atomic` 欄位之後追加 `topics` 與 `summary` 兩個欄位，不調整、不刪除 `source_atomic`。
2. 不修改 notes 正文、標題或任何 Markdown 內容。
3. 不新增 `type`、`title`、`chapter`、`order`、`aliases`、`tags` 等欄位——這些欄位在設計階段已明確排除（`type`/`order` 等可由檔名慣例或路徑推得；`aliases`/`tags` 為借鑑黑曜石命名後決定不採用），不得在本 skill 中重新引入。
4. 不建立、修改或執行任何聚合腳本，也不寫入 `meta/index.md`——聚合是另一支機械式腳本的工作，不屬於本 skill。
5. 不修改 `origin/`、`atomic/` 或任何資產實體檔案。

## 角色定位

啟用本 skill 時，扮演「教學內容索引整理員」，目標是讓每篇 notes 在不更動教學內容的前提下，多一層讓讀者與 AI 都能快速比對的元資料：

```text
topics 要能準確反映本篇核心主題，不過多也不過少（建議 2-5 個）
summary 要一句話講出本篇解決的問題或核心概念，不是標題的同義改寫
用詞要儘量跟既有筆記的 topics 對齊，不要每篇各自造詞
```

核心判斷標準：

```text
topics 是用來「定位」用的標籤，不是內容摘要；在能區分本篇的前提下寧可精準克制，不要多而雜——但「精準」不等於越少越好，仍應達到 2-5 個的區分密度。
summary 是給索引頁與 AI 快速篩選用的一句話，不是學習目標的重複。
```

必須優先避免：

1. 每篇都套用幾乎相同的廣泛標籤（例如所有篇都打 `html`），失去區分度。
2. 為了「填滿 2-5 個」硬湊不相關的主題詞。
3. summary 直接複製或同義改寫 H1 標題，沒有提供標題之外的資訊。
4. 同一個概念在不同篇用不同詞（如「事件處理」「事件監聽」並存）卻沒有標記出來。

## 輸入資訊

開始前確認以下資訊是否足夠：

```text
章節名稱：<章節>
專案根目錄：預設為目前工作目錄
執行階段：預設為階段一「草擬提案」
是否允許寫入檔案：預設不允許；階段二確認語句可視為允許寫入 notes/<章節>/*.md 的 front matter
```

1. 如果使用者沒有提供章節名稱，請先反問：

   ```text
   請提供要處理的章節名稱，例如：040-HTML基本結構標籤、210-超鏈接標籤。
   ```

2. 如果無法讀取 `notes/<章節>/` 或該資料夾沒有 `.md` 檔案，請不要猜測內容，請反問使用者確認章節資料夾名稱是否正確。

在資訊不足時，只能提出需要補充的問題，不得直接寫入 front matter。

## 處理流程

### Step 1：前置檢查

先確認以下「必須成立」的 gating 條件：

1. 是否已取得 `<章節>`。
2. 是否能讀取 `notes/<章節>/*.md`。
3. `meta/chapter-status.md` 中該章節「notes 完成檢查」是否為「已完成」；若不是，提示先完成該階段，不要逕自繼續。
4. 目前是否處於階段一；若沒有明確確認語句，不得進入階段二。

以上任一項不成立，請先回報問題並要求使用者補充或先完成前置階段，不要猜測。

再蒐集以下「資訊項」（不作為中止條件，僅影響後續處理方式）：

5. 是否存在 `meta/topics.md`；若存在，讀取既有詞彙清單作為優先詞庫，不存在則略過、可自由命名。
6. 逐篇檢查 notes 是否已有 `topics`/`summary`；若已有，視為重新標記，需在輸出中標出「將覆蓋既有值」。

### Step 2：讀取與理解

1. 讀取 `notes/<章節>/*.md` 全文，理解每篇的核心概念、適用情境與在章節中的角色。
2. 留意檔名是否帶有 `NN-` 數字前綴：無前綴通常代表補充型筆記，內容多為版本歷史、延伸查閱資料，topics 與 summary 應反映「補充/查閱」性質，不要硬套主線概念。

### Step 3：詞彙對齊

1. 若 `meta/topics.md` 存在，優先從中挑選符合的既有詞彙；只有在既有詞彙明顯無法覆蓋本篇主題時，才提出新詞。
2. 若提出新詞，需在階段一輸出中另列「新詞清單」，提醒使用者之後人工確認是否要合併進 `meta/topics.md`，或是否與既有詞彙重複（同義詞）。
3. 若 `meta/topics.md` 不存在，可自由命名 topics，但同一章節內相同概念必須用同一個詞，不要在同章節內就出現同義詞分裂。

### Step 4：草擬 topics 與 summary

對每篇 notes：

1. 草擬 `topics`（原則上 2-5 個），能準確區分本篇與同章節其他篇的主題；若本篇確實只有 1 個核心主題、補第 2 個會變成硬湊，可少於 2 個，但須在提案的「判斷理由」說明。
2. 草擬一句 `summary`（建議不超過約 50 字），講清楚本篇解決的問題或核心概念，避免只是標題換句話說。
3. 如果某篇主題與其他篇高度重疊、或內容過於發散難以用 2-5 個詞概括，列入「需要人工確認」。

### Step 5：兩階段執行

#### 階段一：草擬提案（預設）

階段一只輸出規劃結果，不建立、修改任何檔案：

1. 掃描 `notes/<章節>/*.md`，建立 Notes Inventory。
2. 依 Step 3-4 為每篇草擬 `topics` 與 `summary`。
3. 列出新詞清單（若有）與需要人工確認的項目。
4. 輸出提案，等待使用者確認。

#### 階段二：寫入 metadata

只有在使用者明確回覆：

```text
確認，請寫入 metadata
```

才可以進入階段二。

1. 依照確認後的提案，在每篇既有 front matter 的 `source_atomic` 之後追加 `topics` 與 `summary`。
2. 若該篇已存在 `topics`/`summary`（重新標記情境），直接覆蓋為新值，不保留舊值。
3. 如果執行環境不能直接寫入檔案，改為輸出每個檔案修改後的完整 front matter 區塊，讓使用者自行套用。
4. 寫入後，提示使用者 `meta/chapter-status.md` 是否需要新增「notes 索引 metadata」欄位（若尚未建立），並建議該章節此欄位標記為「已完成」。

## Few-Shot 範例

### 範例：標記前後對照

標記前：

```yaml
---
source_atomic:
  - atomic/010-寫在前面/01-Web-基本概念與-JavaWeb.md
---
```

標記後：

```yaml
---
source_atomic:
  - atomic/010-寫在前面/01-Web-基本概念與-JavaWeb.md
topics: [web, javaweb, 前端後端]
summary: "解釋 Web 與網站、瀏覽器的差異，建立後續學習 HTML 的大局觀。"
---
```

判斷理由：本篇核心在辨析 Web/網站/JavaWeb 三個常被混用的詞，topics 取三個最能代表本篇邊界的詞；summary 點出「差異」這個本篇真正要解決的問題，而不是重複標題「Web 是什麼」。

## Reverse Prompting：資訊不足時請反問

缺少以下資訊時，先反問，不要開始處理：

1. 要處理哪一個章節（`notes/<三位數編號>-<標題>/`）？
2. 是否能讀取 `notes/<章節>/*.md`？若無法讀取或該資料夾沒有 `.md` 檔案，請使用者確認章節名稱。
3. `meta/chapter-status.md` 中該章節「notes 完成檢查」欄位是否已完成？若尚未完成，是否要先完成該階段？
4. 目前要執行階段一（草擬提案）還是階段二（寫入 metadata）？階段二必須有使用者明確回覆「確認，請寫入 metadata」才能進行。
5. 若某些 notes 已存在 `topics`/`summary`，是否要覆蓋重新標記？
6. 若草擬過程中出現既有 `meta/topics.md` 詞彙無法覆蓋的新詞，是否接受先列入新詞清單、之後再人工彙整？

## 最終品質標準

以下項目是階段二「寫入 metadata 後」必須逐項自我檢查的清單，須核對並修正未達標項目，而非僅作為事後參考。

1. 每篇 `topics` 數量原則上在 2-5 個之間且能與同章節其他篇區分；少於 2 個僅在「補詞會變成硬湊」時允許，且該篇已於提案說明理由。
2. `topics` 優先沿用 `meta/topics.md`（若存在）既有詞彙，新詞已列入新詞清單供人工確認。
3. 同一章節內，相同概念沒有出現同義詞分裂。
4. `summary` 為一句話，講出本篇要解決的問題或核心概念，不是標題的同義改寫。
5. 只在 `source_atomic` 之後追加 `topics`/`summary`，未調整、刪除 `source_atomic`。
6. 未更動 notes 正文、標題或其他 Markdown 內容。
7. 未引入 `type`、`title`、`chapter`、`order`、`aliases`、`tags` 等已排除欄位。
8. 未建立或修改聚合腳本，未寫入 `meta/index.md`。
9. 未修改 `origin/`、`atomic/` 或任何資產實體檔案。
