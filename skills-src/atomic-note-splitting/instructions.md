
# Origin 原始筆記原子化切分

## 定位與適用範圍

本 skill 對應 `meta/chapter-status.md` 章節狀態表中「atomic 切分提案」與「atomic 產生」兩個欄位，是 origin → atomic 轉換階段：

```text
origin/<章節>/*.md
origin/<章節>/assets/
  ↓
資產命名（asset-standardization，另一概念，須先完成或標記為不適用）
  ↓
alt 與連結文字（asset-alt-and-link-text，另一概念，須先完成或標記為不適用）
  ↓
atomic 切分提案（本 skill 階段一）
  ↓
atomic 產生（本 skill 階段二）
  ↓
atomic 內容審查 → notes 生成 → ...
```

`CLAUDE.md` 規定 `origin/` 唯讀，不可修改、覆蓋、刪除。本 skill 不修改 `origin/` 任何內容，只將 `origin/<章節>/*.md` 當作唯一原始資料來源讀取，輸出寫入：

```text
atomic/<章節>/*.md
```

本 skill 不是資產命名或 alt/連結文字整理（屬於另外兩個 skill，且應先於本 skill 完成或標記為不適用），也不是 notes/review/practice/demos/appendix 等下游生成流程；若使用者要求的是資產改名、alt 整理或下游內容生成，應提示改用其他流程。

開始處理前，建議先查閱 `meta/chapter-status.md` 對應章節的「atomic 切分提案」「atomic 產生」欄位，確認狀態為「未開始」「進行中」或「待確認」才需要實際處理；若已是「已完成」，應先確認使用者是否要重新處理，而非預設重跑。同時須確認該章節「資產命名」與「alt 與連結文字」欄位是否為「已完成」或「不適用」，避免在這兩項尚未完成前就進行切分；詳見 Step 1 前置檢查。

## 角色定位

啟用本 skill 時，同時扮演三種角色：

1. **內容切分者**：判斷哪些內容應該拆分、合併、移動、保留或刪重。
2. **Markdown 結構整理者**：調整標題層級、段落順序、清單縮排與資產路徑。
3. **資料安全守門人**：確保不修改、不刪除、不搬移 `origin/` 原始檔與 `assets/` 資產。

本 skill 的工作重點不是把內容改寫成正式教材，而是根據原始資料的語義重新切分、合併、排序與輕度修正，讓後續可以再由其他流程轉成正式 `notes/`、`review/`、`practice/` 或 `demos/` 內容。

必須優先避免：

1. 把原始筆記深度重寫成正式教材。
2. 自行補充原文沒有的新技術知識。
3. 刪除重要概念、程式碼範例、圖片、附件或注意事項。
4. 修改、刪除或搬移 `origin/<章節>/*.md` 原始檔與 `origin/<章節>/assets/` 內的資產檔案。
5. 在 `origin/<章節>/notes/` 建立任何內容（正常情況下不應存在）。
6. 修改 `meta/`、`notes/`、`appendix/`、`demos/`、`practice/`、`review/` 或 `supplements/` 中的內容。
7. 把 `atomic/<章節>/*.md`、`notes/<章節>/*.md` 或其他目錄內容當成原始資料重新整理。

## 輸入資訊

開始前確認以下資訊是否足夠：

```text
章節名稱：<章節>
專案根目錄：預設為目前工作目錄
執行階段：預設為階段一「切分提案」
是否允許寫入檔案：預設不允許；階段二確認語句可視為允許寫入 atomic/<章節>/，但既有檔案仍需再次確認
```

1. 如果使用者沒有提供章節名稱，請先反問：

   ```text
   請提供要處理的章節名稱，例如：040-HTML基本結構標籤、210-超鏈接標籤。
   ```

2. 如果無法讀取 `origin/<章節>/` 或無法取得該路徑下的 Markdown 檔案，請不要猜測內容，請要求使用者提供：

   - 該章節的檔案清單
   - 要處理的 Markdown 原文
   - assets 引用情況

在資訊不足時，只能提出需要補充的問題，不得直接產生 atomic notes。

## 處理流程

### Step 1：前置檢查

處理前請先檢查：

1. 是否已取得 `<章節>`。
2. 是否能讀取 `origin/<章節>/`。
3. 是否存在 `origin/<章節>/*.md`。
4. 是否只會把 `origin/<章節>/*.md` 當作原始資料。
5. 是否不會把既有 `atomic/<章節>/*.md` 當成原始資料。
6. 目前是否處於階段一；若沒有明確確認，不得進入階段二。
7. 該章節「資產命名」與「alt 與連結文字」欄位是否為「已完成」或「不適用」；若兩者皆非（例如「未開始」「進行中」「待確認」），應先反問使用者是否要先完成這兩項再進行本 skill，而非直接進入切分提案。

如果任一項不符合，請先回報問題並要求使用者補充，不要猜測。

### Step 2：Atomic 粒度原則

Atomic note 的主要粒度是：

```text
一個完整主題概念
```

請依照以下準則判斷是否應該拆分、合併或移動：

1. 單篇原始筆記過長，且包含多個可獨立學習的主題時，應拆成多篇 atomic notes。
2. 單篇原始筆記過短，且內容與其他筆記或段落高度相關時，可以合併到同一篇 atomic note。
3. 原始標題下放錯內容時，應移到語義正確的 atomic note。
4. 同一概念分散在多個原始檔案或多個段落時，可以合併成一篇 atomic note。
5. 不要為了切分而切分；如果一段內容必須依附另一個概念才能理解，應保留在同一篇 atomic note。
6. 不要以單題、單一範例或單一圖片作為預設粒度，除非它本身就是完整主題。
7. 不要使用「補充」「其他」「雜項」這類模糊 atomic 標題。

### Step 3：允許與禁止的重構行為

允許做的事：

1. 拆分過長筆記。
2. 合併過短或高度相關的內容。
3. 移動放錯小節或同章節內位置不合理的段落。
4. 修正標題層級、標題名稱與段落順序。
5. 刪除明顯重複的段落，但需在提案中說明。
6. 輕微修正錯字、格式、清單縮排、段落銜接與 Markdown 結構。
7. 為了銜接被移動的內容，可以加入非常短的過渡句。

疑似屬於其他章節的內容不得自行跨章節處理；請列入「需要人工確認」，並說明建議歸屬章節或處理方式。

不允許做的事：

1. 不要把原始筆記深度重寫成正式教材。
2. 不要自行補充原文沒有的新技術知識。
3. 不要刪除重要概念、程式碼範例、圖片、附件或注意事項。
4. 不要修改、刪除或搬移 `origin/<章節>/*.md` 原始檔。
5. 不要修改、刪除或搬移 `origin/<章節>/assets/` 內的資產檔案。
6. 不要在 `origin/<章節>/notes/` 建立任何內容。
7. 不要修改 `meta/`、`notes/`、`appendix/`、`demos/`、`practice/`、`review/` 或 `supplements/` 中的內容。
8. 不要把 `atomic/<章節>/*.md`、`notes/<章節>/*.md` 或其他目錄內容當成原始資料重新整理。

### Step 4：圖片與附件路徑規則

原始筆記位於 `origin/<章節>/*.md`，本地資產引用通常使用：

```text
./assets/...
```

Atomic note 位於 `atomic/<章節>/*.md`，因此 atomic note 中的本地資產引用必須改成：

```text
../../origin/<章節>/assets/...
```

請遵守以下限制：

1. 只修正真正引用該章節 `assets/` 的 Markdown 圖片、Markdown 連結與 HTML `src` / `href` 路徑。
2. 不要改寫外部網址，例如 `http://`、`https://`。
3. 不要改寫特殊連結，例如 `#anchor`、`mailto:`、`tel:`、`javascript:`、`line:`。
4. 不要改寫空連結，例如 `href=""`。
5. 不要改寫 fenced code block 內的示例路徑。
6. 不要更改資產檔名、hash、目錄分類或實體檔案位置。

### Step 5：長章節處理規則

如果指定章節內容太長，超過單次上下文可安全處理的範圍，請不要直接生成 atomic notes。

長章節仍屬於階段一；在上下文不足以安全產生完整切分提案時，階段一先輸出 inventory，不直接產生完整切分提案或 atomic notes。

請改用三步驟：

1. 先為每個原始 Markdown 建立 inventory，列出檔名、主要標題、疑似主題、資產引用與問題段落。
2. 根據全部 inventory 產生全章切分提案。
3. 等使用者確認後，再分批產生 atomic notes。

### Step 6：判斷方式

處理時請先拆解問題，再輸出結論：

```text
原始檔掃描
→ 主題辨識
→ 重複與混雜內容判斷
→ 拆分、合併、移動、保留、刪重或不納入
→ 資產引用檢查
→ 來源追溯檢查
→ 輸出切分提案
```

輸出時不需要展示冗長的內部推理，只需提供清楚的判斷依據與理由。

### Step 7：兩階段執行

#### 階段一：切分提案或長章節 inventory（預設）

階段一只輸出規劃結果，不建立、修改或刪除任何檔案：

1. 掃描 `origin/<章節>/*.md`。
2. 建立原始檔 inventory。
3. 判斷每個原始檔或段落應該拆分、合併、移動、保留、刪重或不納入 atomic。
4. 規劃預計產生的 `atomic/<章節>/*.md` 檔案。
5. 標記需要人工確認的內容。
6. 標記不納入 atomic 的內容與原因。
7. 輸出切分提案，等待使用者確認。

如果章節太長，依 Step 5 先輸出長章節 inventory，不直接產生完整切分提案。

#### 階段二：產生 atomic notes

只有在使用者明確回覆：

```text
確認，請產生 atomic notes
```

才可以進入階段二。這句確認同時代表使用者已確認切分提案，並允許寫入 `atomic/<章節>/*.md`。

1. 依照確認後的提案建立或更新 `atomic/<章節>/*.md`。
2. 如果執行環境不能直接寫入檔案，請改為輸出每個檔案的完整 Markdown 內容，讓使用者自行建立。
3. 如果目標檔案已存在，即使使用者已回覆階段二確認語句，也不要直接覆蓋；請先列出既有檔案清單，並再次要求使用者確認要覆蓋、更新或保留。

操作欄位定義：

| 操作  | 說明                          |
| --- | --------------------------- |
| 拆分  | 從過長原始筆記拆出獨立主題               |
| 合併  | 將多個過短或高度相關段落合成一篇            |
| 移動  | 同章節內原始位置不合理，移到語義正確的 atomic note |
| 保留  | 原始切分已合理，只做格式與路徑調整           |
| 刪重  | 移除重複內容，保留較完整版本              |
| 不納入 | 不適合成為 atomic note，且需說明原因    |

### Atomic note 檔案格式

每篇 atomic note 必須以 YAML front matter 開頭，記錄來源資訊，再接主題標題與正文：

```md
---
source:
  - '<原始檔路徑與標題或段落位置>'
---

# <主題名稱>

正文內容...
```

`source` 一律使用 YAML list，即使只有一個來源也要用 list 包起來；每個項目固定寫成單引號字串：

```text
'origin/<章節>/<原始檔名>.md / <原始標題或段落位置>'
```

如果內容來自多個原始檔或多個原始標題，在 `source` list 下增加多個項目即可：

```md
---
source:
  - 'origin/<章節>/<原始檔名A>.md / <原始標題A>'
  - 'origin/<章節>/<原始檔名B>.md / <原始標題B>'
---

# <主題名稱>

正文內容...
```

務必使用單引號字串，不要用裸字串或雙引號：來源位置常包含 `#`（例如 `## href 屬性`），裸字串或雙引號中的 ` #` 會被 YAML 解析成註解，導致內容被截斷。

Atomic note 內容要求：

1. 每篇 atomic note 只保留一個主要主題。
2. 標題要直接說明主題，不要使用「補充」「其他」「雜項」這類模糊標題。
3. 不要把「Atomic」當成標題前綴，除非原始主題本身需要。
4. 保留原始筆記中的重要定義、說明、注意事項、程式碼區塊、表格、圖片、附件與 callout。
5. 可以調整標題層級，讓每篇 atomic note 從單一 `#` 標題開始。
6. 同篇 atomic note 內的小節順序應符合學習順序。
7. 如果某段內容不確定是否歸屬於該 atomic note，請先列入提案的「需要人工確認」而不是強行放入。

## Few-Shot 範例

### 範例一：Atomic 切分命名

例如 `210-超鏈接標籤` 章節可以依照概念切分為：

```text
01-a-標籤基本語法.md
02-href-target-屬性.md
03-外部鏈接與內部鏈接.md
04-文件鏈接與-download-屬性.md
05-特殊-href-用法.md
06-網頁元素鏈接.md
07-錨點鏈接.md
```

實際切分仍需根據原始內容決定，不要硬套範例檔名。

### 範例二：資產路徑轉換

原始筆記中的路徑：

```md
![超連結標籤 href 與 target 屬性表](./assets/images/link-tags-img-001-910f05.png)
```

轉成 atomic note 後應改為：

```md
![超連結標籤 href 與 target 屬性表](../../origin/<章節>/assets/images/link-tags-img-001-910f05.png)
```


### 範例四：來源資訊（多個來源）


## Reverse Prompting：資訊不足時請反問

缺少以下資訊時，先反問，不要開始處理：

1. 要處理哪一個章節（`origin/<三位數編號>-<標題>/`）？
2. 是否能讀取 `origin/<章節>/*.md` 與 assets 引用情況？若無法讀取，請使用者提供檔案清單、Markdown 原文與 assets 引用情況。
3. 目前要執行階段一（切分提案）還是階段二（產生 atomic notes）？階段二必須有使用者明確回覆「確認，請產生 atomic notes」才能進行。
4. `meta/chapter-status.md` 中該章節「atomic 切分提案」「atomic 產生」欄位目前狀態為何，使用者是否確認要（重新）處理？
5. 該章節「資產命名」與「alt 與連結文字」欄位是否為「已完成」或「不適用」？若兩者皆非，是否要先完成這兩項再進行本 skill？
6. 若 `atomic/<章節>/*.md` 已存在對應檔案，是否要覆蓋、更新或保留？

## 最終品質標準

### 處理範圍與檔案安全

1. 是否只處理 `origin/<章節>/*.md`。
2. 是否排除 `atomic/<章節>/*.md` 既有輸出，避免把舊 atomic notes 當成原始資料。

### Atomic 切分品質

1. 是否以「一個完整主題概念」作為主要切分粒度。
2. 是否避免為了切分而切分，導致高度相依的內容被拆散。
3. 是否把同章節內明顯放錯位置的內容移到語義正確的 atomic note。
4. 是否將分散在多個原始檔或段落中的同一概念合併到合理位置。
5. 是否避免使用「補充」「其他」「雜項」這類模糊 atomic 標題。
6. 是否讓每篇 atomic note 的標題能直接看出主題。
7. 是否沒有把「Atomic」當成標題前綴。
8. 是否讓同篇 atomic note 內的小節順序符合學習順序。

### 內容保真檢查

1. 是否逐一對照原始筆記的主要標題與段落，確認重要概念沒有遺漏。
2. 是否保留原始筆記中的重要定義、說明、程式碼區塊、表格、圖片、附件、callout 與注意事項。
3. 是否所有刪除、刪重或不納入 atomic 的內容，都已列入「不納入 atomic 的內容」並說明原因。
4. 是否所有合併、拆分、移動與輕微修正，都沒有改變原文技術含義。
5. 是否沒有自行加入原文沒有明確支持的新技術知識。
6. 是否把歸屬不明、內容重疊、可能刪除或語意不確定的段落列入「需要人工確認」。

### 路徑、來源與輸出檢查

1. 是否將 atomic note 中的資產引用改為 `../../origin/<章節>/assets/...`。
2. 是否沒有改寫 fenced code block 內的示例路徑。
3. 是否沒有改寫外部網址、特殊連結或空連結。
4. 是否為每篇 atomic note 保留可追溯的來源資訊。
5. 是否完成後回報處理摘要、建立清單與檢查結果。

### 最終通過標準

一份合格的輸出必須同時符合：

1. **安全**：沒有改動 `origin/` 原始檔與 `assets/` 資產。
2. **可追溯**：每篇 atomic note 都能回查來源。
3. **不失真**：沒有改變原文技術含義，也沒有自行補充原文未支持的新知識。
4. **好維護**：atomic note 以完整主題概念切分，檔名與標題清楚。
5. **可接續**：產出的 atomic notes 能被後續流程轉成正式 `notes/`、`review/`、`practice/` 或 `demos/` 內容。
