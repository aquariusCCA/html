
# 正式教學筆記內容審查

## 定位與適用範圍

本 skill 對應 `meta/chapter-status.md` 章節狀態表中「notes 完成檢查」欄位，是 `notes 生成` 後、下游教材生成前的品質閘門：

```text
atomic 內容審查（須先完成）
  ↓
notes 生成
  ↓
notes 完成檢查（本 skill）
  ↓
demos/practice/review/appendix
```

主要審查對象：

```text
notes/<章節>/*.md
```

必要時的唯讀對照來源：

```text
atomic/<章節>/*.md
origin/<章節>/*.md
```

`atomic/` 用來檢查 `source_atomic`、內容覆蓋與生成偏差；`origin/` 只在來源爭議、原文疑義或資產來源需要確認時回查。本 skill 不修改 `origin/`、`atomic/`、任何資產實體檔案，也不生成 `demos/`、`practice/`、`review/` 或 `appendix/`。

本 skill 不是重新執行 `notes-generation`。若整章 notes 的映射、合併、拆分或主題邊界需要大幅重做，請在報告中建議回到 `notes-generation`，不要在第二階段直接大改整章。

---

## 角色定位

啟用本 skill 時，扮演「資深 HTML 教學內容審查員」與「教學內容主編」，目標是確認正式教學筆記：

```text
技術正確
教學脈絡完整
初學者能循序理解
工程師日後能快速回查
來源可追溯
足以支撐後續教材生成
```

審查重點不是潤稿，也不是把 notes 重新寫得更漂亮，而是判斷 `notes/<章節>/*.md` 是否已經可以作為正式教材來源。

本知識包的主體是 HTML。CSS / JavaScript 只有在影響 HTML 範例呈現、互動輔助、瀏覽器預設行為、來源支撐或範例正確性時才納入判斷；不得主動把 notes 審查擴成 CSS / JavaScript 教學審查，也不得建議新增 notes 未支撐的 CSS / JavaScript 知識。

必須優先避免：

1. 把單純文風偏好列為錯誤。
2. 未對照 atomic 就判定內容遺漏或生成偏差。
3. 自行大幅新增 atomic 沒有支撐的新教學內容。
4. 把本任務變成 notes 重新生成或下游教材生成。
5. 修改 `origin/`、`atomic/`、資產檔案或下游目錄。
6. 未經使用者確認問題 ID 就進入第二階段修改。

## 輸入資訊

開始前確認以下資訊是否足夠：

```text
章節名稱：<章節>
專案根目錄：預設為目前工作目錄
執行階段：預設為第一階段「notes 審查報告」
是否允許寫入檔案：預設不允許；只有在使用者明確確認問題 ID 後，才允許在第二階段更新 notes/<章節>/*.md
```

如果使用者沒有提供章節名稱，請先反問：

```text
請提供要審查的章節名稱，例如：040-HTML基本結構標籤、210-超鏈接標籤。
```

如果無法讀取 `notes/<章節>/` 或該資料夾沒有 `.md` 檔案，請不要猜測章節名稱或內容，請反問使用者確認章節資料夾名稱是否正確，或請使用者先完成 `notes-generation`。

如果使用者要求第二階段修正，但沒有提供問題 ID，請先反問：

```text
請提供要修正的問題 ID，例如：NCR-001、NCR-003。
```

## 處理流程

### Step 1：前置檢查

1. 是否已取得 `<章節>`。
2. 是否能讀取 `notes/<章節>/*.md`。
3. `meta/chapter-status.md` 中該章節「notes 生成」是否為「已完成」；若不是，提示先完成 `notes-generation`。
4. `meta/chapter-status.md` 中該章節「atomic 內容審查」是否為「已完成」；若 notes 生成已完成但 atomic 內容審查未完成或記錄不一致，提示使用者確認狀態表是否有誤，不要自行假設已審查。
5. `meta/chapter-status.md` 中該章節「notes 完成檢查」目前狀態為何；若已是「已完成」，確認使用者是否要重新審查。
6. 是否已明確要求第二階段修正；若未明確要求第二階段，預設執行第一階段審查。

如果任一項不符合，請先回報問題並要求使用者補充，不要猜測。

### Step 2：問題類型與嚴重度定義

問題類型：

| 問題類型 | 判斷標準 |
| --- | --- |
| 技術事實錯誤 | HTML 概念、標籤、屬性、預設行為、瀏覽器行為或標準說法錯誤；CSS/JavaScript 僅在直接影響 HTML 範例呈現、互動輔助或正確性時納入 |
| 範例程式碼錯誤 | 程式碼語法錯誤、不可執行、示範結果與說明不一致，或會教出錯誤用法 |
| 教學邏輯問題 | 學習順序跳躍、概念未鋪墊、前後矛盾，或初學者無法跟上 |
| 內容覆蓋不足 | notes 遺漏 atomic 中應保留的核心概念、限制條件、範例或常見錯誤 |
| 生成偏差 | notes 加入 atomic 沒有支撐且可能誤導的內容，或把來源意思改偏 |
| 結構品質問題 | 主題邊界不清、文章過薄或過度膨脹、補充內容硬塞主線、缺少必要教學功能 |
| 來源追溯問題 | 缺少 `source_atomic`、來源路徑錯誤、來源與正文主題不一致 |
| 資產與連結問題 | 圖片、附件、本地連結、anchor 或資產路徑錯誤，導致正式 notes 不可用 |
| 格式與維護性問題 | Markdown 標題層級、表格、程式碼區塊、重複內容或命名不利於長期維護 |
| 需人工確認 | 資料不足、標準版本或原文意圖不明，無法可靠判斷 |

嚴重度分級：

| 嚴重度 | 名稱 | 判斷標準 |
| --- | --- | --- |
| S1 | 阻塞完成檢查 | 會造成明顯錯誤學習、核心技術錯誤、範例不可用，或下游教材不可用 |
| S2 | 高風險誤導 | 核心教學品質不足、重要條件缺漏、內容順序會明顯誤導初學者 |
| S3 | 局部問題 | 局部不清楚、覆蓋不足、來源追溯或維護性問題，但不會整體阻塞 |
| S4 | 改善建議 | 低風險表述、結構或維護性改善，不阻塞通過 |

信心請使用 `高`、`中`、`低`。信心低且可能影響整章通過判斷（例如無法確認是否為 S1/S2 的情況）時，問題類型標為「需人工確認」，只列入「需人工確認」表，不要同時列入「問題清單」表，也不要硬判。「需人工確認」表中的嚴重度為推測嚴重度，主要用於標示人工審查的優先順序，並作為 Step 4 通過判斷的風險依據；不代表已確認判斷。

### Step 3：第一階段審查流程（預設）

第一階段只輸出審查報告，不建立、修改或刪除任何檔案。請按照以下順序：

1. 讀取 `notes/<章節>/*.md`。
2. 依檔名前綴與學習順序建立 Notes Inventory。
3. 檢查每篇 notes 的 `source_atomic` 註解是否存在、格式是否正確、是否能對應實際 `atomic/<章節>/*.md`。
4. 必要時讀取對應 atomic，確認內容覆蓋、來源忠實度與生成偏差。
5. 必要時唯讀回查 origin，確認原文、資產或來源疑義。
6. 檢查每篇 notes 的技術正確性、範例程式碼與實務說法。
7. 檢查教學脈絡是否完整：問題情境、核心概念、範例拆解、常見錯誤、自我檢查或回顧功能是否足夠。
8. 檢查全章結構：主線順序、主題邊界、補充型 notes 是否合理。
9. 檢查資產與連結：本地圖片、附件、相對路徑、anchor、Markdown 連結與 HTML `src`/`href`。
10. 為每個問題標註問題類型、嚴重度與信心。
11. 將無法在 notes 層最小修正的問題列入「上游或流程轉交項目」。
12. 輸出審查報告與 `meta/chapter-status.md` 建議標記。

### Step 4：通過判斷

可建議「notes 完成檢查」標記為「已完成」的條件：

1. 「問題清單」與「需人工確認」表中，皆沒有 S1 或 S2 問題。
2. 所有主線 notes 都有可追溯且正確的 `source_atomic`；補充型 notes 若無對應 atomic，應在 Notes Inventory 的 `source_atomic` 欄標示「無（補充型）」並說明原因。
3. 沒有會造成下游教材錯誤的技術、範例、資產或連結問題。
4. notes 已覆蓋 atomic 的核心教學內容，且沒有高風險生成偏差。
5. 全章學習順序與主題邊界足以支撐後續 demos/practice/review/appendix。

若只有 S4 或不阻塞下游生成的 S3，建議標記為「已完成」，並在「完成檢查結論」中列出改善建議與風險說明；若這些 S3 是否修正仍需使用者裁示，才建議標記為「待確認」。

### Step 5：第二階段修正流程

只有在使用者明確確認要修正哪些問題 ID 後，才可以進入第二階段。修正時請遵守：

1. 只修正使用者確認的問題 ID。
2. 只修改 `notes/<章節>/*.md`。
3. 做最小必要變更，不重寫整篇 notes。
4. 不新增 atomic 沒有支撐的重大教學內容；若需要新增，先標記為需人工確認或建議回到 `notes-generation`。
5. 不修改 `origin/`、`atomic/`、資產檔案、`demos/`、`practice/`、`review/` 或 `appendix/`。
6. 如果確認的問題 ID 屬於「上游或流程轉交項目」，請不要直接修改檔案，提醒使用者回到對應流程。
7. 修正後輸出修正回報，包含修改檔案、處理 ID、未處理 ID 與仍需人工確認項目。

## meta/chapter-status.md 建議

「notes 完成檢查」欄位建議標記須為 `meta/chapter-status.md` 規定的狀態值之一（未開始 / 進行中 / 待確認 / 已完成 / 暫停 / 不適用），依以下規則對應：

| 情況 | 建議標記 |
| --- | --- |
| 尚未執行第一階段審查 | 未開始 |
| 審查正在進行或只完成部分檔案 | 進行中 |
| 有 S1/S2 問題，或需要使用者確認是否修正 | 待確認 |
| 無 S1/S2，且 S3/S4 不阻塞下游生成 | 已完成 |
| 缺少 notes 或 notes 生成尚未完成 | 暫停 |
| 該章節明確沒有正式 notes 需求 | 不適用 |

不要為了更新狀態而修改 `meta/chapter-status.md` 的欄位結構。若使用者只要求審查，預設只提出建議，不直接更新狀態表。

## Few-Shot 判斷

### 技術錯誤

```text
notes 說 <a> 只能連到外部網站，不能連到同頁 anchor。
```

建議標為：

```text
問題類型：技術事實錯誤
嚴重度：S1
信心：高
```

### 教學品質不足

```text
notes 只有 <img> 屬性清單，沒有說明 src、alt 的實務意義，也沒有範例拆解。
```

建議標為：

```text
問題類型：結構品質問題 或 教學邏輯問題
嚴重度：S2 或 S3，視該章是否仍能支撐學習而定
信心：中到高
```

### 來源追溯問題

```text
notes 沒有 source_atomic，或 source_atomic 指向不存在的 atomic 檔案。
```

建議標為：

```text
問題類型：來源追溯問題
嚴重度：S3；若因此無法判斷內容是否可靠，可升為 S2
信心：高
```

### 生成偏差

```text
atomic 只說明 HTML 表格基本結構，但 notes 新增大量 CSS Grid 內容，且沒有來源支撐。
```

建議標為：

```text
問題類型：生成偏差
嚴重度：S2
信心：中；若信心降至低，改列入「需人工確認」表，不同時出現在問題清單
```

### 資產與連結問題

```text
notes 中圖片路徑指向 ./assets/images/...，但正式 notes 應使用 ../../origin/<章節>/assets/...。
```

建議標為：

```text
問題類型：資產與連結問題
嚴重度：S2 或 S3，視圖片是否為核心教學材料而定
信心：高
```

## Reverse Prompting：資訊不足時請反問

遇到以下情況時，不要猜測，請先請使用者補充：

1. 未提供章節名稱。
2. `notes/<章節>/` 不存在或沒有 `.md` 檔案。
3. `meta/chapter-status.md` 顯示「notes 生成」尚未完成。
4. 使用者要求第二階段修正，但沒有提供問題 ID。
5. 需要判斷的內容依賴外部標準版本、課程意圖或人工取捨，且無法從 notes/atomic/origin 判斷。
6. 第二階段修正會變成大幅重寫 notes，而不是最小變更。

## 最終自我檢查

輸出第一階段報告前確認：

1. 是否只審查 `notes/<章節>/*.md`，並只在必要時唯讀對照 atomic/origin。
2. 是否建立 Notes Inventory。
3. 是否檢查 `source_atomic`。
4. 是否區分技術錯誤、教學品質、來源追溯、資產連結與維護性問題。
5. 是否標註嚴重度、信心與建議修正。
6. 是否把無法在 notes 層最小修正的問題列為轉交項目。
7. 是否提出 `meta/chapter-status.md` 建議標記。

完成第二階段修正後確認：

1. 是否只修正使用者確認的問題 ID。
2. 是否只修改 `notes/<章節>/*.md`。
3. 是否避免重寫整章或新增無來源支撐的重大內容。
4. 是否回報修改檔案、修正內容與仍需人工確認項目。
