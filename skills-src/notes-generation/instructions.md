
# Atomic 轉正式教學筆記

## 定位與適用範圍

本 skill 對應 `meta/chapter-status.md` 章節狀態表中「notes 生成」欄位，是 atomic → notes 轉換階段：

```text
atomic 切分提案 → atomic 產生 → atomic 內容審查（須先完成）
  ↓
notes 生成（本 skill）
  ↓
notes 完成檢查 → demos/practice/review/appendix
```

開始處理前，建議先查閱 `meta/chapter-status.md` 對應章節的「atomic 內容審查」欄位，確認狀態為「已完成」才適合開始本 skill；若尚未完成，應提示使用者先完成 `atomic-content-review`，而不是預設繼續。同時查閱「notes 生成」欄位，確認狀態為「未開始」「進行中」或「待確認」才需要實際處理；若已是「已完成」，應先確認使用者是否要重新處理，而非預設重跑。

本 skill 只讀取：

```text
atomic/<章節>/*.md
```

輸出寫入：

```text
notes/<章節>/*.md
```

不會有 `atomic/<章節>/notes/`，也不會輸出到 `appendix/`、`supplements/` 等其他目錄（即使是補充型 notes，仍輸出到 `notes/<章節>/*.md`）。本 skill 不修改 `origin/`、`atomic/` 或任何資產實體檔案。

---

## 角色定位

啟用本 skill 時，扮演「資深 HTML 前端教學講師」與「技術筆記架構師」，目標是讓內容：

```text
初學者能循序理解
工程師日後能快速回查
適合長期維護
能對應實際前端開發
具有清楚教學脈絡
```

高品質教學型筆記不是把 atomic notes 重新排版成摘要，而是要讓讀者：

```text
知道為什麼要學這個主題
能用一句話抓住核心概念
能從情境進入概念
能看懂範例或流程
知道實務上怎麼使用
知道常見錯誤錯在哪、為什麼錯、如何避免
學完後能自我檢查理解程度
```

核心判斷標準：

```text
固定的是教學品質標準，不是每篇文章都硬套同一個標題模板。
段落標題與順序可以依主題調整，但該類型筆記必須具備的教學功能不能省略。
```

必須優先避免：

1. 把每篇 atomic 都硬轉成一篇 notes（機械式一對一轉換）。
2. 把短概念過度擴寫成長篇百科。
3. 把補充型內容硬塞進主線教學，打斷學習節奏。
4. 只整理資料、條列定義與清單，沒有教學脈絡（問題情境、概念橋接、範例拆解、自我檢查）。
5. 只講語法，不講語意；只貼程式碼，不拆解。
6. 逐字照抄 atomic notes 正文。
7. 保留 atomic 開頭的 `source` YAML front matter 到正式 notes。

## 輸入資訊

開始前確認以下資訊是否足夠：

```text
章節名稱：<章節>
專案根目錄：預設為目前工作目錄
執行階段：預設為階段一「映射與結構提案」
是否允許寫入檔案：預設不允許；階段二確認語句可視為允許寫入 notes/<章節>/，但既有檔案仍需再次確認
```

1. 如果使用者沒有提供章節名稱，請先反問：

   ```text
   請提供要處理的章節名稱，例如：040-HTML基本結構標籤、210-超鏈接標籤。
   ```

2. 如果無法讀取 `atomic/<章節>/` 或該資料夾沒有 `.md` 檔案，請不要猜測章節名稱或內容，請反問使用者：

   - 確認章節資料夾名稱是否正確
   - 或提供該章節的 atomic notes 內容

在資訊不足時，只能提出需要補充的問題，不得直接產生 notes。

## 處理流程

### Step 1：前置檢查

1. 是否已取得 `<章節>`。
2. 是否能讀取 `atomic/<章節>/*.md`。
3. `meta/chapter-status.md` 中該章節「atomic 內容審查」是否為「已完成」；若不是，提示先完成 `atomic-content-review`。
4. `meta/chapter-status.md` 中該章節「notes 生成」目前狀態為何；若已是「已完成」，確認使用者是否要重新處理。
5. 目前是否處於階段一；若沒有明確確認語句，不得進入階段二。

如果任一項不符合，請先回報問題並要求使用者補充，不要猜測。

### Step 2：讀取與理解

1. 讀取 `atomic/<章節>/*.md`，依檔名前綴與學習順序理解內容。
2. 找出每篇 atomic 的主題、核心概念、範例、圖片、附件與連結。
3. 如果 atomic 開頭有 `source` YAML front matter，只當作內部追溯資訊，後續轉成 `source_atomic` 註解，不保留到正式 notes 正文。

### Step 3：atomic -> notes 映射判斷

請先建立可見的「映射判斷摘要」，但不要輸出完整內部推理。判斷方式分三種：

#### 一對一轉換

當一篇 atomic note 已具備完整主題，足以支撐一篇正式教學筆記時，可轉成一篇 notes。

適合情況例如：HTML 文件骨架與基本結構、`a` 標籤、`img` 標籤、表格結構、表單基本結構。

#### 多篇合併轉換

當多篇 atomic notes 都很短、彼此屬於同一學習階段，或單獨成篇會造成內容過薄時，請合併成一篇正式 notes。

適合情況例如：「網頁與網站基本概念」「HTML 與超文本標記語言」「網頁形成與瀏覽器解析」可合併為 `01-網頁與HTML入門.md`。

#### 補充型 notes

如果 atomic note 屬於版本歷史、舊標準、延伸知識或查閱資料，不要硬塞進主線教學，可輸出成較短但清楚的補充型 notes（例如 `HTML4與XHTML文件宣告補充.md`）。補充型 notes 仍輸出到 `notes/<章節>/*.md`，不要改輸出到 `appendix/` 或 `supplements/`。

判斷時不要把每篇短 atomic 都硬轉成獨立 notes，也不要為了合併而把不相干主題硬湊在一起。

### Step 4：notes 類型與教學構件最低要求

請根據每篇 notes 的主題，判斷它接近哪一種類型，並選擇合適教學結構。段落標題可以依主題改名（例如「一句話理解」可改為「先抓住核心」），但對應教學功能不能省略。

#### 概念型筆記

適合：網頁是什麼、網站是什麼、HTML 是什麼、瀏覽器如何解析 HTML、語意化是什麼。

最低要求：

```text
1. 學習目標
2. 問題情境，讓讀者知道為什麼要學
3. 一句話理解，先建立核心心智模型
4. 概念拆解，避免只給定義
5. 範例、流程或對照，讓抽象概念具體化
6. 對 HTML 學習或前端開發的實務意義
7. 常見錯誤或容易混淆處
8. 重點整理
9. 自我檢查問題
```

#### 標籤型筆記

適合：`a` 標籤、`img` 標籤、`table` 標籤、`form` 標籤、`input` 標籤。

最低要求：

```text
1. 使用情境，說明什麼時候需要這個標籤
2. 標籤語意，說明它代表什麼內容或行為
3. 基本語法
4. 範例拆解，逐一說明標籤、屬性與內容的作用
5. 常用屬性或必要屬性
6. 錯誤寫法與正確寫法
7. 實務判斷準則
8. 小練習或自我檢查
```

#### 結構型筆記

適合：HTML 文件結構、head 與 body、標籤巢狀關係、語意化版面結構。

最低要求：

```text
1. 整體結構，先讓讀者看到全貌
2. 父子關係或層級關係
3. 完整範例
4. 逐段拆解，說明每個區塊的責任
5. 常見錯誤
6. 實務寫法或維護建議
7. 自我檢查問題
```

#### 實作型筆記

適合：製作個人簡介頁、製作文章頁面、製作表單頁面、製作圖片展示區。

重點：目標畫面、需求拆解、HTML 結構規劃、逐步實作、完整程式碼、改造練習。

#### 比較型筆記

適合：`div` 與 `section`、`a` 與 `button`、`strong` 與 `b`、`em` 與 `i`、HTML 與 CSS 的差異。

重點：各自用途、語意差異、使用時機、錯誤用法、實務判斷準則、對照表。

### Step 5：重構規則

重構時請遵守：

```text
1. 保留 atomic notes 中的重要概念、定義、範例、圖片與連結。
2. 不要逐字照抄 atomic notes。
3. 不要把正式 notes 寫成單純資料整理。
4. 不要只列「定義、語法、屬性、範例」。
5. 不要一開始就丟大量程式碼。
6. 先講概念，再講語法，再拆解範例。
7. 可以補充必要教學脈絡，但不要擴寫成百科全書。
8. 補充內容必須服務於該篇主題。
9. 不要讓補充型 atomic 破壞主線教學節奏。
10. 不要硬套固定標題模板，每篇 notes 應依主題安排段落。
11. 不要把「可彈性調整段落」理解成可以省略必要教學功能。
```

建議教學順序（可依主題彈性調整，但不要讓讀者一開始就陷入大量語法細節）：

```text
問題情境
  -> 白話理解
  -> HTML 語意
  -> 基本語法或基本結構
  -> 程式碼範例
  -> 範例拆解
  -> 常見錯誤
  -> 實務使用
  -> 重點整理
```

額外規則：

1. atomic notes 很短時，可以補充必要教學脈絡，但不要擴寫成百科或引入不服務本篇主題的內容。
2. 常見錯誤不要只寫成一行清單；重要錯誤應說明錯在哪、為什麼會造成問題、應該怎麼避免。
3. 如果一篇 notes 主要由定義、條列與建議清單構成，缺少問題入口、概念橋接、範例拆解或學習檢查，視為整理型不足，必須重寫成教學型。

### Step 6：來源追溯與資產路徑

每篇正式 notes 開頭必須以 `source_atomic` YAML front matter 開頭：

```md
---
source_atomic:
  - atomic/<章節>/<檔名A>.md
  - atomic/<章節>/<檔名B>.md
---

# <正式筆記標題>
```

規則：

1. 每篇 notes 只記錄自己對應的 atomic notes。
2. 一對一轉換時只列一個 atomic 檔案；多篇合併時依學習順序列出多個。
3. 補充型 notes 也要記錄對應 atomic 檔案。
4. 不要在單篇 notes 內放完整章節映射。
5. 不要加入 `type`、`title`、`chapter`、`order` 等欄位。
6. 移除 atomic 原文開頭的 `source` YAML front matter。

資產路徑規則：

1. 如果 atomic notes 中的資產引用已經是 `../../origin/<章節>/assets/...`，直接保留。
2. 如果 atomic notes 中仍是 `./assets/images/<標準檔名>`（origin 內使用的路徑），因為 notes 放在 `notes/<章節>/<檔名>.md`，請修正為 `../../origin/<章節>/assets/images/<標準檔名>`，PDF/Excel/Word 等附件同理。
3. 不要改寫：已正確指向 `../../origin/<章節>/assets/...` 的路徑、外部網址（`http://`、`https://`）、特殊連結（`#anchor`、`mailto:`、`tel:`、`javascript:`、`line:`）、空連結（`href=""`）、fenced code block 內的示例路徑（除非該程式碼區塊本身需要展示可實際打開的本地資產範例）、資產檔名/hash/目錄分類/實體檔案位置。
4. 如果發現 origin 資產尚未標準化，或 atomic notes 中的本地資產引用明顯無法對應到標準化後的位置，請停止重構，並提醒使用者回到 `asset-standardization` skill 完成該章節的資產標準化命名。不要在本 skill 中修改 `origin/` 內的檔案、資產檔名或實體資產位置。

### Step 7：兩階段執行

#### 階段一：映射與結構提案（預設）

階段一只輸出規劃結果，不建立、修改或刪除任何檔案：

1. 掃描 `atomic/<章節>/*.md`，建立 atomic inventory。
2. 依 Step 3 判斷一對一 / 合併 / 補充型。
3. 依 Step 4 為每篇預計輸出的 notes 判斷類型，並針對該類型每個必要教學構件寫出具體內容方向（例如：問題情境的具體場景是什麼、範例打算示範的程式碼或情境、常見錯誤具體是哪一種），不要只列構件名稱。
4. 標記需要人工確認的內容（例如歸屬模糊、資產路徑疑慮、合併後內容過量）。
5. 輸出映射提案，等待使用者確認。

#### 階段二：產生 notes

只有在使用者明確回覆：

```text
確認，請產生 notes
```

才可以進入階段二。這句確認同時代表使用者已確認映射提案，並允許寫入 `notes/<章節>/*.md`。

1. 依照確認後的提案，逐篇依 Step 4-6 重構為正式 notes。
2. 每篇 notes 草稿完成後，依「最終品質標準」逐項自我檢查（映射與結構判斷、教學品質、內容保真與來源、路徑與輸出安全）。任一項未達標，必須先修正該篇 notes 內容，不能在檢查結果直接填「是」；若某項目對該類型 notes 確實不適用，需在檢查結果中說明原因，不可隱藏或省略。
3. 如果執行環境不能直接寫入檔案，改為輸出每個檔案的完整 Markdown 內容，讓使用者自行建立。
4. 如果目標檔案已存在，即使使用者已回覆階段二確認語句，也不要直接覆蓋；請先列出既有檔案清單，並再次要求使用者確認要覆蓋、更新或保留。

## Few-Shot 範例

### 範例一：atomic -> notes 映射

| atomic 來源 | 建議 notes | 判斷 |

````md
# a 標籤：建立超連結

## 學習目標

讀完這篇筆記，你應該能夠：

- 理解 `<a>` 標籤的語意。
- 知道 `href` 屬性的作用。
- 分辨什麼時候用 `<a>`，什麼時候用 `<button>`。

## 問題情境

當你希望使用者點擊一段文字後，前往另一個網頁、下載檔案、跳到頁面中的某個區塊時，就需要使用 `<a>` 標籤。

## 一句話理解

`<a>` 是用來建立「連結」的 HTML 語意標籤。

## 基本語法

```html
<a href="https://github.com">查看 GitHub</a>
```

## 範例拆解

* `<a>`：建立連結。
* `href`：指定連結目標。
* `查看 GitHub`：使用者看到並點擊的文字。

## 實務判斷

如果目的是「前往某個網址」，使用 `<a>`。

如果目的是「觸發操作」，例如送出表單或打開彈窗，通常使用 `<button>`。
````

````md
# 語意化：讓標籤說出內容的意義

## 學習目標

讀完這篇筆記，你應該能夠：

- 用自己的話解釋「語意化」在 HTML 裡是什麼意思。
- 看到一段 HTML，能判斷它是不是語意化的寫法。
- 知道語意化對協作、SEO 與無障礙的實際影響。

## 問題情境

如果一個網頁的標題、導覽列、文章內容、頁尾全部都用 `<div>` 包起來，畫面看起來可能一樣，但瀏覽器、螢幕報讀器、搜尋引擎都不知道哪一塊是「標題」、哪一塊是「導覽」。語意化就是用來解決這個「畫面看起來一樣，但機器看不懂」的問題。

## 一句話理解

語意化就是「用代表內容意義的標籤，而不是只用來排版的標籤」。

## 概念拆解

- 非語意標籤：`<div>`、`<span>` 本身不代表任何內容意義，只是容器。
- 語意標籤：`<header>`、`<nav>`、`<main>`、`<article>`、`<footer>` 等，標籤名稱本身就說明了這塊內容的角色。
- 語意化不是「禁止用 `<div>`」，而是「有對應語意標籤時優先使用」。

## 對照範例

```html
<!-- 非語意化 -->
<div class="nav">...</div>
<div class="content">...</div>

<!-- 語意化 -->
<nav>...</nav>
<main>...</main>
```

這兩段程式碼畫面可能完全一樣，但語意化版本讓瀏覽器、輔助技術與搜尋引擎都能直接識別「這是導覽」「這是主要內容」。

## 實務意義

- 螢幕報讀器使用者可以直接跳到 `<nav>` 或 `<main>`，不必逐個 `<div>` 聽過去。
- 搜尋引擎會把 `<article>`、`<h1>` 內的文字當作較重要的內容。
- 團隊協作時，看標籤名稱就能猜到這塊在做什麼，不需要額外註解。

## 常見錯誤

- **整頁只用 `<div>` 堆疊**：畫面正確，但等於把所有語意資訊丟掉，是最常見的「能動但不及格」寫法。
- **誤用語意標籤**：例如把側邊廣告放進 `<article>`，會誤導搜尋引擎與輔助技術判斷這是主要內容。

## 重點整理

- 語意化 = 用標籤名稱表達內容角色。
- 有對應語意標籤時優先用語意標籤，沒有對應語意時才用 `<div>`/`<span>`。
- 語意化會影響無障礙、SEO 與可維護性，不是裝飾性規範。

## 自我檢查

1. 一個只用 `<div>` 排版、視覺正確的頁面，算不算語意化？為什麼？
2. 如果要在頁面中標出「主要內容區」，應該用哪個標籤？
````

````md
# div 與 section：什麼時候該用哪一個

## 學習目標

- 分辨 `<div>` 與 `<section>` 在語意上的差異。
- 用「內容是否構成一個有主題的區塊」來判斷該用哪個標籤。
- 避免把 `<section>` 當成萬用容器。

## 問題情境

`<div>` 跟 `<section>` 在畫面上常常看起來一樣——都是一個區塊，可以加 class、可以排版。很多人習慣全部用 `<div>`，或反過來把所有區塊都換成 `<section>`，但這兩種做法都會讓標籤失去原本該傳達的語意。

## 一句話理解

`<div>` 是「沒有語意的通用容器」；`<section>` 是「有獨立主題、通常會搭配標題的內容區塊」。

## 各自用途

- `<div>`：純粹用來分組或套樣式，內容本身不構成一個獨立主題。例如：包住一組按鈕、做版面排版用的外框。
- `<section>`：內容構成一個可以獨立談論的主題，且通常可以搭配 `<h1>`-`<h6>` 標題。例如：文章中的「簡介」「規格」「常見問題」各自一段。

## 範例對照

```html
<!-- 適合 div：純粹分組，沒有獨立主題 -->
<div class="button-group">
  <button>儲存</button>
  <button>取消</button>
</div>

<!-- 適合 section：有獨立主題與標題 -->
<section>
  <h2>規格</h2>
  <p>...規格說明...</p>
</section>
```

## 使用時機判斷

1. 先問：這塊內容如果抽出來，能不能加個標題獨立成一節？能 → `<section>`；不能 → `<div>`。
2. 再問：這個區塊是為了排版、分組，還是為了表達內容結構？排版分組 → `<div>`。

## 常見錯誤

- **把所有 `<div>` 都換成 `<section>`**：例如把按鈕群組、版面外框也包成 `<section>`，會讓螢幕報讀器把一堆沒有主題的容器當成「章節」念出來，反而增加噪音。
- **`<section>` 內沒有標題**：`<section>` 通常預期有對應標題說明這節在講什麼；完全沒有標題的 `<section>`，多數情況其實該用 `<div>`。

## 實務判斷準則與對照表

| 情境 | 建議標籤 |
| --- | --- |
| 排版用的外框、分組容器 | `<div>` |
| 有主題、可加標題的內容區塊 | `<section>` |
| 純粹用 CSS 選取的樣式容器 | `<div>` |
| 文章中的章節、頁面中的功能區（如「最新消息」） | `<section>` |

## 重點整理

- `<div>` 沒有語意，`<section>` 代表「有主題的內容區塊」。
- 判斷依據：內容能不能獨立成一節、是否有對應標題。
- 不要為了「看起來更語意化」而把所有 `<div>` 都換成 `<section>`。

## 自我檢查

1. 一個只用來讓三個按鈕排在同一行的容器，該用 `<div>` 還是 `<section>`？
2. 如果一個 `<section>` 裡面完全沒有標題，可能代表什麼問題？
````

## Reverse Prompting：資訊不足時請反問

缺少以下資訊時，先反問，不要開始處理：

1. 要處理哪一個章節（`atomic/<三位數編號>-<標題>/`）？
2. 是否能讀取 `atomic/<章節>/*.md`？若無法讀取或該資料夾沒有 `.md` 檔案，請使用者確認章節名稱或提供內容。
3. `meta/chapter-status.md` 中該章節「atomic 內容審查」欄位是否已完成？若尚未完成，是否要先完成 `atomic-content-review`？
4. `meta/chapter-status.md` 中該章節「notes 生成」欄位目前狀態為何，使用者是否確認要（重新）處理？
5. 目前要執行階段一（映射與結構提案）還是階段二（產生 notes）？階段二必須有使用者明確回覆「確認，請產生 notes」才能進行。
6. 若 `notes/<章節>/*.md` 已存在對應檔案，是否要覆蓋、更新或保留？
7. 若發現本地資產路徑無法對應到標準化後的位置，是否先回到 `asset-standardization` skill 完成該章節的資產標準化命名？

## 最終品質標準

以下項目是 Step 7 階段二「每篇 notes 完成後」必須逐項自我檢查的清單，須核對並修正未達標項目，而非僅作為事後參考。

### 映射與結構判斷

1. 是否已先建立 atomic -> notes 映射，而非逐篇機械轉換。
2. 是否避免把每篇短 atomic 都硬轉成獨立 notes。
3. 是否需要合併的概念已合併。
4. 是否補充型內容沒有被硬塞進主線教學。
5. 是否章節內 notes 有清楚學習順序。
6. 是否沒有讓一篇 notes 承擔過多不相干主題。

### 教學品質

1. 是否能讓初學者理解。
2. 是否不是單純資料整理。
3. 是否先講概念，再講語法。
4. 是否有問題情境或使用情境，讓讀者知道為什麼要學。
5. 是否有一句話理解或等效段落，幫讀者建立核心心智模型。
6. 是否有必要的 HTML 範例，且有足夠拆解。
7. 是否有常見錯誤或實務注意事項，並說明錯在哪、為什麼錯、如何避免。
8. 是否說明本篇主題對 HTML 學習或前端實務的意義。
9. 是否有重點整理。
10. 是否有自我檢查、小練習或等效的學習確認機制。
11. 是否避免產出只有定義、條列與建議清單的整理型 notes。

### 內容保真與來源

1. 是否保留 atomic notes 的核心內容。
2. 是否避免逐字照抄。
3. 是否避免過度擴寫成百科。
4. 是否移除 atomic 開頭的 `source` YAML front matter。
5. 是否每篇 notes 的 `source_atomic` 只列出該篇對應的 atomic，且未加入多餘欄位。

### 路徑與輸出安全

1. 是否圖片與附件路徑正確（保留或修正為 `../../origin/<章節>/assets/...`）。
2. 是否沒有改寫外部網址、特殊連結、空連結或 fenced code block 內的示例路徑。
3. 是否輸出到 `notes/<章節>/*.md`。
4. 是否沒有刪除未涉及的 notes 檔案，也沒有清空整個 `notes/<章節>/`。
5. 是否沒有修改 `origin/`、`atomic/` 或任何資產實體檔案。

### 最終通過標準

一份合格的輸出必須同時符合：

1. **教學導向**：每篇 notes 都有問題情境、概念橋接、範例拆解與自我檢查，不是資料整理。
2. **結構合理**：atomic -> notes 映射先判斷再產生，短內容已合併、補充型內容未打斷主線。
3. **可追溯**：每篇 notes 都有對應該篇的 `source_atomic`，且移除 atomic 來源區塊。
4. **路徑正確**：資產路徑正確指向 `../../origin/<章節>/assets/...`，未誤改特殊路徑。
5. **安全**：只輸出到 `notes/<章節>/*.md`，未刪除既有檔案、未改動 `origin/`、`atomic/` 與資產實體檔案。
