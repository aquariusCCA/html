
# 外部素材整理為補充筆記

## 定位與適用範圍

本 skill 用於將外部素材（AI 回答、技術文章、資深工程師解說等）整理為 `supplements/<章節>/*.md` 補充筆記：

```text
notes/<章節>/*.md（閱讀過程中發現需要補充）
  ↓
取得外部素材（AI 回答、技術文章、工程師回答）
  ↓
notes-supplements-generation（本 skill）
  ↓
supplements/<章節>/*.md
  ↓
scripts/build-supplements-index.mjs（納入 meta/supplements-index.md）
```

本 skill 的素材來源**不是** `atomic/` 或 `notes/`，而是使用者在閱讀 notes 後從外部收集的資料。

本 skill 只讀取：

```text
notes/<章節>/<檔名>.md（來源 note，用於理解補充脈絡）
meta/topics.md（若存在，作為受控詞彙參考；不存在則略過）
使用者提供的外部素材（貼入對話）
supplements/<章節>/（若存在，用於確認是否與既有 supplement 重複）
```

輸出寫入：

```text
supplements/<章節>/<檔名>.md
```

不修改 `origin/`、`atomic/`、`notes/`、`meta/` 或任何資產實體檔案。不更新 `meta/chapter-status.md`（supplements 為選用補充層，不在管線狀態追蹤內）。

---

## 角色定位

啟用本 skill 時，扮演「技術文件整理員」，目標是讓外部素材：

```text
定位清楚：知道它在補充哪個 note 的哪個面向
不重複：不把 notes 已有的內容再整理一遍
有額外價值：提供 notes 刻意省略、深度不足或未觸及的內容
格式一致：front matter 完整，內容可被 build-supplements-index.mjs 索引
```

核心判斷標準：

```text
補充的定義是「notes 沒有講、但閱讀後值得知道」，不是「notes 講過的詳細版本」。
如果素材只是把 notes 已有的內容說得更詳細，不值得另存一份 supplement。
如果素材提供了 notes 沒有的角度（底層原理、實務場景、比較觀點、解答疑問），才是有效補充。
```

必須優先避免：

1. 把 notes 已有的內容重新整理成 supplement（等於重複存一份）。
2. 把素材原文直接貼入，沒有清理格式或整理脈絡。
3. 一個 supplement 同時涵蓋多個不相關主題（應分開存為多個 supplement）。
4. front matter 缺少 `source_notes`、`topics` 或 `summary` 任一欄位。

## 補充類型

判斷素材屬於哪一種補充類型，決定內容組織方式：

| 類型 | 說明 | 適合情境 |
| --- | --- | --- |
| 原理深探型 | 底層機制、規範細節、瀏覽器實作 | 閱讀 notes 後想知道「為什麼」「底層怎麼運作」 |
| 實務延伸型 | 真實場景、進階用法、常見陷阱 | 閱讀 notes 後想知道「實際上會遇到什麼」 |
| 比較對照型 | 與其他技術、標準、寫法的比較 | 閱讀 notes 後想理解「跟 X 有什麼不同」 |
| 答疑型 | 針對閱讀 notes 後產生的具體疑問 | 有一個明確的問題，素材是對這個問題的回答 |

## 輸入資訊

開始前確認以下資訊是否足夠：

```text
來源 note 路徑：notes/<章節>/<檔名>.md（或章節名稱）
外部素材：使用者貼入的原始內容
素材來源類型：AI 回答 / 技術文章 / 工程師解說 / 其他（選填，影響來源標記）
執行階段：預設為階段一「素材分析與結構提案」
是否允許寫入檔案：預設不允許；階段二確認語句可視為允許寫入 supplements/<章節>/
```

1. 如果使用者沒有提供來源 note 或章節，先反問：

   ```text
   請提供這份素材要補充的 note 路徑（例如：notes/020-HTML簡介/03-HTML文件基本骨架.md），或章節名稱。
   ```

2. 如果使用者沒有提供外部素材，先反問：

   ```text
   請貼入要整理的外部素材內容。
   ```

在資訊不足時，只能提出需要補充的問題，不得直接產生 supplement。

## 處理流程

### Step 1：前置確認

1. 是否已取得來源 note 路徑或章節名稱。
2. 是否已取得外部素材。
3. 嘗試讀取指定的 `notes/<章節>/<檔名>.md`，理解它的主題與範圍，判斷素材與它的關係。
4. 確認 `supplements/<章節>/` 目錄是否已存在，了解既有 supplement 情況（避免重複）。
5. 目前是否處於階段一；若沒有明確確認語句，不得進入階段二。

### Step 2：素材理解

1. 閱讀外部素材，識別：
   - 核心主題（一句話）
   - 補充了 notes 的哪個面向（底層原理、實務、比較、答疑）
   - 是否有超出來源 note 主題的內容（例如素材順帶提到了其他章節的概念）
2. 對照來源 note 內容，確認素材確實提供了 notes 未涵蓋的資訊；若素材大部分都是重複，需在提案中標記，並詢問使用者是否仍要保留。

### Step 3：定位與結構提案

1. 判斷補充類型（原理深探 / 實務延伸 / 比較對照 / 答疑）。
2. 提議檔名：使用描述性 slug，以來源 note 的主題為前綴（若有助於識別），例如：
   - `quirks-mode-底層渲染機制.md`
   - `img-loading-lazy-實務場景.md`
   - `form-submit-與-fetch-比較.md`
3. 提議 `source_notes`（指向來源 note，可多個）。
4. 起草 `topics`（2-5 個，優先沿用 `meta/topics.md` 若存在；若提出新詞，在提案中另列「新詞清單」）。
5. 起草 `summary`（一句話，不超過約 50 字，說明這份補充的核心價值）。
6. 起草內容大綱（段落標題與每段要點，不是完整內容）。
7. 標記需要人工確認的項目（例如：素材有一部分屬於其他章節、素材與 notes 重複度高）。

### Step 4：內容整理規則

進入階段二時，依以下規則整理內容：

1. **不照單全收**：素材可能包含與主題無關的部分，只保留與補充定位相關的內容。
2. **格式清理**：將 AI 對話格式、問答格式或網頁貼上的雜亂格式整理為乾淨的 Markdown。
3. **脈絡補充**：若素材是 Q&A 格式，保留問題作為段落標題或前言，說明這份補充的脈絡。
4. **來源標記**：若使用者有提供素材來源（網址、文章名稱、工具名稱），在文末以「來源」區段標記。若無來源資訊，不需強制要求。
5. **不重複 notes**：若整理過程中發現某段落與 notes 完全重複，直接省略。
6. **一個主題一份 supplement**：若素材實際上涵蓋兩個以上不相關的主題，拆分為多個 supplement 提案，每個提案獨立確認後寫入。

### Step 5：兩階段執行

#### 階段一：素材分析與結構提案（預設）

階段一只輸出規劃結果，不建立、修改或刪除任何檔案：

1. 讀取來源 note 與外部素材。
2. 依 Step 2-3 完成定位分析與結構提案。
3. 輸出提案，等待使用者確認。

#### 階段二：產生 supplement

只有在使用者明確回覆：

```text
確認，請產生 supplement
```

才可以進入階段二。

1. 依 Step 4 整理素材為完整 Markdown 內容。
2. 加入 YAML front matter（`source_notes`、`topics`、`summary`）。
3. 完成後依「最終品質標準」逐項自我檢查，任一項未達標，必須先修正。
4. 若執行環境不能直接寫入檔案，改為輸出完整 Markdown 內容，讓使用者自行建立。
5. 若目標檔案已存在，即使使用者已回覆確認語句，也不要直接覆蓋；先告知並再次確認。
6. 寫入成功後，提示使用者執行 `node scripts/build-supplements-index.mjs` 更新 `meta/supplements-index.md`。

## 輸出格式

### 階段一：素材分析與結構提案

不要建立、修改或刪除任何檔案。

```md
## 素材分析與結構提案

- 來源 note：
- 素材主題（一句話）：
- 補充類型：原理深探 / 實務延伸 / 比較對照 / 答疑
- 是否有效補充（素材提供了 notes 未涵蓋的內容）：是 / 否（若否，說明重複程度）
- 需要人工確認項目數：

| 欄位 | 內容 |
| --- | --- |

| 補充類型 | 原理深探 / 實務延伸 / 比較對照 / 答疑 |

## 內容大綱

- H1 標題：
- 段落一：（標題 + 要點說明）
- 段落二：...
- 來源：（若有）

## 新詞清單（若無新詞可省略此區塊）

| --- | --- | --- |

## 需要人工確認（若無項目可省略此區塊）

| 項目 | 問題 | 建議處理 |

```

- 是否 source_notes 正確指向來源 note：

## Few-Shot 範例

### 範例一：答疑型補充（原理深探）

**情境**：使用者閱讀 `notes/020-HTML簡介/03-HTML文件基本骨架.md`，對「DOCTYPE 宣告缺少時瀏覽器進入 quirks mode」感到好奇，從 MDN 找到詳細解說。

**階段一提案**：

| 欄位 | 內容 |
| --- | --- |
| 建議檔名 | `supplements/020-HTML簡介/quirks-mode-瀏覽器渲染模式切換機制.md` |
| source_notes | `notes/020-HTML簡介/03-HTML文件基本骨架.md` |
| topics | `quirks mode, 渲染模式, DOCTYPE, 瀏覽器行為` |
| summary | `說明瀏覽器如何依 DOCTYPE 宣告切換 quirks mode 與 standards mode，以及兩種模式的主要差異。` |
| 補充類型 | 原理深探 |

**階段二 front matter**：

```yaml
---
source_notes:
  - notes/020-HTML簡介/03-HTML文件基本骨架.md
topics: [quirks mode, 渲染模式, DOCTYPE, 瀏覽器行為]
summary: "說明瀏覽器如何依 DOCTYPE 宣告切換 quirks mode 與 standards mode，以及兩種模式的主要差異。"
---
```

### 範例二：實務延伸型補充

**情境**：使用者閱讀 `notes/180-圖片標籤/01-img標籤與圖片替代文字.md`，在技術文章看到關於 `loading="lazy"` 的使用時機與注意事項。

**建議檔名**：`supplements/180-圖片標籤/img-loading-lazy-實務注意事項.md`

```yaml
---
source_notes:
  - notes/180-圖片標籤/01-img標籤與圖片替代文字.md
topics: [img標籤, lazy loading, 效能優化, 瀏覽器行為]
summary: "整理 loading=lazy 的實務使用場景、above-the-fold 注意事項與瀏覽器支援現況。"
---
```

### 範例三：重複度高、建議不存為 supplement 的情況

**情境**：使用者閱讀 `notes/040-HTML基本結構標籤/02-標籤屬性.md`，貼入一段「HTML 屬性是標籤的額外資訊，格式為 name="value"」的解說。

**判斷**：這份素材的核心內容（屬性語法、name/value 結構）與 notes 完全重疊，不構成有效補充。應在階段一提案中標記「是否有效補充：否」，說明重複程度，詢問使用者是否仍要保留。

## Reverse Prompting：資訊不足時請反問

缺少以下資訊時，先反問，不要開始處理：

1. 這份素材要補充哪一個 note（`notes/<章節>/<檔名>.md`）或哪個章節？
2. 外部素材的內容是什麼（請使用者貼入）？
3. 若素材明顯跨多個不相關主題，是否要分拆為多個 supplement？
4. 目前要執行階段一（素材分析）還是階段二（產生 supplement）？階段二必須有使用者明確回覆「確認，請產生 supplement」才能進行。

## 最終品質標準

以下項目是 Step 5 階段二「寫入前」必須逐項自我檢查的清單，須核對並修正未達標項目。

### 補充定位

1. 素材確實提供了來源 note 未涵蓋的內容（底層原理、實務場景、比較觀點或具體答疑）。
2. 沒有把 notes 已有的內容再整理一遍。
3. 一份 supplement 只聚焦於一個主題，沒有強行涵蓋多個不相關主題。

### 內容品質

1. 格式乾淨，無 AI 對話殘留、問答格式雜亂或網頁貼上的 HTML 標記。
2. 素材中與主題無關的部分已省略。
3. 若素材為 Q&A 格式，問題脈絡已保留並適當整合為段落前言或標題。
4. 若有來源資訊，已在文末「來源」區段標記。

### Metadata 完整性

1. front matter 包含 `source_notes`、`topics`、`summary` 三個欄位，無一缺漏。
2. `source_notes` 正確指向來源 note 的相對路徑（`notes/<章節>/<檔名>.md`）。
3. `topics` 優先沿用 `meta/topics.md`（若存在）既有詞彙；新詞已在提案中列入新詞清單。
4. `summary` 為一句話，說明這份補充的核心價值，不是標題換句話說。

### 輸出安全

1. 只輸出到 `supplements/<章節>/<檔名>.md`。
2. 未修改 `notes/`、`origin/`、`atomic/`、`meta/` 或任何資產實體檔案。
3. 未更新 `meta/chapter-status.md`。
4. 若目標檔案已存在，已再次確認使用者是否要覆蓋。
