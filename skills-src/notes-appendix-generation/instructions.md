
# Notes 轉 Appendix 查表資料

## 定位與適用範圍

本 skill 對應 `meta/chapter-status.md` 章節狀態表中「appendix 索引」欄位，是 notes 完成檢查後的下游教材生成階段。appendix 與 demos、practice、review 都各自平行延伸 notes，彼此沒有先後依賴；本 skill 只負責 appendix：

```text
notes 生成 → notes 完成檢查（須先完成）
  ↓
（皆平行延伸 notes，彼此無先後）
appendix 索引（本 skill）
demos 生成
practice 生成
review 生成
```

appendix 是**選用層**，不是每章節必經。只有當該章節有足夠可表格化的查表內容（名詞表、屬性值表、標籤對照表、設定檔範例、速查清單）時才生成；多數內容以敘述為主的章節應整章標記為「不適用」。

開始處理前，應先查閱 `meta/chapter-status.md` 對應章節的「notes 完成檢查」欄位，確認狀態為「已完成」才適合開始本 skill；若尚未完成，應提示使用者先完成 `notes-content-review`，不要預設繼續生成 appendix。

本 skill 主要讀取：

```text
notes/<章節>/*.md
meta/chapter-status.md
```

只輸出或更新：

```text
appendix/<章節>/*.md
```

本 skill 不修改：

```text
origin/
atomic/
notes/
demos/
practice/
review/
supplements/
任何原始資產實體檔案
```

本 skill 不修改 `meta/chapter-status.md`，僅在輸出中提出「appendix 索引」欄位的建議標記，由使用者自行決定是否更新狀態表。

appendix 嚴格對齊章節：每章的 appendix 只放在 `appendix/<章節>/`，章節資料夾名稱（編號+標題）與 `notes/` 一致，不建立跨章節的彙整總表。若某張表天生跨章節（如全站標籤總表），拆進最相關的單一章節，不另開全域目錄。

---

## 角色定位

啟用本 skill 時，扮演「查表資料整理員」與「技術文件審查員」，目標是把正式 notes 中已經出現、適合以表格或清單快速查閱的內容，整理成可獨立查閱的附錄資料。

合格的 appendix 不是把 notes 內容重述一遍，也不是補一份完整的標準參考表，而是要讓讀者：

```text
已經學過該主題後，只為了查某個值或對照而打開
單看一張表、不需要前後文也能用
能從表格回連到對應 notes 取得完整脈絡
看到的每一格資料都來自 notes、可被回溯
```

### 核心紅線（最高優先，違反即不合格）

appendix 的唯一內容來源是 `notes/<章節>/*.md`。**只能把 notes 已經寫出的查表內容重新排成表，一格都不得超出 notes。**

* 不得自行補全 notes 沒有寫出的值。例如 notes 只示範了 3 種 `input type`，appendix 不得擴充成完整的 22 種；只能列 notes 出現過的那些。
* 不得引入 notes 沒有提到的屬性、API、參數、預設值或瀏覽器支援度。
* 規劃或生成時若發現「想列的表，notes 沒寫滿」，不要補，改列入「需要人工確認」，並建議使用者回頭補 notes、或將該表（或整章）標記為不適用。
* 寧可表格不完整，也不得脫離 notes 成為獨立參考資料。

### 其他必須避免

1. 每篇 notes 都機械式生成一張表，導致大量沒有查表價值的附錄。
2. 把整篇 notes 的敘述搬進 appendix；appendix 只放表格、清單或設定檔範例，不放教學敘述。
3. 為純概念、歷史、流程型內容硬做表格。
4. 在表格中加入 notes 沒有支撐的「補充欄位」。
5. 修改 `notes/`、`origin/`、`atomic/` 或其他下游目錄。
6. 建立跨章節彙整總表，或在章節資料夾外輸出。

## 輸入資訊

開始前確認以下資訊是否足夠：

```text
章節名稱：<章節>
專案根目錄：預設為目前工作目錄
執行階段：預設為階段一「Appendix 生成規劃」
是否允許寫入檔案：預設不允許；階段二確認語句可視為允許寫入 appendix/<章節>/
```

如果使用者沒有提供章節名稱，請先反問：

```text
請提供要處理的章節名稱，例如：240-表單標籤、260-字符實體。
```

如果無法讀取 `notes/<章節>/` 或該資料夾沒有 `.md` 檔案，請不要猜測章節名稱或內容，請反問使用者確認章節資料夾名稱是否正確，或請使用者先完成 `notes-generation` 與 `notes-content-review`。

在資訊不足時，只能提出需要補充的問題，不得直接產生 appendix。

## 章節適用性判斷（規劃前先做這步）

appendix 為選用層，第一步不是「切幾張表」，而是判斷「這個章節到底值不值得有 appendix」。

判斷準則：**拿掉前後文、單看一張表仍然可用，且該內容在 notes 中本來就以可表格化的形式出現，才適合生成 appendix。** 否則整章標記為「不適用」。

**主判斷是質性的**：「獨立成 appendix 的查表便利性，是否明顯高於讀者直接回頭翻 notes？」若差異不明顯，傾向不生成。項目數量只是輔助信號——即使有 6、7 個值，若拆出來沒有明顯更好查，也不做。

適合生成（範例）：

* `260-字符實體`：實體名稱 / 數字編碼 / 顯示字元對照
* `050-全局屬性`：屬性 / 值 / 適用範圍 / 範例
* `240-表單標籤`：`input type` 對照、表單元素屬性表
* `080-meta標籤`：常用 meta 名稱 / 用途 / 範例
* `210-超鏈接標籤`：`target` 值、`rel` 值對照

通常不適合（整章標記不適用，範例）：

* `010-寫在前面`、`020-HTML簡介`：以背景與概念敘述為主。
* `060-路徑`、`090-title`、`100-標題`、`140-換行`、`150-水平線` 等單標籤章節：可查表的內容太少，沒有獨立查表價值。

判斷時，若整章只有零星一兩個值可列、湊不成有查表價值的表，傾向標記不適用，不要硬生。

## Appendix 粒度判斷

本 skill 不採固定的一篇 notes 對一張表。請依查表價值選擇以下方式：

| 生成方式 | 適用情況 | 輸出 |
| --- | --- | --- |
| 一對一 | 單篇 notes 含一組明確、可獨立查閱的查表內容 | 一篇 notes 產生一個 appendix 檔 |
| 多對一 | 同一章節內多篇 notes 的查表內容屬於同一主題、合併成一張表更好查 | 多篇 notes 產生一個整合 appendix 檔 |
| 不生成 | 該篇 notes 無可表格化內容，或表格只能重述敘述 | 列入「不生成 Appendix 的 Notes」 |

判斷標準：

1. 若表格只能把 notes 的句子拆行排列、沒有「可對照欄位」，通常不生成。
2. 若同一章節內多篇 notes 講同一類值（例如分散在數篇的表單屬性），合併成一張表更利於查閱，採多對一；跨章節不合併。
3. 每個 appendix 檔要有一個清楚的查表主題，不要把整章不同主題塞進一個大雜燴檔。

## Appendix 內容類型

請依 notes 主題選擇合適類型。類型可混合，但每個 appendix 檔的主類型要明確。

### 名詞 / 術語表

適合：概念密集、出現多個需要對照定義的術語。
欄位範例：術語 | 定義（取自 notes） | 備註。

### 屬性 / 值對照表

適合：全局屬性、標籤屬性、`input type`、`target`/`rel` 值等。
欄位範例：屬性或值 | 用途 | 範例 | 備註。

### 標籤 / 元素對照表

適合：同類標籤分類、語意標籤用途、塊級/行內分類等。
欄位範例：標籤 | 用途 | 範例 | 備註。

### 設定檔 / 程式碼範例速查

適合：notes 中出現、值得快速複製的標準片段（如常用 meta、表單骨架）。
要求：片段必須原樣來自 notes，不得自行擴充功能。

### 速查清單

適合：notes 中以條列出現、適合快速核對的規則或步驟。
要求：清單項目逐條對應 notes，不新增 notes 沒有的項目。

所有類型共同要求：每一欄、每一格的內容都必須能在對應 notes 找到出處；找不到出處的內容不得寫入，列入「需要人工確認」。

## 檔案與路徑規則

### 輸出目錄與檔名

每個 appendix 檔輸出到：

```text
appendix/<章節>/<NN-描述>.md
```

命名規則：

1. 沿用 notes 既有檔名慣例，採 `NN-描述.md`（兩位數序號 + 描述；描述以中文為主，`input`、`type`、`meta` 等技術名詞可保留英文原文），例如 `01-input-type-對照表.md`。
2. 一對一：以對應 notes 檔名（移除 `.md`、移除序號可選）為基礎加上表格主題。
3. 多對一：用能代表共同主題的描述，例如 `01-表單元素屬性表.md`。
4. 同一章節內不得產生重複檔名。
5. 不使用空白、路徑分隔符、冒號、問號等不適合檔名的字元。

### Front matter

每個 appendix 檔開頭加入 YAML front matter，沿用 notes 既有欄位慣例，但來源欄改為 `source_notes`：

```markdown
---
source_notes:
  - notes/240-表單標籤/02-input基礎與文字密碼輸入.md
topics: [input, type, 表單控制項]
summary: "input type 各值的用途與範例速查（僅含 notes 出現過的值）。"
---
```

* `source_notes`：列出本表所有來源 notes 路徑；多對一時列出全部。
* `topics`：2–5 個，盡量沿用對應 notes 既有 topics 用詞，避免同義詞分裂。
* `summary`：一句話，說明這張表查什麼，並可註明資料範圍以 notes 為準。

### 正文結構

front matter 之後：

1. 一個 `#` 標題，標明這是什麼查表。
2. 緊接一句「查表用 + 回連 notes」的引言，強化 appendix 不解釋脈絡的定位，例如：

   ```markdown
   > 查表用。概念與用法說明見 notes/240-表單標籤/02-input基礎與文字密碼輸入.md。
   ```

3. 接著只放表格、清單或程式碼範例，不放教學敘述段落。

範例：

```markdown
---
source_notes:
  - notes/240-表單標籤/02-input基礎與文字密碼輸入.md
topics: [input, type, 表單控制項]
summary: "input type 各值的用途與範例速查（僅含 notes 出現過的值）。"
---

# input type 對照表

> 查表用。概念與用法說明見 notes/240-表單標籤/02-input基礎與文字密碼輸入.md。

| type | 用途 | 範例 | 備註 |
| --- | --- | --- | --- |
| text | 單行文字輸入 | `<input type="text">` | 預設值 |
| password | 密碼輸入，內容遮蔽 | `<input type="password">` | |
```

## 處理流程

本 skill 分兩階段執行：階段一（Appendix 生成規劃，預設）與階段二（產生 appendix）。階段一只輸出規劃、不動任何檔案；只有取得明確確認語句後才進入階段二。

### 階段一：Appendix 生成規劃（預設）

階段一只輸出規劃結果，不建立、修改或刪除任何檔案，最後依「輸出格式」的階段一範本輸出規劃，等待使用者確認。依序執行下列三步。

#### 步驟 1：前置檢查

1. 是否已取得 `<章節>`。
2. 是否能讀取 `notes/<章節>/*.md`。
3. `meta/chapter-status.md` 中該章節「notes 完成檢查」是否為「已完成」；若不是，提示先完成 `notes-content-review`。
4. `meta/chapter-status.md` 中該章節「appendix 索引」目前狀態為何；若已是「已完成」，確認使用者是否要重新處理。
5. 目前是否處於階段一；若沒有明確確認語句，不得進入階段二。

如果任一項不符合，請先回報問題並要求使用者補充，不要猜測。

#### 步驟 2：讀取與理解

1. 讀取 `notes/<章節>/*.md`，依檔名前綴與學習順序理解內容。
2. 建立 Notes Inventory，記錄每篇 notes 的標題、topics/summary、是否含可表格化內容（既有表格、屬性清單、值列舉、條列規則、可複製片段）。
3. 標出每處查表素材在 notes 中的確切位置，作為後續回溯依據。

#### 步驟 3：Appendix 規劃

先做章節適用性判斷；若判定整章不適用，直接於階段一輸出「整章不適用」結論與理由，不再規劃表格。

若判定適合，對每個候選 appendix 規劃：

1. 輸出檔案路徑與檔名。
2. 對應 notes（一對一或多對一）。
3. 內容類型。
4. 表格欄位設計或清單結構。
5. 資料來源回溯：每欄資料來自哪篇 notes 的哪段。
6. 唯一來源缺口：想列但 notes 未涵蓋的內容，列入「需要人工確認」。

「表格欄位設計」必須具體到看得出每欄是什麼、資料從哪來；不能只寫「做一張屬性表」。

### 階段二：產生 appendix

只有在使用者明確回覆：

```text
確認，請產生 appendix
```

才可以進入階段二。這句確認同時代表使用者已確認規劃，並允許寫入 `appendix/<章節>/`。

階段二流程：

1. 依確認後的規劃建立或更新 appendix 檔。
2. 如果目標檔案已存在，不得直接覆蓋；先列出既有項目並要求使用者確認覆蓋、更新或保留。
3. 產生含 front matter 的查表 Markdown。
4. 每個檔案完成後執行靜態檢查。
5. 檢查不通過時先修正，再輸出完成回報。

## 驗證規則

生成後必須驗證，不得省略。appendix 產出為 Markdown 查表資料，沒有瀏覽器驗證；以靜態檢查與來源回溯為主。

### 靜態檢查（必須）

每個 appendix 檔至少確認：

1. `appendix/<章節>/<檔名>.md` 存在且非空。
2. 具備合法 YAML front matter，含 `source_notes`、`topics`、`summary` 三欄。
3. `source_notes` 中每個路徑都指向實際存在的 notes 檔案。
4. Markdown 表格語法正確：表頭、分隔列（`---`）、各列欄數一致，不缺漏管線符號。
5. 正文只含表格、清單或程式碼範例與簡短引言，沒有大段教學敘述。
6. 沒有修改 `notes/`、`origin/`、`atomic/` 或其他下游目錄。
7. 輸出位置在 `appendix/<章節>/`，未建立跨章節彙整檔。

### 來源回溯檢查（必須，最高優先）

逐表、逐欄確認：

1. 表格中每一格資料都能在對應 notes 找到出處。
2. 沒有出現 notes 未涵蓋的值、屬性、API、參數或支援度資訊。
3. 凡是 notes 未涵蓋、卻為了表格完整性想加入的內容，已列入「需要人工確認」而非直接寫入。

若發現任何一格無法回溯到 notes，必須移除該內容或改列「需要人工確認」，不得保留在成品中。

## 輸出格式

### 階段一：Appendix 生成規劃

不要建立、修改或刪除任何檔案。

若步驟 3 判定整章不適用，**使用以下簡短格式直接結束階段一**，不輸出完整規劃表格：

```md
## Appendix 生成規劃

- 章節：
- 章節適用性判斷：建議整章不適用
- 原因：（一句話說明，例如：內容以概念敘述為主，無可獨立查閱的可表格化素材）
- 建議：meta/chapter-status.md「appendix 索引」標記為「不適用」。
```

```md
## Appendix 生成規劃

- 章節：
- 掃描 notes 數：

- 不生成 appendix 的 notes 數：
- 需要人工確認項目數：

| appendix/<章節>/01-主題.md | notes/<章節>/02-主題.md | 一對一 | 屬性值對照表 | type \| 用途 \| 範例 \| 備註 | 取自 02-主題.md 的 input 範例段 |
```

## Few-Shot 判斷

### 範例一：一對一屬性值對照表

```text
notes/240-表單標籤/02-input基礎與文字密碼輸入.md
```

若該 notes 示範了數種 `input type` 與其行為，建議：

```text
輸出檔案：appendix/240-表單標籤/01-input-type-對照表.md
生成方式：一對一
內容類型：屬性值對照表
欄位設計：type | 用途 | 範例 | 備註
來源回溯：每列只列 notes 出現過的 type；notes 未提到的 type 不補。
```

### 範例二：多對一整合表

```text
notes/240-表單標籤/03-單選複選與下拉選項.md
notes/240-表單標籤/04-表單按鈕與提交行為.md
```

若多篇 notes 的表單元素屬性分散，合併更好查，建議：

```text
輸出檔案：appendix/240-表單標籤/02-表單元素屬性表.md
生成方式：多對一
內容類型：屬性值對照表
欄位設計：元素/屬性 | 用途 | 範例 | 備註
來源回溯：逐列標注來自哪篇 notes；跨篇沒有的屬性不補。
```

### 範例三：整章不適用

```text
notes/010-寫在前面/*.md
```

內容以 Web 背景與概念敘述為主，建議：

```text
章節適用性判斷：建議整章不適用
原因：內容以概念與背景敘述為主，沒有可獨立查閱、可表格化的查表素材。
建議：meta/chapter-status.md「appendix 索引」標記為「不適用」。
```

### 範例四：唯一來源缺口（列入人工確認）

```text
notes/050-全局屬性/01-id-class-style-元素識別分類與樣式.md
```

若想做全局屬性速查表，但 notes 只深入講了部分屬性：

```text
規劃：appendix/050-全局屬性/01-全局屬性速查表.md
需要人工確認：notes 未涵蓋的全局屬性是否要納入。
建議處理：只列 notes 已說明的屬性；其餘標注「notes 未涵蓋」，由使用者決定補 notes 或維持不列。
```

## Reverse Prompting：資訊不足時請反問

遇到以下情況時，不要猜測，請先請使用者補充：

1. 未提供章節名稱。
2. `notes/<章節>/` 不存在或沒有 `.md` 檔案。
3. `meta/chapter-status.md` 顯示「notes 完成檢查」尚未完成。
4. 使用者要求第二階段產生 appendix，但尚未明確回覆「確認，請產生 appendix」。
5. 目標 appendix 檔案已存在，且使用者尚未確認覆蓋、更新或保留。
6. 想列的表格內容在 notes 中找不到出處，無法判斷是否該納入。

## 最終品質標準

階段二完成後，必須逐項檢查：

1. 是否只以 `notes/<章節>/*.md` 為內容來源，且每一格都可回溯到 notes。
2. 是否未補全 notes 未涵蓋的值、屬性、API 或支援度（核心紅線）。
3. 是否先做章節適用性判斷，整章無查表價值時正確標記不適用。
4. 是否先建立規劃，而非機械式逐篇生成。
5. 是否正確判斷一對一、多對一與不生成。
6. 是否每個 appendix 檔都有清楚查表主題與正確 front matter（source_notes/topics/summary）。
7. 是否正文只含表格、清單或程式碼範例與簡短引言，未搬入教學敘述。
8. 是否所有 `source_notes` 路徑存在、表格語法正確。
9. 是否輸出在 `appendix/<章節>/`，未建立跨章節彙整檔。
10. 是否未修改 `origin/`、`atomic/`、`notes/` 或其他下游目錄。
11. 是否將所有唯一來源缺口列入「需要人工確認」。
12. 是否提出 `meta/chapter-status.md` 中「appendix 索引」欄位的建議標記。

可建議「appendix 索引」標記為「已完成」的條件：

1. 規劃中應生成的 appendix 都已建立或更新，或整章判定不適用且理由充分。
2. 不生成 appendix 的 notes 都有合理原因。
3. 沒有未解決、會讓表格無法查閱或無法回溯到 notes 的問題。
4. 靜態檢查與來源回溯檢查皆通過。
