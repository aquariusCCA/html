
# 章節最終驗收

## 定位與適用範圍

本 skill 對應 `meta/chapter-status.md` 章節狀態表中「最終驗收」欄位，是整章所有處理階段完成後、章節標記為「已完成」前的**整合閘門（release gate）**：

```text
origin 整理 / 資產命名 / alt 與連結文字
  ↓
atomic 切分提案 / atomic 產生 / atomic 內容審查
  ↓
notes 生成 / notes 完成檢查 / notes 索引元資料
  ↓
demos 生成 / practice 生成 / review 生成 / appendix 索引
  ↓
最終驗收（本 skill）
```

本 skill **不重做各層內部品質審查**。`atomic 內容審查`、`notes 完成檢查` 等欄位已分別由 `atomic-content-review`、`notes-content-review` 等 skill 擔任單層閘門。最終驗收只把整章當成**一份完整交付物**，驗證它整體是否成立、各階段是否真的都過關、跨層是否一致。

主要驗收對象：

```text
meta/chapter-status.md                 該章節整列
meta/chapter-logs/<章節>.md            該章節 log
notes/<章節>/*.md
demos/<章節>/**
practice/<章節>/**
review/<章節>/**
appendix/<章節>/*.md
atomic/<章節>/*.md
origin/<章節>/**
meta/index.md                          聚合索引
```

唯讀對照時，`atomic/`、`origin/` 只在來源追溯鏈或資產來源需要確認時回查。

**核心紅線：本 skill 全程唯讀於教材內容，不修改 `origin/`、`atomic/`、`notes/`、`demos/`、`practice/`、`review/`、`appendix/` 或任何資產實體檔案。** 發現問題一律列為「轉交項目」退回對應層的 skill / 流程處理，不在本任務裡就地修教材。第二階段只允許回填狀態表與章節 log 兩處「狀態記錄」。

---

## 角色定位

啟用本 skill 時，扮演「交付驗收主編」與「發佈品管」，目標是判斷該章節：

```text
所有前置處理階段都已正確完成或有正當理由不適用
每個適用層都有預期產出，且實際可用
跨層來源追溯鏈完整、無斷鏈
連結、資產、anchor、demo 都可解析、可開啟
索引與狀態記錄彼此一致
整章端到端對學習者完整可用
```

必須優先避免：

1. 把本任務變成各層內容的重新審查或重寫。
2. 對教材內容做任何修改（含「順手」修連結、修 front matter、修錯字）。
3. 未對照實際檔案就判定某層「已完成」或「缺漏」。
4. 把單層 skill 應負責的內部品質問題，當成最終驗收自己要修的事。
5. 未經使用者人工核可，就把「最終驗收」標為「已完成」。
6. 為了讓章節看起來完成，而忽略或淡化阻塞問題。

## 輸入資訊

開始前確認以下資訊是否足夠：

```text
章節名稱：<章節>
專案根目錄：預設為目前工作目錄
執行階段：預設為第一階段「最終驗收報告」
是否允許寫入檔案：預設不允許；只有在使用者明確完成人工核可後，才允許在第二階段回填
                  meta/chapter-status.md 該列與 meta/chapter-logs/<章節>.md
```

如果使用者沒有提供章節名稱，請先反問：

```text
請提供要驗收的章節名稱，例如：020-HTML簡介、210-超鏈接標籤。
```

如果無法讀取該章節的對應資料夾，或 `meta/chapter-status.md` 找不到該章節列，請不要猜測，請反問使用者確認章節名稱是否正確。

如果使用者要求進入第二階段回填，但尚未針對第一階段報告做出明確核可，請先反問：

```text
最終驗收需要人工核可後才能回填狀態。請確認是否核可本章通過最終驗收，或指出仍需處理的轉交項目 ID。
```

## 處理流程

### Step 1：前置檢查

1. 是否已取得 `<章節>`。
2. `meta/chapter-status.md` 是否存在該章節列，並讀取整列各欄狀態。
3. `meta/chapter-logs/<章節>.md` 是否存在並可讀取。
4. 「最終驗收」欄目前狀態為何；若已是「已完成」，確認使用者是否要重新驗收。
5. 是否已完成人工核可、要進入第二階段回填；若未核可，預設執行第一階段。

如果任一項不符合，先回報並要求補充，不要猜測。

### Step 2：問題類型與嚴重度定義

問題類型（聚焦整合層，不重列單層內部品質類型）：

| 問題類型 | 判斷標準 |
| --- | --- |
| 前置閘門未完成 | 上游欄位仍為「未開始/進行中/待確認」，或「不適用」缺乏正當理由 |
| 產出缺漏 | 某個非「不適用」的層缺少預期產出檔，或 log 無對應完成記錄 |
| 追溯鏈斷裂 | `source_atomic`/`source_notes` 指向不存在的檔，或來源指標缺漏 |
| 連結與資產失效 | 跨層相對路徑、asset 路徑、anchor 無法解析，或 demo 無法開啟執行 |
| 索引與狀態不一致 | `build-index.mjs --check` 失敗、`meta/index.md` 與 front matter 不一致，或狀態表列與 chapter-log 互相矛盾 |
| Schema 不合規 | 各層 front matter 不符合定案 schema（欄位缺漏或多餘） |
| 全章連貫性問題 | 整章端到端缺口、「不適用」判定不合理、學習路徑明顯不完整 |
| 需人工確認 | 資料不足或牽涉人工取捨，無法可靠判斷 |

嚴重度分級（沿用既有 S1-S4 語彙）：

| 嚴重度 | 名稱 | 判斷標準 |
| --- | --- | --- |
| S1 | 阻塞驗收 | 前置閘門未過、核心產出缺漏、追溯鏈斷裂或索引校驗失敗，章節不可發佈 |
| S2 | 高風險 | 重要連結/資產/demo 失效、狀態嚴重不一致，明顯影響交付可用性 |
| S3 | 局部問題 | 局部不一致、非核心連結問題或 schema 小瑕疵，不整體阻塞 |
| S4 | 改善建議 | 低風險記錄、維護性建議，不阻塞通過 |

信心請使用 `高`、`中`、`低`。信心低且可能影響整章通過判斷時，問題類型標為「需人工確認」，只列入「需人工確認」表，不要同時列入「轉交項目」表，也不要硬判。

### Step 3：第一階段驗收流程（預設）

第一階段只輸出驗收報告，不建立、修改或刪除任何檔案。請按以下順序：

本 skill 所稱「上游 13 欄」固定為：`origin 整理`、`資產命名`、`alt 與連結文字`、`atomic 切分提案`、`atomic 產生`、`atomic 內容審查`、`notes 生成`、`notes 完成檢查`、`notes 索引元資料`、`demos 生成`、`practice 生成`、`review 生成`、`appendix 索引`。`最終驗收` 是本 skill 自身欄位，不列入上游 13 欄。

1. **前置閘門完整性**：逐欄檢查該章上游 13 欄是否皆為「已完成」或有正當理由的「不適用」；列出任何殘留「未開始/進行中/待確認」或無理由的「不適用」。
2. **產出存在性**：對每個非「不適用」的層，確認 `notes/`、`demos/`、`practice/`、`review/`、`appendix/` 等實際有預期產出檔，且 `meta/chapter-logs/<章節>.md` 有對應完成記錄。
3. **追溯鏈完整**：抽查 `notes` 的 `source_atomic`、`demos/practice/review/appendix` 的 `source_notes`（或等義來源欄）是否都能解析到實體檔；列出斷鏈與缺漏。
4. **連結與資產完整**：檢查跨層相對路徑、asset 路徑、anchor 是否可解析；可行時實際開啟 demos 確認可執行。
5. **索引與狀態一致**：執行（或請使用者執行）`scripts/build-index.mjs --check`；確認 `meta/index.md` 與 notes front matter 一致；比對狀態表該列與 chapter-log「狀態摘要」是否自洽。
6. **Schema 合規**：抽查各層 front matter 是否符合定案 schema（例如 notes 的 `topics`/`summary`、review/appendix 的來源欄），列出缺漏或多餘欄位。
7. **全章連貫性（質性）**：通讀章節脈絡，判斷整章是否端到端完整可用、各「不適用」是否合理；此項偏人工，信心不足時列入「需人工確認」。
8. 為每個已確認問題標註問題類型、嚴重度與信心，並列為「轉交項目」，指明應回到哪個層 / 哪支 skill 處理；問題類型為「需人工確認」者只列入「需人工確認」表，不列入「轉交項目」表。
9. 輸出驗收報告、通過/不通過裁決，與 `meta/chapter-status.md`、`meta/chapter-logs/<章節>.md` 的建議標記（僅建議，不在第一階段寫入）。

### Step 4：通過判斷（Definition of Done）

**第一階段「建議通過」須滿足條件 1–8（可在第一階段客觀驗證）：**

1. 上游 13 欄全為「已完成」或有正當理由的「不適用」，無 待確認/進行中/未開始。
2. 每個適用層都有預期產出檔，且 chapter-log 有對應完成記錄。
3. 所有 `source_atomic`/`source_notes` 可解析到實體檔，無斷鏈。
4. `scripts/build-index.mjs --check` 通過，`meta/index.md` 與 front matter 一致。
5. 各層 front matter 符合定案 schema。
6. 跨層連結、anchor、資產路徑可解析，demos 可開啟執行。
7. 「轉交項目」與「需人工確認」表中皆無 S1 或 S2。
8. 狀態表該列可收尾為自洽狀態：`完成率 → 100%`、`整體狀態 → 已完成`、`下一步` 更新。

**進入第二階段前的額外前置條件（須人工確認，非第一階段判斷依據）：**

9. 面向全章連貫性的人工核可已取得。

若只有不阻塞的 S3/S4，可在報告中建議通過並列出改善建議與風險；若這些 S3 是否處理仍需使用者裁示，則建議標記為「待確認」。任何 S1/S2 未結，一律不可通過。

### Step 5：第二階段回填流程

只有在使用者完成人工核可後，才可進入第二階段。回填時請遵守：

1. 只回填兩處：`meta/chapter-status.md` 該章節列、`meta/chapter-logs/<章節>.md`。
2. **不修改任何教材檔案**（`origin/`、`atomic/`、`notes/`、`demos/`、`practice/`、`review/`、`appendix/`、資產）。
3. `meta/chapter-status.md`：將「最終驗收」設為核可後的狀態，並同步更新 `完成率`、`整體狀態`、`下一步`，使整列自洽。
4. `meta/chapter-logs/<章節>.md`：在「已執行流程」追加一筆驗收記錄（含日期、依本 skill 完成最終驗收、核可結論），並更新「待確認與下一步」。
5. 若核可結論是「附帶轉交項目」，不要在本 skill 修教材，提醒使用者回到對應流程。狀態標記依使用者核可意向區分：若使用者明確接受 S3/S4 為已知風險、不另行追蹤，標為「已完成」；若使用者要求仍須追蹤這些轉交項目，標為「待確認」。
6. 回填後輸出回報，包含更新的檔案、最終狀態與仍未結的轉交項目。

## Few-Shot 判斷

### 前置閘門未完成

```text
某章 demos 生成欄為「進行中」，卻要求最終驗收。
```

建議標為：

```text
問題類型：前置閘門未完成
嚴重度：S1
信心：高
轉交：回到 notes-demo-generation 完成 demos，再重啟最終驗收
```

### 追溯鏈斷裂

```text
appendix 某檔 source_notes 指向已不存在的 notes 檔名（notes 後來重命名）。
```

建議標為：

```text
問題類型：追溯鏈斷裂
嚴重度：S1
信心：高
轉交：回到 notes-appendix-generation 修正來源欄
```

### 索引與狀態不一致

```text
build-index.mjs --check 失敗，meta/index.md 與某 notes 的 topics 不一致。
```

建議標為：

```text
問題類型：索引與狀態不一致
嚴重度：S2
信心：高
轉交：回到 notes-index-metadata / 聚合腳本重建索引
```

### 不阻塞的維護性建議

```text
某 review 檔 summary 句尾用詞可更精準，但不影響可用性與追溯。
```

建議標為：

```text
問題類型：全章連貫性問題（輕微）
嚴重度：S4
信心：中
轉交：列為改善建議，不阻塞通過
```

## Reverse Prompting：資訊不足時請反問

遇到以下情況時，不要猜測，請先請使用者補充：

1. 未提供章節名稱。
2. `meta/chapter-status.md` 找不到該章節列，或對應資料夾不存在。
3. 上游欄位狀態與實際檔案明顯矛盾，無法判斷以哪個為準。
4. 需要判斷的全章連貫性依賴課程意圖或人工取捨，無法從現有檔案判定。
5. 使用者要求第二階段回填，但尚未對第一階段報告做出明確核可。

## 最終自我檢查

輸出第一階段報告前確認：

1. 是否全程唯讀於教材內容，未修改任何 origin/atomic/notes/demos/practice/review/appendix 或資產。
2. 是否逐項完成七大驗收面向，並以實際檔案 / 指令輸出為依據。
3. 是否依固定欄位清單確認上游 13 欄的完成情況。
