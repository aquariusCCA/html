
# 本地資產 Alt 與連結文字整理

## 定位與適用範圍

本 skill 對應 `meta/chapter-status.md` 章節狀態表中「alt 與連結文字」欄位，在「origin 整理」「資產命名」欄位完成後進行：

```text
origin/<章節>/*.md
origin/<章節>/assets/
  ↓
資產命名（asset-standardization，另一概念，必須先完成）
  ↓
alt 與連結文字（本 skill）← 必須在 atomic 切分提案 之前完成
  ↓
atomic 切分提案 → atomic 產生 → ...
```

`CLAUDE.md` 規定 `origin/` 唯讀，不可修改、覆蓋、刪除。本 skill 是該規則的明確例外之一，但範圍嚴格限定於：

* 就地更新 `origin/<章節>/*.md` 中本地圖片的 Markdown alt 與 HTML `alt` 屬性。
* 就地更新 `origin/<章節>/*.md` 中本地附件的 Markdown / HTML 連結顯示文字。

除此之外，不得修改 `origin/` 下任何其他內容（資產路徑、實體檔名、教材正文、標題、結構等）。

本 skill 不是「資產標準化命名」（屬於 `asset-standardization` skill，且應先完成），也不是 atomic/notes 等下游生成流程；若使用者要求的是資產改名、路徑整理或下游內容處理，應提示改用其他流程。

開始處理前，建議先查閱 `meta/chapter-status.md` 對應章節的「alt 與連結文字」欄位，確認狀態為「未開始」「進行中」或「待確認」才需要實際處理；若已是「已完成」，應先確認使用者是否要重新檢查，而非預設重跑。同時建議確認該章節「資產命名」欄位已完成，避免在資產路徑尚未標準化前處理 alt/連結文字。

---

## 角色定位

啟用本 skill 時，扮演：

1. 安全的文字描述整理器。
2. 保守的 Markdown / HTML 語法檢查者。
3. 本地圖片 alt 與本地附件連結文字的品質審查者。

不扮演：

1. 教材正文重寫器。
2. 資產檔案整理器。
3. 檔名、路徑或目錄重命名工具。
4. 內容猜測器。

必須優先避免：

1. 把檔名、hash、路徑或副檔名直接當成 alt 或連結文字。
2. 在無法確認圖片或附件用途時猜測描述。
3. 改寫資產路徑、實體檔名。
4. 改寫教材正文、標題、清單、表格內容（除非該文字本身就是 alt 或連結文字）。
5. 破壞 Markdown / HTML 語法。
6. 修改 fenced code block 或行內程式碼中的範例，但 Step 2 所述例外情況（fenced code block 內可唯一對應到 `assets/` 實體檔案的本地資產引用）不在此限。
7. 誤改外部連結、錨點連結、特殊協議連結或空連結。

保守性優先於完整性。錯誤 alt 比空 alt 更糟。不確定時不要猜測。

## 輸入資訊

開始前確認以下資訊是否足夠：

```text
目標章節：origin/<三位數編號>-<章節標題>/
掃描範圍：origin/<章節>/*.md（全部，或指定檔案）
資產根目錄：origin/<章節>/assets/
執行模式：dry-run / apply
```

1. 若使用者沒有指定章節，不要自行掃描整個 `origin/`，先反問使用者。
2. 若使用者沒有指定執行模式，預設使用 `dry-run`，只輸出建議，不修改檔案。

若以上資訊不足，先提出問題，不要直接執行。

## 處理流程

### Step 1：前置檢查

確認：

1. 是否已指定目標章節（`origin/<三位數編號>-<章節標題>/`）。
2. 能讀取該章節目錄，以及章節根目錄下的 `*.md` 文件（章節筆記只放在根目錄，不會有 `origin/<章節>/notes/`）。
3. 能讀取對應的 `assets/` 目錄。
4. 執行模式已指定：`dry-run`（預設）或 `apply`。

任一項無法確認，先反問使用者，不要自行猜測。

### Step 2：掃描範圍與排除規則

只掃描：

```text
origin/<章節>/*.md
```

不掃描：

1. `origin/` 全部章節。
2. 非章節根目錄下的封存檔案。
3. `assets/` 內的實體資產檔案本身。

掃描時必須略過：

1. fenced code block 內的 Markdown 或 HTML 範例。
2. 行內程式碼中的 Markdown 或 HTML 範例。
3. 外部網址，例如 `http://`、`https://`。
4. 特殊連結，例如 `#anchor`、`mailto:`、`tel:`、`javascript:`、`line:`。
5. 空連結，例如 `href=""`。
6. 教材正文、標題、清單、表格內容，除非該文字本身就是圖片 alt 或附件連結文字。

例外：若 fenced code block 內本地圖片 `src` 或本地附件 `href`／連結路徑，依 [`asset-standardization` skill](../asset-standardization/SKILL.md) Step 3、Step 4 的匹配規則可唯一對應到 `origin/<章節>/assets/` 內的實體檔案，則不視為第 1 項所述的教學範例，其圖片 alt 或附件連結文字仍須依本流程的信心等級規則檢查與判斷；無法唯一對應或仍有疑義時，維持第 1 項略過或列入「需要人工確認」。行內程式碼（第 2 項）不適用此例外，一律略過。

### Step 3：找出需要檢查的目標

| 類型 | 範例 | 處理目標 |
| --- | --- | --- |
| Markdown 圖片 alt | `![圖片](./assets/images/a.png)` | 改成有意義的 alt，或在裝飾圖時留空 |
| HTML 圖片 `alt` | `<img src="./assets/images/a.png" alt="圖片">` | 改成有意義的 `alt`，或在裝飾圖時設為 `alt=""` |
| HTML 圖片缺少 `alt` | `<img src="./assets/images/a.png">` | 在能確認用途時補上 `alt` |
| Markdown 本地附件連結 | `[下載](./assets/pdfs/a.pdf)` | 改成可理解的附件連結文字 |
| HTML 本地附件連結 | `<a href="./assets/word/a.docx">下載</a>` | 改成可理解的附件連結文字 |

本地附件包含：

| 類型 | 副檔名 |
| --- | --- |
| PDF | `.pdf` |
| Excel | `.xls`, `.xlsx`, `.csv` |
| Word 文件 | `.doc`, `.docx`, `.odt`, `.rtf` |
| 音訊 | `.mp3`, `.wav`, `.ogg`, `.m4a` |
| 影片 | `.mp4`, `.webm`, `.mov`, `.avi` |
| 壓縮檔 | `.zip`, `.rar`, `.7z`, `.tar`, `.gz` |
| 程式碼檔案 | `.html`, `.css`, `.js`, `.ts`, `.vue`, `.java`, `.sql`, `.json`, `.xml`, `.yml`, `.yaml` |
| 其他本地輔助檔案 | 其他可確認存在於章節 `assets/` 內的檔案 |

### Step 4：判斷品質

以下圖片 alt 可視為低品質，應檢查是否需要改寫：

```text
圖片
圖
截圖
image
image.png
圖片.png
螢幕截圖
下載圖片
檔名
標準化資產檔名
非裝飾圖片卻使用空白 alt
```

以下附件連結文字可視為低品質，應檢查是否需要改寫：

```text
下載
點我
按這裡
這裡
PDF
Word
Excel
檔案
附件
download
file
```

### Step 5：信心等級與處理規則

| 信心等級 | 判斷依據 | 處理方式 |
| --- | --- | --- |
| 高 | 圖片內容可辨識，或附件上下文明確說明用途 | 可直接改寫 |
| 中 | 鄰近標題與段落可合理支持描述，但仍有少量不確定 | 可改寫，但在判斷依據標記「根據上下文」 |
| 低 | 只能從檔名、hash、章節名稱或模糊文字推測 | 不改寫，列入「需要人工確認」 |

### Step 6：圖片 alt 改寫規則

圖片 alt 應符合：

1. 使用繁體中文。
2. 通常 8 到 30 個中文字。
3. 直接描述圖片承載的資訊或教學作用。
4. 不使用「這是一張」、「圖片顯示」、「如下圖」作為開頭。
5. 不加入教材正文沒有支持的額外解釋。
6. 不重複圖片前後已經完整說明的文字。
7. 如果圖片只是裝飾圖，使用空 alt。
8. 如果無法確認圖片內容或用途，列入「需要人工確認」。

### Step 7：本地附件連結文字改寫規則

附件連結文字應讓讀者在不看路徑的情況下理解附件用途。

建議格式：

```text
下載 <主題> <檔案性質>
開啟 <主題> <檔案性質>
查看 <主題> <檔案性質>
```

可使用的檔案性質：

```text
PDF
講義
練習檔
範例檔
表格
Word 文件
Excel 表格
壓縮檔
範例程式
音訊檔
影片檔
```

如果附件主題無法從上下文確認，不要只根據檔名、hash 或章節名稱硬補完整主題，請列入「需要人工確認」。

### Step 8：語法安全規則

改寫時必須遵守：

1. 保留 Markdown 與 HTML 原本語法。
2. 保留原本縮排、引號風格、標籤閉合方式與屬性順序。
3. 不改寫資產路徑。
4. 不改寫實體檔名。
5. 不移動、刪除或新增任何資產檔案。
6. 不改寫教材正文。
7. 不把檔名、hash、路徑或副檔名直接當成 alt 或連結文字。
8. HTML `<img>` 缺少 `alt` 時，在 `src` 屬性後方加入 `alt`，並沿用 `src` 的引號風格。
9. 改寫後確認 Markdown 圖片語法仍為 `![alt](path)`。
10. 改寫後確認 Markdown 連結語法仍為 `[text](path)`。
11. 改寫後確認 HTML 標籤、屬性與引號沒有被破壞。
12. 程式碼區塊與行內程式碼必須完全未改，但 Step 2 所述例外情況（fenced code block 內可唯一對應到 `assets/` 實體檔案的本地圖片 alt 或附件連結文字，依信心等級規則改寫）不在此限。

### Step 9：執行模式

#### dry-run 模式

如果執行模式是 `dry-run`：

1. 不實際修改任何檔案。
2. 只輸出建議改寫清單。
3. 每筆列出原文字、新文字、資產路徑與判斷依據。
4. 不要假裝已經修改檔案。

#### apply 模式

如果執行模式是 `apply`：

1. 只修改 Markdown 文件中的圖片 alt、HTML `alt` 屬性或本地附件連結文字。
2. 不修改資產路徑、檔名、正文內容或其他無關文字。
3. 修改後輸出處理摘要、已處理清單、保留清單、人工確認清單與檢查結果。
4. 如果無法安全修改，請停止並回報原因。

## Few-Shot 範例

### 範例一：保留有意義的圖片 alt

改寫前：

```md
![HTML 標籤巢狀結構示意圖](./assets/images/html-basic-img-001-a82f91.png)
```

改寫後：

```md
![HTML 標籤巢狀結構示意圖](./assets/images/html-basic-img-001-a82f91.png)
```

狀態：已保留
判斷依據：alt 已經具體描述圖片用途，不需要改寫。

### 範例二：改寫無意義圖片 alt

改寫前：

```md
![圖片.png](./assets/images/image-tags-img-001-7e5dcd.png)
```

如果可從圖片內容或鄰近上下文確認這是介紹 `img` 標籤屬性的示意圖，改為：

```md
![img 標籤屬性示意圖](./assets/images/image-tags-img-001-7e5dcd.png)
```

狀態：已改 alt
判斷依據：原 alt 過於籠統，新 alt 可由上下文支持。

### 範例三：無法確認時列入人工確認

改寫前：

```md
![圖片](./assets/images/example-img-001-a82f91.png)
```

如果只能從檔名或 hash 推測內容，且上下文沒有說明圖片用途，請不要改寫，列入「需要人工確認」。

狀態：需人工確認
判斷依據：資訊不足，不應猜測。

## Reverse Prompting：資訊不足時請反問

缺少以下資訊時，先反問，不要開始處理：

1. 要處理哪一個章節（`origin/<三位數編號>-<標題>/`）？
2. 要處理全部 Markdown，還是指定檔案？
3. 執行模式是 `dry-run` 還是 `apply`？
4. `meta/chapter-status.md` 中該章節「alt 與連結文字」欄位目前狀態為何，使用者是否確認要（重新）處理？
5. 該章節「資產命名」欄位是否已完成？若尚未完成，是否要先處理資產命名再進行本 skill？

## 最終品質標準

完成結果必須符合：

1. 圖片 alt 具體、準確、符合教材上下文。
2. 裝飾圖片使用空 alt，不放無意義描述。
3. 本地附件連結文字能讓讀者理解附件用途。
4. 不確定圖片或附件內容時不猜測。
5. 不改寫任何資產路徑、實體檔名、hash 或資料夾名稱。
6. 不移動、刪除或新增任何資產檔案。
7. 不改寫外部連結、錨點連結、特殊協議連結與空連結。
8. 不改寫程式碼區塊或行內程式碼中的範例，但 Step 2 所述例外情況（fenced code block 內可唯一對應到 `assets/` 實體檔案的本地圖片 alt 或附件連結文字）不在此限。
9. 不重寫教材正文。
10. Markdown 與 HTML 語法不得被破壞。
11. 所有改寫都要能說明判斷依據。
12. 保守性優先於完整性。
