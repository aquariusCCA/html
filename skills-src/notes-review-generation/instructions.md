# Notes 轉複習材料

## 定位與適用範圍

本 skill 對應 `meta/chapter-status.md` 章節狀態表中「review 生成」欄位，是 notes 完成檢查後的下游教材生成階段。review 與 demos、practice、appendix 都各自平行延伸 notes，彼此沒有先後依賴；本 skill 只負責 review：

```text
notes 生成 → notes 完成檢查（須先完成）
  ↓
（皆平行延伸 notes，彼此無先後）
review 生成（本 skill）
practice 生成
demos 生成
appendix 索引
```

review 與 practice 容易混淆，先界定清楚：

* **review＝回憶檢索**：考「記不記得」，型式為重點摘要、問答題、填空題（cloze）、翻面卡片，讀者用「想得起來／說得出來」來自我檢測，搭配遺忘曲線在臨界點反覆喚回記憶。
* **practice＝動手應用**：考「會不會用」，題型為應用練習、實作任務、改錯、重構，讀者要動手寫或改 HTML。
* 凡是需要讀者動手寫或改 HTML 才能完成的，一律歸 practice，本 skill **不**重複生成；review 的每一張卡都是「想起來／回答出來」就能完成，不要求動手。

開始處理前，應先查閱 `meta/chapter-status.md` 對應章節的「notes 完成檢查」欄位，確認狀態為「已完成」才適合開始本 skill；若尚未完成，應提示使用者先完成 `notes-content-review`，不要預設繼續生成 review。

本 skill 主要讀取：

```text
notes/<章節>/*.md
meta/chapter-status.md
```

只輸出或更新：

```text
review/<章節>/<NN-主題>.md
```

本 skill 不修改：

```text
origin/
atomic/
notes/
demos/
practice/
appendix/
supplements/
任何原始資產實體檔案
```

本 skill 不修改 `meta/chapter-status.md`，僅在輸出中提出「review 生成」欄位的建議標記，由使用者自行決定是否更新狀態表。

review 嚴格對齊章節：每章的 review 只放在 `review/<章節>/`，章節資料夾名稱（編號+標題）與 `notes/` 一致，不建立跨章節的彙整複習庫。

## 角色定位

啟用本 skill 時，扮演「複習材料設計師」與「來源回溯審查員」，目標是把正式 notes 中值得長期記住的重點，設計成讀者能在遺忘曲線的臨界點反覆喚回、且每次都能自我檢測對錯的複習材料。

合格的 review 不是把 notes 整段抄成卡片，也不是出超綱考題，而是要讓讀者：

```text
讀過該主題後，靠 notes 教過的內容就能回答每一張卡
每張卡都有明確、可對照的標準答案
複習時能對照答案，知道自己記得或記錯
每一張卡的考點與答案都來自 notes、可被回溯
照建議排程在記憶快消退時複習，把短期記憶轉為長期記憶
```

### 核心紅線（最高優先，違反即不合格）

review 的唯一內容來源是 `notes/<章節>/*.md`。**每張卡的考點與答案用到的內容，都不得超出 notes 教過的範圍。**

* 不得出 notes 沒教、只靠常識或其他章節才答得出的卡。例如 notes 只教了 `img` 的 `src`/`alt`，就不得做 `srcset`、`loading` 的問答卡。
* 標準答案只能引用 notes 已說明過的事實、定義、用法；不得在答案裡引入 notes 未涵蓋的內容「順便教」。
* 規劃或生成時若發現「想做的卡，notes 沒有對應內容」，不要硬做，改列入「需要人工確認」，並建議使用者回頭補 notes、改換考點、或不做該卡。
* 寧可卡片簡單，也不得脫離 notes 變成超綱複習。

### 個人進度狀態紅線（重要）

本 skill 只產出「複習材料內容＋建議排程模板」，**不維護任何個人複習進度**。

* front matter 的 `schedule_days` 是「建議的複習間隔模板」，不是某個人的到期日。
* 不得在產出檔寫入 `last_reviewed`、`due_date`、`ease`、`box`、回憶成功次數等逐學習者、隨時間變化的狀態。
* 個人實際進度交給外部 SRS 工具或未來的排程追蹤工具，靠卡片穩定 ID 對應。如此一來產出檔可隨時重生成而不會洗掉任何人的進度。

### 其他必須避免

1. 每篇 notes 都機械式做一份複習材料，導致大量沒有複習價值的湊數卡。
2. 把整段 notes 原文搬進卡片當答案，沒有壓縮成可檢索的考點。
3. 出無法判斷對錯、沒有明確標準答案的開放卡。
4. 把要動手寫或改 HTML 的題放進 review（那是 practice 的事）。
5. 修改 `notes/`、`origin/`、`atomic/` 或其他下游目錄。
6. 建立跨章節彙整複習庫，或在章節資料夾外輸出。

## 輸入資訊

開始前確認以下資訊是否足夠：

```text
章節名稱：<章節>
專案根目錄：預設為目前工作目錄
執行階段：預設為階段一「Review 生成規劃」
是否允許寫入檔案：預設不允許；階段二確認語句可視為允許寫入 review/<章節>/
```

如果使用者沒有提供章節名稱，請先反問：

```text
請提供要處理的章節名稱，例如：180-圖片標籤、240-表單標籤。
```

如果無法讀取 `notes/<章節>/` 或該資料夾沒有 `.md` 檔案，請不要猜測章節名稱或內容，請反問使用者確認章節資料夾名稱是否正確，或請使用者先完成 `notes-generation` 與 `notes-content-review`。

在資訊不足時，只能提出需要補充的問題，不得直接產生 review。

## Review 粒度判斷

本 skill 不採固定的一篇 notes 對一份複習材料。請依複習價值選擇以下方式：

| 生成方式 | 適用情況 | 輸出 |
| --- | --- | --- |
| 一對一 | 單篇 notes 有明確、可檢索的重點 | 一篇 notes 產生一份 review |
| 一對多 | 單篇 notes 含多個彼此獨立、各自值得分開複習的概念群 | 一篇 notes 拆成多份 review |
| 多對一 | 多篇 notes 屬於同一主題、合併成一份複習更連貫 | 多篇 notes 產生一份整合 review |
| 不生成 | 內容無任何值得記憶或檢索的重點 | 列入「不生成 Review 的 Notes」 |

判斷標準：

1. review 的覆蓋面比 practice 廣：即使是概念、版本史、查閱型內容，只要有「該記住的事實或定義」就值得做 review；practice 標為不適用的章節，review 多半仍適用。
2. 若一篇 notes 的不同段落各自是獨立的重點群，且混在一份會讓複習焦點混亂，可一對多。
3. 若多篇短 notes 是同一主題的不同切面，合併複習更連貫，可多對一。
4. 只有當一篇 notes 真的沒有任何值得記憶或檢索的重點（極少數）才列入不生成，不要硬湊。
5. 每份 review 要有清楚的複習主題，不要把整章不同主題塞進一份。

## 卡片型式

review 全部是回憶檢索型卡片。一份 review 內可混用多種型式，每張卡的型式要明確標示。請依 notes 內容選擇合適型式。

> **型式名稱對照**：以下各節標題為完整名稱；front matter `formats` 欄位、卡片標題與卡片 ID 中一律使用短名：問答題 → `問答`、填空題 → `填空`、翻面卡片 → `卡片`、重點摘要 → `重點摘要`。

### 重點摘要

適合：每份 review 開頭，先給可據以重建知識的骨架。
形式：用精簡條列或小表，把該主題的核心重點壓縮成「看一眼就能喚回整段」的提綱，不是原文照抄。
要求：

1. 只摘 notes 已有的重點，壓縮但不新增資訊。
2. 是「提示記憶」用的骨架，不是完整教材；細節留給 notes。

### 問答題

適合：定義、用途、差異、「為什麼」等可主動回憶的考點。
形式：一個問題，摺疊區內放標準答案。
要求：

1. 問題具體、有唯一或明確的標準答案。
2. 答案只用 notes 教過的內容，可回溯。

### 填空題（cloze）

適合：關鍵字、屬性名、標籤名、固定搭配等。
形式：句子中挖空，摺疊區內放被挖掉的答案。用人類可讀寫法，挖空處以 `____`（底線）標示，答案另列。
要求：

1. 挖空處是值得記住的關鍵，不是隨意挖無意義的字。
2. 一句不要挖太多空，避免變成猜謎。
3. 答案明確、可對照。

### 翻面卡片

適合：標籤↔用途、屬性↔效果、名詞↔定義等成對記憶。
形式：正面提示詞、反面答案，反面放摺疊區。
要求：

1. 正面是清楚的提示，反面是對應的標準答案。
2. 一張卡只記一組對應，不要塞多組。

所有型式共同要求：每張卡都要有明確、可對照的標準答案，且考點與答案都能回溯到 notes。每張卡帶一個穩定 ID（見「卡片 ID」）。

## 檔案與路徑規則

### 輸出目錄與檔名

每份 review 輸出到：

```text
review/<章節>/<NN-主題>.md
```

命名規則：

1. 沿用 notes 既有檔名慣例，採 `NN-主題.md`（兩位數序號 + 主題；主題以中文為主，`img`、`input`、`form` 等技術名詞可保留英文原文），例如 `01-img標籤複習.md`。
2. 一對一 / 一對多：以對應 notes 檔名（移除 `.md`）為基礎，一對多在其後加具體子題主題。
3. 多對一：用能代表共同主題的描述，例如 `01-表單元素複習.md`。
4. 同一章節內不得產生重複檔名。
5. 不使用空白、路徑分隔符、冒號、問號等不適合檔名的字元。

### 來源標示與 front matter

每份 review 檔開頭加入 YAML front matter。先放與 siblings 一致的 `source_notes`/`topics`/`summary`，再放一個 `review:` 區塊承載遺忘曲線排程模板：

```markdown
---
source_notes:
  - notes/180-圖片標籤/01-img標籤與圖片替代文字.md
topics: [img, alt, 圖片]
summary: "img 標籤與替代文字的回憶檢索複習材料（重點摘要＋問答＋填空＋卡片）。"
review:
  formats: [重點摘要, 問答, 填空, 卡片]
  card_count: 7
  priority: 高
  schedule_days: [1, 3, 7, 16, 35, 90]
---
```

各欄位規則：

* `source_notes`：列出本份複習所有來源 notes 路徑；多對一時列出全部。
* `topics`：2–5 個，盡量沿用對應 notes 既有 topics 用詞，避免同義詞分裂。
* `summary`：一句話，說明這份複習複什麼、含哪些型式。
* `review.formats`：本份用到的卡片型式，從「重點摘要 / 問答 / 填空 / 卡片」中取用。
* `review.card_count`：本份卡片總數（重點摘要不計入卡數，問答/填空/卡片才計）。
* `review.priority`：`高 / 中 / 低`。代表這份內容在整章的複習權重，供讀者或外部 SRS 工具判斷何時優先複習——基礎、高頻、易混淆的概念給「高」；邊緣、低頻的給「低」。本 skill **不**因 priority 調整 `schedule_days` 序列；優先度差異靠此旗標與每張卡的難度標示（基礎／進階）傳達，實際排程密度由外部工具決定。
* `review.schedule_days`：建議的複習日（離首次學習的天數），是遺忘曲線的擴張間隔模板，預設 `[1, 3, 7, 16, 35, 90]`。這是模板不是個人到期日。

> 排程設計原則：遺忘曲線下記憶會隨時間衰退，必須在「快要忘記的臨界點」複習，把曲線重新拉高、衰退變慢，逐次拉長間隔，最終轉為長期記憶。因此 `schedule_days` 採「先密後疏」的擴張序列。預設序列 `[1, 3, 7, 16, 35, 90]` 適用大多數章節，卡與卡之間的難易差異不靠各自調整序列，而靠 `priority` 與每張卡的難度標示（基礎／進階）來表達複習密度。**只有**當整份 review 的學習週期明顯偏短（例如純查閱型、低記憶負擔）或明顯偏長（例如深度進階主題、需長期內化）時，才調整 `schedule_days`，並在規劃中說明理由。

不得在 front matter 或正文寫入 `last_reviewed`、`due_date`、`ease`、`box` 等個人進度狀態（見「個人進度狀態紅線」）。

### 卡片 ID

每張卡（問答/填空/卡片）帶一個章節內穩定、唯一的 ID，寫在卡片標題旁，格式 `{#<章節數字>-<主題slug>-<型式><序>}`，例如 `{#180-img-q1}`、`{#180-img-c2}`、`{#180-img-f3}`（q=問答、c=卡片、f=填空）。

規則：

1. ID 在同一章節 `review/<章節>/` 內唯一。
2. ID 一旦產生就視為穩定錨點，重生成時若同一張卡仍在，沿用原 ID，不隨意更動，讓外部 SRS 工具能持續對應。
3. 重點摘要不需要 ID。

### 正文結構

front matter 之後：

1. 一個 `#` 標題，標明這是什麼複習材料。
2. 緊接一句「複習用 + 回連 notes」的引言，例如：

   ```markdown
   > 回憶檢索複習。概念與用法說明見 notes/180-圖片標籤/01-img標籤與圖片替代文字.md。
   ```

3. 「重點摘要」段：可據以重建知識的提綱。
4. 各類卡片：每張卡標明型式、難度、ID，問題/正面/挖空句直接顯示，標準答案放 `<details>` 摺疊區。
5. 末尾「建議複習排程」小表：把 front matter 的 `schedule_days` 攤成具體檢查點，說明這是模板、個人進度自行追蹤。

難度標示用「基礎 / 進階」兩級即可，寫在卡片標題旁，不另開 front matter 欄位。

每張卡範本：

```markdown
### 問答 1 · 基礎 · {#180-img-q1}

img 標籤的 `alt` 屬性是用來做什麼的？

<details>
<summary>答案</summary>

提供圖片的替代文字：圖片載入失敗或螢幕報讀時會用它，內容要描述圖片本身而非寫「圖片」這類無資訊的字。

</details>
```

建議複習排程小表範本：

```markdown
## 建議複習排程

> 以下為遺忘曲線複習模板（離首次學習的天數），個人實際進度請自行或用 SRS 工具追蹤。

| 第幾次 | 第 1 次 | 第 2 次 | 第 3 次 | 第 4 次 | 第 5 次 | 第 6 次 |
| --- | --- | --- | --- | --- | --- | --- |
| 距首讀 | 第 1 天 | 第 3 天 | 第 7 天 | 第 16 天 | 第 35 天 | 第 90 天 |
```

## 處理流程

本 skill 分兩階段執行：階段一（Review 生成規劃，預設）與階段二（產生 review）。階段一只輸出規劃、不動任何檔案；只有取得明確確認語句後才進入階段二。

### 階段一：Review 生成規劃（預設）

階段一只輸出規劃結果，不建立、修改或刪除任何檔案，最後依「輸出格式」的階段一範本輸出規劃，等待使用者確認。依序執行下列三步。

#### 步驟 1：前置檢查

1. 是否已取得 `<章節>`。
2. 是否能讀取 `notes/<章節>/*.md`。
3. `meta/chapter-status.md` 中該章節「notes 完成檢查」是否為「已完成」；若不是，提示先完成 `notes-content-review`。
4. `meta/chapter-status.md` 中該章節「review 生成」目前狀態為何；若已是「已完成」，確認使用者是否要重新處理。
5. 目前是否處於階段一；若沒有明確確認語句，不得進入階段二。

如果任一項不符合，請先回報問題並要求使用者補充，不要猜測。

#### 步驟 2：讀取與理解

1. 讀取 `notes/<章節>/*.md`，依檔名前綴與學習順序理解內容。
2. 建立 Notes Inventory，記錄每篇 notes 的標題、topics/summary、核心重點、可檢索的事實/定義/差異、易混淆點。
3. 找出每篇 notes 中值得長期記住的重點，判斷可轉成哪種卡片型式。
4. 區分哪些內容適合 review（回憶）、哪些只適合 practice（動手）或不值得做卡。

#### 步驟 3：Review 規劃

對每個候選 review 規劃：

1. 輸出檔案路徑與檔名。
2. 對應 notes（一對一 / 一對多 / 多對一 / 不生成）。
3. 卡片型式組成與張數（重點摘要＋問答×N＋填空×N＋卡片×N）。
4. 每張卡的考點、難度、標準答案方向。
5. 排程模板：`priority`（高/中/低）與 `schedule_days`（預設 `[1, 3, 7, 16, 35, 90]`，必要時說明調整理由）。
6. 來源回溯：每張卡考點與答案出自哪篇 notes 的哪段。
7. 唯一來源缺口：想做但 notes 沒有對應內容的卡，列入「需要人工確認」。

「卡片型式與考點」必須具體到看得出要考什麼、答案是什麼、怎麼判斷對錯；不能只寫「做幾張卡」或「複習圖片」。

### 階段二：產生 review

只有在使用者明確回覆：

```text
確認，請產生 review
```

才可以進入階段二。這句確認同時代表使用者已確認規劃，並允許寫入 `review/<章節>/`。

階段二流程：

1. 依確認後的規劃建立或更新 review 檔。
2. 如果目標檔案已存在，不得直接覆蓋；先列出既有項目並要求使用者確認覆蓋、更新或保留。更新既有檔時，沿用仍存在卡片的原 ID。
3. 產生含 front matter（含 `review:` 排程區塊）的複習 Markdown，每張卡帶穩定 ID 與摺疊標準答案，末尾附建議複習排程小表。
4. 每份 review 完成後執行靜態檢查與來源回溯檢查。
5. 檢查不通過時先修正，再輸出完成回報。

---

## 驗證規則

生成後必須驗證，不得省略。

### 靜態檢查（必須）

每份 review 至少確認：

1. `review/<章節>/<NN-主題>.md` 存在且非空。
2. 具備合法 YAML front matter，含 `source_notes`、`topics`、`summary` 三欄，以及 `review:` 區塊（含 `formats`、`card_count`、`priority`、`schedule_days`）。
3. `source_notes` 中每個路徑都指向實際存在的 notes 檔案。
4. `review.card_count` 與正文實際卡片數一致；`review.schedule_days` 是遞增的天數序列。
5. 每張卡（問答/填空/卡片）都有型式標示、難度、唯一 ID，以及 `<details>` 摺疊的標準答案；每張卡 ID 在章節內唯一。
6. 有「重點摘要」段與「建議複習排程」小表。
7. `<details>`、表格、fenced code block 語法正確，巢狀 code block 未破壞 `<details>` 結構。
8. front matter 與正文都沒有寫入 `last_reviewed`、`due_date`、`ease`、`box` 等個人進度狀態。
9. 沒有修改 `notes/`、`origin/`、`atomic/` 或其他下游目錄。
10. 輸出位置在 `review/<章節>/`，未建立跨章節彙整複習庫。

### 來源回溯檢查（必須，最高優先）

逐卡確認：

1. 每張卡考點都能在對應 notes 找到出處。
2. 標準答案只用 notes 已說明過的內容，沒有引入 notes 未涵蓋的事實或技術。
3. 每張卡都有明確、可對照的標準答案，能判斷對錯。
4. 沒有把要動手寫或改 HTML 的題誤放進 review。
5. 凡是 notes 沒有對應內容、卻為了做卡想加入的考點，已列入「需要人工確認」而非直接寫入。

若發現任何一張卡考點或答案無法回溯到 notes，必須改卡或移除，不得保留在成品中。

## Few-Shot 判斷

### 範例一：一對一含多種卡片型式

```text
notes/180-圖片標籤/01-img標籤與圖片替代文字.md
```

若該 notes 教了 `src`、`alt`、替代文字寫法與常見錯誤，建議：

```text
輸出檔案：review/180-圖片標籤/01-img標籤複習.md
生成方式：一對一
卡片組成：重點摘要×1、問答×3（基礎2/進階1）、填空×2（基礎）、卡片×2（基礎）
各卡考點：q1 alt 用途；q2 為何要替代文字；q3 替代文字怎麼寫才好；f1 挖 src 屬性名；f2 挖 alt 屬性名；c1 src↔指定圖片來源；c2 alt↔替代文字。
priority：高（圖片是高頻基礎）；schedule_days：[1,3,7,16,35,90]
來源回溯：每張卡只用 notes 教過的 src/alt；notes 沒教的 srcset/loading 不做。
```

### 範例二：多對一整合複習

```text
notes/240-表單標籤/02-input基礎與文字密碼輸入.md
notes/240-表單標籤/03-單選複選與下拉選項.md
notes/240-表單標籤/04-表單按鈕與提交行為.md
```

若多篇是同一主題的不同切面，建議：

```text
輸出檔案：review/240-表單標籤/01-表單元素複習.md
生成方式：多對一
卡片組成：重點摘要×1、問答×4、填空×3、卡片×3
各卡考點：涵蓋 input type、單選複選差異、下拉、按鈕 type 與提交行為，全部來自三篇 notes。
priority：高；schedule_days：[1,3,7,16,35,90]
來源回溯：每張卡對應三篇 notes 教過的元素；未教的驗證屬性不做卡。
```

### 範例三：概念章節仍可生成 review

```text
notes/020-HTML簡介/02-HTML5版本與特性.md
```

即使是版本史與特性概覽（practice 標為不適用）：

```text
輸出檔案：review/020-HTML簡介/02-HTML5版本與特性複習.md
生成方式：一對一
卡片組成：重點摘要×1、問答×2、填空×1、卡片×2
理由：版本史/特性有「該記住的事實」，正適合 review 的回憶檢索；雖無動手面（practice 不適用），但記憶價值高。
```

### 範例四：不生成 review

```text
某篇純引導語、無任何需記憶重點的 notes
```

```text
原因：通篇是過場引導，沒有任何值得記憶或檢索的事實/定義/用法。
建議：列入不生成 review，不要硬湊湊數卡。
```

## Reverse Prompting：資訊不足時請反問

遇到以下情況時，不要猜測，請先請使用者補充：

1. 未提供章節名稱。
2. `notes/<章節>/` 不存在或沒有 `.md` 檔案。
3. `meta/chapter-status.md` 顯示「notes 完成檢查」尚未完成。
4. 使用者要求第二階段產生 review，但尚未明確回覆「確認，請產生 review」。
5. 目標 review 檔案已存在，且使用者尚未確認覆蓋、更新或保留。
6. 想做的卡考點或答案在 notes 中找不到對應內容，無法判斷是否該做。
7. 使用者要求本 skill 維護個人複習進度/到期日——說明這超出本 skill 範圍（只產內容＋排程模板），個人進度交給外部 SRS 工具。

## 最終品質標準

階段二完成後，必須逐項檢查：

1. 是否只以 `notes/<章節>/*.md` 為內容來源，且每張卡考點與答案都可回溯到 notes。
2. 是否未出超出 notes 教過範圍的考點或答案（核心紅線）。
3. 是否正確區分 review（回憶）與 practice（動手），未把動手題放進 review。
4. 是否先建立規劃，而非機械式逐篇做卡。
5. 是否正確判斷一對一、一對多、多對一與不生成。
6. 是否每份 review 都有清楚主題與正確 front matter（source_notes/topics/summary 與 review 排程區塊）。
7. 是否每張卡都有型式、難度、唯一 ID、明確標準答案，且有重點摘要與建議複習排程小表。
8. 是否所有 `source_notes` 路徑存在、`<details>`/表格/code block 語法正確、`card_count` 與實際卡數一致。
9. 是否未在 front matter 或正文寫入任何個人複習進度狀態（核心紅線）。
10. 是否輸出在 `review/<章節>/`，未建立跨章節彙整複習庫。
11. 是否未修改 `origin/`、`atomic/`、`notes/` 或其他下游目錄。
12. 是否將所有唯一來源缺口列入「需要人工確認」。
13. 是否提出 `meta/chapter-status.md` 中「review 生成」欄位的建議標記。

可建議「review 生成」標記為「已完成」的條件：

1. 規劃中應生成的 review 都已建立或更新。
2. 不生成 review 的 notes 都有合理原因。
3. 沒有未解決、會讓卡片無法判斷對錯，或考點超出 notes 的問題。
4. 靜態檢查與來源回溯檢查皆通過。
