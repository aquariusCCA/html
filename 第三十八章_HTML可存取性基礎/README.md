# 第三十八章｜HTML 可存取性基礎

> 章節定位：用 HTML 本身建立可被鍵盤、螢幕閱讀器、搜尋工具與自動化工具理解的頁面基礎  
> 建議閱讀方式：先掌握「語意化 HTML 優先」，再理解 ARIA 是什麼、它補充哪一層資訊，以及什麼情況才需要使用 ARIA

## 本章在學什麼

這一章整理 **HTML 層面的可存取性基礎**。

可存取性不是只服務螢幕閱讀器使用者，也包含：

- 只能使用鍵盤操作頁面的人。
- 暫時無法使用滑鼠的人。
- 使用螢幕閱讀器、語音控制、放大工具的人。
- 在手機、小螢幕、弱光或高干擾環境下瀏覽的人。
- 需要讓搜尋引擎、自動化測試或輔助工具理解頁面結構的情境。

本章的核心觀念是：

- **先寫對 HTML，再考慮 ARIA。**
- 標題階層、連結文字、圖片替代文字、表單標籤、表格標頭會影響輔助技術理解頁面。
- 鍵盤焦點與操作順序會影響不使用滑鼠的使用者。
- ARIA 是 **Accessible Rich Internet Applications** 的縮寫，是用來補充可存取性語意、狀態與關係的屬性規範。
- ARIA 主要影響的是 **accessibility tree**，也就是瀏覽器提供給輔助技術理解頁面的資訊。
- ARIA 可以補角色、名稱、狀態與關係，但不會自動補齊鍵盤操作、焦點管理或元件行為。
- 工具檢查只能輔助，仍需要人工檢查頁面操作流程是否合理。

## ARIA 快速定位

ARIA 可以理解成：

> 當原生 HTML 無法完整描述元件的角色、名稱、狀態或關係時，補給輔助技術看的語意資訊。

但要特別注意：

- ARIA 不是 CSS，不負責畫面外觀。
- ARIA 不是 JavaScript，不會自動補互動行為。
- ARIA 不是萬能修補工具，不能把錯誤 HTML 直接變成完整可用元件。
- 若原生 HTML 已經能表達語意，通常應優先使用原生 HTML。

例如：

```html
<button type="button">儲存</button>
```

通常比下面這種寫法更好：

```html
<div role="button">儲存</div>
```

因為 `<button>` 內建語意、可聚焦能力、鍵盤操作與按鈕行為；`role="button"` 主要只補語意，不會自動補完整行為。

## 易混淆主題

| 易混淆點 | 正確理解 |
|---|---|
| 看得見 vs 可理解 | 畫面上能看見，不代表輔助技術能正確理解。 |
| 可存取性 vs 螢幕閱讀器 | 可存取性不只和螢幕閱讀器有關，也包含鍵盤、焦點、結構、文字、錯誤提示與操作流程。 |
| ARIA 是什麼 | ARIA 是補充 accessibility tree 語意、狀態與關係的屬性規範，不是畫面樣式或行為框架。 |
| ARIA vs HTML | 能用正確 HTML 表達時，通常先用 HTML。 |
| `alt` vs `title` | 圖片替代文字不是滑鼠提示文字。 |
| `placeholder` vs `label` | placeholder 不是穩定欄位名稱，不能取代 label。 |
| `role="button"` vs `<button>` | `role` 主要補語意，不會自動補齊鍵盤行為、焦點管理與表單行為。 |
| `tabindex` vs 正確 DOM 順序 | 不要依賴正整數 `tabindex` 修補混亂結構，應優先調整 HTML 原始順序。 |
| 工具檢查 vs 人工檢查 | Lighthouse 等工具可以輔助，但不能保證頁面對所有使用者都真的好用。 |

- 能判斷 ARIA 何時是合理補充，何時是在掩蓋錯誤 HTML。
