# HTML 元素的內容模型與基本巢狀規則

## 學習目標

讀完這篇筆記，你應該能夠：

- 理解 HTML 巢狀不是「想包什麼就包什麼」。
- 分辨排版上的 block、inline 與 HTML 內容模型不是同一件事。
- 用 `div` 與 `span` 理解 flow content、phrasing content 的基本差異。
- 遇到不確定的巢狀關係時，知道要回到個別元素的 content model 判斷。

## 問題情境

初學 HTML 時，很容易把規則記成：

- 塊級元素裡可以放塊級元素與行內元素。
- 行內元素裡只能放行內元素，不能放塊級元素。

這個說法可以幫你建立第一層直覺，但它不夠精準。真正決定一個元素能放什麼內容的，不是只看它在畫面上像 block 還是 inline，而是看這個元素在 HTML 規範中的 content model。

## 一句話理解

HTML 巢狀規則要看元素允許的內容模型；block、inline 只能幫助理解排版，不能取代合法性判斷。

## 先看整體結構

可以先用兩個常見容器建立直覺：

| 元素 | 常見角色 | 可放內容的直覺 |
| --- | --- | --- |
| `div` | 通用區塊容器 | 可包含多數 flow content，例如文字、行內元素、其他區塊元素 |
| `span` | 通用行內容器 | 通常只應包含 phrasing content，例如文字、`em`、`strong`、`a`、`input` 等 |

這個對照不是要你背死 `div` 與 `span`，而是要理解：不同元素有不同內容模型，不能只靠視覺排版判斷。

## `div`：通用 flow 容器

`div` 是沒有特定語意的通用區塊容器。它常用來包住一組內容，以便分組、排版或套用 CSS。

```html
<div>
  <span>尚硅谷1</span>
  <input type="text">
  <div>尚硅谷2</div>
</div>
```

這段範例中：

- 外層 `div` 是一個通用容器。
- `span` 是行內語意內容，可以放在 `div` 裡。
- `input` 是表單控制項，也可以作為 flow content 的一部分。
- 內層 `div` 是另一個區塊容器，也可以放在外層 `div` 裡。

所以在一般情況下，`div` 可以包住很多不同類型的內容。不過這不代表所有塊級元素都像 `div` 一樣自由。

## `span`：通用 phrasing 容器

`span` 是沒有特定語意的行內容器，常用來包住文字中的某一小段內容。

```html
<span>
  <span>尚硅谷1</span>
  <input type="text">
  <span>尚硅谷2</span>
</span>
```

這段範例中，外層 `span` 包住的是行內語意內容。這種情境通常是合理的。

但下面這種寫法就不適合：

```html
<span>
  <div>這是一個區塊</div>
</span>
```

問題不只是 `div` 會換行，而是 `span` 的內容模型不適合放 `div` 這類區塊內容。瀏覽器可能會嘗試修正 DOM，但你的原始 HTML 已經不是良好的結構。

## 內容模型比 block/inline 更重要

常見教學會用「塊級元素」與「行內元素」描述巢狀規則，但這只是簡化版本。實際上你應該掌握三件事：

1. `div` 這類通用容器可以包含多數 flow content。
2. `span` 這類行內容器通常只應包含 phrasing content。
3. 特定元素會有自己的限制，例如 `p`、`h1` 到 `h6`、`a` 都不能只用 block/inline 規則概括。

也就是說，不是所有塊級元素都能放任何內容；也不是所有行內元素都有完全相同的限制。

## 常見錯誤

### 錯誤一：把 `div` 的自由度套用到所有塊級元素

`div` 可以包很多內容，因為它是通用 flow 容器。但 `p` 代表段落，`h1` 到 `h6` 代表標題，它們都有更明確的內容限制。

如果你用「塊級元素幾乎什麼都能放」來理解所有元素，就很容易寫出 `p` 裡放 `div`、`h1` 裡放 `h2` 這類錯誤結構。

### 錯誤二：把所有行內元素都當成同一種限制

`span` 通常只能包 phrasing content，但 `a` 是特殊例子。`a` 使用 transparent content model，在某些情況下可以包住區塊內容。

不過 `a` 也不是完全無限制，它不能包含另一個 `a`，也不能包含按鈕、表單控制項等互動內容。

### 錯誤三：只要瀏覽器有顯示，就認為 HTML 合法

瀏覽器會盡量修正錯誤 HTML，讓頁面可以顯示。但「瀏覽器幫你修正」不等於「原始寫法正確」。

錯誤巢狀可能造成：

- DOM 結構和你想像的不一樣。
- CSS 選擇器套不到預期元素。
- JavaScript 取得的節點結構不符合原始碼。
- 輔助技術或搜尋引擎理解錯誤。

## 實務寫法

寫 HTML 巢狀時，可以照這個順序判斷：

1. 先問：這個標籤的語意是否適合目前內容？
2. 再問：這個標籤的內容模型是否允許放入目前的子元素？
3. 如果只是為了排版或分組，考慮用 `div` 或 `span`，再用 CSS 控制外觀。
4. 如果遇到 `p`、標題、連結、表單控制項等常見限制元素，要特別檢查巢狀是否合法。

## 重點整理

- HTML 巢狀是否合法，要看元素的 content model。
- block/inline 是排版直覺，不是完整的合法性規則。
- `div` 可包含多數 flow content，但不能把這個特性套到所有塊級元素。
- `span` 通常只適合包 phrasing content。
- 瀏覽器能顯示錯誤 HTML，不代表原始結構正確。

## 自我檢查

1. 為什麼不能只用「塊級元素」與「行內元素」判斷所有巢狀規則？
2. `div` 可以包住另一個 `div`，是否代表 `p` 也可以包住 `div`？為什麼？
3. 如果你想包住一句話中的某幾個字來加樣式，應該優先考慮 `div` 還是 `span`？
4. 瀏覽器會修正錯誤巢狀，這對 CSS 或 JavaScript 可能造成什麼影響？
