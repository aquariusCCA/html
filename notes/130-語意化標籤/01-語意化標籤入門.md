# 語意化標籤入門

## 學習目標

讀完這篇筆記，你應該能夠：

- 用自己的話說明「語意化」在 HTML 裡代表什麼。
- 認出常見 HTML5 語意化標籤的基本用途。
- 理解語意化對閱讀、維護、SEO 與輔助技術的影響。
- 避免把語意化誤解成「不能用 `<div>`」或「只為了 SEO」。

## 問題情境

假設一個頁面有頁首、導覽列、主要文章、側邊補充資訊與頁尾，但整個頁面都用 `<div>` 包起來：

```html
<div class="header">...</div>
<div class="nav">...</div>
<div class="content">...</div>
<div class="footer">...</div>
```

人類可能可以靠 class 名稱猜出每一塊的用途，但瀏覽器、搜尋引擎與螢幕閱讀器看到的仍然是一堆沒有明確語意的通用容器。

語意化標籤要解決的問題就是：讓 HTML 不只「排得出畫面」，也能「說得出內容的角色」。

## 一句話理解

語意化就是使用能表達內容意義的 HTML 標籤，而不是只用通用容器來包版面。

## 語意化不是裝飾，而是結構

HTML 的任務不是讓畫面變漂亮，而是描述內容結構。樣式可以交給 CSS，但內容的角色應該盡量由 HTML 本身表達。

例如：

- `<header>` 代表頁面或區塊的開頭內容。
- `<nav>` 代表導覽連結。
- `<main>` 代表頁面的主要內容，通常一個頁面只會有一個主要的 `<main>` 區域。
- `<article>` 代表可獨立閱讀或分發的完整內容，例如文章、新聞、部落格貼文。
- `<section>` 代表具有共同主題的一段內容，通常會搭配標題。
- `<aside>` 代表與主要內容相關，但可以分離閱讀的補充內容。
- `<footer>` 代表頁面或區塊的結尾資訊。

這些標籤的外觀不一定特別，但它們讓頁面結構更容易被理解。

## 語意化版面範例

一個常見頁面可以被拆成多個角色清楚的區塊：

![HTML5 語意化版面區塊示意圖](../../origin/130-語意化標籤/assets/images/semantic-tags-img-001-fe28a0.png)

這張圖可以把語意化想成搭積木：每一塊不只是「一個盒子」，而是有自己的內容責任。

例如：

- 頁面最上方的標題、Logo 或介紹內容可以放在 `<header>`。
- 主要選單可以放在 `<nav>`。
- 文章或主要教學內容可以放在 `<main>` 裡。
- 單篇文章、新聞或卡片內容可以用 `<article>`。
- 同一主題下的一段區塊可以用 `<section>`。
- 側邊推薦、補充說明或廣告可以用 `<aside>`。
- 版權資訊、聯絡方式或頁尾連結可以放在 `<footer>`。

## 語意化帶來什麼好處

語意化的好處不只一個，也不只服務搜尋引擎。

第一，程式碼更容易閱讀與維護。當團隊看到 `<nav>`、`<article>`、`<footer>` 時，比看到一串 `<div>` 更容易理解頁面結構。

第二，樣式遺失時仍保留清楚結構。即使 CSS 沒有載入，語意化 HTML 仍能讓內容按合理順序呈現。

第三，有助於搜尋引擎理解頁面。SEO 是 Search Engine Optimization，也就是搜尋引擎優化。語意化結構可以幫助搜尋引擎判斷頁面中哪些內容是導覽、主要內容或文章區塊，但搜尋排名仍受內容品質、網站權重、效能、外部連結等多種因素影響，不能只靠換標籤解決。

第四，輔助技術能更好地解析頁面。螢幕閱讀器可以利用語意結構提供導覽提示，讓使用者快速跳到導覽列、主要內容或頁尾，而不是逐個沒有語意的容器聽過去。

## 語意化與上下文

「上下文」指的是一段文字或內容前後相連的語言環境。單看一句話，有時很難理解意思；放回整篇文章，就能知道它正在回應什麼問題。

HTML 標籤也有類似概念。單看一段 `<div>`，它可能是導覽、卡片、文章，也可能只是排版容器；但如果使用 `<nav>`、`<article>` 或 `<section>`，標籤本身就提供了內容角色的上下文。

語意化標籤不是取代內容本身，而是幫內容補上結構線索。

## 常見錯誤

### 錯誤一：整頁只用 `<div>`

```html
<div class="nav">...</div>
<div class="main">...</div>
<div class="footer">...</div>
```

這樣的頁面可能看起來正常，也能用 CSS 排出想要的畫面，但 HTML 結構本身沒有清楚表達「哪裡是導覽」「哪裡是主要內容」「哪裡是頁尾」。

比較好的寫法是：

```html
<nav>...</nav>
<main>...</main>
<footer>...</footer>
```

### 錯誤二：以為語意化只是為了 SEO

SEO 是語意化的受益場景之一，但不是唯一目的。語意化同時影響可維護性、輔助技術、其他裝置解析與團隊協作。

如果只從 SEO 角度理解語意化，很容易忽略它最基本的任務：讓 HTML 結構本身更有意義。

### 錯誤三：以為語意化標籤只能用一次

有些語意化標籤可以在同一頁多次出現。例如一個頁面可以有多篇 `<article>`，每篇文章裡也可以有自己的 `<header>` 與 `<footer>`。

但這不代表所有語意化標籤都能任意重複使用。像 `<main>` 代表頁面的主要內容區，通常不應在同一個頁面中放多個。

重點不是「能不能用多次」，而是每次使用時內容角色是否合理。

### 錯誤四：完全不用 `<div>`

語意化不是禁止 `<div>`。當某個元素只是為了排版、分組或套樣式，沒有特別的內容語意時，使用 `<div>` 仍然合理。

實務上應該先問：這一塊內容有沒有明確角色？如果有，就優先選擇合適的語意化標籤；如果沒有，`<div>` 就是合適的通用容器。

## 實務注意事項

現代瀏覽器與行動端瀏覽器通常能良好支援 HTML5 語意化標籤。

如果專案需要支援非常舊的瀏覽器，例如 IE9，可能需要確認這些新元素是否能正確以區塊元素呈現。不過在現代前端開發中，通常應依專案目標瀏覽器與相容性需求決定是否額外處理。

## 重點整理

- 語意化是用標籤表達內容角色，而不是只用標籤包版面。
- `<header>`、`<nav>`、`<main>`、`<article>`、`<section>`、`<aside>`、`<footer>` 都是常見語意化標籤。
- 語意化能改善可讀性、可維護性、SEO、輔助技術解析與樣式失效時的內容結構。
- 語意化不是禁止 `<div>`，而是在有明確內容角色時優先使用更合適的標籤。
- 選標籤時要根據內容意義，不是根據畫面長相。
