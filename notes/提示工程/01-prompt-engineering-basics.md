# 提示工程教學筆記：骨架與招式

## 一、提示工程的本質

提示工程不是單純「把問題問清楚」而已，而是把你的需求整理成 AI 可以理解、可以執行、可以檢查的指令。

一個好的 Prompt，本質上是在做三件事：

```text
1. 告訴 AI：你要做什麼
2. 告訴 AI：你要在什麼背景下做
3. 告訴 AI：怎樣的答案才算好
```

所以提示工程可以理解成：

```text
需求分析 + 指令設計 + 輸出控制
```

它不只是寫一句問題，而是把「角色、任務、背景、限制、格式、範例、標準」組合起來，讓 AI 更穩定地產出你想要的結果。

---

# 二、Prompt 的骨架

提示工程最核心的骨架是：

```text
角色 + 任務 + 上下文 + 限制條件 + 輸出格式 + 範例 + 評估標準
```

這七個元素不是每次都要全部使用，而是依照任務複雜度選擇需要的元素。

任務越簡單，Prompt 可以越短。

任務越複雜，Prompt 就越需要完整的骨架。

---

## 1. 角色 Role

角色是指定 AI 要用什麼身份、專業角度或思考方式回答。

例如：

```text
你是一位資深 Java 後端工程師。
你是一位前端架構師。
你是一位嚴格的 Code Reviewer。
你是一位技術面試官。
```

角色的作用不是裝飾，而是讓 AI 知道回答時要採用哪種判斷標準。

例如同樣是分析一段程式碼：

```text
你是一位初學者教學老師
```

AI 會比較偏向用簡單方式解釋。

```text
你是一位資深架構師
```

AI 會比較偏向討論可維護性、擴充性、架構風險。

```text
你是一位嚴格 Code Reviewer
```

AI 會比較偏向指出 bug、壞味道、命名問題、可讀性問題。

所以 Role 的重點是：

```text
讓 AI 從正確的角度看問題。
```

---

## 2. 任務 Task

任務是 Prompt 的核心。

它告訴 AI 你到底要它完成什麼事。

例如：

```text
請幫我解釋策略模式。
請幫我優化這段 Java 程式碼。
請幫我設計一個 Vue 表單元件。
請幫我分析這段 SQL 查詢效能問題。
```

任務要避免太模糊。

不好的寫法：

```text
幫我看一下。
```

比較好的寫法：

```text
請幫我檢查這段 Java 程式碼是否違反開閉原則，並提供重構建議。
```

差別在於，後者明確說明了：

```text
檢查對象：Java 程式碼
檢查重點：開閉原則
期待結果：重構建議
```

Task 的重點是：

```text
讓 AI 明確知道要完成哪一件事。
```

---

## 3. 上下文 Context

上下文是讓 AI 知道目前的背景情境。

例如：

```text
我目前使用 Java 8。
我有兩年 Vue 開發經驗，但後端經驗較少。
這是一個銀行內部報表系統。
這段程式碼會用在大量資料匯出 Excel 的場景。
```

上下文越清楚，AI 的答案越貼近你的實際需求。

例如你問：

```text
請幫我設計一個報表匯出功能。
```

AI 可能會給你很通用的答案。

但如果你補上上下文：

```text
這是一個銀行內部報表系統，使用 Java 8、Spring MVC、MyBatis，
資料量可能有數萬筆，最後需要匯出 Excel。
```

AI 的回答就會更接近真實專案，可能會考慮：

```text
分頁查詢
記憶體壓力
Excel 欄位組裝
Apache POI
資料來源封裝
錯誤處理
權限控管
```

Context 的重點是：

```text
讓 AI 不要用通用答案，而是用符合你情境的答案。
```

---

## 4. 限制條件 Constraints

限制條件是規定 AI 回答時要遵守的邊界。

例如：

```text
請使用 Java 8。
不要使用 Spring Boot。
不要使用太複雜的框架。
程式碼需要加上註解。
請用初學者能理解的方式說明。
請不要只講概念，要搭配實際業務場景。
```

限制條件可以避免 AI 回答太發散。

例如：

```text
請幫我寫一個策略模式範例。
```

AI 可能會使用最新 Java 語法。

但如果你補上：

```text
請使用 Java 8，不要使用 lambda，不要使用 Spring。
```

AI 的答案就會更符合你的學習環境。

Constraints 的重點是：

```text
控制 AI 不要超出你能理解、能使用、能落地的範圍。
```

---

## 5. 輸出格式 Format

輸出格式是指定 AI 的答案要怎麼呈現。

例如：

```text
請用表格整理。
請分成：概念、範例、總結。
請先給流程，再給程式碼。
請用 Markdown 筆記格式輸出。
請用「問題 → 原因 → 解法 → 程式碼」的格式回答。
```

輸出格式很重要，因為同一個內容，用不同格式呈現，理解難度會差很多。

例如你可以要求：

```text
請分成以下段落：
1. 這個問題是什麼
2. 為什麼會發生
3. 如何解決
4. 完整程式碼
5. 總結
```

這樣 AI 的答案會比較穩定，也比較容易變成筆記。

Format 的重點是：

```text
控制答案的結構，讓結果更好閱讀、好整理、好使用。
```

---

## 6. 範例 Examples

範例是給 AI 參考的輸出樣式。

範例的重點不是教使用者怎麼問，而是讓 AI 看見「理想答案或處理方式長什麼樣子」。

例如：

```text
請參考下面格式回答：

1. 問題是什麼
2. 為什麼會發生
3. 如何解決
4. 完整程式碼
5. 實務注意事項
```

也可以直接提供一組範例：

```text
輸入：
請解釋單例模式。

輸出：
1. 單例模式解決什麼問題
2. 使用場景
3. Java 實作
4. 優缺點
5. 常見錯誤
```

Examples 的好處是，AI 很擅長模仿範例。

你給它什麼結構，它就比較容易照著那個結構回答。

Examples 的重點是：

```text
讓 AI 模仿你想要的回答方式。
```

---

## 7. 評估標準 Criteria

評估標準是告訴 AI 什麼樣的答案才算好。

例如：

```text
請優先考慮：
1. 可維護性
2. 可擴充性
3. 初學者能理解
4. 能對應實際專案
```

這會影響 AI 的取捨。

例如你要求：

```text
請優先考慮執行效能。
```

AI 可能會給你偏效能優化的答案。

如果你要求：

```text
請優先考慮程式碼可讀性與初學者理解。
```

AI 可能會避免太抽象、太進階的寫法。

Criteria 的重點是：

```text
讓 AI 知道答案要往哪個品質方向靠近。
```

---

# 三、七大元素的實務優先順序

實務上不一定每次都要把七個元素全部寫出來。

最核心的是四個元素：

```text
任務 Task
上下文 Context
限制條件 Constraints
輸出格式 Format
```

可以簡化成一句話：

```text
在什麼背景下，請你完成什麼任務，遵守哪些限制，並用什麼格式輸出。
```

這就是日常使用 AI 最實用的 Prompt 公式。

---

## 簡化模板

```text
我目前在【上下文】。

請你幫我【任務】。

請遵守以下限制：
1. 【限制條件 1】
2. 【限制條件 2】

請用以下格式輸出：
1. 【輸出格式 1】
2. 【輸出格式 2】

好的答案需要符合：
1. 【評估標準 1】
2. 【評估標準 2】
```

---

## 範例：學習 Java 設計模式

```text
我目前正在學 Java 8，後端經驗較少。

請你幫我解釋策略模式，並用銀行報表系統作為例子。

限制條件：
1. 請使用 Java 8
2. 不要使用 Spring
3. 程式碼需要加註解
4. 用初學者能理解的方式說明

輸出格式：
1. 先說明策略模式解決什麼問題
2. 再給實際業務流程
3. 再給 Java 程式碼
4. 最後說明這樣設計的好處

評估標準：
1. 能對應實際專案
2. 程式碼好維護
3. 初學者看得懂
```

這個 Prompt 比單純問：

```text
請解釋策略模式。
```

更好，因為它包含了：

```text
上下文：正在學 Java 8，後端經驗較少
任務：解釋策略模式
限制條件：Java 8、不使用 Spring、加註解
輸出格式：概念、流程、程式碼、好處
評估標準：實際專案、好維護、看得懂
```

---

# 四、骨架與招式的關係

前面七大元素可以稱為 Prompt 的「骨架」。

```text
角色 + 任務 + 上下文 + 限制條件 + 輸出格式 + 範例 + 評估標準
= 骨架
```

而 CoT、Few-Shot、Persona 這些技巧，可以理解成讓骨架更好用的「招式」。

```text
CoT、Few-Shot、Persona
= 讓這個骨架更好用的招式
```

也就是說，CoT、Few-Shot、Persona 不是另外一套分類，而是七大元素中的進階用法。

---

# 五、招式一：Persona Prompting

## 1. Persona 是什麼？

Persona Prompting 就是角色提示。

它對應到七大元素中的：

```text
角色 Role
```

也就是讓 AI 扮演某種角色，從特定角度回答問題。

例如：

```text
你是一位資深 Java 後端工程師。
你是一位前端架構師。
你是一位嚴格的 Code Reviewer。
你是一位技術面試官。
```

---

## 2. Persona 的作用

Persona 可以讓 AI 的回答更有方向。

例如你問：

```text
請幫我看這段程式碼。
```

如果沒有指定角色，AI 可能只是簡單解釋程式碼。

但如果你寫：

```text
你是一位嚴格的 Code Reviewer，請幫我檢查這段 Java 程式碼。
```

AI 會更傾向指出：

```text
命名問題
重複程式碼
職責不清
可讀性問題
可維護性問題
潛在 bug
```

---

## 3. Persona 範例

```text
你是一位資深 Java 後端工程師，也是嚴格的 Code Reviewer。

我目前正在學 Java 8，後端經驗較少。

請你幫我檢查這段程式碼是否違反開閉原則。

限制條件：
1. 請用初學者能理解的方式說明
2. 不要只講概念，要指出實際問題
3. 如果需要重構，請使用 Java 8

輸出格式：
1. 程式碼目前的問題
2. 違反了哪些設計原則
3. 如何重構
4. 重構後程式碼
5. 重構後的好處
```

這裡的 Persona 是：

```text
資深 Java 後端工程師
嚴格 Code Reviewer
```

它會影響 AI 的回答角度。

---

## 4. Persona 使用原則

Persona 不要只寫得很帥，而要跟任務有關。

不好的寫法：

```text
你是全世界最強的工程師。
```

比較好的寫法：

```text
你是一位熟悉 Java 8、Spring MVC、MyBatis 的資深後端工程師。
```

原因是後者更具體，AI 更容易知道要用什麼標準回答。

Persona 的實務原則是：

```text
角色要具體
角色要符合任務
角色要能影響判斷標準
```

---

# 六、招式二：Few-Shot Prompting

## 1. Few-Shot 是什麼？

Few-Shot Prompting 是少樣本提示。

它對應到七大元素中的：

```text
範例 Examples
```

做法是在正式任務之前，先給 AI 一到三個範例，讓它模仿你的格式、語氣和邏輯。

---

## 2. Few-Shot 的作用

AI 很擅長模仿範例。

所以當你希望答案有固定格式時，Few-Shot 很有用。

例如你希望 AI 幫你整理技術筆記，每次都用同樣格式：

```text
1. 這是什麼
2. 解決什麼問題
3. 使用場景
4. 程式碼範例
5. 常見錯誤
6. 總結
```

你可以先給 AI 一個範例，它之後就更容易照著這個格式輸出。

---

## 3. Few-Shot 範例

```text
請你之後都參考以下格式整理技術筆記。

範例：

主題：策略模式

1. 這是什麼
策略模式是一種把不同演算法封裝起來，讓它們可以互相替換的設計模式。

2. 解決什麼問題
它解決大量 if-else 判斷造成程式碼難以維護的問題。

3. 使用場景
當系統中有多種計算規則、匯出規則、折扣規則時，可以考慮使用策略模式。

4. 程式碼範例
使用 Java 介面定義策略，再由不同類別實作不同規則。

5. 常見錯誤
不要為了使用設計模式而使用設計模式，如果規則很少且不會變動，簡單 if-else 反而更直覺。

6. 總結
策略模式的核心是把變動的規則抽出去，讓主流程保持穩定。

正式任務：
請用同樣格式幫我整理「工廠模式」。
```

這種寫法會比單純說：

```text
請幫我整理工廠模式。
```

更穩定。

---

## 4. Few-Shot 使用原則

Few-Shot 適合用在以下情境：

```text
想固定回答格式
想固定語氣
想固定分析邏輯
想讓 AI 產出類似風格的內容
```

例如：

```text
整理技術筆記
整理面試題
產生 Code Review 報告
產生學習計畫
產生固定格式的 commit message
```

Few-Shot 的實務原則是：

```text
範例不要太多
範例要接近正式任務
範例格式要清楚
範例品質要高
```

# 七、招式三：CoT 類提示

## 1. CoT 是什麼？

CoT 是 Chain-of-Thought，常被翻譯成「思考鏈提示」。

它常見的口訣是：

```text
請一步一步思考。
```

但在實務使用上，更推薦把它改寫成：

```text
請先拆解問題，再給出結論。
請列出關鍵判斷依據。
請用可檢查的步驟說明。
請先分析流程，再提供程式碼。
```

因為真正有價值的不是要求 AI 寫出所有內部推理，而是讓 AI 的答案變得更有結構、更容易檢查。

---

## 2. CoT 對應七大元素中的哪一個？

CoT 類提示通常對應到：

```text
限制條件 Constraints
輸出格式 Format
評估標準 Criteria
```

例如：

```text
請先分析問題，再給解法。
```

這是輸出格式。

```text
請列出關鍵判斷依據。
```

這是限制條件。

```text
答案需要能讓我檢查每一步是否合理。
```

這是評估標準。

---

## 3. CoT 的作用

CoT 類提示適合用在複雜任務，例如：

```text
數學題
邏輯推理
程式碼除錯
架構設計
流程設計
需求分析
決策比較
```

例如你問：

```text
請幫我設計一個外匯交易審核流程。
```

AI 可能會直接給結果。

但如果你要求：

```text
請先拆解業務流程，再整理狀態，接著定義狀態流轉規則，最後給程式設計方向。
```

AI 的答案會更容易理解，也更符合開發思維。

---

## 4. CoT 範例

```text
我正在學習如何把銀行業務流程轉成程式設計。

請你幫我設計一個「外匯交易審核流程」。

請按照以下步驟回答：
1. 先描述真實業務流程
2. 再整理可能的狀態
3. 再定義狀態流轉規則
4. 再說明前端要如何根據狀態顯示畫面
5. 再說明後端要如何根據狀態控制流程
6. 最後給一個簡化版 Java 程式碼

限制條件：
1. 使用初學者能理解的方式
2. 不要一開始就講設計模式
3. 先講流程，再講程式
4. 程式碼要加註解

評估標準：
1. 能對應實際業務場景
2. 能幫我理解「流程 → 狀態 → 狀態流轉規則 → 程式」
3. 程式碼不要過度複雜
```

這裡的 CoT 技巧不是單純叫 AI「一步一步想」，而是明確規定 AI 要按照哪幾步拆解問題。

---

# 八、骨架與招式的整合範例

下面是一個結合七大元素與三種招式的完整 Prompt。

```text
你是一位資深 Java 後端工程師，也是一位擅長教初學者的技術講師。  // Role + Persona

我目前有兩年 Vue 前端經驗，正在學 Java 8，後端經驗較少。
我想理解如何把真實業務流程轉成後端程式設計。  // Context

請你幫我設計一個「銀行報表匯出流程」的後端範例。  // Task

請遵守以下限制：
1. 使用 Java 8
2. 不要使用 Spring Boot
3. 可以使用簡化版程式碼
4. 程式碼需要加註解
5. 不要一開始就套設計模式，先讓我理解流程  // Constraints

請按照以下格式回答：
1. 真實業務流程
2. 系統狀態設計
3. 狀態流轉規則
4. Java 類別設計
5. 完整範例程式碼
6. 可以如何重構成更好維護的版本  // Format + CoT

請參考以下說明風格：
- 先講白話概念
- 再對應到實際業務
- 最後才給程式碼  // Examples

好的答案需要符合：
1. 初學者能理解
2. 能對應實際專案
3. 程式碼好維護
4. 能幫我理解「流程 → 狀態 → 程式」  // Criteria
```

這個 Prompt 裡面：

```text
Persona：
資深 Java 後端工程師、技術講師

Few-Shot：
提供想要的說明風格

CoT：
要求按照流程、狀態、規則、設計、程式碼的順序拆解
```

這就是：

```text
七大元素是骨架
進階技巧是招式
```

---

# 九、常見錯誤

## 錯誤一：只有任務，沒有上下文

不好的 Prompt：

```text
請幫我解釋 MyBatis。
```

問題是 AI 不知道你目前程度，也不知道你想用在哪裡。

比較好的 Prompt：

```text
我目前正在學 Java Web，已經了解 JDBC，但剛開始接觸 MyBatis。

請你用初學者能理解的方式解釋 MyBatis 解決什麼問題，
並用「原生 JDBC → MyBatis」的差異來說明。
```

---

## 錯誤二：角色太空泛

不好的 Prompt：

```text
你是世界頂尖工程師。
```

比較好的 Prompt：

```text
你是一位熟悉 Java 8、Spring MVC、MyBatis、銀行內部系統的資深後端工程師。
```

角色越具體，回答越容易對準需求。

---

## 錯誤三：要求一步一步思考，但沒有指定步驟

普通 Prompt：

```text
請一步一步思考。
```

比較好的 Prompt：

```text
請按照以下順序分析：
1. 先判斷問題類型
2. 再指出可能原因
3. 再提供解法
4. 最後給完整程式碼
```

這樣比單純要求「一步一步思考」更穩定。

---

## 錯誤四：沒有評估標準

不好的 Prompt：

```text
請幫我優化這段程式碼。
```

問題是「優化」可能代表很多意思：

```text
效能優化
可讀性優化
可維護性優化
架構優化
語法簡化
```

比較好的 Prompt：

```text
請幫我優化這段 Java 程式碼。

優先順序如下：
1. 可讀性
2. 可維護性
3. 可擴充性
4. 效能

請不要為了效能犧牲可讀性。
```

---

# 十、實務結論

提示工程可以分成兩層：

```text
第一層：骨架
角色 + 任務 + 上下文 + 限制條件 + 輸出格式 + 範例 + 評估標準

第二層：招式
CoT、Few-Shot、Persona
```

骨架解決的是：

```text
Prompt 是否完整
AI 是否知道要做什麼
AI 是否知道背景與限制
AI 是否知道答案要長什麼樣子
```

招式解決的是：

```text
回答是否更穩定
推理是否更清楚
格式是否更一致
角色角度是否更精準
```

所以可以用一句話總結：

```text
七大元素是 Prompt 的結構骨架；
CoT、Few-Shot、Persona 是讓這個骨架更好用的技巧。
```

真正會使用 AI 的人，不只是問問題，而是會設計問題。

也就是：

```text
普通使用者：把問題丟給 AI

進階使用者：把背景、任務、限制、格式、標準一起交給 AI

高階使用者：根據任務難度，選擇合適的骨架與招式
```

提示工程的核心能力，就是把模糊需求整理成清楚指令。

這和寫程式很像：

```text
需求越清楚，程式越好設計。
Prompt 越清楚，AI 越容易給出好答案。
```
