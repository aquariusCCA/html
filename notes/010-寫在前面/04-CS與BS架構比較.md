# C/S 與 B/S 架構：為什麼網頁不用安裝就能用

## 學習目標

- 分辨 C/S 架構與 B/S 架構的核心差異。
- 理解 B/S 架構的三層模型（表現層、邏輯層、數據層）。
- 知道接下來要學的 HTML，在 B/S 架構中對應哪一層。

## 問題情境

你可能用過需要先「安裝」才能用、並且會連線到服務端的桌面客戶端，例如 QQ；也用過打開瀏覽器、輸入網址就能直接使用的服務，例如各種網站。這背後其實對應兩種不同的軟體架構：C/S 架構與 B/S 架構。接下來要學的 HTML，主要是為哪一種架構服務的？

## 一句話理解

C/S 架構需要先安裝專屬的客戶端程式才能使用；B/S 架構只要有瀏覽器就能使用，大部分邏輯都交給伺服器處理。

## C/S 架構：需要安裝客戶端

C/S（Client/Server，客戶端/伺服器）架構的軟體有一個特點：使用者要用之前，需要先下載並安裝一個專屬的客戶端程式，並透過這個客戶端連線到服務端。常見例子有 QQ 這類桌面客戶端軟體。

要注意的是，C/S 架構下的客戶端**不只是負責顯示畫面**，它通常也會處理一部分運算與業務邏輯——也就是說，客戶端也分擔了一部分原本該由伺服器做的事。

![C/S 架構客戶機與服務器雙向請求示意圖](../../origin/010-寫在前面/assets/images/cs-bs-architecture-img-001-0f94cf.png)

**C/S 架構的優點**

- 客戶端可以提供較豐富的介面、操作體驗與本機能力。
- 在特定內網、專用設備或受控環境中，較容易配合權限與部署策略做安全控管（但實際安全性仍取決於整體設計與實作）。
- 客戶端可以承擔部分運算或快取資料，某些場景下回應速度較快（仍取決於網路、伺服器、客戶端與資料量）。

**C/S 架構的缺點**

- 需要安裝或更新專屬客戶端，部署與維護成本較高。
- 使用者群通常較固定；如果要面向大量不確定的使用者，安裝門檻會比較高。
- 一旦需要升級，所有客戶端的程式都要跟著更新，維護成本高。

## B/S 架構：只要有瀏覽器就能用

B/S（Browser/Server，瀏覽器/伺服器）架構，使用者不需要特別安裝任何程式，只要有 Web 瀏覽器即可使用。**這也正是接下來要學習的前端開發所對應的架構**：大部分業務邏輯交給伺服器（後台）實現，前端主要負責資料渲染、發送請求等相對較少的邏輯。

B/S 架構常可用三層模型來理解：

![B/S 架構界面、邏輯與數據層示意圖](../../origin/010-寫在前面/assets/images/cs-bs-architecture-img-002-fc4d53.png)

- **表現層**：負責使用者介面、互動與結果展示——也就是接下來要學的 HTML/CSS/JavaScript 在瀏覽器中呈現的部分。
- **邏輯層**：主要在伺服器端處理業務邏輯、請求與回應。
- **數據層**：負責資料儲存、查詢與管理，通常由伺服器端的邏輯層存取。

**B/S 架構的優點**

- 客戶端無需安裝，有 Web 瀏覽器即可使用。
- 可以直接部署在廣域網路上，透過權限控制讓多個使用者訪問，互動性較強。
- 升級只需更新伺服器，不需要逐一升級每個客戶端，可以隨時更新版本而不必讓使用者重新下載。

**B/S 架構的缺點**

- 在不同瀏覽器之間，顯示與行為的一致性不一定理想（這正是「瀏覽器與渲染引擎」「Web 標準與三層分離」要進一步討論的問題）。
- 要做到接近 C/S 程式那樣豐富的操作體驗，需要花費不少設計與開發成本。
- 在速度與安全性上，往往需要投入較大的設計成本，是 B/S 架構常見的挑戰之一。

## 對照表

| 比較項目 | C/S 架構 | B/S 架構 |
| --- | --- | --- |
| 是否需要安裝 | 需要安裝專屬客戶端 | 不需要，有瀏覽器即可 |
| 邏輯分布 | 客戶端也承擔部分運算與邏輯 | 主要邏輯在伺服器，瀏覽器端較輕量 |
| 升級方式 | 需逐一更新每個客戶端 | 只需更新伺服器 |
| 跨平台/跨裝置 | 通常需要為不同平台分別開發/安裝 | 只要有瀏覽器即可使用 |
| 適合情境 | 需要控制安裝對象、效能要求較高的內部系統 | 面向大量、不確定的一般使用者 |

> 補充：以上的優缺點並非絕對規則。實際上，大量運算可以放在伺服器端、原生客戶端，或瀏覽器端，架構的選擇取決於效能、安全性、資料存放位置、授權控制與使用體驗等多方面需求。例如有些軟體需要嚴格控制「哪些電腦可以安裝或執行」，這類情境就可能更適合 C/S 架構。

## 對 HTML 學習的實務意義

接下來要學的 HTML（之後還會加上 CSS、JavaScript），正是 B/S 架構中**表現層**的具體實作。瀏覽器會把寫好的 HTML/CSS/JavaScript 解析、渲染成使用者看到的畫面——這也是為什麼「前端開發」幾乎等於「B/S 架構下的表現層開發」。理解這一點，有助於之後判斷：哪些工作該由 HTML/CSS/JS（表現層）負責，哪些屬於伺服器端（邏輯層、數據層）的範疇。

## 常見錯誤

- **以為 B/S 架構一定比 C/S 架構「更好」或「更差」**：兩者是針對不同需求的架構選擇，沒有絕對的優劣，需要依據情境（使用者規模、是否需要控制安裝、效能與安全需求等）來判斷。
- **以為前端（表現層）完全不需要處理任何邏輯**：B/S 架構強調主要邏輯在伺服器端，但前端仍會有少量邏輯，例如表單輸入檢查、畫面上的互動效果——這些之後會在 JavaScript 的學習中遇到。

## 重點整理

- C/S 架構：需要安裝客戶端，客戶端也分擔部分運算與邏輯，適合受控環境或需要豐富本機操作的場景。
- B/S 架構：只需瀏覽器，邏輯主要在伺服器，可拆解為表現層、邏輯層、數據層三層。
- 接下來要學的 HTML/CSS/JavaScript，對應 B/S 架構中的「表現層」。
