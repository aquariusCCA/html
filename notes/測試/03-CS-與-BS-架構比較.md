# C/S 與 B/S 架構比較

## 學習目標

讀完這篇筆記，你應該能夠：

- 說明 C/S 架構與 B/S 架構分別是什麼。
- 比較兩種架構在安裝、更新、運算分工與使用情境上的差異。
- 理解 B/S 架構為什麼與 Web 前端學習密切相關。
- 避免把「瀏覽器應用」簡化成「沒有邏輯的畫面」。

## 問題情境

有些軟體需要下載安裝，例如桌面版 Office、某些聊天軟體或企業內部工具。有些系統只要打開瀏覽器就能使用，例如網站後台、線上文件、購物網站或雲端服務。

兩者背後常見的架構思路不同：

- 需要安裝專用客戶端的，常被歸類為 C/S 架構。
- 主要透過瀏覽器使用的，常被歸類為 B/S 架構。

理解這組差異，可以幫助你知道 Web 前端是在什麼架構背景下工作。

## 一句話理解

C/S 架構依賴專用客戶端與服務器協作；B/S 架構把瀏覽器當作主要客戶端，使用者通常不需要額外安裝專用程式。

## C/S 架構

C/S 是 Client/Server，也就是客戶端 / 服務器架構。

這類軟體通常需要先下載並安裝客戶端。使用者透過這個客戶端操作系統，客戶端再和服務器交換資料。

![C/S 架構客戶機與服務器雙向請求示意圖](../../origin/第01章_寫在前面/assets/images/cs-bs-architecture-img-001-0f94cf.png)

C/S 架構中的客戶端通常不只是顯示畫面，也可能負責：

- 複雜操作界面。
- 本機資料處理。
- 部分業務邏輯。
- 離線能力。
- 硬體或系統能力整合。

因此在某些高運算、需要本機能力、需要特殊授權控管或企業內部受控環境中，C/S 架構仍然有優勢。

## C/S 架構的優點

| 優點 | 說明 |
|---|---|
| 操作體驗可高度客製 | 專用客戶端可以設計更複雜的界面與本機互動 |
| 可整合本機能力 | 比較容易使用本機檔案、硬體、系統權限或離線資料 |
| 部分場景響應快 | 客戶端可承擔運算，減少每次操作都依賴遠端回應 |
| 受控環境管理明確 | 在企業內部或特定設備中，客戶端版本與權限可被集中控管 |

這些優點不是說 C/S 一定比 Web 快或安全，而是它在部署模式與本機能力整合上有不同條件。

## C/S 架構的缺點

| 缺點 | 影響 |
|---|---|
| 需要安裝 | 使用者必須先取得並安裝客戶端 |
| 更新成本高 | 客戶端版本需要更新，可能出現使用者版本不一致 |
| 跨平台成本高 | Windows、macOS、Linux 或行動系統可能需要不同版本 |
| 不適合未知大量使用者 | 面向公開使用者時，安裝門檻可能降低使用意願 |

如果每次升級都要讓大量使用者更新客戶端，維護成本會變高。

## B/S 架構

B/S 是 Browser/Server，也就是瀏覽器 / 服務器架構。

使用者通常不需要安裝專用客戶端，只要有 Web 瀏覽器就能使用系統。網站、後台管理系統、線上文件、雲端工具，都常見於這種模式。

B/S 架構與前端學習關係密切，因為瀏覽器端的頁面通常由 HTML、CSS、JavaScript 組成。

## B/S 架構的三層結構

與傳統 C/S 架構相比，B/S 架構常被理解成三層：

![B/S 架構界面、邏輯與數據層示意圖](../../origin/第01章_寫在前面/assets/images/cs-bs-architecture-img-002-fc4d53.png)

| 層級 | 常見責任 | 對前端學習的意義 |
|---|---|---|
| 表現層 | 顯示畫面、接收使用者操作、呈現結果 | HTML、CSS、JavaScript 的主要工作區 |
| 邏輯層 | 處理業務規則、權限、流程與 API | 前端常透過請求與它互動 |
| 數據層 | 儲存、查詢與管理資料 | 通常不直接由瀏覽器存取 |

入門時可以先理解：瀏覽器負責表現層，服務器負責主要邏輯與資料存取。不過現代前端也會承擔不少互動邏輯，不能把前端理解成完全沒有邏輯。

## B/S 架構的優點

| 優點 | 說明 |
|---|---|
| 使用門檻低 | 有瀏覽器即可訪問，不必安裝專用客戶端 |
| 更新方便 | 多數更新在服務器端或網站資源更新，使用者重新整理即可取得新版本 |
| 適合公開訪問 | 可透過域名與權限控制服務大量使用者 |
| 跨平台能力好 | 不同作業系統只要有相容瀏覽器，就有機會使用同一套系統 |

這也是 Web 應用普及的重要原因：使用者取得服務的成本低，開發者更新系統的速度也快。

## B/S 架構的缺點

| 缺點 | 影響 |
|---|---|
| 受瀏覽器差異影響 | 不同瀏覽器與版本可能有呈現或功能差異 |
| 複雜互動成本高 | 要做到接近原生桌面軟體的體驗，需要更多前端工程能力 |
| 依賴網路與服務端 | 網路品質、服務器性能與 API 設計會影響體驗 |
| 安全與性能需完整設計 | 權限、資料保護、請求控制與前端效能都需要被考慮 |

B/S 不是「簡單版軟體」，而是把很多能力放在瀏覽器、Web 標準、服務器與網路協作中完成。

## C/S 與 B/S 對照

| 比較項目 | C/S 架構 | B/S 架構 |
|---|---|---|
| 全稱 | Client/Server | Browser/Server |
| 使用入口 | 專用客戶端 | Web 瀏覽器 |
| 是否需要安裝 | 通常需要 | 通常不需要 |
| 更新方式 | 更新客戶端與服務端 | 多數情況更新服務端或網站資源 |
| 客戶端能力 | 可深度整合本機能力 | 受瀏覽器與 Web API 限制 |
| 常見場景 | 桌面軟體、企業內部系統、重度本機工具 | 網站、Web App、後台系統、雲端服務 |
| 前端關聯 | 不一定使用 Web 前端 | HTML、CSS、JavaScript 是核心 |

## 實務判斷

選擇架構時，不是單純問哪個比較新，而是看需求。

比較適合 C/S 的情境：

- 需要大量本機運算或離線能力。
- 需要深度整合硬體或作業系統能力。
- 使用者群固定，安裝與更新可被管理。
- 有特殊授權或企業內部管控需求。

比較適合 B/S 的情境：

- 希望使用者打開瀏覽器即可使用。
- 需要快速更新與公開訪問。
- 使用者設備與作業系統差異大。
- 主要功能可以透過 Web 技術與服務器完成。

## 常見錯誤與混淆

### 以為 C/S 只能用在局域網

C/S 架構也可以用於網際網路服務。重點是是否使用專用客戶端與服務器協作，不是網路範圍一定只能在局域網。

避免方式：用「是否需要專用客戶端」與「客戶端 / 服務器分工」來判斷，而不是只看網路位置。

### 以為 B/S 的前端只負責顯示資料

B/S 中瀏覽器端確實常負責表現層，但現代前端也會處理路由、表單驗證、狀態管理、即時互動與資料請求。

避免方式：把 B/S 前端理解為「在瀏覽器中運行的應用層」，而不是靜態畫面。

### 以為 B/S 一定比 C/S 更好

B/S 使用方便、更新快速，但在重度本機能力、離線能力、硬體整合或特殊效能需求中，C/S 仍可能更合適。

避免方式：根據需求判斷架構，不要用流行程度判斷優劣。

## 對 HTML 學習的意義

HTML 學習主要發生在 B/S 架構的表現層。你寫的 HTML 會在瀏覽器中被解析，和 CSS、JavaScript 一起組成使用者看到與操作的界面。

理解 C/S 與 B/S 後，你會更容易知道：

- 為什麼學前端要關心瀏覽器。
- 為什麼網站不用安裝也能使用。
- 為什麼前端需要和後端 API 溝通。
- 為什麼跨瀏覽器差異會是 Web 開發問題。

## 重點整理

- C/S 是 Client/Server，通常需要安裝專用客戶端。
- B/S 是 Browser/Server，通常透過瀏覽器訪問。
- B/S 常分為表現層、邏輯層、數據層。
- HTML、CSS、JavaScript 主要工作在 B/S 的瀏覽器端。
- 兩種架構沒有絕對優劣，要依使用情境判斷。
