# C/S 架構與 B/S 架構比較

## 學習目標

讀完這篇筆記，你應該能夠：

- 理解 C/S 架構與 B/S 架構的基本差異。
- 說明 B/S 架構的三層結構。
- 判斷兩種架構在安裝、更新、互動與維護上的差異。

## 問題情境

有些軟體需要下載安裝，例如桌面版通訊軟體、辦公軟體或特定企業系統。有些服務只要打開瀏覽器就能使用，例如網站、後台系統或線上工具。

這背後常可以用 C/S 架構與 B/S 架構來理解。

## C/S 架構是什麼

C/S 是 Client/Server，也就是客戶端 / 服務器架構。

這類軟體通常需要先下載並安裝客戶端。使用時，客戶端會和服務器溝通，並可能在本機處理一部分運算、互動或業務邏輯。

常見例子包括傳統桌面軟體、需要安裝的企業系統、特定通訊軟體或高度依賴本機能力的應用。

![C/S 架構客戶機與服務器雙向請求示意圖](../../origin/第01章_寫在前面/assets/images/cs-bs-architecture-img-001-0f94cf.png)

## C/S 架構的特點

C/S 架構的客戶端不只是顯示畫面，它也可能處理大量互動、運算、資料快取、硬體整合或本機檔案操作。

因此，某些高運算、需要離線能力、需要硬體整合或需要特殊授權控管的場景，仍可能適合使用 C/S 架構。

## C/S 架構的優點

- 客戶端界面和操作可以做得很豐富。
- 較容易整合本機能力，例如檔案、裝置或特殊硬體。
- 在某些直接連線或受控環境中，回應速度可以很快。
- 在受控環境中，設備與使用者控管可能更集中。

安全性不會因為採用 C/S 架構就自動變好，仍取決於認證、授權、加密、更新與整體系統設計。

## C/S 架構的缺點

- 使用者通常需要安裝客戶端。
- 版本更新可能需要每台客戶端都同步升級。
- 維護成本較高。
- 不適合面向大量未知使用者的公開服務，因為安裝門檻較高。

## B/S 架構是什麼

B/S 是 Browser/Server，也就是瀏覽器 / 服務器架構。

廣義來看，B/S 仍然可以視為 Client/Server 思維的一種。這裡把 C/S 和 B/S 對照，是為了區分「專用客戶端」與「瀏覽器作為客戶端」這兩種常見型態。

B/S 架構的系統通常不需要使用者特別安裝專用客戶端，只要有 Web 瀏覽器就可以使用。前端主要在瀏覽器中呈現畫面、處理互動與發送請求，主要資料與業務邏輯通常由服務器端處理。

## B/S 架構的三層結構

和需要安裝專用客戶端的 C/S 架構相比，B/S 架構常用三層來理解：

![B/S 架構界面、邏輯與數據層示意圖](../../origin/第01章_寫在前面/assets/images/cs-bs-architecture-img-002-fc4d53.png)

| 層級 | 主要職責 |
| --- | --- |
| 表現層 | 處理使用者看到的畫面、互動與查詢結果輸出，通常由瀏覽器中的前端負責。 |
| 邏輯層 | 處理應用邏輯、請求流程、權限與業務規則，通常由服務器端負責。 |
| 數據層 | 負責資料儲存、查詢與管理，通常由邏輯層存取。 |

這種分層可以幫助系統把畫面、邏輯與資料管理分開，降低維護難度。

## B/S 架構的優點

- 使用者不需要安裝專用軟體，有瀏覽器即可使用。
- 適合放在廣域網上，讓不同地方的使用者訪問。
- 更新通常集中在服務器與前端資源，不需要使用者手動下載新版本。
- 適合公開網站、後台管理系統、線上工具等場景。

## B/S 架構的缺點

- 需要考慮不同瀏覽器與版本的相容性。
- 若要做到接近原生客戶端的體驗，可能需要更多前端工程與效能優化。
- 速度與安全性需要透過系統設計、快取、權限、加密與部署策略共同處理。

## C/S 與 B/S 對照

| 比較項目 | C/S 架構 | B/S 架構 |
| --- | --- | --- |
| 全稱 | Client/Server | Browser/Server |
| 使用入口 | 專用客戶端 | Web 瀏覽器 |
| 是否安裝 | 通常需要安裝 | 通常不需要安裝專用軟體 |
| 更新方式 | 多個客戶端都可能需要更新 | 通常集中更新服務器與網頁資源 |
| 本機能力 | 較容易深度整合 | 受瀏覽器能力與權限限制 |
| 常見場景 | 桌面軟體、企業內部系統、專用工具 | 網站、Web App、後台系統、線上服務 |

## 實務判斷

如果目標是讓大量使用者快速透過網址使用服務，B/S 架構通常更符合 Web 開發的方向。

如果應用需要大量本機能力、離線操作、特殊硬體整合或受控部署環境，C/S 架構仍有它的價值。

## 重點整理

- C/S 架構通常需要安裝客戶端，客戶端可能承擔較多運算與互動。
- B/S 架構透過瀏覽器訪問服務，適合 Web 系統與公開線上服務。
- B/S 架構常用表現層、邏輯層、數據層三層來理解。
- 兩種架構沒有絕對好壞，重點是看使用場景、維護成本與使用者需求。
