# contenteditable 屬性

> 所屬章節：第五章｜全局屬性
> 關鍵字：`contenteditable`、可編輯內容、行內編輯、rich-text editing、`true`、`false`、`plaintext-only`、繼承、HTML 全局屬性
> 建議回查情境：想知道怎麼讓元素內容可直接修改、想分清 `true` / `false` / `plaintext-only` 的差別、想理解為什麼沒寫值時有時還是能編輯、想知道 `contenteditable` 和表單欄位有什麼不同

## 本節導讀

`contenteditable` 是 HTML 的全局屬性之一。
它的作用是：**讓一般 HTML 元素的內容，可以直接在頁面上被使用者修改。**

例如，原本一個普通的 `<div>` 或 `<p>` 只是顯示文字；
加上 `contenteditable` 之後，使用者就能直接點進去輸入、刪除、修改內容。

但真正要學懂它，不能只停在「可編輯 / 不可編輯」這麼簡單。
這個屬性真正容易搞混的地方有三個：

1. 這個元素現在到底能不能編輯
2. 可編輯時，是一般可編輯，還是純文字可編輯
3. 如果自己沒寫值，會不會繼承父元素的可編輯狀態

你可以先記一句最核心的話：

> `contenteditable` 不是單純的開關，而是控制「元素內容是否可編輯、用哪種方式編輯、以及是否會繼承父層編輯狀態」的屬性。

## 30 秒複習入口

* `contenteditable` 可以讓一般 HTML 元素變成可直接編輯的區域。
* `contenteditable="true"`：可編輯，通常允許富文字行為。
* `contenteditable="false"`：不可編輯。
* `contenteditable="plaintext-only"`：可編輯，但只保留純文字。
* 如果元素沒寫這個屬性，或值無效，結果常會看父元素。
* 可編輯不代表資料已經保存，保存仍要靠 JavaScript 或後端處理。

## 速查區

### 核心概念

* `contenteditable` 控制的是：**元素內容能不能直接在頁面上被修改**
* 它常用在：

  * 行內編輯
  * 簡易筆記區
  * 留言草稿區
  * 富文字編輯器的基礎可編輯區

### 關鍵規則

* 它不是單純只有 `true` / `false`
* 至少要先掌握三種常見值：

  * `true`
  * `false`
  * `plaintext-only`
* 沒有設定或值無效時，常會繼承父元素的狀態
* 它讓元素進入「可編輯模式」，但**不會自動幫你保存資料**

### 常見混淆點

* `contenteditable` 不是表單元件
* `contenteditable="true"` 不等於你完成了一個真正可用的編輯器
* 可編輯不代表資料已提交或已儲存
* `plaintext-only` 的重點不是「也能編輯」，而是「不保留富文字格式」
* 父層可編輯時，子層就算沒寫，也可能跟著可編輯

### 一句話抓核心

* `contenteditable` 的重點不是只背 `true` 或 `false`，而是判斷：**這個元素是不是可編輯區、是什麼編輯模式、以及是否受父層影響。**

## 這篇在解決什麼問題？

有時候我們不想讓使用者另外打開表單欄位來修改資料，
而是希望他直接在畫面上點一下某段文字，就能開始編輯。

例如：

* 點標題直接改標題
* 點備註區直接修改備註
* 在頁面中直接輸入一段簡單內容

這時候，`contenteditable` 就很有用。

所以你可以把它理解成：

> 把一般元素變成「可以直接在頁面上編輯內容的區域」。

---

# 1. `contenteditable` 是什麼？

`contenteditable` 是 HTML 全局屬性，
用來控制某個元素的內容，是否可以被使用者直接修改。

例如：

```html
<p contenteditable="true">這段文字可以直接修改。</p>
```

這裡的 `<p>` 原本只是普通段落，
但加上 `contenteditable="true"` 後，使用者就可以直接點進去改字。

## 最白話的理解

你可以把它想成：

* 原本這只是一塊顯示內容的區域
* 加上 `contenteditable` 後，瀏覽器把它當成一塊「可直接編輯」的內容區

---

# 2. 它和 `<input>`、`<textarea>` 有什麼不同？

這是很容易混淆的地方。

## `<input>` / `<textarea>`

* 是原生表單輸入元件
* 天生就是拿來輸入資料的
* 有固定的輸入行為與表單語意

## `contenteditable`

* 不是表單欄位
* 它是讓**一般元素**變成可編輯
* 例如把 `<div>`、`<p>`、`<section>` 變成可直接改內容的區域

### 對照理解

* `<textarea>`：本來就是輸入框
* `contenteditable`：把普通元素暫時變成像可編輯區的東西

所以要記住：

> `contenteditable` 改變的是元素的編輯狀態，不是把元素變成真正的表單控制項。

---

# 3. `contenteditable` 不是只有 `true` 和 `false`

很多教材會先把它教成：

* `true`：可編輯
* `false`：不可編輯

這樣入門沒錯，但還不夠完整。

因為 `contenteditable` 更準確地說，是一種**列舉屬性**，
至少要先掌握三種常見值。

---

## 3-1. `contenteditable="true"`

表示元素內容可編輯。

```html
<div contenteditable="true">
  這裡可以直接修改。
</div>
```

### 這代表什麼？

* 使用者可以點進去輸入文字
* 可以刪除、修改內容
* 在許多情況下，貼上內容時也可能保留部分格式

所以你可以把 `true` 理解成：

> 進入一般可編輯模式，通常允許富文字相關行為。

---

## 3-2. `contenteditable="false"`

表示元素內容不可編輯。

```html
<p contenteditable="false">這段不能直接改。</p>
```

這種寫法有兩種常見理解方式：

* 單獨存在時：就是不可編輯
* 放在父層可編輯區裡時：可以把某塊內容排除在可編輯範圍外

---

## 3-3. `contenteditable="plaintext-only"`

表示元素內容可編輯，但只保留純文字。

```html
<div contenteditable="plaintext-only">
  這裡可以輸入，但只想保留純文字。
</div>
```

### 這個值最重要的地方是什麼？

不是「它也能編輯」，而是：

> 它重點在於不想保留富文字格式。

例如你從別的網站複製一段內容，裡面原本可能有：

* 粗體
* 顏色
* 連結
* 字型大小
* 背景色

貼到 `plaintext-only` 的可編輯區時，重點通常是：

* 只留下文字本身
* 不保留那些格式資訊

這對以下情境很有用：

* 備註欄
* 簡短說明區
* 留言輸入
* 不想處理富文字清理的地方

---

# 4. 沒有寫值，或值無效時，會怎樣？

這是最容易被忽略，但非常重要的一點。

如果元素本身：

* 沒有設定 `contenteditable`
* 或設定了無效值

結果通常**不是直接等於不可編輯**，
而是會看父元素是不是可編輯。

也就是說，它常有**繼承父層狀態**的行為。

---

## 例子：父層可編輯，子層沒寫

```html
<div contenteditable="true">
  這整塊可編輯
  <p>這個段落雖然沒寫，但通常也會跟著可編輯。</p>
</div>
```

### 這個例子在說什麼？

* 父層 `<div>` 是可編輯區
* 子層 `<p>` 自己沒寫 `contenteditable`
* 所以它通常會跟著父層一起進入可編輯狀態

所以判斷時不要只看自己，還要看父層。

---

# 5. 怎麼判斷某個元素到底能不能編輯？

可以用這個順序思考：

1. 先看元素自己有沒有寫 `contenteditable`
2. 如果是 `true`，它可編輯
3. 如果是 `false`，它不可編輯
4. 如果是 `plaintext-only`，它可編輯，但以純文字模式理解
5. 如果沒寫或值無效，再往父層看有沒有可編輯設定

你可以把它記成一句話：

> 先看自己，自己沒有明確答案，再看父層。

---

# 6. 常見例子

## 例子 1：最基本的可編輯與不可編輯

```html
<p contenteditable="true">這段文字可以直接修改。</p>
<p contenteditable="false">這段文字不能直接修改。</p>
```

### 重點

* 第一段可編輯
* 第二段不可編輯
* 這是最基本的入門例子

---

## 例子 2：純文字模式

```html
<div contenteditable="plaintext-only">
  在這裡輸入內容時，只想保留純文字。
</div>
```

### 重點

* 可以輸入、刪除、修改
* 但貼上時，通常不希望保留外部格式
* 這在很多「只收文字」的情境很實用

---

## 例子 3：父層可編輯、子層排除

```html
<div contenteditable="true">
  這整塊內容可以編輯，
  <span contenteditable="false">但這一小段不行。</span>
</div>
```

### 重點

* 父層整塊可編輯
* 子層明確寫 `false`
* 所以那一小段被排除在可編輯範圍外

這是實務上非常重要的觀念，
因為它代表 `contenteditable` 不只是簡單開關，還會影響巢狀區域的編輯範圍。

---

## 例子 4：父層可編輯、子層未設定

```html
<div contenteditable="true">
  外層可編輯
  <span>這段通常也會跟著可編輯。</span>
</div>
```

### 重點

* 子層沒有明寫
* 所以常會繼承父層狀態
* 這就是為什麼你不能只看某個元素自己有沒有寫值

---

# 7. `contenteditable` 不等於資料已經保存

這是最需要特別提醒的地方之一。

```html
<div contenteditable="true">這裡可以改</div>
```

這只能表示：

* 使用者可以改內容

但**不代表**：

* 資料已經存進資料庫
* 資料已經送到後端
* 重整頁面後內容一定還在

也就是說：

> 可編輯，只是讓內容能被改；
> 至於要不要保存、怎麼保存，還要另外用 JavaScript、表單送出或 API 處理。

---

# 8. `contenteditable="true"` 不等於完整編輯器完成

這也是初學者很容易誤判的地方。

很多人看到元素能輸入，就以為自己已經做出編輯器了。
其實還差很遠。

真正可用的編輯器，往往還要處理很多事情，例如：

* 貼上內容的清理
* 富文字格式控制
* 游標與選取範圍
* Enter 換行行為
* 資料儲存
* 驗證與安全性處理
* 內容同步到後端

所以：

> `contenteditable` 只是讓編輯成為可能，
> 不是自動幫你完成整套編輯器功能。

---

# 9. 常見混淆點 / 易錯點

## 錯誤一：把它當成普通布林屬性

不夠完整。

因為除了 `true`、`false`，還要知道：

* `plaintext-only`
* 以及沒寫值時可能會看父層

---

## 錯誤二：以為可編輯就等於已儲存

不對。

可編輯只是畫面上的互動能力，
資料保存仍是另外一件事。

---

## 錯誤三：以為它只給 `<input>` 或 `<textarea>` 用

不對。

它更常見的用途，反而是讓一般元素可編輯。

例如：

* `<div>`
* `<p>`
* `<section>`
* `<span>`

---

## 錯誤四：忽略巢狀結構的影響

如果父層可編輯，子層沒寫值，
子層常常也會跟著進入可編輯狀態。

如果子層明確寫 `false`，
又可能會變成局部不可編輯區。

所以判斷時不能只看單一元素。

---

# 10. 一句話總結

`contenteditable` 的本質是：

> 讓一般 HTML 元素的內容可以直接在頁面上編輯，並且透過不同值控制可編輯與否、純文字模式，以及是否受父層狀態影響。
