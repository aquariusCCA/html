# 兼容性問題

> 所屬章節：第三章｜兼容性問題  
> 關鍵字：瀏覽器兼容性、相容性、`X-UA-Compatible`、`IE=Edge`、`renderer`、雙核瀏覽器、`webkit`、`Trident`、`html5shiv`、HTML5 Shiv、條件註解、Conditional Comment、舊版 IE  
> 建議回查情境：看到舊專案中的 `X-UA-Compatible`、`renderer`、`html5shiv` 不知道在做什麼；想理解舊版 `IE` 為什麼常常需要額外處理；想知道這些做法在現代前端還是否重要

---

## 本節導讀

這一節介紹的是**早期前端常見的瀏覽器兼容性處理方式**。

在舊時代的前端開發中，不同瀏覽器，尤其是不同版本的 `Internet Explorer`，對 `HTML`、`CSS`、新標籤和頁面渲染規則的支援差異很大。  
因此，同一份頁面在不同瀏覽器中，可能會出現：

- 顯示結果不一樣
- 某些樣式失效
- 新標籤無法正常辨識
- 瀏覽器使用了不預期的渲染模式

這篇的重點，就是理解幾種在舊專案、舊教材、歷史文章裡很常看到的相容性做法，包括：

- `X-UA-Compatible`
- `renderer`
- `html5shiv`
- `IE Conditional Comment`

你要先有一個整體觀念：

> 這些做法大多是為了解決**舊版瀏覽器**的問題，尤其是舊版 `IE`。  
> 它們不是現代前端專案的日常必備設定，而是偏**歷史性相容方案**。

補充：本文中的「兼容性」與「相容性」視為同義，這裡依章節命名保留「兼容性問題」這個標題。

---

## 你會在這篇學到什麼

- `X-UA-Compatible` 是什麼？它在控制什麼？
- `renderer` 這個 `meta` 標籤想解決什麼問題？
- 為什麼舊版 `IE` 需要 `html5shiv`？
- `IE Conditional Comment` 是什麼？常見判斷詞怎麼看？

---

## 30 秒快速掌握

- `X-UA-Compatible`：控制**舊版 IE 用哪種文件模式解讀頁面**
- `renderer`：提示**某些雙核瀏覽器優先用哪個渲染核心**
- `html5shiv`：讓**舊版 IE 能辨識 HTML5 新標籤**
- `IE Conditional Comment`：只讓**特定版本的 IE** 載入某些程式碼

你可以先這樣記：

> `X-UA-Compatible` 看的是 **IE 怎麼解讀頁面**  
> `renderer` 看的是 **瀏覽器想用哪個核心畫頁面**  
> `html5shiv` 解決的是 **舊版 IE 不認得 HTML5 新標籤**

---

## 速查區

### 核心概念

這篇主要在處理三種舊時代常見的兼容性問題：

1. **舊版 IE 的文件模式問題**
2. **雙核瀏覽器可能切換不同渲染核心**
3. **舊版 IE 不認得 HTML5 語意化標籤**

---

### 一句話對比

- `X-UA-Compatible`：管的是 **舊版 IE 用什麼文件模式解析頁面**
- `renderer`：管的是 **部分雙核瀏覽器傾向使用哪個核心渲染頁面**
- `html5shiv`：補的是 **舊版 IE 對 HTML5 新標籤的辨識能力**

---

### 常見錯誤觀念

- 看到這些標籤，就以為現代前端專案都還必須加上。
- 把 `renderer` 當成所有瀏覽器都支援的標準能力。
- 把 `html5shiv` 當成所有瀏覽器都需要的通用補丁。
- 把 `X-UA-Compatible` 跟 `renderer` 混為一談，以為它們在做同一件事。

---

## 這篇在解決什麼問題？

早期瀏覽器對網頁標準的支援並不一致，尤其是 `IE6`、`IE7`、`IE8`、`IE9` 等版本，常常會有以下問題：

- 同一份 HTML，在不同版本 IE 中解析方式不同
- 瀏覽器可能進入舊的相容模式，而不是較新的標準模式
- HTML5 新增的語意化標籤無法被舊版 IE 正常辨識
- 某些雙核瀏覽器可能會切換不同渲染核心，導致畫面表現不同

因此，早期前端工程師常會加上一些額外設定，盡量讓頁面在舊環境中更穩定。

---

# 1. 添加 Meta 資訊，讓瀏覽器使用較合適的渲染方式

## 1.1 `X-UA-Compatible`：讓 IE 使用較新的文件模式

`X-UA-Compatible` 是一個 `meta` 標籤，主要用來告訴 **舊版 Internet Explorer**：

> 這個頁面應該用哪種文件模式來解讀與渲染。

### 為什麼需要它？

舊版 `IE` 在不同文件模式下，對 `HTML`、`CSS`、盒模型、排版與某些標準的支援會不一樣。  
有些情況下，瀏覽器可能會因為環境或設定，退回比較舊的相容模式，導致頁面顯示異常。

因此，早期專案常透過 `X-UA-Compatible` 提示 IE：

> 盡量使用你目前可用的**最新文件模式**來解析頁面。

最常見的寫法是：

```html
<meta http-equiv="X-UA-Compatible" content="IE=Edge">
````

### `IE=Edge` 是什麼意思？

這裡的 `Edge` 不是現在的 Microsoft Edge 瀏覽器名稱。
它的意思是：

> 讓 IE 使用它能提供的**最高等級文件模式**來渲染頁面，而不要退回較舊的相容模式。

### 用白話理解

你可以把它想成：

> `X-UA-Compatible` 是在告訴舊版 IE：
> 「請用你比較新的規則來看這個頁面，不要用太舊的方式解讀。」

### 補充說明

* 這是針對**舊版 IE** 的歷史性相容方案
* 現代主流瀏覽器通常不理會它
* 若要使用，通常會建議盡量放在 `head` 前段，避免 IE 太晚讀到

---

## 1.2 `renderer`：提示雙核瀏覽器優先使用哪個渲染核心

某些早期常見的瀏覽器，特別是部分國內雙核瀏覽器，可能同時具備兩套渲染核心，例如：

* `WebKit`
* `IE Trident`

這代表同一個頁面進入瀏覽器後，可能會面臨一個問題：

> 到底要用哪一個核心來渲染這個頁面？

這時候，舊教材裡常見一個做法，就是加入：

```html
<meta name="renderer" content="webkit">
```

它的目的，是提示某些雙核瀏覽器：

> 優先使用 `WebKit` 核心來渲染頁面。

### 用白話理解

如果說 `X-UA-Compatible` 是在管：

> **IE 要用哪種規則來解讀頁面**

那麼 `renderer` 比較像是在提示：

> **某些雙核瀏覽器想優先用哪個核心來畫頁面**

### 要注意的地方

* `renderer` **不是通用的 HTML 標準**
* 它主要出現在特定瀏覽器的相容性語境中
* 不是所有瀏覽器都會理會這個標籤

所以看到它時，不要直接理解成「所有瀏覽器都支援的正式功能」，而要先聯想到：

> 這多半是在處理某些特定雙核瀏覽器的歷史兼容問題

---

## 1.3 `X-UA-Compatible` 和 `renderer` 的差別

這兩個東西很容易混，建議直接這樣記：

### `X-UA-Compatible`

重點是：

> **舊版 IE 要怎麼解讀頁面**

它處理的是**文件模式**問題。

### `renderer`

重點是：

> **某些雙核瀏覽器想用哪個核心渲染頁面**

它處理的是**渲染核心選擇**問題。

### 最簡單的記法

* `X-UA-Compatible`：偏向「**規則怎麼看**」
* `renderer`：偏向「**頁面怎麼畫**」

---

# 2. 使用 `html5shiv` 讓舊版 IE 認得 HTML5 新標籤

## 2.1 為什麼需要 `html5shiv`？

`HTML5` 引入了很多語意化標籤，例如：

* `header`
* `nav`
* `section`
* `article`
* `footer`

對現代瀏覽器來說，這些標籤很正常。
但對舊版 `IE`，特別是 `IE6`、`IE7`、`IE8` 而言，這些標籤可能是「未知元素」。

### 未知元素會有什麼問題？

問題不只是「看不懂名字」而已。
更麻煩的是：

* 瀏覽器可能無法正確建立這些元素
* 可能無法正常套用樣式
* 可能影響版面流程與區塊顯示

也就是說，舊版 IE 不認得這些標籤時，常會導致畫面表現不正常。

---

## 2.2 `html5shiv` 在做什麼？

`html5shiv` 是一段 JavaScript，作用是：

> 先幫舊版 IE 建立 HTML5 的新元素，讓瀏覽器能辨識它們，進一步比較正常地套用樣式與參與排版。

你可以把它想成一個「補認識新標籤」的工具。

### 白話版理解

不是 `header`、`section` 這些標籤不能寫，
而是舊版 IE 看到它們時會想：

> 「這是什麼？我不熟。」

`html5shiv` 的作用，就是先幫它補上這層認知，讓它比較知道怎麼處理這些元素。

---

## 2.3 `html5shiv` 的典型寫法

因為這個需求主要出現在舊版 `IE`，所以早期常搭配條件註解，只在特定版本 IE 載入：

```html
<!--[if lt IE 9]>
<script src="./js/html5shiv.min.js"></script>
<![endif]-->
```

這段的意思是：

> 如果瀏覽器版本是 **IE9 以下**，才載入 `html5shiv`

這樣可以避免其他不需要的瀏覽器也去載入這段舊相容程式碼。

# 3. IE Conditional Comment：舊版 IE 專用的條件註解

## 3.1 什麼是條件註解？

`IE Conditional Comment` 是舊版 `IE` 特有的語法機制。

* 對一般瀏覽器來說，它看起來像普通註解
* 對舊版 `IE` 來說，它會被當成「可以判斷版本條件的程式碼區塊」

因此，前端工程師可以利用它做到：

* 只在某些 IE 版本中載入 CSS
* 只在某些 IE 版本中載入 JS
* 只對某些 IE 版本顯示特定內容

---

## 3.2 為什麼常跟 `html5shiv` 一起出現？

因為 `html5shiv` 主要是給舊版 `IE` 用的。
如果讓所有瀏覽器都載入，不但沒必要，也會讓頁面多做一些不需要的事。

所以常見做法是：

> 用條件註解把 `html5shiv` 限定在舊版 IE 中才載入

這就是為什麼早期教材很常把兩者放在一起講。

---

## 3.3 常見判斷詞

### 對照表

* `lt`：less than，小於
* `lte`：less than or equal，小於等於
* `gt`：greater than，大於
* `gte`：greater than or equal，大於等於
* `!`：not，非

### 記憶方式

你可以把它們想成「版本比較運算子」：

* `lt / lte`：看的是「比它小」
* `gt / gte`：看的是「比它大」
* `!`：排除某個條件

---

## 3.4 範例

```html
<!--[if IE 8]>僅 IE8 可見<![endif]-->
<!--[if gt IE 8]>僅 IE8 以上可見<![endif]-->
<!--[if lt IE 8]>僅 IE8 以下可見<![endif]-->
<!--[if gte IE 8]>IE8 以上可見<![endif]-->
<!--[if lte IE 8]>IE8 及以下可見<![endif]-->
<!--[if !IE 8]>非 IE8 的 IE 可見<![endif]-->
```

# 4. 常見混淆點整理

## 混淆 1：`X-UA-Compatible` 和 `renderer` 是一樣的嗎？

不是。

* `X-UA-Compatible` 解決的是 **IE 文件模式**
* `renderer` 解決的是 **部分雙核瀏覽器的核心選擇**

它們關心的問題不同。

---

## 混淆 2：`html5shiv` 是不是所有瀏覽器都要用？

不是。

它主要是為了解決 **舊版 IE 對 HTML5 新標籤的支援問題**。
現代瀏覽器通常不需要這種補丁。

---

## 混淆 3：條件註解是不是一般 HTML 註解的進階版？

不是。

它不是通用 HTML 機制，而是**舊版 IE 的特殊能力**。
離開那個歷史語境，它就不是現代前端的常用方案。

---

## 混淆 4：看到這些寫法，代表這是現代前端必備設定嗎？

通常不是。

如果你在專案裡看到這些東西，很多時候代表：

* 你正在看舊專案
* 你正在看舊教材
* 你正在處理歷史遺留的相容性需求

---

# 5. 總結

這篇介紹的三個重點，可以這樣收斂：

* `X-UA-Compatible`：讓**舊版 IE** 盡量用較新的文件模式解析頁面
* `renderer`：提示**部分雙核瀏覽器** 優先用哪個核心渲染頁面
* `html5shiv`：讓**舊版 IE** 能辨識 HTML5 新標籤，並較正常地套用樣式與參與排版
* `IE Conditional Comment`：只讓**特定版本的 IE** 載入某些兼容性程式碼

你不用把它們當成現代前端必背配置，
但當你閱讀舊專案、舊文章、老教材時，理解這些概念會很有幫助。
