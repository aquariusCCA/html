details / summary：三種常見錯誤對照

每一組都是「錯誤寫法」對「正確寫法」。實際展開試試看， 對照畫面差異與旁邊的「錯在哪」說明。

錯誤 1：只寫 details，沒有 summary

✗ 錯誤寫法

這是一段補充說明。

<details>
  <p>這是一段補充說明。</p>
</details>

錯在哪：沒有 <summary>，瀏覽器會自動補一個預設標題 （通常顯示「Details」或「詳細資料」）。這個標題不一定符合內容，讀者也難以理解。

✓ 正確寫法

補充說明

這是一段補充說明。

<details>
  <summary>補充說明</summary>
  <p>這是一段補充說明。</p>
</details>

明確提供 <summary>，標題語意清楚。

錯誤 2：summary 放在 details 外面

✗ 錯誤寫法

文章概要

文章詳細內容。

<summary>文章概要</summary>
<details>
  <p>文章詳細內容。</p>
</details>

錯在哪：<summary> 不在 <details> 裡， 就不再是這組可展開內容的摘要標題，點它也不會控制展開；而 details 因為少了 summary， 又會回到自動補預設標題的情況。

✓ 正確寫法

文章概要

文章詳細內容。

<details>
  <summary>文章概要</summary>
  <p>文章詳細內容。</p>
</details>

<summary> 放在 <details> 裡，且為第一個子元素。

錯誤 3：用 details 硬做複雜互動元件

✗ 不適合

把主導覽選單、分頁切換或需要記錄狀態的操作面板，全部塞進巢狀 details：

主選單
產品
分類 A

巢狀越疊越深，焦點管理與鍵盤操作都會出問題。

錯在哪：<details> 只適合「一個標題 + 一段可選擇查看的內容」。 多層級選單、分頁、需要動畫與焦點控制的元件，硬用 details 會難以維護也不易操作。

✓ 適合

單純的「常見問題」展開，正是 details 的拿手情境：

運費怎麼計算？

滿額免運，未滿額酌收固定運費。

可以退貨嗎？

商品到貨七天內可申請退貨。

內容是「標題＋可選擇查看的詳細資訊」，用 <details> 既簡單又語意清楚。 複雜互動則改用完整的 HTML + CSS + JavaScript 設計。
