# 相容性寫法判斷小工具

## 1. Demo 目標

這個 demo 用來示範：

- 相容性寫法應根據專案瀏覽器支援範圍決定。
- 何時可能需要 `X-UA-Compatible`、`renderer`、`html5shiv`。
- 為什麼新專案通常不應無條件照抄舊版 IE 相容程式碼。

  - 維護舊系統

  - 根據支援範圍決定相容策略

## 3. 如何執行

方式一：直接打開檔案

1. 找到 `index.html`
2. 使用瀏覽器開啟
3. 勾選不同專案條件，觀察建議與 HTML 片段如何改變

方式二：使用 VS Code Live Server

1. 在 demo 資料夾上按右鍵
2. 選擇 `Open with Live Server`
3. 在瀏覽器中查看結果

## 4. 觀察重點

1. 勾選「需要支援 IE8 或更舊版本」後，建議會加入 IE 相關相容處理。
2. 同時需要支援 IE8 且使用 HTML5 語意化元素時，才建議加入 `html5shiv` 條件註解。
3. `renderer` 只在目標環境包含特定雙核瀏覽器時才有討論價值。

## 5. 程式碼重點

### HTML

- 使用 checkbox 表示專案條件。
- 使用 `pre code` 顯示建議 HTML 片段。

### CSS

- 樣式只用來分隔條件、建議與程式碼區塊。

### JavaScript

- `renderDecision()` 根據勾選條件產生建議與 HTML 片段。
- 勾選「只支援現代瀏覽器」時，會自動取消「需要支援 IE8 或更舊版本」。

## 6. 修改練習

請嘗試修改以下內容，觀察結果差異：

1. 新增一個「企業內部舊系統」條件，並讓建議文字更保守。
2. 修改 `renderer` 的建議文字，補上專案註解規則。
3. 將產生的 HTML 片段改成只輸出 `head` 區塊。

## 7. 常見錯誤

| 錯誤 | 可能造成的問題 | 修正方式 |
|---|---|---|
| 新專案照抄所有舊相容寫法 | 增加無用程式碼與維護成本 | 先確認專案瀏覽器支援範圍 |
| 移除舊程式碼前沒有測試 | 舊系統使用者可能遇到回歸問題 | 確認支援版本、客戶情境與測試覆蓋 |
| 以為 meta 可以解決所有相容問題 | CSS、JavaScript、API 支援仍可能不同 | 搭配 fallback、polyfill 與實際測試 |
