# 第四十四章｜結構化資料

> 章節定位：理解網頁如何用「機器可讀的資料格式」補充描述頁面內容，讓搜尋引擎與資料工具更容易理解頁面中的文章、商品、課程、活動等資訊。  
> 一句話心智模型：**結構化資料不是拿來顯示畫面的 HTML，而是把頁面上已經存在的內容，整理成搜尋工具更容易讀懂的資料說明書。**  
> 建議閱讀方式：先理解 JSON-LD 放在哪裡、描述什麼，再比較它和語意化 HTML、`meta`、Open Graph 的差異。

## 本章在學什麼

這一章是 HTML 進階選修章，主軸是理解「結構化資料」在網頁中的角色。

重點不是保證 SEO 排名，也不是背完整 Schema.org 規格，而是理解：

- 結構化資料是給搜尋引擎、資料工具等機器讀的補充描述。
- JSON-LD 常透過 `<script type="application/ld+json">` 放進 HTML。
- JSON-LD 裡的內容應該描述頁面上真實、可見、存在的內容。
- Schema.org 提供常見實體類型與屬性，例如文章、商品、課程、活動、FAQ。
- 結構化資料可以輔助機器理解，但不能取代語意化 HTML。
- 加了結構化資料不代表一定會提升排名，也不保證搜尋結果會出現特殊樣式。

## 快速判斷：什麼時候需要結構化資料？

| 情境 | 是否適合使用 JSON-LD | 原因 |
|---|---:|---|
| 文章頁、部落格文章、新聞稿 | 適合 | 可以用 `Article` 描述標題、作者、發布日期 |
| 商品詳情頁 | 適合 | 可以用 `Product` 描述商品名稱、價格、評價等真實資訊 |
| 課程介紹頁 | 適合 | 可以用 `Course` 描述課程名稱、提供者等資訊 |
| 活動頁 | 適合 | 可以用 `Event` 描述活動時間、地點、名稱 |
| FAQ 頁 | 視情況適合 | 可以用 `FAQPage` 描述頁面上真的存在的問答 |
| 普通練習頁、單純排版頁 | 不一定需要 | 先把語意化 HTML、標題、內容結構做好更重要 |
| 頁面上沒有的評分、價格、評論 | 不應該使用 | 結構化資料不能偽造頁面不存在的內容 |

## 適合快速回查的主題

- 結構化資料是什麼
- JSON-LD 怎麼放進 HTML
- `type="application/ld+json"` 是什麼
- `@context` 和 `@type` 是什麼
- Schema.org 扮演什麼角色
- 結構化資料和語意化 HTML 差在哪
- 結構化資料和 `meta description`、Open Graph 差在哪
- 為什麼結構化資料必須和頁面可見內容一致
- 什麼情況需要 / 不需要加 JSON-LD
- 實務上怎麼檢查結構化資料是否有效

## 易混淆主題

- 結構化資料 vs 可見內容：前者給機器讀，後者給使用者讀，兩者應該描述同一件事。
- JSON-LD vs JavaScript 程式：JSON-LD 放在 `<script>` 裡，但本質是資料，不是頁面互動邏輯。
- Schema.org vs Google rich results：Schema.org 是通用詞彙表；Google 是否顯示特殊搜尋結果，還要看平台規則與頁面品質。
- SEO 保證 vs 搜尋理解：結構化資料有助於理解，不保證排名或特殊展示。
- 結構化資料 vs 語意化 HTML：前者是額外補充資料，後者是頁面本身的結構品質。

## 學完後你應該能做到

- 看懂 `<script type="application/ld+json">` 的用途。
- 判斷一段 JSON-LD 是描述文章、商品、課程還是活動。
- 說明 `@context`、`@type` 的基本意思。
- 判斷 JSON-LD 是否和頁面可見內容一致。
- 分辨語意化 HTML、`meta description`、Open Graph、JSON-LD 的差異。
- 知道加結構化資料前，應先確認頁面內容本身是完整、真實、可見的。
