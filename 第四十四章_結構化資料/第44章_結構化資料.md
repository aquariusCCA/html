> - 建議回查情境：想知道 JSON-LD 怎麼放進 HTML、結構化資料和 meta / Open Graph 差在哪、為什麼不能亂寫不存在的內容時

## 本節導讀

HTML 本身是給瀏覽器解析並呈現給使用者的文件。  
但有時候我們還希望搜尋引擎或其他工具更明確知道：

- 這頁是一篇文章。
- 這是產品名稱、價格和評分。
- 這是課程資訊。
- 這是活動時間。

結構化資料就是用機器可讀的格式補充描述頁面內容。

最常見方式之一是 JSON-LD：

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "HTML 學習筆記",
  "datePublished": "2026-04-26"
}
</script>
```

## 30 秒複習入口

- 結構化資料是給機器讀的補充描述。
- JSON-LD 常用 `<script type="application/ld+json">` 放進 HTML。
- 這個 `<script>` 裡是資料，不是要執行的 JavaScript 程式。
- Schema.org 提供常見類型與屬性詞彙。
- 結構化資料應該和頁面可見內容一致。
- 它有助於搜尋工具理解內容，但不保證排名或特殊展示。
- 它不能取代語意化 HTML。

## 正文筆記

### 1. JSON-LD 基本寫法

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "HTML 學習筆記",
  "author": {
    "@type": "Person",
    "name": "小明"
  },
  "datePublished": "2026-04-26"
}
</script>
```

這段放在 HTML 裡，但不是普通 JavaScript 程式。  
它是一段 JSON-LD 格式的資料。

`type="application/ld+json"` 告訴瀏覽器和工具：這裡是一段 linked data JSON。

### 2. `@context` 和 `@type`

`@context` 通常指定詞彙表：

```json
"@context": "https://schema.org"
```

可以先理解成：

> 下面這些欄位要用 Schema.org 的語彙來解釋。

`@type` 指定這個資料描述的是什麼類型：

```json
"@type": "Article"
```

如果是商品，可能是：

```json
"@type": "Product"
```

### 3. 結構化資料要和可見內容一致

不要在 JSON-LD 裡寫頁面上不存在或不符合事實的內容。

例如頁面上沒有評分，就不要硬寫：

```json
"aggregateRating": {
  "@type": "AggregateRating",
  "ratingValue": "5",
  "reviewCount": "999"
}
```

結構化資料是幫助機器理解頁面，不是用來偽造內容。

### 4. 結構化資料和語意化 HTML 差在哪？

語意化 HTML 是頁面結構本身：

```html
<article>
  <h1>HTML 學習筆記</h1>
  <time datetime="2026-04-26">2026 年 4 月 26 日</time>
</article>
```

JSON-LD 是額外的機器可讀描述：

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "HTML 學習筆記",
  "datePublished": "2026-04-26"
}
</script>
```

兩者可以互補。  
不要因為寫了 JSON-LD，就把 HTML 寫得沒有語意。

### 5. 結構化資料和 Open Graph 差在哪？

Open Graph 常用於社群分享預覽：

```html
<meta property="og:title" content="HTML 學習筆記">
<meta property="og:description" content="整理 HTML 常用標籤與語意。">
<meta property="og:image" content="https://example.com/cover.jpg">
```

JSON-LD 更像是描述頁面中的實體資料：

```json
{
  "@type": "Course",
  "name": "HTML 入門課程",
  "provider": {
    "@type": "Organization",
    "name": "Frontend Notes"
  }
}
```

簡單記：

- Open Graph 偏社群分享卡片。
- JSON-LD 結構化資料偏機器理解頁面內容。

### 6. 放在哪裡？

JSON-LD 通常可以放在 `<head>`，也可以放在 `<body>`。  
實務上常放在 `<head>`，方便和其他 metadata 集中管理。

```html
<head>
  <title>HTML 學習筆記</title>
  <script type="application/ld+json">
  {
    "@context": "https://schema.org",
    "@type": "Article",
    "headline": "HTML 學習筆記"
  }
  </script>
</head>
```

### 7. 常見錯誤

常見錯誤包括：

- JSON 格式錯誤，例如多了尾逗號。
- 欄位和頁面可見內容不一致。
- 使用了不適合頁面內容的 Schema 類型。
- 以為加了結構化資料就保證 SEO 排名提升。
- 把 JSON-LD 當成會執行的 JavaScript。

### 8. 實務上怎麼檢查？

實務使用時應該：

- 先確認頁面可見內容完整。
- 再補 JSON-LD。
- 使用搜尋平台提供的測試工具驗證。
- 定期檢查平台支援的 schema 類型與規則。

這是進階主題，初學先會看懂基本寫法和用途即可。

## 常見問法

### JSON-LD 是 JavaScript 嗎？

不是用來執行頁面邏輯的 JavaScript。  
它是放在 script 標籤中的 JSON 格式資料。

### 寫了結構化資料就會有搜尋特殊結果嗎？

不保證。

它可能幫助搜尋引擎理解內容，但是否顯示 rich result 由搜尋平台的規則和判斷決定。

### 結構化資料可以和頁面內容不同嗎？

不應該。

它應該描述頁面上真實可見的內容。

### 有 JSON-LD 還需要語意化 HTML 嗎？

需要。

語意化 HTML 是頁面本身的結構品質，JSON-LD 是額外資料描述。
