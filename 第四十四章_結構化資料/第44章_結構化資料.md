# 結構化資料

> - 所屬章節：第四十四章｜結構化資料  
> - 關鍵字：結構化資料、structured data、Schema.org、JSON-LD、`application/ld+json`、搜尋引擎、rich results、metadata  
> - 建議回查情境：想知道 JSON-LD 怎麼放進 HTML、結構化資料和語意化 HTML / `meta` / Open Graph 差在哪、為什麼不能亂寫不存在的內容時

## 本節導讀

HTML 本身是給瀏覽器解析，並把內容呈現給使用者看的文件。

但有些時候，我們還希望搜尋引擎或資料工具更明確知道：

- 這頁是一篇文章。
- 這是文章標題、作者和發布日期。
- 這是產品名稱、價格和評分。
- 這是課程資訊。
- 這是活動名稱、時間和地點。

這時候就可以使用**結構化資料**。

一句話理解：

> 結構化資料就是把頁面上已經存在的內容，用機器比較容易理解的格式重新描述一次。

最常見方式之一是 JSON-LD：

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "HTML 學習筆記",
  "datePublished": "2026-04-26"
}
</script>
```

這段資料不會直接顯示在畫面上，但搜尋引擎或其他工具可以讀取它，進一步理解頁面內容。

## 30 秒複習入口

- 結構化資料是給機器讀的補充描述。
- 它應該描述頁面上真實存在、使用者也看得到的內容。
- JSON-LD 常用 `<script type="application/ld+json">` 放進 HTML。
- 這個 `<script>` 裡是資料，不是要執行的 JavaScript 程式。
- Schema.org 提供常見類型與屬性詞彙，例如 `Article`、`Product`、`Course`、`Event`。
- `@context` 表示使用哪套詞彙表，常見是 `https://schema.org`。
- `@type` 表示這段資料描述的是哪一種東西，例如文章、商品、課程。
- 結構化資料可以輔助搜尋工具理解內容，但不保證排名或特殊搜尋結果。
- 結構化資料不能取代語意化 HTML。
- 實務上應先做好頁面內容，再補 JSON-LD，最後用測試工具驗證。

## 正文筆記

### 1. 結構化資料是什麼？

結構化資料是網頁中提供給機器讀取的補充資訊。

它通常用來描述頁面中的「實體」與「屬性」。

例如一篇文章可能有：

- 文章標題
- 作者
- 發布日期
- 修改日期
- 文章類型

一個商品頁可能有：

- 商品名稱
- 商品圖片
- 商品價格
- 商品評分
- 商品庫存狀態

這些內容人類可以直接從畫面上理解，但機器不一定能準確判斷哪一段是標題、哪一段是價格、哪一段是評分。

結構化資料的作用，就是用更明確的資料格式，把這些資訊重新標示出來。

### 2. JSON-LD 基本寫法

JSON-LD 常放在 `<script type="application/ld+json">` 裡：

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "HTML 學習筆記",
  "author": {
    "@type": "Person",
    "name": "小明"
  },
  "datePublished": "2026-04-26"
}
</script>
```

這段放在 HTML 裡，但不是普通 JavaScript 程式。

它是一段 JSON-LD 格式的資料。

`type="application/ld+json"` 的意思是告訴瀏覽器和工具：

> 這個 `<script>` 裡面放的是 linked data JSON，不是用來操作 DOM 或執行互動邏輯的 JavaScript。

### 3. `@context` 和 `@type`

#### `@context`

`@context` 通常用來指定這段 JSON-LD 要使用哪一套詞彙表：

```json
"@context": "https://schema.org"
```

可以先理解成：

> 下面這些欄位，例如 `headline`、`author`、`datePublished`，要用 Schema.org 的規則來解釋。

#### `@type`

`@type` 用來指定這段資料描述的是什麼類型的東西。

例如文章：

```json
"@type": "Article"
```

例如商品：

```json
"@type": "Product"
```

例如課程：

```json
"@type": "Course"
```

簡單記：

- `@context`：使用哪一套語彙表。
- `@type`：這筆資料是在描述哪一種東西。

### 4. 結構化資料要和可見內容一致

結構化資料不是拿來憑空增加內容，也不是拿來欺騙搜尋引擎。

它應該描述頁面上真實存在、使用者也能看到的內容。

#### 可見內容與 JSON-LD 的對應

假設頁面上有這段內容：

```html
<article>
  <h1>HTML 學習筆記</h1>
  <p>作者：小明</p>
  <time datetime="2026-04-26">2026 年 4 月 26 日</time>
</article>
```

那 JSON-LD 可以對應成：

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "HTML 學習筆記",
  "author": {
    "@type": "Person",
    "name": "小明"
  },
  "datePublished": "2026-04-26"
}
</script>
```

對照表：

| 頁面可見內容 | JSON-LD 欄位 |
|---|---|
| `<h1>HTML 學習筆記</h1>` | `"headline": "HTML 學習筆記"` |
| `作者：小明` | `"author": { "name": "小明" }` |
| `<time datetime="2026-04-26">` | `"datePublished": "2026-04-26"` |

#### 不應該這樣做

如果頁面上沒有評分，就不要硬寫：

```json
"aggregateRating": {
  "@type": "AggregateRating",
  "ratingValue": "5",
  "reviewCount": "999"
}
```

因為這樣會造成 JSON-LD 描述的內容和頁面可見內容不一致。

簡單記：

> 頁面沒有的東西，不要只寫在 JSON-LD 裡。

### 5. 結構化資料和語意化 HTML 差在哪？

語意化 HTML 是頁面本身的結構。

例如：

```html
<article>
  <header>
    <h1>HTML 學習筆記</h1>
    <p>作者：小明</p>
    <time datetime="2026-04-26">2026 年 4 月 26 日</time>
  </header>

  <p>這是一篇整理 HTML 基礎觀念的文章。</p>
</article>
```

這段 HTML 的重點是：

- `<article>` 表示這是一篇獨立文章。
- `<h1>` 表示主標題。
- `<time>` 表示時間。
- 使用者、瀏覽器、輔助技術和搜尋工具都能從 HTML 結構中理解內容。

JSON-LD 則是額外補充的機器可讀描述：

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "HTML 學習筆記",
  "author": {
    "@type": "Person",
    "name": "小明"
  },
  "datePublished": "2026-04-26"
}
</script>
```

兩者可以互補，但不能互相取代。

也就是說：

- 不要因為寫了 JSON-LD，就把 HTML 寫得沒有語意。
- 不要只依賴 JSON-LD 來描述頁面內容。
- 先把頁面內容與 HTML 結構寫清楚，再補結構化資料。

### 6. 結構化資料和 `meta` / Open Graph 差在哪？

#### `meta description`

`meta description` 通常用來提供搜尋摘要參考：

```html
<meta name="description" content="整理 HTML 常用標籤、語意化結構與基礎觀念。">
```

它通常是一段簡短文字，用來描述整個頁面。

#### Open Graph

Open Graph 常用於社群分享預覽：

```html
<meta property="og:title" content="HTML 學習筆記">
<meta property="og:description" content="整理 HTML 常用標籤與語意。">
<meta property="og:image" content="https://example.com/cover.jpg">
```

重點是當頁面被分享到社群平台時，可以顯示合適的標題、描述和圖片。

#### JSON-LD

JSON-LD 更像是在描述頁面中的實體資料：

```json
{
  "@context": "https://schema.org",
  "@type": "Course",
  "name": "HTML 入門課程",
  "provider": {
    "@type": "Organization",
    "name": "Frontend Notes"
  }
}
```

簡單記：

- 語意化 HTML：頁面本身的結構。
- `meta description`：給搜尋摘要參考。
- Open Graph：給社群分享卡片使用。
- JSON-LD：給搜尋引擎與資料工具理解頁面實體與屬性。

### 7. 什麼時候需要結構化資料？

不是每個頁面都一定要加 JSON-LD。

適合使用的情境：

- 文章頁：可以使用 `Article`。
- 商品詳情頁：可以使用 `Product`。
- 課程頁：可以使用 `Course`。
- 活動頁：可以使用 `Event`。
- FAQ 頁：可以使用 `FAQPage`。
- 麵包屑導覽：可以使用 `BreadcrumbList`。

不一定需要使用的情境：

- 普通練習頁。
- 單純排版頁。
- 沒有明確實體資料的頁面。
- 頁面內容還沒有整理清楚的頁面。

不應該使用的情境：

- 頁面沒有評分，卻在 JSON-LD 裡硬寫評分。
- 頁面沒有評論，卻在 JSON-LD 裡硬寫評論數。
- 頁面沒有商品價格，卻在 JSON-LD 裡硬寫價格。
- 使用和頁面內容不符的 Schema 類型。

實務順序應該是：

1. 先讓頁面內容完整、真實、可見。
2. 再用語意化 HTML 寫出合理結構。
3. 最後補 JSON-LD，幫助機器理解。

### 8. JSON-LD 放在哪裡？

JSON-LD 通常可以放在 `<head>`，也可以放在 `<body>`。

實務上常放在 `<head>`，方便和其他 metadata 集中管理：

```html
<head>
  <title>HTML 學習筆記</title>
  <meta name="description" content="整理 HTML 常用標籤與語意化觀念。">

  <script type="application/ld+json">
  {
    "@context": "https://schema.org",
    "@type": "Article",
    "headline": "HTML 學習筆記",
    "datePublished": "2026-04-26"
  }
  </script>
</head>
```

如果是 Vue、React 這類前端框架專案，還要注意：

- JSON-LD 最終是否真的出現在 HTML 裡。
- 搜尋工具是否能讀到該段資料。
- 如果頁面高度依賴前端動態渲染，SEO 需求較高時，通常會考慮 SSR、SSG 或其他可讓搜尋工具穩定讀取內容的方案。

初學階段先掌握基本觀念即可：

> JSON-LD 可以放在 HTML 裡，但它不是用來控制畫面互動，而是用來描述頁面資料。

### 9. 實務檢查流程

實務上加入結構化資料時，可以照下面流程檢查。

#### 第一步：先檢查頁面內容

確認：

- 頁面是否真的有這些內容。
- 使用者是否能在畫面上看到主要資訊。
- 標題、作者、日期、價格、評分等資訊是否真實。
- 頁面本身的 HTML 結構是否清楚。

#### 第二步：選擇合適的 Schema 類型

例如：

| 頁面內容 | 可能使用的 Schema 類型 |
|---|---|
| 文章 | `Article` |
| 商品 | `Product` |
| 課程 | `Course` |
| 活動 | `Event` |
| FAQ | `FAQPage` |
| 麵包屑 | `BreadcrumbList` |

#### 第三步：撰寫 JSON-LD

先從最基本、最確定的欄位開始寫。

例如文章可以先寫：

- `@context`
- `@type`
- `headline`
- `author`
- `datePublished`

不要一開始就加太多不確定的欄位。

#### 第四步：檢查 JSON 格式

常見格式錯誤：

```json
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "HTML 學習筆記",
}
```

上面最後一個欄位後面多了逗號，這在 JSON 中是不合法的。

正確寫法：

```json
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "HTML 學習筆記"
}
```

#### 第五步：使用工具驗證

可以使用：

- Google Rich Results Test：檢查是否符合 Google rich result 相關條件。
- Schema Markup Validator：檢查 Schema.org 結構是否合理。
- 瀏覽器 DevTools：確認 JSON-LD 是否真的輸出到 HTML 中。

工具驗證通過不代表一定會有特殊搜尋結果，只代表格式與部分條件比較沒有問題。

### 10. 常見錯誤與修正方向

| 常見錯誤 | 問題 | 修正方向 |
|---|---|---|
| JSON 多了尾逗號 | JSON 格式不合法 | 移除最後一個屬性後面的逗號 |
| 頁面沒有評分，JSON-LD 卻寫評分 | 內容不一致 | 移除 `aggregateRating`，或先讓頁面真的有評分內容 |
| 商品頁誤用 `Article` | Schema 類型不符合內容 | 改用 `Product` |
| 以為加了 JSON-LD 就會提升排名 | 對 SEO 的期待錯誤 | 理解它是輔助理解，不是排名保證 |
| 只寫 JSON-LD，不重視 HTML 結構 | 忽略頁面本身品質 | 先做好語意化 HTML，再補 JSON-LD |
| 把 JSON-LD 當作 JS 程式 | 誤解 `<script>` 用途 | 記住它是資料，不是互動邏輯 |

## 常見問法

### JSON-LD 是 JavaScript 嗎？

不是用來執行頁面邏輯的 JavaScript。

它是放在 `<script>` 標籤中的 JSON 格式資料。

雖然外層使用 `<script>`，但 `type="application/ld+json"` 已經表明這裡放的是 linked data JSON，而不是一般 JavaScript。

### 寫了結構化資料就會有搜尋特殊結果嗎？

不保證。

結構化資料可能幫助搜尋引擎理解內容，但是否顯示 rich result，還取決於搜尋平台規則、頁面品質、內容是否符合資格，以及搜尋引擎自己的判斷。

### 結構化資料可以和頁面內容不同嗎？

不應該。

它應該描述頁面上真實、可見、存在的內容。

如果頁面沒有的內容只出現在 JSON-LD 裡，這就是不一致。

### 有 JSON-LD 還需要語意化 HTML 嗎？

需要。

語意化 HTML 是頁面本身的結構品質，JSON-LD 是額外資料描述。

比較好的做法是：

1. 先把 HTML 結構寫清楚。
2. 再補上 JSON-LD。
3. 讓兩者描述同一份真實內容。

### 每個頁面都需要 JSON-LD 嗎？

不需要。

如果只是普通練習頁、單純排版頁，或沒有明確文章、商品、課程、活動、FAQ 等實體資料，通常可以先不加。

初學時更重要的是先把：

- 標題層級
- 語意化標籤
- 頁面內容
- `title`
- `meta description`
- Open Graph

這些基礎做好。

### Vue 專案也可以加 JSON-LD 嗎？

可以。

但要注意 JSON-LD 最後是否真的被輸出到頁面的 HTML 裡。

如果是 SEO 需求高的頁面，通常要進一步考慮 SSR、SSG，或其他能讓搜尋工具穩定讀取頁面內容的做法。

1. 主要給搜尋引擎、資料工具等機器讀。
2. 常用 `<script type="application/ld+json">`。
3. `@context` 表示使用哪套詞彙表；`@type` 表示這段資料描述的是哪一種東西。
4. 應該一致，JSON-LD 應描述頁面上真實、可見、存在的內容。
5. Open Graph 偏社群分享卡片；JSON-LD 偏機器理解頁面中的實體與屬性。
6. 因為它只是輔助搜尋工具理解內容，是否排名提升或顯示特殊結果，仍取決於搜尋平台規則和頁面品質。
7. 因為語意化 HTML 是頁面本身的結構品質，JSON-LD 只是額外補充資料。
8. 不應該，因為頁面沒有的內容不應該只寫在 JSON-LD 裡。
9. 不一定，普通練習頁通常先做好 HTML 結構、標題、內容即可。
10. 可以用 Google Rich Results Test、Schema Markup Validator、瀏覽器 DevTools 檢查。

## 回查關鍵句

- 結構化資料是給機器讀的補充描述，不是給使用者看的主要內容。
- JSON-LD 是資料，不是頁面互動邏輯。
- `@context` 看詞彙表，`@type` 看資料類型。
- 頁面沒有的內容，不要只寫在 JSON-LD 裡。
- 先做好語意化 HTML，再補結構化資料。
- 結構化資料有助於理解，不保證排名或 rich result。
